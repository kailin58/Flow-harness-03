# 用户端订单列表结构决策记录（2026-04-09）

## 背景

在 `meeting/5-1-2-2履约与售后服务块规划.md` 中，`订单列表` 已被定义为履约与售后服务块的总入口，但如果不继续拆分，后续很容易把它重新做成一个简单流水列表，无法稳定承接物流、售后、客服等动作。

## 本次决策

本次新增：

- `meeting/5-1-2-2-1订单列表与筛选规划.md`

统一把订单列表定义为：

- 用户下单后服务闭环的总入口
- 重点承担“找单、看状态、进动作”
- 通过顶部状态分组和卡片快捷动作提高处理效率

## 关键口径

- 订单列表不是交易首页的延续，而是成交后的服务检索页
- 订单列表不是后台工单视图，不暴露内部审核、履约协同和仲裁过程字段
- 订单列表不是经营分析页面，不承接复杂图表和多角色聚合

## 当前冻结的一期状态分组

- 全部
- 待支付
- 待发货
- 待收货
- 已完成
- 售后中

## 当前冻结的一期核心动作

- 去支付
- 查看详情
- 查看物流
- 确认收货
- 申请售后
- 联系客服
- 再次购买

## 原因

- 用户在订单列表中的核心诉求是快速找到订单并完成下一步动作
- 如果状态分组过多，会把底层状态机复杂度直接暴露给用户
- 如果列表卡片没有快捷动作，用户查物流、发售后、联系客服都需要多跳，体验会明显变差

## 影响范围

- 后续细化订单详情、物流查询、售后申请时，应默认以订单列表作为上层入口
- 顶部状态分组与卡片动作的显隐判断，应由后端真相源和业务规则统一驱动
- Story 或页面稿后续细化时，不应突破本次冻结的状态分组与动作优先级

## 风险

- 如果后续业务侧希望把更多状态直接暴露为顶部分组，可能会破坏当前用户心智
- 如果订单卡片承载过多信息或动作，会重新退化成复杂列表，不利于快速处理

## 后续建议

- 继续细化 `订单详情`
- 再细化 `物流查询`
- 然后细化 `售后申请`
