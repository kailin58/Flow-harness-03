# 用户端订单详情结构决策记录（2026-04-09）

## 背景

在 `履约与售后服务块` 下继续细化时，`订单详情` 很容易被实现成一个静态字段展示页，导致物流、售后、客服动作入口分散，无法稳定承接用户下单后的服务路径。

## 本次决策

本次新增：

- `meeting/5-1-2-2-2订单详情规划.md`

统一把订单详情定义为：

- 单笔订单的核心服务汇聚页
- 负责收口订单状态、关键信息、物流入口、售后入口、客服入口和核心动作
- 重点解决“看清这一单，并完成这一单相关的下一步动作”

## 当前冻结的页面结构

订单详情一期先按 6 段结构冻结：

1. 订单状态总览区
2. 商品与金额信息区
3. 履约与收货信息区
4. 服务入口区
5. 订单基础信息区
6. 底部动作区

## 当前冻结的一期核心动作

- 去支付
- 确认收货
- 申请售后
- 查看物流
- 联系客服
- 再次购买

## 原因

- 用户查看订单详情的核心诉求不是“看字段”，而是“看状态 + 做动作”
- 如果没有稳定的服务入口区，物流、售后、客服动作会继续四散在不同页面和文案里
- 订单详情必须成为订单列表、物流页、售后页、客服链路之间的中间汇聚点

## 影响范围

- 后续细化物流查询、售后申请、售后进度时，应默认以订单详情作为重要上游入口
- 前端页面稿和 Story 细化时，不应把订单详情退化成纯信息展示页
- 订单状态和动作显隐必须与后端规则、真实状态口径一致

## 风险

- 如果后续继续往详情页塞过多复杂说明，页面会重新变重
- 如果详情页承载过多本该在专页完成的流程，会重新和物流页、售后页产生边界冲突

## 后续建议

- 继续细化 `物流查询`
- 再细化 `售后申请`
- 然后细化 `售后进度`
