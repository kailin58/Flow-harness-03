# meeting 文档可开源/对标分析（2026-04-09）

## 结论摘要

基于 `meeting/` 下现有文档，ACmodus 规划中的能力可以分成三类：

1. **适合直接借助免费且可商用开源底座**：账号与权限、客服工作台、AI 调用网关与观测、前台商城骨架、轻量发货/WMS、内容资产管理底座。
2. **适合“开源做局部 + 商业产品做形态对标”**：供应商协同、渠道店铺管理、内容创造端、分销、积分。
3. **核心规则仍应自研，只适合参考产品形态**：结算中台、复杂数据权限、跨渠道同步规则、佣金/积分核算口径、AI 治理中的成本/合规/路由策略。

## 可直接找开源底座的部分

### 1. 账号与权限中心 / 权限中台

- 对应文档：
  - `meeting/账号体系与权限规划.md`
  - `meeting/账号与权限中心菜单规划.md`
  - `meeting/账号与权限中心对外接口草案.md`
- 适合原因：
  - 需求已经非常接近标准 IAM / AuthN / AuthZ / 审计中心形态。
  - 文档强调统一身份、统一鉴权、统一审计、统一安全策略，和成熟 IAM 产品模型高度一致。
- 开源候选（免费且可商用）：
  - `Keycloak`：Apache-2.0，适合 SSO、登录、安全策略、会话、身份联邦。
  - `Casbin`：Apache-2.0，适合 RBAC / ABAC / 资源动作鉴权。
  - `OpenFGA`：Apache-2.0，适合关系型授权与可扩展授权查询。
  - `OpenSearch`：Apache-2.0，适合审计检索与权限变更日志查询。
- 结论：
  - **这是最值得优先引入开源底座的模块。**
  - 但“供应商范围、字段权限、审批权限、高危动作二次校验”仍要按你们业务规则做自研编排。

### 2. 客服工作台

- 对应文档：
  - `meeting/5-4客服工作台入口.md`
  - `meeting/6端域边界.md`
- 适合原因：
  - 会话、工单、分派、升级、留痕，本质上就是 helpdesk / omni-channel support 的标准形态。
- 开源候选（免费且可商用）：
  - `Chatwoot`：MIT，适合会话聚合、坐席工作台、标签、协同。
  - `Apache Answer`：Apache-2.0，可作为知识库/FAQ/Q&A 底座。
- 商业对标：
  - `Zendesk`
  - `Freshdesk`
  - `Intercom`
- 结论：
  - **客服工作台可明显参考开源。**
  - 但“订单/售后/物流上下文联查、平台客服与供应商协同边界、退款审批边界”必须自研。

### 3. AI 模型管理中心的调用层与观测层

- 对应文档：
  - `meeting/AI模型管理中心规划.md`
- 适合原因：
  - 文档中的“统一调用接口、路由、限额、调用日志、成本归集、观测”与 LLM Gateway / LLMOps 工具高度吻合。
- 开源候选（免费且可商用）：
  - `LiteLLM`：MIT，适合统一模型网关、路由、预算/限流。
  - `Langfuse`：MIT，适合调用观测、Tracing、Prompt 版本、评测。
  - `OpenTelemetry`：Apache-2.0，适合统一埋点与链路观测。
- 结论：
  - **AI 中心的调用层最容易借助开源。**
  - 但“模型标签口径、敏感数据出域规则、成本归集口径、灰度策略、业务场景路由”还是平台规则，不是现成开源能替代的。

### 4. 用户统一入口的商城骨架

- 对应文档：
  - `meeting/5-1用户统一入口.md`
  - `meeting/5-1-1用户端增强模块规划.md`
- 开源候选（免费且可商用）：
  - `Medusa`：MIT，适合 headless commerce 基础骨架。
  - `Bagisto`：MIT，适合作为多商家/商城后台能力参考。
- 结论：
  - 商品浏览、购物车、下单这类通用交易骨架可借鉴开源。
  - 但你们的“推广员、积分、穿搭对话、衣橱、行程规划、多凭证合并”明显超出标准商城，需要自研。

### 5. 发货执行端 / 轻量 WMS

- 对应文档：
  - `meeting/5-5发货执行入口.md`
- 开源候选（免费且可商用）：
  - `ModernWMS`：Apache-2.0。
  - `OpenBoxes`：EPL-1.0，可商业使用。
- 商业对标：
  - `Veeqo`
  - `ShipStation`
  - `Peoplevox`
- 结论：
  - 如果一期只是“扫码拣货、打包、打印面单、发货确认、异常上报”，**可以强参考开源 WMS**。
  - 但如果后续进入波次、路径优化、复杂仓配调度，就要转向更强的 WMS 产品形态设计。

## 适合“开源做局部 + 商业对标做整体”的部分

### 6. 供应商协同入口

- 对应文档：
  - `meeting/5-3供应商协同入口.md`
  - `meeting/6端域边界.md`
- 开源候选：
  - `Mercur`（基于 Medusa 的多商家 marketplace）：公开资料显示为 MIT，可用于 vendor panel / vendor dashboard 参考。
  - `Bagisto` 多商家方案：MIT，可参考 seller dashboard、商品提报、订单协同。
- 商业对标：
  - `SAP Ariba`
  - `Coupa`
  - `Jaggaer`
- 结论：
  - 供应商协同入口能找到开源“卖家中心/商家后台”参考。
  - 但你们文档里的重点不是通用卖家后台，而是“`supplier_id` 数据隔离 + 发货协同 + 售后协同 + 对账确认”，这一层仍偏自研。

### 7. 内容创造端 / AI 资产中心

- 对应文档：
  - `meeting/5-2平台后台入口.md`
  - `meeting/AI模型管理中心规划.md`
  - `meeting/20260407.md`
- 开源候选：
  - `Payload CMS`：MIT，可作为素材、版本、草稿、审核流、媒体资产底座参考。
- 商业对标：
  - `Bynder`
  - `Brandfolder`
  - `Adobe Experience Manager Assets`
- 结论：
  - “素材管理、版本、审核、分发”能找到 DAM/CMS 对标。
  - 但“AI 生图任务、版权与合规、模型调用成本归集、渠道素材下发”要自研业务编排。

### 8. 外部渠道与店铺管理端

- 对应文档：
  - `meeting/5-2平台后台入口.md`
  - `meeting/6端域边界.md`
  - `meeting/20260407.md`
- 开源候选：
  - `UCP (Universal Commerce Protocol)`：Apache-2.0，更适合做协议/模型参考，不是现成成品。
  - 一些轻量连接器或库存同步项目可作为局部参考，但难以直接支撑“多平台多店 + 授权 + 商品/库存/订单同步 + 客服消息回写”。
- 商业对标：
  - `ChannelEngine`
  - `Lengow`
  - `ChannelAdvisor / CommerceHub`
- 结论：
  - **这块更适合做商业产品对标，不适合指望一个成熟免费开源成品直接落地。**
  - 开源更多只能用在适配层、同步任务、标准协议层。

### 9. 分销（推广员端）

- 对应文档：
  - `meeting/6端域边界.md`
  - `meeting/7一期范围冻结.md`
  - `meeting/20260407.md`
- 开源候选：
  - `Refferq`：MIT，可作为 affiliate/referral 平台的基础参考。
- 商业对标：
  - `Impact`
  - `PartnerStack`
  - `Refersion`
- 结论：
  - 推广码、推广链接、佣金概览这类前台与运营后台可参考开源/商业产品。
  - 但你们的一次奖励、归因窗口、退款冲正、供应商结算顺序、风控黑名单，都是强业务规则，仍需自研。

### 10. 积分商城 / 忠诚度

- 对应文档：
  - `meeting/6端域边界.md`
  - `meeting/7一期范围冻结.md`
  - `meeting/20260407.md`
- 开源候选：
  - 目前能搜到的 MIT 忠诚度项目偏小而轻，例如 `Stampee`，更像小商户数字集点卡，不足以直接覆盖你们的积分商城与积分账本。
- 商业对标：
  - 更适合直接参考 loyalty/CRM 产品设计思路，而不是押注某个成熟开源框架。
- 结论：
  - **积分前台展示可参考现成会员/忠诚度产品。**
  - **积分账本、抵现、退单回冲、等级口径必须自研。**

## 更适合只做对标、不建议直接找开源替代的部分

### 11. 结算中台

- 对应文档：
  - `meeting/20260407.md`
  - `meeting/6端域边界.md`
- 原因：
  - 涉及平台、供应商、分销、营销补贴、AI 成本、物流费用的统一计费编排。
  - 强依赖你们自己的账务口径、事件顺序、冲正规则、账单周期。
- 结论：
  - 可以参考 ERP / 财务结算产品，但**不建议直接找开源替代**。

### 12. 权限中的数据权限 / 字段权限 / 审批权限编排

- 原因：
  - 登录认证可借 IAM，基础 RBAC 可借 AuthZ。
  - 但你们要求的供应商、品牌、店铺、渠道、组织、字段、审批的组合授权，已经是平台规则系统，不是标准 IAM 产品能直接配置完事。

### 13. 用户端增强模块

- 对应文档：
  - `meeting/5-1-1用户端增强模块规划.md`
- 包括：
  - 对话模块
  - 衣橱模块
  - 行程规划模块
- 结论：
  - 只能做产品形态和交互路径对标。
  - 没有成熟开源成品可以直接替代“穿搭顾问 + 个人衣橱 + 行程规划 + 商城联动”。

## 综合判断

### 最适合优先找开源底座的 5 个方向

1. `账号与权限中心`
2. `客服工作台`
3. `AI模型管理中心` 的调用层/观测层
4. `发货执行端`
5. `商城主链路骨架`

### 最适合优先做商业对标的 5 个方向

1. `外部渠道与店铺管理端`
2. `供应商协同入口`
3. `内容创造端 / AI资产中心`
4. `分销`
5. `积分商城`

### 不要期待开源直接替代的 4 类核心规则

1. `结算中台`
2. `跨渠道同步规则与主数据口径`
3. `佣金 / 积分 / 售后冲正规则`
4. `细粒度数据权限 / 字段权限 / 审批权限`

## 对后续技术选型的建议

建议后续在 `meeting/9技术选型与工程底座.md` 里按下面方式落地：

1. 先区分“买底座 / 借开源 / 必须自研规则引擎”。
2. 对每个端域只保留 1~2 个开源候选，不要做大而全的工具清单。
3. 许可证优先级建议：`MIT > Apache-2.0 > EPL/LGPL/OSL`。
4. 任何涉及账务、结算、价格、归因、数据范围的部分，都不要因为“有开源”就放弃自研规则真相源。

## 如果现在就要定技术选型方向

建议先按下面口径定：

1. `账号与权限`：优先采用“开源底座 + 自研规则编排”。
2. `客服工作台`：优先采用“开源工作台 + 自研订单/售后上下文”。
3. `AI 模型中心`：优先采用“开源网关/观测 + 自研治理策略”。
4. `商城、发货、内容资产`：优先“借骨架，不借真相源”。
5. `分销、积分、渠道、结算`：优先“参考形态，自研规则”。

## 框架选型 Shortlist（可直接开会用）

|---|---|---|---|---|---|---|
| `Keycloak` | Apache-2.0 | 是 | 账号与权限中心、统一登录 | 成熟度高，SSO/OIDC/SAML/MFA 能力完整 | 管理模型偏重，细粒度业务权限仍需外接 | 是，P0 |
| `Casdoor` | Apache-2.0 | 是 | 账号与权限中心、认证门户 | UI 较完整，身份源和登录方式丰富 | 生态与成熟度通常不如 Keycloak 稳 | 是，P1 备选 |
| `Authelia` | Apache-2.0 | 是 | 登录安全、二次验证 | 适合补强登录保护和 MFA | 更偏 access proxy，不是完整 IAM 中台 | 是，P2 局部能力 |
| `Ory Kratos` | Apache-2.0 | 是 | 身份认证、账号注册恢复 | 身份流和认证流程清晰 | 权限治理和后台治理台要自己补 | 是，P2 备选 |
| `Casbin` | Apache-2.0 | 是 | 授权引擎、RBAC/ABAC | 轻量、易嵌入、资源动作模型清楚 | 数据权限/审批权限编排仍要自己设计 | 是，P0 |
| `OpenFGA` | Apache-2.0 | 是 | 关系型授权 | 很适合“主体-资源-关系”模型 | 上手门槛比普通 RBAC 高 | 是，P0 |
| `SpiceDB` | Apache-2.0 | 是 | 细粒度授权 | Zanzibar 风格成熟，适合复杂关系授权 | 运营复杂度比 Casbin 更高 | 是，P1 |
| `Chatwoot` | MIT | 是 | 客服工作台 | 会话、坐席、标签、协同能力成型 | 订单/售后/物流上下文仍需自研集成 | 是，P0 |
| `Apache Answer` | Apache-2.0 | 是 | 客服知识库、FAQ | 适合知识沉淀与检索 | 不是完整客服系统 | 是，P1 |
| `LiteLLM` | MIT | 是 | AI 统一调用网关 | 多模型统一接口、路由、预算/限流好用 | 治理规则和敏感数据策略仍要自研 | 是，P0 |

| `SigNoz` | MIT（社区部分） | 是 | 观测平台 | 可接 OTel，UI 比较完整 | 仓库含企业版目录，选型时要分清社区能力 | 是，P1 |

## 不建议作为主选型的原因汇总

- `GPL/BSL/开放核心但企业版边界复杂` 的项目，不是不可以用，而是不适合作为当前主方案第一选择。
- `更像参考项目而不是生产底座` 的项目，适合借交互和模型，不适合直接承担业务真相源。
- `与当前规划边界不完全一致` 的项目，比如 marketplace seller center，不等于供应商协同平台。
