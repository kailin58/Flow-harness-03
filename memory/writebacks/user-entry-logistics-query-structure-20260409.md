# 用户端物流查询结构决策记录（2026-04-09）

## 背景

在 `履约与售后服务块` 下继续细化时，`物流查询` 如果不单独冻结结构，很容易继续被弱化成订单详情页里的一个入口，无法完整承接包裹、轨迹、签收和预计送达等履约可见性需求。

## 本次决策

本次新增：

- `meeting/5-1-2-2-3物流查询规划.md`

统一把物流查询定义为：

- 用户查看单笔订单履约进度的完整进度页
- 重点承接包裹、运单、轨迹、预计送达、签收状态
- 看完物流后可顺畅进入售后或客服后续路径

## 当前冻结的页面结构

物流查询一期先按 5 段结构冻结：

1. 履约状态总览区
2. 包裹与运单区
3. 物流轨迹区
4. 签收与预计送达区
5. 服务衔接区

## 当前冻结的一期核心范围

- 履约状态总览
- 包裹与运单基础信息
- 物流轨迹时间线
- 预计送达提示
- 签收状态展示
- 多包裹基础查看
- 联系客服入口
- 售后入口或售后进度入口

## 原因

- 用户在物流页的核心诉求是“货在哪里、什么时候到、是否已签收”
- 多包裹和轨迹变化如果只在订单详情做摘要，信息容易不够用
- 物流页需要和售后、客服形成清晰衔接，但不能直接吞并后两者的职责

## 影响范围

- 订单详情后续只保留物流摘要与入口，不承接完整轨迹展示
- 售后申请和客服入口后续应默认把物流页作为重要上游上下文来源
- 多包裹、签收、预计送达等口径应统一在物流页收口

## 风险

- 如果后续把更多复杂异常处理直接塞进物流页，容易和售后、客服重新抢边界
- 如果多包裹场景没有继续细化，后续仍可能在展示层出现理解成本

## 后续建议

- 继续细化 `售后申请`
- 再细化 `售后进度`
- 视需要再拆 `包裹与运单` 和 `物流轨迹` 子分册
