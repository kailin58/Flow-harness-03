# 用户入口搜索下方模块快捷栏决策记录（2026-04-09）

## 背景

在补齐 `meeting/5-50-用户入口结构图解.md` 与 `meeting/5-1-2-1-1首页承接规划.md` 时，用户进一步明确首页顶部搜索入口下方还需要一条新的模块快捷栏：

- 该栏位于统一 Chat / 搜索入口下方
- 当前先放 `天气`、`商城`、`衣橱`、`行程规划`
- 后续还需要继续增加新的模块

如果不单独冻结这条栏位，后续首页新增模块时仍会反复改动首页主结构说明，导致“模块扩展”与“首页壳子变化”重新混在一起。

## 本次决策

统一新增：

- 首页搜索下方模块快捷栏

统一口径为：

- 该栏是首页内的稳定容器
- 当前首批先挂 `天气`、`商城`、`衣橱`、`行程规划`
- `商城` 点击后按用户身份分流：会员进入客户端商城，分销进入自己商城的首页
- 后续模块可持续新增，不预设固定上限
- 当前不预设固定数量上限，但当前端模块超出首屏可视容量时，应统一保持单栏展示并通过横向滑动承接，不因模块增多改成纵向堆叠或新增新栏位
- 平台后台只负责配置该栏中的模块内容、顺序、显隐、身份命中规则与跳转，不改写展示形态

## 原因

- 搜索入口下方确实需要一个更贴近首页顶部的轻量模块层，用于承接天气、商城和增强模块快捷入口
- 如果把这类模块继续散落在活动区、增强露出区或备注里，后续结构会越来越不清楚
- 允许后续持续新增模块，但仍要求前台层级稳定，能避免每加一个模块就重写首页结构
- 用固定栏位 + 后台配置的方式，可以同时满足“持续扩展”和“主链路稳定”两个目标
- 对分销用户来说，首页需要一个更直接的身份化承接入口，避免再从其他层级绕回自己的店铺首页

## 影响范围

- `meeting/5-50-用户入口结构图解.md` 需要把该栏画进首页结构图
- `meeting/5-1用户统一入口.md` 需要把该栏定义为首页内稳定容器
- `meeting/5-1-2-1-1首页承接规划.md` 需要把该栏纳入首页分区、优先级和一期范围
- 后续首页新增模块时，应默认优先判断是否挂在这条快捷栏中

## 风险

- 如果后续把模块数量无限扩展为多行堆叠，仍可能重新挤压商品主内容区
- 如果天气、商城、衣橱、行程规划在首页直接展开完整功能，会破坏“首页只做入口承接”的原则
- 如果后台配置缺少排序、互斥和兜底约束，模块栏仍可能出现语义混乱

## 后续建议

- 后续补一份“首页搜索下方模块快捷栏配置规则”文档
- 明确该栏默认采用单栏横向滑动形态
- 后续新增模块时，优先回答“是否适合挂在搜索下方模块快捷栏”，再决定是否进入其他露出区
