# meeting/9 数据库体现范围说明（2026-04-09）

## 结论

`meeting/9技术选型与工程底座.md` 不应只在技术栈列表里简单出现 `MySQL 8.4`，还应补充一节“数据库与存储底座建议”。

但该文档只需要体现**架构级数据库口径**，不需要下沉到表结构、字段、索引、DDL 设计细节。

## 建议在 meeting/9 中体现的内容

- `MySQL` 是核心业务真相源，承接订单、库存、售后、结算、积分、分销等强一致数据
- 本地事务边界：业务写入、幂等记录、outbox 事件写入必须在同一事务内完成
- `Redis` 只承担缓存、幂等辅助、限流、会话辅助，不作为业务真相源
- `OpenSearch` 用于审计检索、搜索、知识检索，不承担核心交易写模型
- 一期以单库/分模块 schema 治理为主，不预设大规模分库分表
- 数据库变更通过 migration 管理，schema 真源仍以 `db/migrations/` 为准

## 不建议写进 meeting/9 的内容

- 具体表设计
- 字段清单
- 索引细节
- 分表规则细节
- SQL 优化细节

## 原因

`meeting/9` 的定位是“技术选型与工程底座”，如果完全不说明数据库职责，会导致：

- `MySQL / Redis / OpenSearch / Kafka` 的职责边界不清
- “业务真相源”只停留在应用层，缺少存储层落点
- 后续讨论一致性、审计、读写分层时容易反复

同时，如果写得太细，又会和 `db/migrations/` 以及后续详细设计文档职责重叠。
