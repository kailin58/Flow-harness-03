# 用户端履约与售后服务块决策记录（2026-04-09）

## 背景

在细化 `meeting/5-1-2用户端细化块规划.md` 时，发现用户端关于 `订单`、`物流`、`售后`、`退款` 的内容仍然容易被混入 `交易主链路块`，导致下单前转化路径与下单后服务路径混在一起。

## 本次决策

本次继续新增：

- `meeting/5-1-2-2履约与售后服务块规划.md`

统一把用户端下单后的服务闭环收口为 `履约与售后服务块`，主要承接：

1. 订单列表
2. 订单详情
3. 物流与包裹查询
4. 售后申请入口
5. 售后进度与结果展示

## 关键边界

- `交易主链路块` 负责浏览、搜索、详情、加购、结算、下单、支付
- `履约与售后服务块` 负责订单查看、物流查看、售后发起、售后跟踪、必要时转客服
- `智能导购与服务分流块` 负责理解问题、解释问题、做分流
- `客服工作台端` 负责人工处理、内部协作、分派升级和留痕

## 原因

- 下单前与下单后属于两种完全不同的用户任务
- 如果不单独成块，物流、退款、退货、换货、售后进度等能力会被弱化成订单详情页的附属说明
- 售后与物流都需要更清晰地和客服协同、后台真相源做边界隔离

## 影响范围

- 后续细化用户端时，`订单列表`、`订单详情`、`物流查询`、`售后入口`、`售后进度` 应默认归这个块
- 交易主链路文档后续不再吞并下单后的服务路径
- 客服工作台端与用户端服务入口的边界会更清晰

## 风险

- 如果后续把所有“订单相关页面”再次一股脑归回交易主链路，会导致边界再次混乱
- 如果售后类型扩展过快，`退款`、`退货退款`、`换货`、`争议处理` 之间仍可能需要再次细分

## 后续建议

- 先细化 `订单列表与筛选`
- 再细化 `订单详情`
- 接着细化 `物流查询` 与 `售后申请`
- 最后细化 `售后进度`
