# 用户统一入口细化块结构决策记录（2026-04-09）

## 背景

在 `meeting/5-1用户统一入口.md` 已冻结 IA、`meeting/5-1-1用户端增强模块规划.md` 已冻结增强模块定位后，继续往下细化时，出现一个组织问题：

- 是直接开始写页面或功能清单
- 还是先补一层“块级结构”
- 是立即建立文件夹
- 还是沿用现有 `meeting/` 编号平铺体系

## 本次决策

本次先新增：

- `meeting/5-1-2用户端细化块规划.md`

统一把用户统一入口继续拆成 8 个细化块：

1. 交易主链路块
2. 履约与售后服务块
3. 智能导购与服务分流块
4. 用户资产沉淀块
5. 场景规划块
6. 增长与权益块
7. 账户与身份块
8. 流量承接与活动块

同时决定：

- 当前先沿用 `meeting/` 下的编号平铺文件方式
- 暂不立即迁移到独立文件夹
- 后续按 `5-1-2-1` 到 `5-1-2-8` 的方式逐块展开
- 用户入口层只冻结稳定结构，高变活动位、增长位、导购位统一交由平台后台配置

## 原因

- 现有文档结构已经形成 `5-1`、`5-1-1` 的连续编号关系
- 当前阶段更适合先冻结“块边界”，而不是直接进入页面与功能细节
- 立即迁移到文件夹会增加已有引用和阅读路径的回改成本
- 用户端目前最需要的是承上启下的中间层，避免“入口、模块、功能、页面”四层混谈
- 用户下单后的 `物流`、`履约进度`、`售后` 服务如果只零散挂在订单页里，会在后续细化时被弱化，因此提升为独立块更利于形成完整用户服务闭环
- 如果首页高变运营入口继续直接写死在 IA 中，后续活动和增长需求仍会反向侵入主链路，因此需要把高变入口从结构层剥离到配置层

## 影响范围

- 后续细化 `用户统一入口` 时，应默认先按块展开
- `对话模块`、`衣橱模块`、`行程规划模块` 分别落入块级结构，不再孤立讨论
- 增长、权益、账号、安全、活动流量等能力也有了统一挂载层
- `订单列表`、`订单详情`、`物流查询`、`售后入口与进度` 统一收口到 `履约与售后服务块`
- 后续首页活动、增长位和导购位应以“固定坑位 + 平台后台配置”推进，而不是持续修改入口结构文档

## 风险

- 如果后续直接跳过块级层去写页面稿，文档之间仍可能重新出现边界重叠
- 如果后续 `5-1-*` 文档数量快速膨胀，平铺方式会逐渐变长，届时需要再评估是否迁移为目录
- 如果后台配置边界定义不清，仍可能出现运营入口借配置名义突破主链路优先级

## 后续建议

- 优先细化 `交易主链路块` 与 `履约与售后服务块`
- 再细化 `智能导购与服务分流块`
- 再细化 `用户资产沉淀块` 与 `场景规划块`
- 待 `5-1-*` 文档数量进一步增长后，再决定是否整体迁移到 `meeting/5-1用户统一入口/` 目录形态
- 后续如继续补首页或活动相关文档，应先定义配置后台约束，再细化前台露出位
