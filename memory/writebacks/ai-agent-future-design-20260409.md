# AI 能力与 Agent 化未来设计建议（2026-04-09）

## 结论摘要

基于当前 `meeting/` 中的 5 个一级入口、9 个业务端域和 `AI模型管理中心` 规划，项目**非常适合增加 AI 能力**，也**可以引入 Agent 框架**，但前提是：

1. **先把 AI 做成平台能力，而不是先把所有业务做成 Agent。**
2. **先做“AI-ready / Agent-ready”架构，再逐步让少数高价值流程 Agent 化。**
3. **交易、支付、结算、权限裁定、价格真相源等核心链路，不应直接交给自主 Agent 决策。**

一句话判断：

- **可以加 AI**
- **可以加 Agent**
- **但应该采用“规则真相源 + Agent 辅助编排”的未来设计**

## 当前项目最适合增加的 AI 能力

### 1. 用户统一入口

最适合加的 AI 能力：

- 穿搭顾问对话
- 商品理解与推荐解释
- 搜索意图理解与 query rewrite
- 衣橱识别与搭配建议
- 行程场景生成穿搭建议
- 售后问题分流
- 智能客服入口前置问询

适合原因：

- 用户端已经明确有 `对话模块`、`衣橱模块`、`行程规划模块`
- 这些能力天然适合 LLM + 多模态模型增强
- 同时又不会直接替代交易真相源

### 2. 客服工作台

最适合加的 AI 能力：

- 会话摘要
- 工单自动分类
- 辅助回复草稿
- 意图识别与情绪识别
- 订单/售后上下文摘要
- 升级工单建议
- 知识库召回与答案生成

适合原因：

- 客服场景高频、重复、文本密集
- 文档中已明确客服端是模型消费端
- 价值直接体现在处理效率和留痕质量上

### 3. 内容创造端

最适合加的 AI 能力：

- 文生图 / 图生图
- 商品卖点文案生成
- 素材变体生成
- 渠道适配文案改写
- 主图/详情图建议
- 素材质量检测
- 违规与版权风险辅助识别

适合原因：

- 这是最天然的 AI 生产场景
- 你们已经单独规划了内容创造端和 AI模型管理中心

### 4. 平台后台入口

最适合加的 AI 能力：

- 运营 Copilot
- 商品审核辅助
- 售后审核辅助
- 风险规则解释
- 报表摘要
- 异常单归因解释
- 审计日志智能检索

适合原因：

- 后台操作信息密度高、跨模块跳转多
- AI 适合做“理解、汇总、建议”，不适合直接替代真相源

### 5. 供应商协同入口

最适合加的 AI 能力：

- 商品提报表单辅助填写
- 图片/属性自动识别补全
- 售后协同建议
- 对账差异解释
- 发货异常原因归类

适合原因：

- 供应商端有大量结构化录入与协同沟通
- AI 可以提升提报质量和协同效率

### 6. 发货执行入口

最适合加的 AI 能力：

- 异常上报自动归类
- 面单/包裹 OCR
- 拣货异常语音转结构化
- 现场操作助手

适合原因：

- 该端是轻执行端，不适合大模型重交互
- 但很适合小而快的识别、归类、辅助输入能力

## 哪些能力适合 Agent 化

不是所有 AI 都要做成 Agent。更适合 Agent 的，是“跨步骤、跨系统、需要工具调用和状态管理”的任务。

### 适合做 Agent 的场景

#### 1. 客服协同 Agent

可做：

- 读取订单、售后、物流上下文
- 生成回复草稿
- 判断是否需要升级
- 自动创建工单草稿
- 推荐知识库答案

不应直接做：

- 直接退款
- 直接改价
- 直接修改订单关键字段

定位：

- `客服副驾驶 Agent`

#### 2. 内容生产 Agent

可做：

- 读取商品信息
- 按模板生成图文素材
- 调用不同模型生成多个版本
- 自动进入审核流
- 按渠道要求导出不同版本

定位：

- `内容生产编排 Agent`

#### 3. 运营 Copilot Agent

可做：

- 解释报表
- 汇总某类异常订单
- 给出处理建议
- 生成活动配置草稿
- 帮运营定位问题入口

定位：

- `运营分析 Agent`

#### 4. 渠道同步异常处理 Agent

可做：

- 识别同步失败原因
- 归类为授权问题/字段问题/库存问题
- 自动形成补偿任务草稿
- 推荐处理顺序

定位：

- `渠道运维 Agent`

#### 5. 商品提报辅助 Agent

可做：

- 从图片、标题、说明中抽取属性
- 补全类目字段
- 检查缺失项
- 生成提报草稿

定位：

- `商品提报 Agent`

## 哪些能力不建议直接 Agent 化

以下能力可以被 AI 辅助，但**不应交给自主 Agent 做最终决策**：

- 价格真相源
- 库存真相源
- 支付回调核心链路
- 退款最终裁定
- 佣金核算
- 积分账本变更
- 结算出账
- 权限裁定真相源
- 高风险导出审批

原因：

- 这些链路需要强规则、可审计、可回放、可追责
- Agent 可以参与“建议”和“草稿”，不能替代确定性引擎

## 是否值得引入 Agent 框架

结论：**值得，但不要一开始就重型多 Agent 化。**

### 值得引入的原因

- 你们的系统天然是多工作台、多角色、多上下文、多工具调用场景
- 已经规划了 `AI模型管理中心`，具备统一模型入口条件
- 后续会有客服、内容、运营、供应商、渠道等多个 AI 消费端
- 如果不提前设计 Agent 层，后面容易出现每个模块各自写一套 prompt + tool calling

### 不建议一开始重型化的原因

- 一期主线还是交易、供应商、客服、发货闭环
- 重型多 Agent 编排容易过早增加复杂度
- 没有 eval、trace、审批、审计、回放前，Agent 容易不可控

## 推荐的未来架构分层

建议在现有 `AI模型管理中心` 之上，再抽一层 `Agent Runtime / Agent Platform`：

### 第一层：规则真相源层

包括：

- 订单中心
- 库存中心
- 价格中心
- 售后中心
- 分销与佣金中心
- 积分与权益中心
- 权限中台
- 结算中台

职责：

- 所有确定性规则最终都在这里落地

### 第二层：AI 模型治理层

即当前规划中的：

- `AI模型管理中心`

职责：

- 模型注册
- 标签
- 路由
- 配额
- 成本
- 语料
- 统一调用接口

### 第三层：Agent Runtime 层

建议新增能力：

- Agent 注册中心
- Tool Registry
- Prompt Registry
- Session / Memory 管理
- Planner / Executor
- Human-in-the-loop 审批
- Trace / Replay
- Eval / Benchmark

职责：

- 把“模型调用”升级为“可调用工具、可分步执行、可审计回放”的 Agent 执行层

### 第四层：业务 Agent 层

建议未来逐步建设：

- 客服副驾驶 Agent
- 内容生产 Agent
- 运营分析 Agent
- 商品提报 Agent
- 渠道运维 Agent

### 第五层：业务工作台层

包括：

- 用户统一入口
- 平台后台入口
- 供应商协同入口
- 客服工作台入口
- 发货执行入口

## Agent 框架是否有开源选择

有，而且可以考虑，但应按用途选，不要盲目追热度。

### 适合重点关注的 Agent / AI 应用框架

- `LangGraph`
  - 开源
  - 适合有状态、多步骤、可中断恢复的 Agent 工作流
  - 很适合客服协同、内容编排、渠道异常处理

- `Mastra`
  - 开源
  - 更偏现代 TypeScript Agent 应用开发
  - 如果你们前后端未来偏 Node/TS，会比较顺手

- `AutoGen`
  - 开源
  - 适合多 Agent 协作研究和复杂编排
  - 更适合探索，不一定适合作为一期生产主骨架

- `CrewAI`
  - 开源
  - 更偏角色式协作 Agent
  - 适合 PoC，不一定适合作为高可控生产底座

### 更适合你们的组合，不是纯 Agent 框架，而是“Agent 运行栈”

建议组合思路：

- 模型网关：`LiteLLM`
- 观测评测：`Langfuse`
- Agent 编排：`LangGraph` 或 `Mastra`
- 长流程事务编排：`Temporal`
- 检索：`OpenSearch`
- 权限：`Casbin/OpenFGA`

这套组合比“单押一个万金油 Agent 框架”更适合企业级未来设计。

## 面向未来的设计建议

### 一期建议做

- 统一模型网关
- 统一 AI 调用协议
- Prompt Registry 预留
- Tool Registry 预留
- Trace / Audit 预留
- Human approval 节点预留
- 客服副驾驶 PoC
- 内容生产 Agent PoC

### 一期预留但不重做

- Session memory
- Agent 配置中心
- Eval 数据集
- A/B 路由
- 多 Agent 协作

### 二期再做

- 通用 Agent 平台
- 跨工作台共享 Agent 身份
- 自主规划型 Agent
- 多 Agent 协同编排
- 智能运维 Agent
- 复杂知识图谱与长期记忆

## 最重要的设计原则

1. **Agent 只能编排，不应成为业务真相源。**
2. **所有高风险动作必须保留人工确认或规则引擎兜底。**
3. **所有 Agent 调用都必须可审计、可回放、可评测。**
4. **先做单 Agent + 工具调用，再做多 Agent 协作。**
5. **先做高频低风险场景，再逐步触达复杂场景。**

## 最终判断

你们这个项目非常适合做成一个：

`交易与中台规则稳定 + AI 能力统一治理 + Agent 逐步进入各工作台协作`

的未来架构。

最佳路径不是：

- “把系统全部改造成 Agent 系统”

而是：

- “把系统设计成对 Agent 友好、对未来可扩展、对核心规则仍可控的企业应用”
