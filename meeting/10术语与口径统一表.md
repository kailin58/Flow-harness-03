# 10术语与口径统一表

## 目的

本文件用于统一当前 `meeting/` 目录中最容易混淆的概念，避免后续继续出现：

- 把一级入口当成业务端域
- 把治理工作域当成新业务端域
- 把增强模块当成独立系统
- 把中台能力当成前台工作台
- 把 Epic、模块、端域、入口混成一层

统一目标：

- 对外沟通时，口径稳定
- 内部排期时，颗粒度一致
- 技术实现时，边界清楚

## 当前冻结口径

当前统一采用：

- `5个一级入口`
- `9个业务端域`
- `若干治理工作域`
- `若干端内增强模块`
- `若干后台支撑能力`

当前没有新增第 10 个业务端域。

## 外部渠道平台范围口径

当前涉及第三方渠道、渠道店铺、外部平台导流时，平台范围统一表述为：

- 如 `淘宝`、`抖音`、`小红书`、`京东`、`唯品会`、`快手`、`百度` 等第三方平台

统一说明：

- 这是平台范围口径，不等于一期全部同时接入
- 用户统一入口承接这些平台的导流、回站与参数透传
- `外部渠道与店铺管理端` 负责这些平台的接入、授权、能力矩阵与映射治理
- 一期试点平台数量与顺序，以 `meeting/7一期范围冻结.md` 和 `meeting/8一期优先级排序.md` 为准

## 核心术语定义

| 术语 | 定义 | 当前是否冻结 | 典型例子 | 是否影响“9个业务端域”数量 |
|---|---|---|---|---|
| 一级入口 | 面向用户或岗位的一级访问入口，强调访问形态与角色感知 | 是 | 用户统一入口、平台后台入口 | 否 |
| 业务端域 | 服务某类角色并承接相对完整业务流程的业务工作域 | 是 | C端交易端、客服工作台端 | 是 |
| 治理工作域 | 负责统一治理、配置、审计、能力输出的工作域 | 是 | 账号与权限中心、AI模型管理中心 | 否 |
| 增强模块 | 某个业务端域内部的增强交互或资产模块 | 是 | 对话模块、衣橱模块、行程规划模块 | 否 |
| 后台支撑能力 | 后端真相源、规则源、执行引擎、中台能力 | 是 | 订单中心、库存中心、结算中台 | 否 |
| Epic | 一期排期与交付视角下的建设切片 | 是 | E1 交易主链路、E10 账号治理台 | 否 |
| 后端模块 | 代码与运行时的工程模块 | 是 | `trade`、`inventory`、`iam-auth` | 否 |
| Agent Runtime | AI/Agent 运行与编排能力层 | 是 | Tool Registry、Prompt Registry、Trace | 否 |
| 一级导航 / 一级菜单 | 某个一级入口内**最顶层**的导航分组：用户端习惯称「一级导航」（如底部 Tab），后台与协同工作台习惯称「一级菜单」（如侧栏顶层项）；**含义等价**，仅语境差异 | 是 | 用户端 5 Tab、平台后台侧栏 | 否 |
| 核心子入口 | 用户统一入口内除底部一级导航外的重要模块入口，须明确挂载在首页、「我的」或业务页等位置，避免与主导航抢位 | 是 | 推广中心、积分商城、对话入口 | 否 |
| 主导航（极简） | 发货执行等轻量入口的顶层路径，强调 Tab/侧栏**数量上限**与首屏心智，不展开为多级业务后台 | 是 | 发货任务 / 异常 / 我的 | 否 |

## 最容易混淆的几组概念

### 1. 一级入口 vs 业务端域

- 一级入口强调“从哪里进入”
- 业务端域强调“承接什么业务流程”

例子：

- `平台后台入口` 是一级入口
- `平台运营后台端`、`内容创造端`、`外部渠道与店铺管理端` 是挂在该入口下的业务端域

### 2. 业务端域 vs 治理工作域

- 业务端域服务某类角色的完整工作流
- 治理工作域负责统一治理、配置、审计、能力输出

例子：

- `内容创造端` 是业务端域
- `AI模型管理中心` 是治理工作域

### 3. 业务端域 vs 增强模块

- 业务端域可独立讨论职责边界
- 增强模块只在某个端域内部增强体验或能力

例子：

- `C端交易端` 是业务端域
- `对话模块`、`衣橱模块`、`行程规划模块` 是其增强模块

### 4. 业务端域 vs Epic

- 业务端域是业务分层
- Epic 是排期分层

例子：

- `平台运营后台端` 是业务端域
- `E10 账号与权限中心治理台` 是一期建设项

### 5. 治理工作域 vs 后台支撑能力

- 治理工作域通常会有平台后台中的管理界面
- 后台支撑能力更多是后端真相源或中台服务，不一定直接暴露为独立页面

例子：

- `账号与权限中心` 是治理工作域
- `权限与组织中台` 是后台支撑能力

## 快速判断规则

讨论某个新能力时，建议按下面顺序判断：

1. 它是不是一个新的访问入口？
2. 它是不是一个新的业务端域？
3. 它是不是已有端域下的治理工作域？
4. 它是不是已有端域下的增强模块？
5. 它是不是后台支撑能力？

### 判断为“新业务端域”前，至少要满足

- 有独立服务对象
- 有完整主流程
- 与现有端域边界明显不同
- 不是单纯治理、配置、审计能力
- 不是现有端域内部增强模块

只要不满足以上条件，就不要轻易新增业务端域数量。
