## 核心决策一览

### 组织架构（横向）

**架构定型：固定4层组织（槽位全部写死）+ Deep Agent 执行树（无限深）**

```
固定组织层（4层，槽位数全部写死，通信规则严格约束）：
  L0  CEO     ：1个（写死）
  L1  VP      ：11个（写死，不可新增第12个）
  L2  总监    ：每VP固定11个槽位，全平台 11×11 = 121个（写死）
  L3  经理    ：每总监固定11个槽位，全平台 11×11×11 = 1331个（写死）

  槽位状态只有两种：✅ 激活（active） | ⏸ 预设占位（placeholder）
  数量写死 = 不可新增槽位，只允许在现有槽位上切换激活状态

执行树层（经理以下，树状结构，深度无限，不写死）：
  deep_agent     ← 可嵌套内部节点，深度执行者，可孵化子节点或挂载预设流程
  sub_deep_agent ← 叶节点，原子执行单元，不再向下分发
```

- **升级路径**：3层 → 4层（加VP）→ 固定4层+槽位写死+Deep Agent执行树（最终定型）
- VP 层由平台统一治理（共11个VP位，不允许企业新增VP）
- VP01 数字技术VP：11个总监槽位（6个已激活：Explore/Plan/General/Inspector/Research/DigitalOps；5个预设占位）
- 所有VP总监槽位固定11个，各VP预设总监方向按域规划，详见：[01_3层升4层架构.md](./01_3层升4层架构.md)
- 写死VP（7个）：VP01/VP02/VP03/VP06/VP07/VP08/VP10，企业不可删改
- 可选VP（4个）：VP04/VP05/VP09/VP11，企业可激活或停用，不可新增第12个
- 企业只能在现有槽位内激活/停用总监或经理，不可新增槽位
- 经理以下的 `deep_agent` 树不限层数，`sub_deep_agent` 为叶节点，任意节点可挂载预设流程
- 组织能力树写死：**VP(11) → 总监(11/VP) → 经理(11/总监) → deep_agent树 → Skills**
- 治理约束写死：Skills 调用受 L2/L5 约束；deep_agent 输出必须经 L7/L8；进化只在 L11、上线只经 L10
- COMM_RULES 基于 level（固定组织层规则严格；执行树内部节点只和直接上下级通信）
- **节点身份规范（新增写死规则）**：
  - 每个节点（VP/总监/经理/deep_agent）必须声明三要素：人设（Persona）/ 专业（Domain）/ 技能（Skills）
  - Skills 槽位：预设11个（起始基准），不设上限，可按需扩展
  - Skills 归属调用层级：VP级 / 总监级 / 经理级 / 执行树级，下级不可调用上级未开放的 Skill
  - **deep_agent 职责唯一原则**：同一父节点下，每个 deep_agent 工作职责互不重叠，职责唯一保障专业化积累

```

│  L5  安全策略层   准入/合规检查（原L2）               │
│  L6  执行监控层   Monitor/AutoRetry（原L3）           │
├─────────────────────────────────────────────────────┤
│  L7  Inspector层  质检/审查（原L4）                  │
│  L8  质量门禁层   通过/拦截（原L5）                   │
│  L9  反馈闭环层   复盘/优化（原L6）                   │
│  L10 版本发布层   候选/审批/灰度/回滚                 │ ← 新增
│  L11 进化实验层   沙盒自优化/实验评测/隔离            │ ← 新增
└─────────────────────────────────────────────────────┘
```

**两维度说明：**
```

  固定4层组织（槽位写死）：
```

## 子层规划（系统预设不污染）

目标：用户Agent可进化，系统预设保持稳定且不被自动污染。

### 层级结构（含子层）

```
L1 入口扩展层
L2 权限管理层
  ├── L2-SYS              （系统预设身份与权限域）
  └── L2-USER             （用户Agent身份与权限域）
L3 任务编排层
  └── L3-SKILL-ASSEMBLY   （deep_agent 能力档案组装：三层 Skill 按优先级合并为最终清单）
L4 协作编排层
L5 安全策略层
L6 执行监控层
  ├── L6-SYS              （系统执行监控与告警）
  ├── L6-USER             （用户执行监控与告警）
  └── L6-SKILL-RESOLVER   （执行中 Skill 版本派发 + 调用日志记录）
L7 Inspector层
L8 质量门禁层
L9 反馈闭环层
L10 版本发布层
  ├── L10-SYS-RELEASE     （系统预设发布通道）
  ├── L10-USER-RELEASE    （用户Agent发布通道）
  └── L10-SKILL-PROMOTE   （Skill 升层审批：用户训练/自进化能力升级到预设层的专用流程）
L11 进化实验层
  ├── L11-USER-TRAIN      （用户主动训练配置的验证通道；管控重点：RBAC + 质量验证）
  └── L11-USER-EVO        （agent 自进化沙盒；管控重点：代码沙盒隔离 + 严格质量门禁）
```

### 子层优先级（先规划）

1) **L2 权限管理层（第一优先）**
- 规划子层：`L2-SYS` / `L2-USER`
- 作用：先切清身份与作用域边界（谁能做什么）。

2) **L11 进化实验层（第二优先）**
- 规划子层：`L11-USER-TRAIN` / `L11-USER-EVO`
- 作用：区分「用户主动训练」和「agent自进化」两条不同风险等级的通道；系统预设默认不进入任何子层。

3) **L10 版本发布层（第三优先）**
- 规划子层：`L10-SYS-RELEASE` / `L10-USER-RELEASE` / `L10-SKILL-PROMOTE`
- 作用：统一发布门禁；`L10-SKILL-PROMOTE` 专门处理能力升层审批，任何进化结果先过评测+审批再写入预设层。

4) **L6 执行监控层（第四优先）**
- 规划子层：`L6-SYS` / `L6-USER` / `L6-SKILL-RESOLVER`
- 作用：运行态隔离监控、限流、告警与审计分治；`L6-SKILL-RESOLVER` 在热路径上按优先级规则派发正确的 Skill 版本。

5) **L3 任务编排层（第五优先）**
- 规划子层：`L3-SKILL-ASSEMBLY`
- 作用：任务分配时预先组装 deep_agent 的完整能力档案（pre-flight），避免执行中重复解析。

### 核心原则（先写死）
- 进化只在用户隔离域发生，不允许直接写入系统预设域。
- 系统预设只接受 L10 发布层合入，不接受实验层直连。
- 任何发布必须可追踪、可审批、可回滚。
- L11-USER-TRAIN 与 L11-USER-EVO 共用 L10 发布门禁，但进入 L10 前的验证强度不同（训练通道低于进化通道）。

## 企业级能力分层归属（标准口径）

- **L1 入口扩展层**：注册/登录入口、请求归一化、租户识别、限流防刷。
- **L2 权限管理层**：角色体系、指挥权分配、租户与组织隔离、审批权限。
- **L3 任务编排层**：任务拆解、Agent分配、模型路由与调度策略；子层 `L3-SKILL-ASSEMBLY` 负责在任务分配时组装 deep_agent 的完整能力档案（三层 Skill 按优先级合并）。
- **L4 协作编排层**：多指挥者协调、跨Agent通信、冲突仲裁、状态机推进。
- **L5 安全策略层**：数据访问策略执行、合规检查、敏感操作拦截（deny-by-default）。
- **L6 执行监控层**：运行监控、告警、重试、熔断、分域监控（SYS/USER）；子层 `L6-SKILL-RESOLVER` 负责执行中按优先级规则派发正确的 Skill 版本并记录调用日志。
- **L7 Inspector层**：目标对齐检查、完整性验证、质量评分、风险扫描、安全漏洞检测。
- **L8 质量门禁层**：通过/打回决策，不达标结果不得进入发布链路。
- **L9 反馈闭环层**：复盘沉淀、经验回写、优化建议管理。
- **L10 版本发布层**：候选版本、审批、灰度发布、正式发布、回滚；子层 `L10-SKILL-PROMOTE` 专门处理 Skill 升层审批（用户训练/自进化能力 → 预设层）。
- **L11 进化实验层**：子层 `L11-USER-TRAIN`（用户主动训练配置验证，管控重点RBAC+质量）和 `L11-USER-EVO`（agent自进化沙盒，管控重点代码隔离+严格门禁）；进化结果不可直达生产，必须经 L10 发布链路。

## L2 与 L5 职责区分（关键说明）

> 两层最容易混淆，必须区分清楚再实施，否则实现时职责必然交叉。

| 维度 | L2 权限管理层 | L5 安全策略层 |
|------|-------------|-------------|
| 核心问题 | **Who can do What**（谁有权限做什么） | **How to do safely**（怎么做才合规） |
| 检查对象 | 用户身份 / 角色 / 租户归属 | 操作内容 / 文件路径 / 命令参数 |
| 典型场景 | 这个用户/租户有没有权限发起这类任务 | 这个文件读取操作是否在白名单内 |
| 实现机制 | RBAC / 租户隔离 / 审批授权 | Deny-by-Default / 白名单 / 合规规则 |
| 触发时机 | 任务进入编排前（L2 先于 L3） | 每次工具调用/操作执行时（L5 伴随 L6） |

**记忆口诀**：L2 管"人"，L5 管"事"。

## 各层详解（11层完整版）

### L1 — 入口扩展层

```
职责：任务进入系统的第一道门
当前实现：pass-through（直接放行，空插件链，零副作用）

未来可插入的能力：
  ├── API认证（REST API上线时接入）
  ├── 频率限制（防滥用/防刷）
  ├── 请求格式归一化（兼容不同输入格式）
  ├── A/B测试路由（不同用户走不同流程）
  └── 任何无法预见的入口逻辑

实现机制：插件钩子链
  LAYER1_HOOKS = []              // 当前为空，注册即生效，不需要改主流程
  MULTIMODAL_NORMALIZER_HOOK     // 多模态归一化占位（当前空实现）
    输入：任意格式（图片/视频/音频/文档）
    输出：统一文本 + 结构化描述（供后续层处理）
    未来插入：图片描述生成 / 音频转文字 / 视频帧分析
```

### L2 — 权限管理层

```
职责：Who can do What（谁能做什么）
当前实现：pass-through（单用户/开发场景直接放行）

完整实现目标：
  ├── RBAC 角色权限控制
  │   CEO / VP / 总监 / 子Agent 各层操作权限边界
  ├── 租户隔离
  │   SaaS场景下租户A不可访问租户B的数据/Agent/结果
  ├── 操作授权
  │   敏感操作（支付/删除/部署）需要额外审批
  └── BYOK API Key 管理
      用户自带模型Key的权限范围控制与隔离
```

### L3 — 任务编排层

```
职责：CEO的核心调度逻辑，支持两种路由模式

模式A：传统编排模式（步骤明确的任务）
  step1_analyze   分析任务类型、目标、风险等级
  step2_decompose 拆解为子任务列表
  step3_assign    分配给对应VP（不直接分配到总监）
                  VP再负责分配给自己的总监

模式B：蜂群路由模式（范围大/边界模糊/步骤不可预知的任务）
  step1_analyze   识别任务为「蜂群类任务」（见判断规则）
  step2_skip      不拆解，跳过 decompose
  step3_swarm     整包路由给对应总监：
                    { goal, expected_result, constraints }
                  总监接收后启动蜂群，蜂群内部自拆解

蜂群类任务判断规则（满足全部三条）：
  ① task.type in [scan / explore / batch_process / discovery]
  ② task.scope == "large"（范围大）
  ③ task.steps == "unknown"（步骤不可预知）

关键变更：step3_assign / step3_swarm 路由目标均为 VP（不跳过VP层）
          VP是本域对外的唯一出口，不可被绕过
```

#### L3-SKILL-ASSEMBLY — 能力档案组装子层

```
触发时机：step3_assign 完成后，任务正式下发给 deep_agent 之前（pre-flight）

职责：
  1. 读取目标 deep_agent 的身份（id / 所属经理 / 激活层级）
  2. 从三个来源加载该 deep_agent 的 Skill 清单：
       预设能力层  ← 平台写死的通用底座 Skill
       用户训练层  ← 该租户/用户为此 deep_agent 配置的训练扩展
       自进化层    ← 该 deep_agent 自己进化产生的 Skill（若有）
  3. 按优先级规则合并，生成最终能力档案（Skill Profile）：
       规则1：预设层 + 用户训练层叠加，用户训练层可对预设层的同一参数点精确覆盖
              （覆盖仅限该参数点，预设层其余部分保留）
       规则2：自进化层优先级最低，仅在预设层和用户训练层均未定义某 Skill 时生效
       规则3：同一 Skill 若三层均有定义，最终生效版本 = 预设+用户训练叠加版
  4. 将 Skill Profile 随任务一起下发给 deep_agent

输出：{ task, skillProfile: { skills: [...], resolvedAt: timestamp } }
隔离保证：三层 Skill 的原始定义互不污染，Skill Profile 是只读快照
```

### L4 — 协作编排层

```
职责：多Agent并行执行时的协调管理

Task State Machine（完整7状态）：
  PENDING → DELIBERATING → PLANNING → EXECUTING
  → INSPECTING → REVIEWING → DONE / FAILED

核心组件：
  ① VP协调器
     多VP并行时的进度同步
     跨VP通信请求路由（CEO强制主持）

  ② 冲突检测器
     发现Agent输出冲突时介入
     触发 DeliberationEngine（商议引擎）解决冲突

  ③ 死锁检测器
     某状态停留超过阈值 → 自动告警 → 触发人工介入

  ④ 看板数据收集
     实时收集所有Agent状态 → 供终端/Web看板展示
```

#### L4-SWARM-COORDINATOR — 蜂群协作编排子层

```
适用：总监启动蜂群模式时激活（传统编排模式不激活此子层）
完整设计见：08_蜂群牧羊犬协议.md

五个组件：

① Wave Scheduler（波次调度器）
   管理探索波 → 执行波的时序
   task_pool 出现足够任务块时，自动通知经理孵化执行波工蜂
   参数：首波数量、扩展批次大小、最大并发

② Blackboard Task Manager（任务池管理器）
   维护 task_pool.* 的并发锁（防多工蜂同时认领同一任务块）
   管理任务块状态流转：open → in_progress → done / blocked
   超时未完成的 in_progress 任务块自动重置为 open

③ Spawn Arbiter（增殖请求仲裁器）
   接收工蜂的增殖申请，检查是否超出规模上限
   批准 → 通知经理孵化；拒绝 → 写拒绝原因到黑板
   规模上限（写死）：单经理 ≤20个agent，蜂群总计 ≤200个，深度 ≤5层

④ Sheepdog Report Handler（牧羊犬上报处理器）
   接收牧羊犬上报，判断处理层级（经理 or 总监）
   触发对应处理：孵化/终止工蜂 / 通知总监

⑤ Convergence Judge（终止条件评估器）
   持续计算 swarm_status.coverage_pct 等指标
   终止条件满足 → 触发收敛流程，写入 swarm_status.*
```

**状态机超时与回滚配置表：**

| 状态 | 超时阈值 | 超时后动作 | 可回退到 |
|------|---------|----------|---------|
| PENDING | 120s | fail | — |
| DELIBERATING | 300s | escalate | PENDING |
| PLANNING | 600s | retry（1次）| DELIBERATING |
| EXECUTING | 3600s | escalate | PLANNING |
| INSPECTING | 300s | retry（2次）| EXECUTING |
| REVIEWING | 600s | escalate | INSPECTING |
| DONE / FAILED | 终态 | — | 不可回退 |

```
escalate = 触发 Human-in-the-Loop 流程
死锁判定：任务在非终态停留超过 timeout × 3 且无任何状态日志写入
          → 强制写 S3 审计记录 + 发 escalate 通知
```

#### L6-SKILL-RESOLVER — Skill 运行时派发子层

```
触发时机：deep_agent 在执行中实际调用某个 Skill 的瞬间（runtime 热路径）

职责：
  1. 接收调用请求：{ agentId, skillId, params }
  2. 从该 deep_agent 的 Skill Profile（由 L3-SKILL-ASSEMBLY 预先组装）中
     查找 skillId 对应的最终版本实现
  3. 若 Skill Profile 中不存在该 skillId，按以下顺序 fallback：
       a. 预设层全局查找
       b. 仍未找到 → 返回 SKILL_NOT_FOUND，记录告警
  4. 派发到正确版本的 Skill 实现并执行
  5. 记录调用日志（agentId / skillId / 使用版本层 / 耗时 / 结果）
     → 日志供 L7 Inspector 审查，同时写入 S3 审计库

与 L3-SKILL-ASSEMBLY 的分工：
  L3-SKILL-ASSEMBLY = pre-flight 组装（一次性，任务开始前）
  L6-SKILL-RESOLVER = runtime 派发（每次 Skill 调用时）
```

#### L10-SKILL-PROMOTE — Skill 升层审批子层

```
职责：处理「用户训练层 / 自进化层的 Skill → 预设能力层」的升级流程

触发条件（任意一条）：
  - 人工判定某 Skill 已足够成熟，发起升层提案
  - L11-USER-TRAIN 或 L11-USER-EVO 完成验证后自动生成升层提案

审批流程：
  1. 生成升层提案（skillId / 来源层 / 当前版本 / 质量报告摘要）
  2. L7 Inspector 补充安全漏洞检测报告
  3. L8 质量门禁通过
  4. 人工审批（支持双人确认，来自自进化层的 Skill 强制双人）
  5. 审批通过 → 写入预设能力层，版本号递增，更新 S4 规则库
  6. 审批拒绝 → 该 Skill 继续停留在原层级，记录拒绝原因

升层后的效果：
  下次 L3-SKILL-ASSEMBLY 组装能力档案时，该 Skill 进入预设层清单
  已有使用此 Skill 的 deep_agent 在下次任务时自动获得新版本

隔离保证：
  升层操作不影响正在执行的任务（当前任务的 Skill Profile 不变）
  升层结果仅对新任务生效
```

## Deep Agent 分层归属与治理链

定位：Deep Agent 是执行能力体，不是总框架；总框架仍由11层治理链路承载。

- **主归属层：L3 任务编排层**
  - 负责 Deep Agent 的分配、路由、降级和成本/SLA权衡。
  - 子层 `L3-SKILL-ASSEMBLY` 在任务下发前组装 deep_agent 的完整能力档案。
- **协同层：L4 协作编排层**
  - 负责多 Deep Agent 并行协作、冲突仲裁和状态推进。
- **运行层：L6 执行监控层**
  - 负责运行健康、时延、成本、失败重试与熔断反馈。
  - 子层 `L6-SKILL-RESOLVER` 在执行中按优先级派发正确的 Skill 版本。
- **治理链：L7 -> L8 -> L10 -> L11**
  - L7 质量检查（含自进化产出的安全漏洞检测）
  - L8 门禁决策
  - L10 发布与回滚；子层 `L10-SKILL-PROMOTE` 处理 Skill 升层审批
  - L11 进化实验（`L11-USER-TRAIN` 用户训练通道 / `L11-USER-EVO` 自进化沙盒）

执行红线：
- Deep Agent 不得绕过 L2/L5 权限与安全策略。
- Deep Agent 输出不得绕过 L7/L8 直接进入生产。
- 进化结果不得直写生产，必须经 L10 发布链路。
- 自进化层产生的 Skill 优先级最低，不得覆盖预设层或用户训练层已定义的 Skill。

## deep_agent 四层能力模型

> **补充日期**：2026-04-14 | **性质**：新增设计决策——deep_agent 自进化能力架构

### 每个 deep_agent 的能力由四层叠加组成

```
┌─────────────────────────────────────────────────────────┐
│  总能力层（Aggregated Output）                           │
│  → 对外输出的最终能力视图，由下方三层聚合生成            │
│  → 由 L3-SKILL-ASSEMBLY（pre-flight）+                  │
│     L6-SKILL-RESOLVER（runtime）共同维护                │
├─────────────────────────────────────────────────────────┤
│  自进化层（Self-Evolution）                              │
│  → agent 自己写代码产生的 Skill                          │
│  → 在 L11-USER-EVO 沙盒内生成，优先级最低               │
├─────────────────────────────────────────────────────────┤
│  用户训练配置层（User Training）                         │
│  → 用户/租户主动配置的训练扩展                           │
│  → 在 L11-USER-TRAIN 通道验证，RBAC 隔离                │
├─────────────────────────────────────────────────────────┤
│  预设能力层（Platform Preset）                           │
│  → 平台写死的通用底座 Skill                              │
│  → 只有 L10-SKILL-PROMOTE 审批通过后才能更新             │
└─────────────────────────────────────────────────────────┘
```

### Skill 聚合优先级规则（写死，不可在运行时修改）

```
规则1（叠加）：预设层 + 用户训练层叠加，形成"稳定能力组合"
  - 用户训练层可对预设层同一 Skill 的同一参数点精确覆盖
  - 覆盖仅限该参数点，预设层其余参数保留
  - 两层合并后视为整体，优先级高于自进化层

规则2（自进化层最低）：
  - 仅当"稳定能力组合"中不存在某 Skill 时，自进化层的版本才生效
  - 若三层均定义了同一 Skill，最终生效 = 规则1的叠加结果（自进化版本忽略）

规则3（层间隔离）：
  - 三层的原始定义互不污染，总能力层是只读快照
  - 修改任一层需走对应的治理流程，不可直接写入总能力层

优先级速查：
  预设 + 用户训练（叠加）  >  自进化
  用户训练（参数级）       >  预设（同参数点）
```

### Skill 升层路径（能力晋升流程）

```
用户训练层 Skill 升级到预设层：
  L11-USER-TRAIN 验证通过
    → L7 质量检查
    → L8 质量门禁
    → L10-SKILL-PROMOTE（单人审批）
    → 写入预设层，版本号递增

自进化层 Skill 升级到预设层：
  L11-USER-EVO 验证通过
    → L7 质量检查（含安全漏洞检测）
    → L8 质量门禁
    → L10-SKILL-PROMOTE（强制双人审批）
    → 写入预设层，版本号递增

升层生效规则：
  - 升层操作不影响正在执行的任务（当前 Skill Profile 快照不变）
  - 仅对升层完成后新创建的任务生效
  - 升层后原层级的 Skill 条目保留（可回滚）
```

### 四层与11层工作流的对应关系

## Skills 生命周期与分层约束

定位：Skills 为平行能力插件，由 Agent 挂载与调用，不构成新的组织层级。

生命周期：
- 注册（Registry）-> 沙盒验证（L11）-> 质量门禁（L8）-> 发布（L10）-> 运行监控（L6）-> 回滚/下线

分层约束：
- **L3**：Skills 路由与调用策略（由编排统一调度）。
- **L2**：Skills 调用权限控制（角色/租户/作用域）。
- **L5**：Skills 运行安全策略（命令/文件/API 白名单）。
- **L8/L10**：Skills 发布门禁、版本发布与回滚。
- **L11**：用户新增或变更 Skills 的实验与验证域。

治理要求：
- 用户可新增 Skills，但仅在 USER 域生效，不得污染 SYS 预设域。
- Skills 必须版本化、可审计、可回滚。
- 高风险 Skills 默认拒绝，需显式审批放行。

### Skills VP 归属模型

Skills 按"调用者归属"分三级，不按功能分类：

```
平台级 Skill（Platform Skill）
  归属：VP02 平台治理VP 负责注册与发布
  调用权：任意VP、任意总监均可调用
  示例：文件读写、代码执行、HTTP请求

VP级 Skill（VP Skill）
  归属：对应VP自己负责注册与发布
  调用权：仅该VP及其下属总监可调用
  示例：CRM数据读取（归 VP07 销售增长VP）

总监级 Skill（Director Skill）
  归属：总监负责注册，所属VP审批发布
  调用权：仅该总监及其子Agent可调用
  示例：特定数据库查询模板
```

**Skills 依赖管理：**
```
每个 Skill 声明 dependencies[]
发布时由 L8 质量门禁层做依赖树检查
禁止循环依赖
上游 Skill 升级时，自动通知所有依赖它的 Agent
```

## 第三维度：数据生命周期管理

> **补充日期**：2026-04-13  
> **性质**：架构补全（不修改现有11层，不修改组织层级，正交补充）  
> **结论**：现有两维（组织横向 + 工作流纵向）缺少数据管理维度，需补充四类存储库

---

### 背景：为什么需要第三维度

现有11层是**纵向工作流**，解决"任务如何流转"。  
以下8类数据贯穿整个任务生命周期，跨越多个工作流层，没有专属归属：

| 数据条目 | 现有层覆盖 | 覆盖程度 |
|---------|-----------|---------|
| 用户需求 | L1接收 + L3解析 | ⚠️ 有接收无持久化存储 |
| 任务目标 | L3拆解分配 | ⚠️ 运行时存在，无持久化 |
| 中间结果 | L6捕获 + L7检查 | ❌ 无专属存储层 |
| 规则约束 | L2权限 + L5安全 | ⚠️ 规则分散两层，无统一注册表 |
| 工具输出 | L6收集 + L7质检 | ⚠️ 流过即消，无结构化归档 |
| 审批状态 | L8门禁 + L10发布 | ⚠️ 两处分散，无统一审批状态链 |
| 异常记录 | L6告警/重试 | ❌ 只有实时告警，无持久异常库 |
| 修改历史 | L10版本 + L9复盘 | ❌ 无全链路变更审计轨迹 |

**结论**：缺的不是工作流层，是数据的归属与治理。

```
现有两维：
  横向（组织）：CEO → VP → 总监 → 子Agent
  纵向（流程）：L1 → L2 → ... → L11

新增第三维：
  数据层（Data）：S1上下文库 / S2制品库 / S3审计库 / S4规则库
```

三个维度互不干扰，任意一层均可按权限读写对应存储库。

### S3 分布式写入保障原则（缺口8）

> **补充日期**：2026-04-14
> **性质**：实现约定——"写入不可拒绝"在分布式场景下的具体保障机制
> **状态**：✅ 原则已定，实施时遵守

**问题背景**："写入不可拒绝"是 S3 的核心约束，但在分布式场景下（存储服务暂时不可用、网络抖动、节点重启等），若处理不当，会导致主流程被 S3 写入阻塞、审计失败中断业务流程，或业务逻辑误依赖 S3 返回值产生错误控制流。

**三条保障原则（写入架构文档，实现时遵守）：**

**原则1：S3写入必须异步，不阻塞主流程**
```
主流程（L3/L4/L6/L8/L10 等任意层）
  │
  ├─ 执行业务逻辑（同步，主流程）
  │
  └─ 触发 S3 审计写入（异步，fire-and-forget）
       → 不等待写入结果，不阻塞下一步业务动作
```

**原则2：S3写入失败时的兜底链路**
```
S3 写入请求
  │
  ├─ 成功 → 正常归档
  │
  └─ 失败（S3不可用 / 超时 / 网络错误）
       │
       ├─ Step1: 写本地缓冲文件（Local Buffer）
       │          路径：<工作目录>/.audit_buffer/<timestamp>_<event_id>.json
       │          格式：与 S3 审计记录相同结构（保证后续可无损回写）
       │
       ├─ Step2: 定时重试（指数退避）
       │          1s → 2s → 4s → 8s → ... → 上限 N 次（默认10次）
       │          重试成功 → 将本地缓冲文件内容写入 S3 → 删除缓冲文件
       │
       └─ Step3: 重试超过 N 次 → 触发系统告警
                  告警级别：P1（高优先级）
                  告警内容：未同步的缓冲文件数量 + 最早未同步时间戳
                  ⚠️ 告警不触发业务流程中断，主流程继续执行
```

**原则3：S3 是"后置记录"，不是"前置控制"**
```
正确写法：业务决策先行 → 异步写 S3（后置记录）
错误写法：等待 S3 写入结果 → 基于结果决定下一步业务逻辑

任何层在写 S3 时，不可以：
  × 等待 S3 写入返回值再决定下一步业务逻辑
  × 把 S3 写入成功作为业务流程推进的前提条件
  × 因为 S3 不可用而拒绝继续执行主流程
```

**本地缓冲文件一致性保障（实现时遵守）：**
- 缓冲文件采用原子写入（先写临时文件再重命名），防止写入中途崩溃导致文件损坏。
- 系统启动时自动扫描缓冲目录，恢复未同步的历史记录。
- 缓冲文件本身不作为"已审计"凭证，S3 写入成功后方视为审计完成。

## 经验库 → 进化闭环（三段式路径）

### 第一段：经验沉淀（L9 → S2）

```
任务完成后，L9 反馈闭环层生成「经验条目」写入 S2 制品库：
  task_type          任务类型
  success            true / false
  failure_layer      失败发生在哪一层（如有）
  correction_trace   人工修正记录（如有）
  final_result       最终正确输出
  confidence_delta   本次任务使系统置信度提升/下降值
```

### 第二段：实验触发（S2 → L11）

```
触发条件（满足任意一个）：
  - 同类任务累计失败率超阈值（如 > 20%）
  - 人工修正记录累计超N条（如 > 10条）
  - 管理员手动触发实验

L11 进化实验层从 S2 拉取该类型经验条目
→ 生成优化提案（Proposal）

提案内容范围（低风险优先）：
  · Prompt 模板变更
  · 工具调用顺序调整
  · Agent 路由策略调整
  · 检索策略优化
  （不直接涉及核心代码）
```

### 第三段：提案发布（L11 → L8 → L10）

```
L11 在隔离沙盒中验证提案效果（新策略 vs 旧策略 A/B对比）
    ↓
L8 质量门禁层评估（新策略评分 > 旧策略才通过）
    ↓
L10 版本发布层：审批 → 灰度上线 → 全量
    ↓
全量后同步更新 S4 规则库对应规则版本
```

**闭环保障原则：**
```
进化只在 L11-USER-EVO 隔离沙盒发生
系统预设域不参与进化
任何进化结果不可直写生产，必须经 L10 发布链路
```

## 工具接入层（Tool Integration Layer）

> **补充日期**：2026-04-13  
> **性质**：架构扩展——Tool Port 的完整规划  
> **核心命题**：未来最关键的能力不是"模型会不会答"，而是"系统能不能拿到正确上下文并执行正确动作"

---

### 定位：在六边形架构中的位置

工具接入层是**Tool Port 的具体实现集合**，是 Agent 与外部世界之间的标准化桥梁。

```
内核（CEO + VP + 编排逻辑）
        ↓ 只能通过 Tool Port 调用
   ┌────────────────────────┐
   │      Tool Port         │  ← 统一接口契约，内核永不直连外部
   └────────────┬───────────┘
                ↓
   ┌────────────────────────────────────────────────┐
   │              工具接入层（Tool Adapters）         │
   │  T1文件  T2知识库  T3数据库  T4企业系统          │
   │  T5 API  T6搜索   T7表格   T8自动化平台         │
   └────────────────────────────────────────────────┘
                ↓
        外部真实系统
```

**关键原则**：适配器变化不影响内核；内核升级不破坏适配器；工具故障不传导到 Agent 层。

### 8类工具适配器规划

按接入特性分为四组：

#### 第一组：本地资源类

**T1 文件系统适配器（FileSystemAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 读写本地/容器文件、目录遍历、文件监听、临时文件管理 |
| 核心风险 | 路径穿越攻击、意外删除、写入生产配置 |
| 权限策略 | 白名单目录制（Deny-by-Default），读/写/删 三级独立授权 |
| 失败处理 | 不存在返回空而非报错；写失败必须记录审计事件（S3可用则异步写入S3，S3不可用则写本地缓冲文件，后台重试回写；不可因S3不可用而阻塞主流程） |
| 与现有层关系 | L5 安全策略层负责路径白名单校验；S2 制品库归档工具输出 |

---

#### 第二组：知识检索类

**T2 知识库适配器（KnowledgeBaseAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 向量检索、全文检索、文档片段召回、知识图谱查询 |
| 核心价值 | 为 Agent 提供领域专有上下文，解决模型训练数据时效问题 |
| 权限策略 | 按知识域隔离（租户A的知识库不可被租户B检索） |
| 质量机制 | 召回结果置信度低于阈值时，标记为"低置信参考"而非直接使用 |
| 与现有层关系 | 召回结果写入 S1 上下文库供全链路使用；L7 Inspector 做来源核查 |

**T6 搜索引擎适配器（SearchAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 互联网实时搜索、内网搜索、代码库搜索、文档搜索 |
| 核心风险 | 搜索结果噪音、幻觉注入、敏感信息泄露给外部搜索引擎 |
| 权限策略 | 外网搜索需显式开启；内网搜索按角色隔离 |
| 失败处理 | 搜索超时返回空集，不阻断主流程；结果必须标注来源与时间戳 |
| 与现有层关系 | 结果写入 S1 上下文库；L5 安全策略层过滤敏感查询词 |

#### 第三组：数据存取类

**T3 数据库适配器（DatabaseAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 关系型DB查询/写入、NoSQL读写、时序数据库、向量数据库 |

| 操作约束 | 写操作必须在事务内；影响行数超过阈值需二次确认 |
| 与现有层关系 | 写操作触发 S3 审计记录；L8 门禁可设置数据变更审批规则 |

| 项目 | 说明 |
|------|------|

| 典型场景 | 财务报表生成、数据分析输出、业务指标汇总 |
| 权限策略 | 按文件/Sheet/单元格区域三级授权 |

| 项目 | 说明 |

| 与现有层关系 | 异步状态变更触发 S3 审计；L4 协作编排层管理异步等待状态 |

### 工具注册表（Tool Registry）

所有工具适配器上线前必须在注册表中登记，注册表存储在 **S4 规则库**。

**必填字段：**

| 字段 | 说明 |
|------|------|
| `tool_id` | 全局唯一标识 |
| `category` | T1-T8 分类 |
| `risk_level` | 低/中/高/极高（影响调用审批策略） |
| `allowed_callers` | 允许调用的 Agent 层级（level）列表 |
| `operations` | 支持的操作列表及各操作的读/写/管理级别 |
| `rate_limit` | 频率限制（次/分钟） |
| `timeout_ms` | 超时阈值 |
| `fallback` | 超时或失败时的降级策略 |
| `idempotent` | 是否幂等（影响重试策略） |
| `audit_required` | 是否强制写 S3 审计（高风险操作必须为 true） |

### 工具调用标准链路

Agent 发起工具调用的完整路径：

```
Agent（任意层级）
  │
  ├─ 1. 查询工具注册表（S4）→ 确认工具存在且版本有效
  │
  ├─ 2. L2 权限检查 → 调用方 level 是否在 allowed_callers 中
  │
  ├─ 3. L5 安全策略检查 → 操作是否在白名单，参数是否合规
  │
  ├─ 4. 风险等级判断
  │     低/中风险 → 直接执行
  │     高风险    → 需 L8 门禁审批后执行
  │     极高风险  → 需 L10 发布链路人工审批
  │
  ├─ 5. Tool Port → 路由到对应 Adapter → 调用外部系统
  │
  ├─ 6. 结果处理
  │     成功 → 输出写 S2 制品库 → 返回 Agent
  │     失败 → 触发 L6 重试/熔断逻辑（业务决策先行）
  │             + 异步写 S3 异常记录（后置记录，不阻塞L6决策）
  │
  └─ 7. 审计写入（risk_level=高时必须）→ 异步写 S3 审计库（不等待返回值）
```

### 工具治理规则

#### 分级审批策略

| 风险等级 | 典型操作 | 审批要求 |
|---------|---------|---------|
| 低 | 只读查询、搜索、知识库检索 | 直接执行，事后审计 |
| 中 | 文件写入、数据库写入、API调用 | L5 合规检查通过即可执行 |
| 高 | 企业系统写入、批量数据变更 | L8 门禁审批，同步通知责任人 |
| 极高 | 财务操作、合同签署、生产DB删除 | L10 人工审批，双人确认，留痕 |

#### 稳定性治理

| 机制 | 说明 | 管理层 |
|------|------|-------|
| 熔断器 | 连续失败超阈值暂停调用，自动恢复检测 | L6 执行监控 |
| 限流 | 按工具/租户/Agent 三级限流 | L2 权限层 |
| 超时隔离 | 工具超时不阻断主流程，降级返回 | L6 执行监控 |
| 重试策略 | 仅幂等操作允许自动重试，指数退避 | L6 执行监控 |
| 版本管理 | 适配器变更必须版本化，旧版本保留 | L10 版本发布 |

**现状**：Tool Port 当前为内部定义接口。  
**方向**：MCP（Model Context Protocol）作为外部工具接入的标准化协议优先落地。

```
MCP 协议的价值：

```

> **补充日期**：2026-04-14

---

### 补全背景

原有 escalate 骨架（见 03_Agent协作保障.md）写于架构早期，仅覆盖 L4 超时一个触发源，且只定义了"冻结→通知→等待"三步骨架，缺少：

- 触发来源的完整收口（L10、Tool Port 后加的两个触发源未覆盖）
- 对话判定机制（意图不清晰时如何多轮对话直至收敛）
- 长上下文处理方案（无限轮次对话如何保持 O(1) 上下文）
- 标准化报文结构（escalation 5字段协议）
- 人工操作权边界（人工可以做什么、不可以做什么）

---

### 核心设计结论

**意图漏斗协议**：每轮对话不保留原文，只提取结构化"意图 delta"并更新累加器。

```
上下文预算（每轮恒定，O(1)）：
  escalation_snapshot  ≈ 500 tokens  （固定）
+ intent_accumulator   ≈ 300 tokens  （有界结构体）
+ last_exchange        ≈ 200 tokens  （仅最后1轮原文）
─────────────────────────────────────
总计                   ≈ 1000 tokens （与轮次无关）
```

对话阶段状态机：
```
ESCALATED → PRESENTING → CLARIFYING（多轮）→ CONFIRMING → EXECUTING → CLOSED
                                  ↑                 ↓
                              （未清晰）       （人工修正）
                                  │
                          round >= max_rounds
                                  ↓
                         FORCE_CONFIRMING（强制 Y/N）
```

| 触发源 | 层级 | 是否为新增（原骨架未覆盖） |
|--------|------|--------------------------|

| 置信度低于阈值 | L7 Inspector 层 | **新增** |

### 人工操作权边界（最关键补充）

```
人工可以：覆盖 L8 门禁决策 / 修改任务状态 / 终止任务 / 强制降级 / 追加 decision_options
人工不可以：绕过 S3 审计写入 / 直接修改 L11 进化结果 / 删除历史累加器记录 / 跨租户操作
```

## 经验进化闭环协议（缺口7）

> **补充日期**：2026-04-14
> **性质**：数据层补全——定义"经验沉淀→实验触发→上线"的完整闭环路径
> **完整文档**：[07_经验进化闭环协议.md](./07_经验进化闭环协议.md)
> **状态**：✅ 文档已完成

---

### 补全背景

现有文档中各层责任范围已覆盖 L9、L11、L8、L10 的基本层中责任，但缺少以下三个关键路径：

- L9 将经验条目写入 S2 的具体路径（数据结构 + 写入流程）
- S2 → L11 的触发条件（失败率阈值、修正记录数等）
- L11 → L8 → L10 → S4 的评审与发布闭环

---

### 核心设计结论

**三个关键决策：**

1. **经验条目写 S2（record_type = experience）**：S2 内部按子类型区分保留策略，经验条目永久保留（L11 读取来源）；S3 作为审计轨迹，不混入学习数据。
2. **correction_trace 存引用不存副本**：L9 将此次任务涉及的 escalation_id 列表写入 correction_trace 字段，S3 保留完整修正数据的唯一来源。
3. **失败率统计归 L9**：L9 在每次任务结束后聚合查询 S2 历史经验条目，从 S4 读取触发阈值，判定是否需要触发 L11。

**完整闭环路径：**
```
任务完成
  ↓
L9 反馈闭环层
  ├─ 从 S3 查询本任务的 escalation 记录 → 获取 correction_trace（escalation_id 列表）
  ├─ 生成 experience_entry（6字段）
  ├─ 写入 S2（record_type=experience，永久保留）
  └─ 聚合查询 S2 同类历史经验条目，从 S4 读取触发阈值
        ├─ 失败率 > 阈值 → 触发 L11
        ├─ 修正记录数 > 阈值 → 触发 L11
        └─ 未达阈值 → 结束本轮

L11 进化实验层（被触发后）
  ├─ 从 S2 拉取该类型全部 experience 条目
  ├─ 生成 Proposal（Prompt变更 / 路由规则变更 / 工具调用顺序变更）
  └─ 沙盒验证 Proposal 效果

L8 质量门禁层（进化评审模式）
  └─ 新策略 vs 旧策略对比评分 → 推荐采用 or 否决

L10 版本发布层
  ├─ 人工审批 → 灰度上线 → 全量
  └─ 全量后写入 S4：更新对应规则的版本号
```

---

### 关键数据结构

**experience_entry（S2 写入，L11 读取）：**

```json
{
  "record_type": "experience",
  "task_type": "任务类型（如 code-generation / research / review）",
  "success": true,
  "failure_layer": null,
  "correction_trace": ["ESC-20260414-0023"],
  "final_result": "最终正确输出的简要描述",
  "confidence_delta": 0.05,
  "task_id": "T-889",
  "created_at": "2026-04-14T10:30:00Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `record_type` | string | 固定为 `experience`，用于 S2 子类型过滤 |
| `task_type` | string | 任务类型，L11 按此分组聚合 |
| `success` | bool | 本次任务是否最终成功 |
| `failure_layer` | string \| null | 失败发生在哪一层，成功时为 null |
| `correction_trace` | string[] | 人工介入的 escalation_id 列表，S3 中可查全量修正记录 |
| `final_result` | string | 最终正确输出的摘要，L11 训练准确度参考 |
| `confidence_delta` | float | 本次任务使系统置信度提升/下降多少（L7 Inspector 反馈来源） |

---

### L11 触发条件参数（存入 S4 规则库）

| 参数 | 默认值 | 可配置 | 说明 |
|------|--------|--------|------|
| `evo_trigger.failure_rate_threshold` | 0.20 | ✅ | 同类任务历史失败率超过此阈值触发 L11 |
| `evo_trigger.correction_count_threshold` | 10 | ✅ | 同类任务中人工修正记录累计超过此数触发 |
| `evo_trigger.min_sample_size` | 20 | ✅ | 样本量不足时不触发，避免少量样本错误判断 |
| `evo_trigger.manual` | — | — | 管理员任意手动触发 L11，无需阈值 |

---

### L8 进化评审模式（区别于生产门禁模式）

| 模式 | 输入 | 评估方式 | 输出 |
|------|------|------|------|
| 生产门禁模式 | 单个任务输出 | 判断是否达标 | 通过 / 打回 |
| **进化评审模式** | **Proposal + 历史经验条目 + 展开结果** | **新策略 vs 旧策略对比评分** | **推荐采用 / 否决** |

L8 进化评审模式由 L11 主动调用，传入三项内容：提案内容、历史样本、沙盒执行指标。L8 返回评分和建议，不直接决定是否上线（最终决定权在 L10 人工审批）。

## S3 分布式写入保障（缺口8）

> **补充日期**：2026-04-14
> **性质**：实现约定补全——"写入不可拒绝"原则的分布式场景保障机制
> **状态**：✅ 原则已定

---

### 补全背景

S3 审计库的核心约束"写入不可拒绝"早在数据层设计时已确立，但原有描述只给出了目标，缺少在分布式场景下的具体保障机制。此外，多处文档对 S3 写入的描述存在以下问题：

- **隐式同步语义**：将"写 S3"作为内联步骤写在流程箭头中，读者实现时容易写成阻塞调用。
- **循环依赖**：T1 适配器原描述"写失败必须写 S3 审计记录"——当 S3 本身不可用时，该句话逻辑上无法执行。
- **前置控制误用**：工具调用链路步骤6原描述"写 S3 → 触发 L6 重试/熔断"，使业务决策依赖于 S3 写入结果，违反 S3 后置记录定位。

---

### 三条核心原则

1. **S3写入必须异步**，不阻塞主流程（fire-and-forget）。
2. **S3写入失败时的兜底链路**：写本地缓冲文件 → 指数退避重试（上限N次）→ 超限触发P1告警（不中断业务）。
3. **S3是后置记录，不是前置控制**：任何层不得等待 S3 写入返回值来决定下一步业务逻辑。

详见本文档"S3 分布式写入保障原则（缺口8）"章节（位于 S3 审计库定义紧后方）。
