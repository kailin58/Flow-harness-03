# 主题四：长期架构

---

## 框架选择

### 结论：官方SDK + 纯Node.js

```
运行时：      Node.js（保持，不迁移）
LLM调用：     官方SDK（最稳定，官方维护）
              openai / @anthropic-ai/sdk / @google/generative-ai
本地模型：    直接 fetch 调用（Ollama API简单，不需要SDK）
不引入：      LangChain / AutoGen / CrewAI（避免框架绑定）
```

### 为什么不用 LangChain
```
❌ 版本更新频繁，破坏性变更多
❌ 太重，大量用不到的功能
❌ 框架本身也在试错，不够稳定
❌ 会限制我们自研架构的灵活性

官方SDK优势：
✅ 跟模型同步更新
✅ 极轻量，只做一件事
✅ 官方支持，问题有保障
✅ 换模型只换SDK，不改架构
```

## 六边形架构（Ports & Adapters）

```
              外部LLM（Claude/GPT/Gemini/本地）
                    ↕  LLM Port
外部协议     ←→  [ Adapter 层 ]  ←→   内核
(MCP/A2A)         ↕  Tool Port         CEO
                  ↕  Storage Port    + VP层
              外部存储/工具            + 通信规则
              (DB/文件/API)           + 编排逻辑
```

### 5个端口

| 端口 | 作用 | 现在 | 未来 |
|---|---|---|---|
| LLM Port | 模型调用 | MockAdapter | Claude/GPT/Gemini/Local Adapter |
| Protocol Port | 外部Agent通信 | 内存消息 | MCP Adapter / A2A Adapter |
| Storage Port | 数据持久化 | MemoryAdapter | File / SQLite / CloudDB |
| Tool Port | 外部工具调用 | 无 | Browser / File / API / MCP工具 |
| Gateway Port | 用户接入 | CLI | REST API / WebSocket / SDK |

### 核心原则
```
内核（CEO + VP层 + 通信规则）永远不直接接触外部
所有外部依赖通过 Adapter 隔离
外部变化（模型升级/协议迭代/存储切换）只影响 Adapter
内核永远稳定
```

## 协议支持

### MCP（Model Context Protocol）— 先做
```
Anthropic 推出，Cursor 已在使用
作用：
  ← 让外部工具接入我们的 Agent
  → 我们的 Agent 调用外部 MCP 工具

直接受益：Research Agent 可以调用任意 MCP 工具
          （浏览器、数据库、文件系统等）
```

### A2A（Agent-to-Agent）— 占位
```
Google 推出，2025年发布
作用：跨系统 Agent 通信标准
现状：生态还在建设，等成熟再实现
现在：预留接口占位，不实现
```

## 防外部变化策略

### 外部变化类型
```
LLM模型：   GPT-4 → GPT-5 / Claude升级 / 本地模型崛起
协议标准：  MCP进化 / A2A推广 / 未来新标准
工具平台：  浏览器API变 / OS接口变 / 云服务变
合规要求：  AI治理法规 / 数据安全要求
```

### 应对原则
```
变化快的 → 全部放 Adapter 层（外面）
不变的   → 放内核（CEO规则/通信规则/VP层级）

版本化：每个 Adapter 支持多版本
        旧版本保留（老用户不受影响）
        新版本并行（新功能逐步迁移）
```

## 不重复造轮子原则

每次设计新功能前，Research Agent 执行：
```
1. web_search  "xxx pattern existing frameworks 2026"
2. doc_lookup  相关官方文档
3. 输出对比表：现有方案 vs 我们的需求 gap
4. 结论：借用 / 改造 / 原创
5. 写入 KnowledgeBase（external命名空间）
   └→ Plan Agent 读取后再做方案设计
```

### 已调研结论
```
AutoGen 嵌套GroupChat  → 可借鉴三方会话机制实现
MetaGPT 发布订阅      → KnowledgeBase已参考（已实现）
CrewAI manager        → 参考VP层任务委托模式

我们原创的部分：
  ✦ 4层严格层级 + 跳级DENY规则
  ✦ 跨VP必须逐级上报的升级链
  ✦ 冻结常量表驱动通信路由
  ✦ 副总作为域出口的唯一性设计
```
