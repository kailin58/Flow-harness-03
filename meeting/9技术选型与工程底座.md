# 9技术选型与工程底座

## 目的

在 `5个一级入口`、`9个业务端域边界`、`一期范围冻结`、`一期优先级排序` 已经明确后，本文件用于冻结：

- 前后端技术栈
- 模块化单体结构
- 工程基建
- API 与事件约定
- AI/Agent 的接入方式

目标不是追求“最新技术最多”，而是形成一套：

`交易主链路稳定 + 中后台可扩展 + AI/Agent 面向未来可接入`

的工程底座。

## 总体结论

建议采用：

`核心交易与中台规则 = Java 21 + Spring Boot 3.5 模块化单体`

`前端工作台 = Vue 3 / TypeScript 分端建设`

`AI / Agent 能力 = 独立侧车服务，统一接入 AI模型管理中心`

`异步一致性 = Outbox + Kafka`

一句话：

**一期先稳住交易真相源与中台规则，AI 和 Agent 采用旁路增强，不直接侵入高风险核心链路。**

## 口径补充

本文件讨论的是技术底座，不改变前序文档已经冻结的业务层口径。

当前仍然是：

- `5个一级入口`
- `9个业务端域`

以下能力虽然在技术方案中单独建设，但**不新增为业务端域**：

- `账号与权限中心`
- `AI模型管理中心`
- `Agent Runtime`
- `对话模块`
- `衣橱模块`
- `行程规划模块`

统一理解：

- `账号与权限中心`、`AI模型管理中心` 属于治理工作域
- `Agent Runtime` 属于平台技术能力层
- `对话模块`、`衣橱模块`、`行程规划模块` 属于 `C端交易端` 的增强模块

因此，技术架构在扩展，但业务端域数量不变。

## 技术栈冻结

### 1. 后端核心业务栈

建议冻结为：

- `Java 21`
- `Spring Boot 3.5`
- `Spring Modulith` 或同等模块化单体约束方案
- `MySQL 8.4`
- `Kafka`（Outbox 模式）
- `Redis`（缓存、幂等、限流、会话辅助）

采用原因：

- 适合承载订单、库存、售后、分销、积分、权限、结算等强规则域
- Java 生态更适合企业级事务、模块边界、审计、批处理、并发控制
- 模块化单体适合一期快速形成稳定边界，又比微服务更容易控复杂度

### 数据库与存储底座建议

建议补充冻结以下口径：

- `MySQL 8.4` 是核心业务真相源，承接订单、库存、售后、分销、积分、结算、权限等强一致数据
- 业务写入、幂等记录、Outbox 事件写入必须保持在同一本地事务内
- `Redis` 只承担缓存、幂等辅助、限流、会话辅助，不作为核心业务真相源
- `OpenSearch` 主要承接审计检索、搜索、知识检索，不承担核心交易写模型
- 一期优先做模块边界与 schema 治理，不预设重型分库分表
- 详细表结构、字段、索引设计不在本文件展开，数据库 schema 真源以后续详细设计与正式 migration 为准

采用原因：

- 技术底座如果不说明存储职责边界，`MySQL`、`Redis`、`OpenSearch`、`Kafka` 的分工容易反复讨论
- “业务真相源”需要在存储层落地，否则一致性、审计、读写职责很难统一口径
- 一期重点应是先冻结事务边界和数据职责，而不是过早进入复杂拆库拆表设计

### 2. 前端技术栈

#### 用户统一入口

建议冻结为：

- `Vue 3`
- `TypeScript`
- `Nuxt 3`

采用原因：

- 用户端已明确存在 SEO/预渲染诉求
- `Nuxt 3` 更适合承接 H5、落地页、活动页、商品详情页的 SSR/预渲染能力

#### 平台后台 / 供应商协同 / 客服工作台

建议冻结为：

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Element Plus`

采用原因：

- 中后台场景强调表单、表格、权限显隐、工作流页面
- Vue 生态在中后台开发效率、人员获取和组件成熟度上较均衡

#### 发货执行入口

建议冻结为：

- `uni-app`

采用原因：

- 发货端需要小程序、PDA、极简 Web 的多形态适配
- 重点是扫码、打印、异常上报，不需要复杂中后台框架

### 3. AI / Agent 侧车栈

建议冻结为：

- `Node 24`
- `TypeScript`
- `LiteLLM`
- `Langfuse`
- `Temporal`
- `LangGraph` 或 `Mastra`

采用原因：

- AI/Agent 迭代快，TS 生态接新模型、新框架更快
- 与核心交易后端解耦，可以独立演进
- 更适合承接工具调用、Prompt 编排、Agent Runtime、评测与追踪

## 为什么采用“核心后端 + AI 侧车”双栈

不建议把所有东西都堆进同一套技术栈里。

建议边界如下：

- `Java 栈`：负责业务真相源、规则引擎、事务、审计、事件
- `Node/TS 栈`：负责 AI 网关、Prompt 编排、Agent Runtime、内容生成、辅助决策

这样做的好处：

- 核心交易稳定性不受 AI 迭代速度影响
- AI 侧可以快速替换模型、框架、Agent 运行方式
- 一期不会因为追求 Agent 化而拖慢主链路交付

## 总体架构建议

### 分层架构

建议采用 5 层结构：

1. `终端与工作台层`
2. `BFF / 接入层`
3. `业务真相源层`
4. `AI模型治理层`
5. `Agent Runtime 层`

### 建议结构图

```text
用户统一入口 / 平台后台 / 供应商协同 / 客服工作台 / 发货执行
                │
                ▼
         BFF / API Gateway
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
业务真相源层   AI模型管理中心   Agent Runtime
(Java)         (AI治理)        (TS侧车)
    │           │           │
    │           └────┬──────┘
    │                ▼
    │          LiteLLM / Langfuse
    │                │
    ▼                ▼
订单/库存/售后/...   模型供应商 / 本地模型
```

## 模块化单体建议

### 核心业务模块

建议后端按模块化单体拆成以下主模块：

- `iam-auth`
- `iam-permission`
- `catalog`
- `inventory`
- `pricing`
- `trade`
- `payment`
- `aftersale`
- `member`
- `promotion-distribution`
- `points`
- `settlement`
- `logistics`
- `channel-hub`
- `channel-sync`
- `cs-support`
- `ai-model`
- `ai-asset`
- `system-audit`

### 模块依赖原则

- `交易主链路模块` 只能依赖基础中台和公共能力，不能反向依赖 AI 侧能力
- `AI 模块` 只能消费业务读模型和受控工具接口，不能直接写核心真相表
- `渠道模块` 不得绕过商品、库存、订单真相源直接篡改核心事实
- `客服模块` 不得绕过售后、订单规则直接执行业务最终裁定

## API 分层建议

建议分为三类：

### 1. 同步业务 API

用于：

- 查询
- 创建草稿
- 提交动作
- 审批触发

建议形式：

- `REST + OpenAPI`

### 2. 内部能力 API

用于：

- 鉴权
- 数据范围查询
- 审计写入
- AI 模型统一调用
- Agent 工具调用

建议形式：

- 内部 `REST/RPC`

### 3. 事件接口

用于：

- OrderPaid
- RefundApproved
- SupplierBillGenerated
- PointsGranted
- CommissionAccrued
- AccountScopeChanged

建议形式：

- `Outbox + Kafka`

## BFF 与网关建议

建议为不同入口保留独立 BFF：

- `bff-user`
- `bff-admin`
- `bff-supplier`
- `bff-cs`
- `bff-fulfillment`

采用原因：

- 各入口的权限模型、页面聚合、读模型结构不同
- BFF 可以减少前端直接拼多个后端接口
- AI 能力可以在 BFF 层进行受控注入，而不污染核心域服务

## 权限与安全底座建议

建议组合：

- `Keycloak`：统一登录、会话、OIDC、MFA
- `Casbin` 或 `OpenFGA`：授权决策引擎
- `OpenSearch`：审计检索

冻结原则：

- 登录认证与统一身份先冻结
- 角色、资源、动作、数据范围、字段范围、审批权限统一建模
- 后端所有高风险动作必须二次校验，不信任前端按钮显隐

建议补充消息内容治理能力：

- 客服、运营、供应商等外发消息统一经过内容治理层
- 统一维护 `违禁词`、`敏感词`、`辱骂词`、`导流词`、`联系方式识别规则`
- 对 `政治敏感`、`辱骂攻击`、`第三方通讯导流`、`手机号/微信号/QQ/邮箱/链接/二维码外发` 等高风险内容默认禁止发送
- 人工输入、快捷话术、知识引用、AI 生成建议必须共用同一套发送校验规则
- 规则引擎需支持版本化、灰度发布、审计留痕、误杀申诉与回滚

## AI 技术底座建议

### AI模型管理中心

一期必须冻结：

- 模型注册
- 能力标签
- 路由
- 配额
- 密钥管理
- 统一调用接口
- 调用日志

建议采用：

- 网关：`LiteLLM`
- 观测：`Langfuse`
- 统一埋点：`OpenTelemetry`

### AI 接入原则

- 各工作台只能通过统一 AI 调用层消费能力
- 不允许业务模块直接连接第三方模型
- 不允许各模块自建独立 Prompt 和独立日志口径而完全失管

## Agent 技术底座建议

### 结论

建议做 `Agent-ready`，但一期不做“重型多 Agent 平台”。

### 一期建议预留的 Agent Runtime 能力

- Tool Registry
- Prompt Registry
- Session 管理
- Trace / Replay
- Human approval
- Eval 数据集接口预留

### 一期建议 PoC 的 Agent

- `客服副驾驶 Agent`
- `内容生产编排 Agent`

### 二期建议扩展的 Agent

- `运营分析 Agent`
- `商品提报 Agent`
- `渠道异常处理 Agent`

### 建议框架

- `LangGraph`：优先用于有状态、多步骤 Agent 流程
- `Mastra`：作为 TS Agent 应用备选
- `Temporal`：承接长流程编排、补偿、可恢复执行

## 检索与知识底座建议

建议采用：

- `OpenSearch`

用于承接：

- 审计日志检索
- 工单与客服知识检索
- 商品与内容搜索辅助
- AI RAG 的统一检索底座

客服知识沉淀建议补充冻结：

- 客服知识库优先采用“结构化知识条目 + 检索索引”的方式建设，不建议一开始重押复杂 RAG
- 知识来源应覆盖 FAQ、标准话术、售后 SOP、渠道差异说明、争议案例、优秀工单总结
- 会话、工单、升级协同、质检复盘应作为持续知识来源，而不是只靠人工单独维护
- 每条知识应至少具备分类、适用范围、版本、负责人、更新时间、审核状态
- 高风险知识必须经过审核后才能被客服引用或被 AI 辅助消费
- AI 只能通过统一知识检索层消费知识，不允许绕过治理直接读取散落文档或聊天记录

建议演进路径：

1. 一期先建基础知识条目库与检索能力
2. 二期再接相似案例召回、RAG、知识推荐与客服副驾驶 Agent

## 可观测性建议

建议冻结：

- 指标：`Prometheus`
- 链路：`OpenTelemetry`
- 日志：结构化 JSON 日志
- 检索与分析：`OpenSearch`
- AI 观测：`Langfuse`

重点要求：

- 所有关键动作带 `trace_id`
- 所有高风险动作带 `operator_id`
- 所有 Agent 执行链可回放
- 所有模型调用必须可查成本、成功率、时延和降级记录

建议环境：

- `dev`

- 环境全隔离

## 技术选型最终建议

### 核心建议

1. 核心业务真相源采用 `Java 21 + Spring Boot 3.5 + MySQL 8.4 + Kafka`。
2. 中后台工作台采用 `Vue 3 + TypeScript + Vite`。
3. 用户统一入口采用 `Nuxt 3` 承接 SEO 与 H5 体验。
4. AI 与 Agent 采用 `Node 24 + TypeScript` 侧车架构独立演进。
5. 权限采用“统一身份 + 统一授权 + 统一审计”模式。
6. AI 采用“统一模型治理 + 统一调用网关 + 统一观测”模式。
7. Agent 采用“单 Agent 先行、人工确认兜底、可回放可评测”的渐进式路线。

### 明确不建议

- 一期直接拆成大量微服务
- 一期直接重押多 Agent 协作
- 让 Agent 直接成为价格、库存、结算、退款、权限的最终裁定者
- 各模块各自直连模型、各自维护 Prompt 和日志

## 当前阶段结论

- 技术底座应以“模块化单体 + AI 侧车”作为主方案
- 一期先完成交易、供应商、客服、发货闭环所需的稳定底座
- AI 和 Agent 从高频低风险场景进入，不阻塞主链路
- 这套方案既能支撑一期交付，也能为未来 AI/Agent 深化留出足够空间
