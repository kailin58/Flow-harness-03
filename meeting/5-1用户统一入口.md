# 用户统一入口

## 入口定位

用户统一入口是整个前台的唯一主入口，承接普通用户、会员、推广员相关的交易与权益能力。

采用统一前台，而不是把商城、积分、推广拆成多个独立前台系统。

## 服务对象

- 普通用户
- 注册会员
- 已开通或申请中的推广员

## 挂载的业务端域

- C端交易端
- 推广员端
- 积分商城端

## 口径澄清

本入口当前挂载的**业务端域**仍然只有 3 个：

- C端交易端
- 推广员端
- 积分商城端

以下能力虽然在用户端中单独规划，但**不新增为业务端域**：

- `对话模块`
- `衣橱模块`
- `行程规划模块`

统一口径：

- 三者都属于 `C端交易端` 的增强模块
- 三者当前用于增强理解、沉淀与编排能力
- 三者不单独计入业务端域数量

## 用户端新增模块（先规划，不展开功能）

## 当前阶段范围（仅 IA）

本文件在本阶段**仅冻结信息架构（IA）**：入口形态、底部「建议一级导航」、核心子入口清单及其挂载约定、登录默认落地原则。页面流程、组件级交互、埋点与运营配置细节在后续阶段展开。

下列内容**不在本阶段展开**：登录协议与风控细则、深链路由规范、各子入口的视觉稿与 A/B 方案等。

## 一期边界切片（与范围冻结对齐）

为避免“先写进 IA、后续无限扩张”，本入口补充一期边界口径：

**一期必须做（MVP）**

- 交易主链路闭环：浏览、搜索、详情、加购、下单、支付、订单查询、售后入口
- 推广中心极简入口（按资格显隐）
- 会员权益基础展示与积分商城基础入口
- 客服入口（商品页/订单页/我的页可达）
- 承接外部渠道导流、回站与来源参数透传

**一期预留但不展开**

- 活动会场增强形态与复杂运营编排
- 首页增长实验位的细粒度策略
- 对话模块、衣橱模块、行程规划模块的完整功能细化与最终页面布局

**二期再做**

- 短剧产品形态
- 重内容化消费路径

## 高变入口配置化原则

为避免首页活动位、增长入口、导购入口持续膨胀后重新干扰交易主链路，用户统一入口采用“主链路结构固定，高变入口配置化”的统一原则：

- 用户统一入口只冻结稳定 IA，不写死具体活动内容和短期运营入口
- 底部一级导航与核心交易路径固定，不因单次活动、增长策略或导购方案调整而改动
- 首页中的活动位、增长位、导购位统一按“固定坑位 + 平台后台配置”处理
- 平台后台负责配置显隐、排序、跳转目标、命中人群、投放时间和失效回收
- 任一配置化入口都不得突破交易主链路的固定层级和优先级
- 当前台配置失效、活动结束或策略回撤时，应自动回退到稳定默认态，不影响主链路可用性

## 首页搜索下方模块快捷栏

- 首页在统一 Chat 入口栏下方新增一个模块快捷栏，作为首页内的稳定容器
- 该栏用于承接可扩展模块入口，后续可持续新增，不预设固定上限
- 当前首批先挂 `天气`、`商城`、`衣橱`、`行程规划`
- `商城` 点击后按用户身份分流：会员进入客户端商城，分销进入自己商城的首页
- 平台后台负责配置该栏中的模块内容、顺序、显隐与跳转
- 该栏不新增底部 Tab，不形成独立前台系统，不替代交易主链路
- 该栏只承接快捷进入和轻量状态提示，不在首页内展开完整功能
- 当模块数量超过首屏可视容量时，应保持单栏展示并通过横向滑动承接，不新增新的首页栏位

### 模块快捷栏准入与治理规则（IA 约束）

- 新增模块必须满足“服务交易主链路或会员权益主路径”，不承接纯短期活动入口
- 快捷栏采用“固定坑位 + 配置化内容”模式，不预设固定数量上限；超出首屏可视容量时保持单栏横向滑动承接，不新增首页结构层级
- 排序优先级遵循“交易保障 > 权益承接 > 增长引导”，不得覆盖交易主路径固定入口
- 支持按人群策略命中不同模块排序，但策略失效时必须自动回退默认排序
- `商城` 分流异常时统一兜底到客户端商城首页，并保留来源参数以便后续归因补偿

## 外部导流参数与归因最小口径（IA 约束）

统一承接外部平台导流时，至少保留以下来源字段：

- `source_platform`
- `campaign_id`
- `content_id`
- `promoter_id`
- `landing_page`
- `trace_id`

约束说明：

- 来源参数在“落地页 -> 浏览 -> 加购 -> 下单 -> 支付回传”链路中不得无故丢失
- 如发生参数缺失，系统应回退到“可识别最小来源集”并打归因补偿标记，避免静默丢单归因

## 一级导航与核心子入口挂载（IA）

以下约定用于避免「子入口」与底部 Tab 抢位；**不新增**独立前台系统。具体曝光位（首页金刚区、浮层等）可配置，但 **IA 归属**如下表冻结。

| 核心子入口 | IA 挂载约定 |
| --- | --- |
| 消息中心 | 首页顶部图标入口 +「我的」内二级入口；承接订单/售后/推广/积分/系统公告等通知；**不作为**底部 Tab 独立项 |
| 对话入口 | 首页主曝光位（可配置开关）+ 商品/订单等交易页服务分流入口；**不作为**底部 Tab 独立项 |
| 衣橱入口 | 默认在「我的」内；可选经配置在首页露出二级入口，**不作为**底部 Tab 独立项 |
| 行程规划入口 | 默认在「我的」内；可选经配置在首页露出二级入口，**不作为**底部 Tab 独立项 |
| 推广中心 | 「我的」内；推广员资格未达成时按规则显隐 |
| 会员权益 | 「我的」内；页面内含「积分商城」子 Tab，展示会员等级、权益概览、积分余额与积分兑换入口；积分商城可选经配置在首页露出快捷入口，**不作为**独立核心子入口 |
| 账号安全 | 「我的」内；多端凭证合并提示挂此 |
| 客服入口 | 商品页、订单页、「我的」等多点汇聚；**不作为**底部 Tab 独立项 |
| 活动会场入口 | 首页/会场位（可配置开关）；与交易主链路冲突时以配置降级 |

底部「建议一级导航」承载交易主路径：**首页、分类/搜索、购物车、订单、我的**。

## 登录默认落地（IA）

- 建议登录成功后的**默认落地**为「首页」，保持商城主交易场优先。
- 强业务引导（如未支付订单、售后待办）建议以**弹窗/浮层/首页模块**呈现，是否打断默认落地由运营策略配置；IA 上不单独增加第 6 个底部 Tab。

## 游客态与登录合并最小规则（IA 约束）

- 游客态允许浏览、搜索、加购，触发登录后必须执行购物车与来源参数合并
- 合并优先级遵循“已支付与风控约束 > 库存与价格有效性 > 用户操作最近性”
- 同一商品冲突时按规则合并数量并提示结果，不允许静默覆盖
- 登录合并失败时需提供重试与兜底提示，不得直接清空游客态购物车与来源信息
- 合并完成后，订单归因链路与会员身份链路应保持一致可追踪

## 关键设计要求

- 用户只感知一个前台系统
- 商城仍然是主入口与主交易场
- 对话模块、衣橱模块、行程规划模块的定位与归属详见 `meeting/5-1-1用户端增强模块规划.md`
- 推广员能力按资格显隐
- 积分商城作为会员权益的子页面，不做独立登录体系，不做独立核心子入口
- 消息中心承接订单状态、售后进度、推广佣金、积分变动、系统公告等通知，支持按类型分类与未读标记
- 客服入口可从商品页、订单页、我的页进入
- 推广中心、活动会场入口采用配置化开关，避免频繁发版；积分商城首页快捷入口同样可配置
- 首页搜索下方模块快捷栏支持后续继续扩展模块，但只能在既定栏位内配置，不改写首页整体结构
- 模块快捷栏中的 `商城` 入口按用户身份命中不同承接页：会员进入客户端商城，分销进入自己商城的首页
- 用户端第一优先级始终是交易主链路，增长入口不能压过交易主链路
- 用户入口层只承接稳定结构，不直接承接短期运营编排；高变活动入口统一交由平台后台配置
- 前台页面承接的是入口容器与优先级约束，不是某个具体活动本身

## 关键补充说明

- 推广员准入入口在“我的/推广中心”下承载，且准入流程必须强绑定学习平台：先学习、再测验、再签署协议、再提交审核
- 会员分享、推广员分享、活动会场流量最终仍回到同一用户统一入口
- 用户统一入口应统一承接淘宝、抖音、小红书、京东、唯品会、快手、百度等外部渠道的导流、回站与参数透传，不把第三方渠道流量单独拆成新的前台入口
- 短视频素材导流页属于用户统一入口的流量承接页，不单独算一级入口
- 短剧与短视频素材必须分清：一期只承接短视频素材导流，不在同期引入完整短剧产品形态
- 对话模块、衣橱模块、行程规划模块的定位、归属与模块间关系详见 `meeting/5-1-1用户端增强模块规划.md`
- 游客可浏览商品、加购；登录后需合并购物车上下文与来源参数，不得丢失归因链路
- 手机号、微信、支付宝、小程序 `openid` 的多端凭证合并提示，建议挂在“我的/账号安全”
- 用户统一入口只承接会员账号自助能力，如账号安全、凭证绑定、登录态管理；不承接员工账号、供应商账号、角色权限治理
- 投流落地页、商品详情页等有 SEO 需求的页面，可采用 SSR 或预渲染；交易页优先稳定渲染策略

## 本入口不负责

- 不负责商品、库存、订单、积分、分销规则的真相源
- 不负责后台规则配置
- 不负责结算、物流、权限等中后台能力的独立管理
