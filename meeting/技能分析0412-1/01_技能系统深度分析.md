## 二、审计类技能

### 2.1 /audit - 技术质量审计

**触发场景**: 可访问性检查、性能审计、技术质量审查

**核心功能**: 运行系统性技术质量检查，生成带 P0-P3 严重性评分的报告

**检测维度**:

| 维度 | 检查项 | 评分标准 (0-4) |
|------|--------|----------------|
| **Accessibility** | 对比度、ARIA、键盘导航、语义 HTML、Alt 文本、表单 | 0=无法访问 → 4=优秀 (WCAG AA+) |
| **Performance** | 布局抖动、昂贵动画、优化缺失、包大小 | 0=严重问题 → 4=优秀优化 |
| **Theming** | 硬编码颜色、深色模式、Token 一致性 | 0=无主题 → 4=完整 Token 系统 |
| **Responsive** | 固定宽度、触摸目标、水平滚动、断点 | 0=仅桌面 → 4=流畅响应 |
| **Anti-Patterns** | AI slop 特征、设计反模式 | 0=AI slop 画廊 → 4=无 AI 特征 |

**评分等级**:
- 18-20: Excellent (轻微打磨)
- 14-17: Good (解决弱维度)
- 10-13: Acceptable (需要大量工作)
- 6-9: Poor (重大翻修)
- 0-5: Critical (根本问题)

### 2.2 /critique - UX 设计审查

**触发场景**: 设计审查、评估、反馈

**核心功能**: 从 UX 视角评估设计，包括视觉层次、信息架构、情感共鸣、认知负荷

**双重评估机制**:

```
┌────────────────────────────────────────────────────┐
│                 Critique 评估流程                   │
├────────────────────────────────────────────────────┤
│                                                     │
│  ┌─────────────────┐    ┌─────────────────┐       │
│  │   Assessment A  │    │  Assessment B   │       │
│  │   LLM 设计审查   │    │  自动化检测      │       │
│  └────────┬────────┘    └────────┬────────┘       │
│           │                      │                 │
│           │    ┌─────────────────┘                 │
│           │    │                                   │
│           ▼    ▼                                   │
│     ┌─────────────────┐                           │
│     │   综合评估报告   │                           │
│     └─────────────────┘                           │
│                                                     │
└────────────────────────────────────────────────────┘
```

**Nielsen's 10 启发式评分 (每项 0-4 分)**:

| # | 启发式原则 | 说明 |
|---|-----------|------|
| 1 | 系统状态可见性 | 用户知道正在发生什么 |
| 2 | 系统与现实匹配 | 使用用户熟悉的语言 |
| 3 | 用户控制与自由 | 用户有"紧急出口" |
| 4 | 一致性与标准 | 遵循平台惯例 |
| 5 | 错误预防 | 防止错误发生 |
| 6 | 识别而非回忆 | 减少用户记忆负担 |
| 7 | 灵活性与效率 | 为高级用户提供加速器 |
| 8 | 美学与极简设计 | 不包含无关信息 |
| 9 | 错误恢复 | 帮助用户从错误中恢复 |
| 10 | 帮助与文档 | 提供必要的帮助 |

**Persona 测试**: 自动选择 2-3 个相关 Persona 进行红队测试

## 三、质量提升类技能

### 3.1 /polish - 最终打磨

**触发场景**: 发布前的最后一道关卡、设计系统对齐

**核心功能**: 精细修复对齐、间距、一致性和微小细节问题

**检查清单**:

| 维度 | 检查项 |
|------|--------|
| **视觉对齐** | 像素完美对齐、间距一致性、光学对齐、响应式一致性 |
| **排版** | 层次一致性、行长度 (45-75ch)、行高、寡妇/孤儿处理 |
| **颜色对比** | WCAG 标准、Token 使用、主题一致性、色调中性色 |
| **交互状态** | default, hover, focus, active, disabled, loading, error, success |
| **微交互** | 平滑过渡 (150-300ms)、一致缓动、60fps、减少动画偏好 |
| **内容** | 术语一致性、大小写一致性、语法、标点 |
| **图标图片** | 风格一致、大小适当、对齐正确、Alt 文本、Retina 支持 |
| **表单** | 标签一致、必填指示、错误消息、Tab 顺序 |
| **边缘情况** | 加载、空、错误、成功、长内容、无内容、离线状态 |
| **响应式** | 所有断点、触摸目标 44x44px、可读文本、无水平滚动 |
| **性能** | 快速初始加载、无布局偏移、平滑交互、优化图片 |
| **代码质量** | 移除 console.log、注释代码、未使用导入、类型安全 |

### 3.2 /distill - 精简简化

**触发场景**: 简化、清理、减少噪音、移除元素

**核心功能**: 剥离至本质，移除不必要的复杂性

**设计哲学**: "伟大的设计是简单、强大、干净的"

### 3.3 /clarify - UX 文案优化

**触发场景**: 文本混淆、标签不清、错误消息不佳

**核心功能**: 改进 UX 文案、错误消息、微文案、标签和说明

**核心原则**:
- 按钮标签: 永不使用 "OK"、"Submit"、"Yes/No"
- 错误消息: 回答 (1) 发生了什么？(2) 为什么？(3) 如何修复？
- 不责怪用户: "请输入..." 而非 "您输入了无效..."

### 3.4 /optimize - 性能优化

**触发场景**: 性能问题、慢、卡顿

**核心功能**: 诊断和修复加载速度、渲染、动画、图片、包大小问题

### 3.5 /harden - 强化健壮性

**触发场景**: 边缘情况、错误处理、国际化

**核心功能**: 增强错误处理、引导流程、国际化、边缘情况处理

### 3.6 /delight - 愉悦体验

**触发场景**: 添加打磨、个性、动效、微交互

**核心功能**: 添加愉悦时刻、个性元素、意外惊喜

## 四、视觉调整类技能

### 4.1 /animate - 动效添加

**触发场景**: 添加动画、过渡、微交互、动效设计

**核心功能**: 添加有目的的动效，改善可用性和愉悦感

**动效原则**:

```
┌─────────────────────────────────────────────────────┐
│                   动效决策树                         │
├─────────────────────────────────────────────────────┤
│                                                      │
│  需要动效？                                          │
│      │                                               │
│      ├── 状态变化？                                  │
│      │       └── 是 → 入场/退出/反馈动画            │
│      │                                               │
│      ├── 引导注意？                                  │
│      │       └── 是 → 强调动画                      │
│      │                                               │
│      └── 纯装饰？                                    │
│              └── 考虑移除 → 或保持微妙              │
│                                                      │
│  选择持续时间：                                      │
│      │                                               │
│      ├── 即时反馈 → 100-150ms                       │
│      ├── 状态变化 → 200-300ms                       │
│      ├── 布局变化 → 300-500ms                       │
│      └── 入场动画 → 500-800ms                       │
│                                                      │
│  选择缓动：                                          │
│      │                                               │
│      ├── 入场 → ease-out-quart/quint/expo           │
│      ├── 退出 → ease-in                             │
│      └── 切换 → ease-in-out                         │
│                                                      │
│  ❌ 永不使用: bounce、elastic 缓动                   │
│  ❌ 永不动画: width, height, padding, margin        │
│  ✅ 只动画: transform, opacity                       │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 4.2 /colorize - 颜色引入

**触发场景**: 设计单调、缺乏温暖、需要更多颜色

**核心功能**: 添加战略性颜色，使界面更具吸引力和表现力

**色彩原则**:
- 使用 OKLCH 而非 HSL
- 色调中性色（永不纯灰）
- 60-30-10 规则（视觉重量）

### 4.3 /bolder - 大胆化

**触发场景**: 设计平淡、通用、缺乏个性

**核心功能**: 放大安全或无聊的设计，增加视觉冲击力

### 4.4 /quieter - 柔和化

**触发场景**: 设计过于大胆、过于响亮、压倒性

**核心功能**: 降低视觉强度，保持质量

### 4.5 /typeset - 排版修复

**触发场景**: 字体问题、可读性、文字层次

**核心功能**: 修复字体选择、层次、大小、粗细、可读性

**字体选择流程**:

```
Step 1: 写下 3 个品牌词汇（具体，非"现代"、"优雅"）
        例: "温暖、机械、有主见" 或 "冷静、临床、谨慎"

Step 2: 列出你通常会选择的前 3 种字体 → 拒绝它们
        禁止列表: Inter, Roboto, Fraunces, Syne, IBM Plex...

Step 3: 浏览字体目录，想象字体作为物理对象
        打字机色带、手绘招牌、1970s 大型机手册...

Step 4: 交叉检查结果
        如果最终选择符合你的反射模式，返回 Step 3
```

### 4.6 /layout - 布局修复

**触发场景**: 布局感觉不对、间距问题、视觉层次、拥挤 UI

**核心功能**: 改善布局、间距和视觉节奏

**间距系统**:
- 基础单位: 4pt（不是 8pt）
- 刻度: 4, 8, 12, 16, 24, 32, 48, 64, 96px
- 使用语义命名 (`--space-sm`) 而非值命名 (`--spacing-8`)

## 五、高级技能

### 5.1 /adapt - 设备适配

**触发场景**: 响应式设计、移动布局、断点、视口适配

**核心功能**: 适配不同屏幕尺寸、设备、上下文或平台

**适配原则**:
- 移动优先（min-width 查询）
- 内容驱动断点
- 检测输入方式而非仅屏幕尺寸
- 安全区域处理（notch）

### 5.2 /overdrive - 超凡效果

**触发场景**: 需要惊艳、非凡、技术突破

**核心功能**: 添加技术上非凡的效果——着色器、弹簧物理、滚动驱动揭示、60fps 动画

### 5.3 /shape - UX/UI 规划

**触发场景**: 编码前的规划阶段

**核心功能**: 在编码前运行结构化发现访谈，生成设计简报

## 六、技能间依赖关系

### 6.1 调用链

```
impeccable (核心)
    │
    ├── 被所有技能调用 → 上下文收集协议
    │
    └── 三种模式:
            ├── craft: 形状-然后-构建流程
            ├── teach: 设计上下文设置
            └── extract: 提取可复用组件

audit ──────────────┐
                    │
critique ───────────┼───→ 问题发现
                    │
                    ▼
polish ◄────────────┴───→ 问题修复
    │
    ├── distill ─── 简化
    ├── clarify ── 文案
    ├── optimize ─ 性能
    └── harden ─── 健壮性

视觉调整:
    animate ◄─── colorize ───► bolder
        │              │          │
        └──────────────┴──────────┘
                       │
                       ▼
                   quieter (调节)

layout ◄─────────► typeset
    │                  │
    └───────┬──────────┘
            │
            ▼
         adapt (设备适配)
```

### 6.2 推荐工作流

**完整审计流程**:
```
/audit → 发现问题
↓
/critique → UX 审查
↓
选择修复命令 (根据优先级)
↓
/polish → 最终打磨
```

**视觉调整流程**:
```
/colorize → 添加颜色
    ↓
/bolder 或 /quieter → 调整强度
    ↓
/animate → 添加动效
    ↓
/polish → 最终打磨
```

## 八、关键设计原则总结

### 8.1 绝对禁令

| 禁止项 | 原因 | 替代方案 |
|--------|------|----------|
| `border-left/right > 1px` | 最易识别的 AI 特征 | 全边框、背景色调、或无指示 |
| 渐变文字 | 装饰性，AI 特征 | 纯色、粗细、大小 |
| Inter/Roboto 等 | 数百万网站使用 | 独特字体选择流程 |
| 纯黑/纯白 | 不自然 | 色调中性色 |
| 灰色在彩色上 | 看起来褪色 | 背景色的深色调 |
| 弹跳缓动 | 过时俗气 | 指数缓动 |
| 卡片嵌套 | 视觉噪音 | 间距、排版、分隔线 |

### 8.2 必须执行

- 使用 OKLCH 色彩空间
- 色调中性色
- 4pt 间距系统
- 语义 Token 命名
- 移动优先响应式
- `prefers-reduced-motion` 支持
- WCAG AA 对比度
- 44x44px 触摸目标
- 仅动画 transform/opacity
