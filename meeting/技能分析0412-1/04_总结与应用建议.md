## 一、项目核心价值

### 1.1 设计哲学核心

> **AI Slop Test**: 如果你把这个界面给某人看并说"AI 做了这个"，他们会立即相信你吗？如果是，那就是问题所在。

一个独特的界面应该让人问"这是怎么做的？"而不是"哪个 AI 做的这个？"

### 1.2 三大核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                    Impeccable 核心架构                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌────────────────┐                                         │
│  │  设计原则系统   │ ────→ 告诉 AI 应该做什么               │
│  │  (impeccable)  │        排版、颜色、布局、动效...        │
│  └────────────────┘                                         │
│           │                                                  │

│  │  18 个命令系统  │ ────→ 引导 AI 执行特定设计任务         │
│  │                │        审计、打磨、优化...              │
```

## 二、关键设计原则速查

### 2.1 绝对禁令

| 禁止 | CSS 模式 | 原因 | 替代 |
|------|----------|------|------|
| ❌ 侧条纹边框 | `border-left: 3px solid` | 最易识别的 AI 特征 | 全边框、背景色调、无指示 |
| ❌ 渐变文字 | `background-clip: text` + gradient | 装饰性 AI 特征 | 纯色、粗细、大小 |
| ❌ 过度使用字体 | Inter, Roboto, 等 | 数百万网站使用 | 独特字体选择流程 |
| ❌ 纯黑/纯白 | `#000`, `#fff` | 不自然 | 色调中性色 |
| ❌ 灰色在彩色上 | 灰色文字在彩色背景 | 看起来褪色 | 背景色的深色调 |
| ❌ 弹跳缓动 | `bounce`, `elastic` | 过时俗气 | 指数缓动 |
| ❌ 卡片嵌套 | 卡片中的卡片 | 视觉噪音 | 间距、排版、分隔线 |

### 2.2 必须执行

| 必须 | 说明 |
|------|------|
| ✅ OKLCH 色彩空间 | 感知均匀，HSL 不是 |
| ✅ 色调中性色 | 所有中性色带品牌色调 (chroma 0.005-0.015) |
| ✅ 4pt 间距系统 | 8pt 太粗糙，使用 4, 8, 12, 16, 24, 32... |
| ✅ 语义 Token 命名 | `--space-sm` 不是 `--spacing-8` |
| ✅ 移动优先响应式 | `min-width` 查询 |
| ✅ 减少动画支持 | `prefers-reduced-motion` |
| ✅ WCAG AA 对比度 | 正文 4.5:1，大文本 3:1 |
| ✅ 44x44px 触摸目标 | 最小触摸尺寸 |
| ✅ 仅动画 transform/opacity | 其他属性导致布局抖动 |

### 2.3 设计数值速查

| 参数 | 值 |
|------|-----|
| **正文最小字号** | 14px (理想 16px) |
| **行高范围** | 1.5 - 1.7 |
| **最大行长度** | 65-75ch |
| **最小内边距** | 8px (理想 12-16px) |
| **类型刻度比** | ≥1.25 |
| **动效 - 即时反馈** | 100-150ms |
| **动效 - 状态变化** | 200-300ms |
| **动效 - 布局变化** | 300-500ms |
| **动效 - 入场** | 500-800ms |
| **对比度 - 正文** | ≥4.5:1 |
| **对比度 - 大文本** | ≥3:1 |

## 三、实际应用建议

### 3.1 安装与配置

**推荐方式**: 从网站下载
```
https://impeccable.style
```

**Claude Code 安装**:
```bash
# 项目级
cp -r dist/claude-code/.claude your-project/

# 或全局
cp -r dist/claude-code/.claude/* ~/.claude/
```

**首次使用**:
```
/impeccable teach
```
这会收集项目的设计上下文。

### 3.2 工作流推荐

#### 新项目启动流程

```
1. /impeccable teach     → 收集设计上下文
2. /impeccable craft     → 形状-然后-构建流程
3. /audit                → 技术质量检查
4. /critique             → UX 设计审查
5. /polish               → 最终打磨
```

#### 现有项目优化流程

```
1. /audit                → 发现问题
2. /critique             → UX 审查
3. [根据报告选择修复命令]
4. /polish               → 最终打磨
```

#### 视觉调整流程

```
/colorize → 添加颜色
    ↓
/bolder 或 /quieter → 调整强度
    ↓
/animate → 添加动效
    ↓
/typeset + /layout → 排版和布局
    ↓
/polish → 最终打磨
```

### 3.3 CLI 使用场景

**开发时扫描**:
```bash
# 扫描目录
npx impeccable detect src/

# 仅正则，快速
npx impeccable detect --fast --json .
```

**CI/CD 集成**:
```bash
# 在 CI 中运行
npx impeccable detect --json src/ > report.json
# 退出码 0 = 干净，2 = 有问题
```

**浏览器实时检测**:
```bash
npx impeccable live
# 注入: http://localhost:PORT/detect.js
```

### 3.4 命令选择指南

| 场景 | 命令 |
|------|------|
| 不确定问题是什么 | `/audit` |
| 需要 UX 反馈 | `/critique` |
| 发布前准备 | `/polish` |
| 设计太复杂 | `/distill` |
| 文案不清楚 | `/clarify` |
| 性能问题 | `/optimize` |
| 缺少错误处理 | `/harden` |
| 需要动效 | `/animate` |
| 需要颜色 | `/colorize` |
| 设计太无聊 | `/bolder` |
| 设计太大胆 | `/quieter` |
| 字体问题 | `/typeset` |
| 布局问题 | `/layout` |
| 需要愉悦感 | `/delight` |
| 设备适配 | `/adapt` |
| 惊艳效果 | `/overdrive` |
| 规划阶段 | `/shape` |

## 四、扩展与定制

### 4.1 添加自定义反模式

1. **创建夹具** (`tests/fixtures/antipatterns/{rule-id}.html`)
2. **编写失败测试**
3. **添加规则定义** (在 `ANTIPATTERNS` 数组)
4. **实现纯检查函数**
5. **添加两个适配器** (浏览器 + Node)
6. **接入两个循环**
7. **添加 SKILL.md DON'T 行**
8. **运行构建**
9. **运行测试**
10. **浏览器验证**

### 4.2 适配新平台

1. 创建新转换器 (`scripts/lib/transformers/{platform}.js`)
2. 在 `scripts/build.js` 中注册
3. 创建输出目录 (`dist/{platform}/`)
4. 测试构建输出

### 4.3 定制设计原则

编辑 `source/skills/impeccable/SKILL.md`:
- 添加品牌特定的 DON'T 规则
- 调整颜色/字体建议
- 添加行业特定的指导

## 五、技术亮点

### 5.1 构建系统

```
source/ → [transformers] → dist/
         │
         ├── cursor.js      → .cursor/
         ├── claude-code.js → .claude/
         ├── gemini.js      → .gemini/
         └── codex.js       → .codex/
```

**关键特性**:
- 单一真相源 (`source/`)
- 自动占位符替换
- 前置元数据转换
- 跨平台一致性验证

### 5.2 检测器架构

```
┌─────────────────┐
│   Universal     │
│   Detector      │
└────────┬────────┘
         │
    ┌────┴────┬─────────────┐
    ▼         ▼             ▼
┌───────┐ ┌───────┐   ┌───────────┐
│ Node  │ │Browser│   │  Chrome   │
│(jsdom)│ │ (DOM) │   │ Extension │
└───────┘ └───────┘   └───────────┘
```

**双适配器模式**:
- 浏览器: `getComputedStyle` + `getBoundingClientRect`
- Node: `window.getComputedStyle` + `parseFloat(style.width)`

### 5.3 TDD 流程

```
Fixture → Test → Rule → Adapter → Build → Verify
  │        │       │        │        │        │
  └────────┴───────┴────────┴────────┴────────┘
           每一步都必须通过才能继续
```

## 六、与 Flow Harness 集成建议

### 6.1 潜在集成点

| Flow Harness 组件 | Impeccable 集成 |
|-------------------|-----------------|
| **知识库** | 添加设计原则作为知识类型 |
| **反馈循环** | 使用 CLI 输出作为反馈信号 |
| **模式库** | 整合反模式检测 |
| **度量系统** | 添加设计质量指标 |

### 6.2 推荐集成方案

```javascript
// 在 Flow Harness 中集成 Impeccable
const impeccable = {
  // 作为知识源
  knowledge: {
    principles: 'impeccable/skills/impeccable/SKILL.md',
    references: 'impeccable/skills/impeccable/reference/'
  },
  
  // 作为反馈机制
  feedback: {
    detect: 'npx impeccable detect --json',
    patterns: 'impeccable/src/detect-antipatterns.mjs'
  },
  
  // 作为度量
  metrics: {
    antiPatternCount: '检测到的反模式数量',
    healthScore: 'audit 健康分数'
  }
};
```

### 6.3 扩展建议

1. **自定义反模式**: 添加项目/团队特定的设计规则
2. **度量追踪**: 将检测分数纳入 Flow Harness 度量系统
3. **反馈循环**: 在代码审查时自动运行检测
4. **知识积累**: 将修复方案记录为经验

## 七、总结

### 7.1 项目定位

Impeccable 是一个**前端设计技能系统**，通过：
- 正面指导（设计原则）
- 反面警示（反模式检测）
- 引导操作（18 个命令）

帮助 AI 编程助手生成更具设计感、更独特的前端界面。

### 7.2 核心创新

1. **可编程检测**: 不只是文档，而是可执行的检测工具
2. **双面向导**: 同时告诉做什么和不做什么
3. **上下文感知**: teach 模式收集项目特定设计方向
4. **多平台输出**: 一次编写，多平台可用

### 7.3 应用价值

| 角色 | 价值 |
|------|------|
| **开发者** | 快速获得专业级设计指导 |
| **设计师** | 将设计系统转化为可执行规则 |
| **团队** | 统一设计语言和质量标准 |
| **AI 工具** | 提供具体、可操作的设计知识 |

### 7.4 最终建议

1. **从 teach 开始**: 确保每个项目都有设计上下文
2. **善用检测器**: 在 CI/CD 中集成，持续监控
3. **定制化**: 根据团队风格添加自定义规则
4. **工作流化**: 建立标准的设计审查流程
