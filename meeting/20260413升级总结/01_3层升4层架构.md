# 主题一：3层 → 4层架构升级

---

## 层级变化

```
旧（3层）          新（4层）
─────────          ─────────────────────
L0: CEO            L0: CEO
L1: 5总监          L1: 副总（VP层）   ← 新增
L2: 子Agent        L2: 总监           ← 原L1下移
                   L3: 子Agent
```

VP01  数字技术VP（Digital）

### VP治理规则

```
写死VP（7个，企业不可删改）：VP01 / VP02 / VP03 / VP06 / VP07 / VP08 / VP10
可选VP（4个，企业可激活或停用）：VP04 / VP05 / VP09 / VP11
总计：11个VP，不允许企业新增第12个

可选VP激活/停用规则：
  停用后 → 其下属总监和子Agent进入冻结状态，任务不可路由到该VP
  再次激活 → 恢复原有配置，历史数据保留
```

## CEO 预处理管道与路由机制

### 两条入口路线

**路线1：CEO 自动路由（默认）**

```
用户输入（自然语言）
    ↓
识别 user_id → L2权限层查询该用户的VP访问范围
    ↓
[节点1] 需求精确化
[节点2] 任务拆解
    ↓
CEO 路由决策（Step1-5 能力标签匹配）
    ↓
路由到匹配VP → VP → 总监 → 子Agent
```

**路线2：用户显式指定（@ 或 \ 前缀）**

```
用户输入 + @VP名 或 \VP指令
    ↓
L2权限层校验：该用户是否有权访问此VP
  ├─ 无权限 → 拦截并提示用户
  └─ 有权限 ↓
[节点1] 需求精确化（仍然执行）
[节点2] 任务拆解（仍然执行）
    ↓
跳过 CEO 路由决策（目标VP已明确）
    ↓
直接路由到指定VP → VP → 总监 → 子Agent
```

> **路线2 完整预检流程（方案C：转交VP前预检，不匹配直接提示用户）**
>
> ```
> 用户输入 @VP名 或 \VP指令
>     ↓
> 预检1：L2 权限校验（该用户是否有权访问此VP？）
>   ├─ 无权限 → 拦截，返回：
>   │    "你无权访问 @{VP名}
>   │     你当前可访问的VP（共N个）：
>   │       · VP编号  VP名称
>   │       · ...
>   │     请重新指定 或 直接输入任务由系统自动路由"
>   └─ 有权限 ↓
> 预检2：能力匹配校验（该VP的能力标签是否覆盖本次任务类型？）
>   ├─ 不匹配 → 拦截，返回：
>   │    "@{VP名} 不支持「{任务类型}」类任务
>   │     该VP支持的能力范围：{能力标签列表}
>   │     推荐改用：@{匹配度最高的VP名}
>   │     或直接输入任务由系统自动路由"
>   └─ 匹配 ↓
> [节点1] 需求精确化
> [节点2] 任务拆解
>     ↓
> 直接路由到指定VP → VP → 总监 → 子Agent
> ```
>
> **两类提示的数据来源：**
> - 用户可访问VP列表 → L2 权限层（按 user_id 查询）
> - VP能力标签 → S4 规则库（VP自己注册的能力范围）
> - 推荐VP → 复用 Step2 能力标签匹配逻辑

### CEO 路由决策树（Step1-5，仅路线1执行）

```
Step1：读任务元数据
  → 任务类型标签（TaskType）
  → 需要能力标签（RequiredCapabilities）
  → 风险等级（RiskLevel）

Step2：匹配能力标签 → VP列表
  能力标签注册在 S4 规则库
  每个VP声明自己覆盖哪些能力

Step3：
  单VP命中 → 直接分配
  多VP命中 → 按"主责VP"原则选最高匹配度1个，其余作协同VP
  零VP命中 → 路由到 VP01（数字技术VP），走通用兜底路径

Step4：风险等级 ≥ 高 → 路由前先经 L2权限层确认CEO有授权

Step5：分配完成 → 写入 S1上下文库（任务分配记录）

核心原则：CEO 不做"猜"，只做"匹配"
  能力标签由VP自己声明并注册到S4
  新增VP只需更新注册表，不需要改CEO逻辑
```

---

### CEO 三种工作模式

```
模式A：接收输入模式（主动）
  触发：用户发起新对话/任务
  动作：驱动预处理管道 → 执行路由决策（路线1）或权限校验（路线2）

模式B：全局监听模式（被动）
  触发：任务执行中
  动作：只读取L4看板数据，不介入VP/总监内部决策
  CEO知道：每个VP的当前状态、整体进度、是否有风险

模式C：介入模式（触发式）
  触发条件（满足任意一个才介入）：
    ① VP之间结论冲突，需要CEO仲裁
    ② L4状态机触发 escalate（死锁/超时）
    ③ 用户发起新一轮对话（目标可能改变）
    ④ 执行中发现高风险操作，风险等级升级
  动作：重新介入决策，可调整路由或终止任务
```

### 预处理管道节点定义

每个会话/任务默认挂载两个节点，顺序固定：

```
[节点1] 需求精确化
  输入：用户模糊描述
  输出：结构化、精确的任务意图
  Sub-functions：
    · 意图识别
    · 歧义消解
    · 上下文补全（读 S1 历史）
    · 约束条件提取

[节点2] 任务拆解
  输入：精确的任务意图
  输出：可分配的子任务列表（含能力标签）
  Sub-functions：
    · 子任务生成
    · 依赖关系识别
    · 能力标签标注（RequiredCapabilities）
    · 优先级排序

[节点N] 平台管理员可添加更多节点...
```

**节点标准字段：**

```
node_id         唯一标识
name            节点名称
order           执行顺序（管道内位置）
status          active / disabled
sub_functions[]
  ├── func_id
  ├── func_name
  ├── enabled     true / false（可单独开关）
  └── config{}    可配置参数（如：歧义消解置信度阈值）
input_schema    节点接收的数据格式
output_schema   节点输出的数据格式
on_failure      skip / retry / escalate
```

### 配置权限矩阵

| 操作 | 平台管理员 | VP | 总监 | 用户 |
|------|-----------|-----|------|------|
| 新增/删除节点 | ✅ | ❌ | ❌ | ❌ |
| 调整节点全局顺序 | ✅ | ❌ | ❌ | ❌ |
| 全局开关节点 | ✅ | ❌ | ❌ | ❌ |
| 全局开关 sub_function | ✅ | ❌ | ❌ | ❌ |
| 修改 sub_function 全局参数 | ✅ | ❌ | ❌ | ❌ |
| 开关自己会话内的 sub_function | ✅ | ❌ | ❌ | ✅ |
| 修改自己会话内的参数 | ✅ | ❌ | ❌ | ✅ |

**用户配置规则：**

```
范围：只能调整管理员已开放的 sub_function 参数
限制：不能新增节点 / 不能关闭整个节点 / 不能改变节点顺序
数据归属：每个用户配置互相隔离，不共享、不污染
存储：用户配置 → S4 规则库（USER域，user_id 作为隔离键）
      用户上下文 → S1 上下文库（USER域，user_id + session_id 双键）

日志与追回：
  每次配置变更自动写入 S3 审计库（永久保留，不可删除）
  删除/重置操作不物理删除，状态标记为「已删除」
  用户或管理员可从审计日志找到历史版本并恢复
  恢复操作本身也写一条审计记录
```

## 通信规则（4层版）

### 直接允许
```
CEO      ↔  VP        （上下级直接沟通）
VP       ↔  自己的总监 （上下级直接沟通）
总监     ↔  自己的子Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
VP ↔ VP                   → CEO 强制主持
同VP下总监 ↔ 总监          → 所属VP强制主持
同总监下子Agent ↔ 子Agent  → 所属总监强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信    → DENY，必须上报自己的VP，走 VP↔VP 通道（CEO主持）
任何跳级通信        → DENY（CEO不能直接找总监，VP不能直接找子Agent）
```

### 跨VP通信正确链路
```
IT总监A 需要和 运营总监B 沟通：
  IT总监A → 上报 IT VP
               └→ IT VP ↔ 运营VP（CEO三方主持）
                              └→ 运营VP 传达 运营总监B
```
**设计原则**：VP是本域对外的唯一出口，不可被绕过（否则VP架空）

## 关键代码改动

### COMM_RULES：从"基于Agent ID" → "基于level"

```javascript
// 现在（错误方向）：点名道姓，加VP要改规则
match: ctx => ctx.fromId === 'supervisor' && ctx.toId === 'explore'

// 改后（正确）：基于层级，加任何VP自动生效
