# 主题一：3层 → 4层架构升级

---

## 层级变化

```
旧（3层）          新（4层）
─────────          ─────────────────────
L0: CEO            L0: CEO
L1: 5总监          L1: 副总（VP层）   ← 新增
L2: 子Agent        L2: 总监           ← 原L1下移
                   L3: 子Agent
```

---

## VP层设计

### 写死（系统内置，所有用户都有）
```
数字VP（Digital VP）
  ├── Explore Agent    总监1  代码探索
  ├── Plan Agent       总监2  规划架构
  ├── General Agent    总监3  执行实施
  ├── Inspector Agent  总监4  质检审查
  └── Research Agent   总监5  研究查资料
```
**定位**：多行业通用的技术基础底座，任何业务都需要这5个能力。

### 占位（用户自建，按业务需要激活）

| VP | 定位 | 预设总监方向 |
|---|---|---|
| 智能VP | AI/ML能力 | 数据分析、模型训练、知识图谱、AI研究 |
| 通用力VP | 共享执行服务 | 进程管理、调度、系统集成、监控 |
| 生产VP | 上线运维 | 部署、CI/CD、配置、故障处理 |
| 销售运营VP | 市场增长 | CRM、营销、客户支持、收入分析 |
| 产品研发VP | 产品设计 | 需求、原型、路线图、用户研究 |
| 业务支持VP | 内部职能 | HR、法务、财务、合规 |
| 内容VP | 内容品牌 | 内容创作、审核、SEO、社媒 |

## 通信规则（4层版）

### 直接允许
```
CEO      ↔  副总      （上下级直接沟通）
副总     ↔  自己的总监 （上下级直接沟通）
总监     ↔  自己的子Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
副总 ↔ 副总              → CEO 强制主持
同VP下总监 ↔ 总监         → 所属副总强制主持
同总监下子Agent ↔ 子Agent  → 所属总监强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信    → DENY，必须上报自己的副总，走 副总↔副总 通道
任何跳级通信        → DENY（CEO不能直接找总监，副总不能直接找子Agent）
```

### 跨VP通信正确链路
```
IT总监A 需要和 运营总监B 沟通：
  IT总监A → 上报 IT副总
               └→ IT副总 ↔ 运营副总（CEO三方主持）
                              └→ 运营副总 传达 运营总监B
```
**设计原则**：副总是本域对外的唯一出口，不可被绕过（否则副总架空）

## 关键代码改动

### COMM_RULES：从"基于Agent ID" → "基于level"

```javascript
// 现在（错误方向）：点名道姓，加VP要改规则
match: ctx => ctx.fromId === 'supervisor' && ctx.toId === 'explore'

// 改后（正确）：基于层级，加任何VP自动生效
match: ctx => ctx.fromLevel === 0 && ctx.toLevel === 1
```

### CORE_AGENTS：新增VP层条目
```javascript
// 数字VP（写死）
{ id: 'vp_digital', level: 1, role: 'VP', parentId: 'supervisor' }

// 5总监下移到 level 2
{ id: 'explore', level: 2, parentId: 'vp_digital' }
// ... 其他总监同理
```
