# 主题一：3层 → 4层架构升级

---

## 层级变化

```
旧（3层）          新（4层）
─────────          ─────────────────────
L0: CEO            L0: CEO
L1: 5总监          L1: 副总（VP层）   ← 新增
L2: 子Agent        L2: 总监           ← 原L1下移
                   L3: 子Agent
```

## VP层设计（11个VP，总数固定）

### 系统保障域（写死，平台控制，企业不可删改）

```
VP01  数字技术VP（Digital）
  ├── Explore Agent      总监1  代码/系统探索
  ├── Plan Agent         总监2  规划架构设计
  ├── General Agent      总监3  执行实施
  ├── Inspector Agent    总监4  质检审查
  ├── Research Agent     总监5  研究查资料
  └── DigitalOps Agent   总监6  数字化运营
  定位：多行业通用技术底座，任何业务都需要这6个能力

VP02  平台治理VP（Governance）
  预设总监方向：权限管理、审计合规、配额管理、策略治理

VP03  数据智能VP（Data）
  预设总监方向：知识库、向量检索、数据分析、报表
```

### 业务能力域——写死VP（平台预设，企业不可删改）

| 编号 | VP名称 | 预设总监方向 |
|------|--------|------------|
| VP06 | 内容工厂VP（Content） | 内容创作、审核、多模态处理 |
| VP07 | 销售增长VP（Growth） | CRM、营销自动化、市场调研、市场分析 |
| VP08 | 产研中心VP（Product） | 需求、原型、路线图、设计 |
| VP10 | 供应链中心VP（Supply） | 工厂管理、进度管理、物流、仓储 |

### 业务能力域——可选VP（企业可激活或停用，不可新增第12个）

| 编号 | VP名称 | 预设总监方向 |
|------|--------|------------|
| VP04 | 智能体工厂VP（AgentOps） | Agent创建、测试、发布、Agent学院、版本管理、工具适配器管理 |
| VP05 | 未来增长VP（Frontier） | T1-T8前沿技术、趋势探索 |
| VP09 | 业务支持VP（BizOps） | HR、法务、财务、合规、培训 |
| VP11 | 外部协作VP（External） | 第三方Agent接入、A2A协议管理 |

### VP治理规则

```
写死VP（7个，企业不可删改）：VP01 / VP02 / VP03 / VP06 / VP07 / VP08 / VP10
可选VP（4个，企业可激活或停用）：VP04 / VP05 / VP09 / VP11
总计：11个VP，不允许企业新增第12个

可选VP激活/停用规则：
  停用后 → 其下属总监和子Agent进入冻结状态，任务不可路由到该VP
  再次激活 → 恢复原有配置，历史数据保留
```

## 通信规则（4层版）

### 直接允许
```
CEO      ↔  副总      （上下级直接沟通）
副总     ↔  自己的总监 （上下级直接沟通）
总监     ↔  自己的子Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
副总 ↔ 副总              → CEO 强制主持
同VP下总监 ↔ 总监         → 所属副总强制主持
同总监下子Agent ↔ 子Agent  → 所属总监强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信    → DENY，必须上报自己的副总，走 副总↔副总 通道
任何跳级通信        → DENY（CEO不能直接找总监，副总不能直接找子Agent）
```

### 跨VP通信正确链路
```
IT总监A 需要和 运营总监B 沟通：
  IT总监A → 上报 IT副总
               └→ IT副总 ↔ 运营副总（CEO三方主持）
                              └→ 运营副总 传达 运营总监B
```
**设计原则**：副总是本域对外的唯一出口，不可被绕过（否则副总架空）

## 关键代码改动

### COMM_RULES：从"基于Agent ID" → "基于level"

```javascript
// 现在（错误方向）：点名道姓，加VP要改规则
match: ctx => ctx.fromId === 'supervisor' && ctx.toId === 'explore'

// 改后（正确）：基于层级，加任何VP自动生效
match: ctx => ctx.fromLevel === 0 && ctx.toLevel === 1
```

### CORE_AGENTS：新增VP层条目
```javascript
// VP01 数字技术VP（写死）
{ id: 'vp_digital', level: 1, role: 'VP', parentId: 'supervisor' }

// VP01 下 6 个总监（level 2）
{ id: 'explore', level: 2, parentId: 'vp_digital' }
// ... 其他 5 个总监同理（共 6 个：Explore/Plan/General/Inspector/Research/DigitalOps）
```
