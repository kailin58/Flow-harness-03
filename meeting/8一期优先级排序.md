# 一期优先级排序

## 目的

在 `7一期范围冻结.md` 已明确"做什么/不做什么"后，本文件解决**做的顺序**问题：

- 哪些先启动
- 哪些可并行
- 哪些必须等前置完成后才能启动
- 资源有限时优先砍哪些

避免出现：

- 所有端域同时铺开，资源打散
- 依赖项没做完，消费方空转
- 底座没稳就堆业务功能

## 排序原则

### 1. 底座先行

权限、鉴权、审计、数据隔离是所有业务端域的前置。不做完底座就写业务 CRUD，后续必须回头补权限，返工量极大。

### 2. 交易主链路优先

平台的核心价值链是"用户下单 → 支付 → 发货 → 确认收货"。这条链路不通，其他一切增长、内容、分销能力无处挂载。

### 3. 消费端跟着供给端

供应商协同是"商品从哪来、库存谁维护、谁发货"的根基。C 端展示的商品依赖供应商提报，发货依赖供应商协同。

### 4. 增长闭环滞后半拍

分销与积分依赖交易主链路（订单事件、价格快照、确认收货事件）。交易还没跑通就做分销，佣金计算无数据源。

### 5. 渠道同步可晚启动

外部渠道需要商品中心、库存中心、订单中心全部跑通后才有实际同步对象。可以晚半个迭代启动。

### 6. AI 能力不阻塞交易

AI 素材生成、模型管理中心属于增强能力，可在交易主链路跑通后并行低优接入，不作为主链路阻断项。

## 依赖关系

```
E4 权限与审计底座
├── E1 交易主链路（依赖 E4 的登录、会员鉴权）
│   ├── E3 供应商协同 MVP（依赖 E1 的商品/库存模型 + E4 的数据隔离）
│   │   └── E9 渠道店铺 + 同步（依赖 E1 商品 + E3 库存 + E4 权限）
│   ├── E2 售后最小闭环（依赖 E1 的订单模型）
│   │   └── E8 客服工单一期（依赖 E2 售后 + E1 订单上下文）
│   ├── E5 一级分销站内主链路（依赖 E1 的 OrderPaid 事件 + 价格快照）
│   ├── E6 积分闭环 MVP（依赖 E1 的订单 + 价格引擎）
│   └── E7 结算与对账可读（依赖 E1/E2/E3/E5/E6 的计费事件）
├── E10 账号与权限中心治理台（依赖 E4 底层能力）
└── E11 AI模型管理中心一期（弱依赖 E4 权限，可并行）
```

## 推荐迭代顺序

### 第一波：底座 + 交易骨架（Sprint 1~3）

| 优先级 | Epic | 启动条件 | 并行关系 |
|---|---|---|---|
| P0 | E4 权限与审计底座 | 无前置 | 独立启动 |
| P0 | E10 账号与权限中心治理台 | 与 E4 同步推进 | 与 E4 并行（共用底层） |
| P0 | E1 交易主链路 | E4 完成登录 + 基础 RBAC | E4 完成后立即启动 |

目标：Sprint 3 结束时，平台可登录、可配权限、可完成"浏览→加车→下单→支付→确认收货"。

### 第二波：供给 + 售后 + 客服（Sprint 3~5）

| 优先级 | Epic | 启动条件 | 并行关系 |
|---|---|---|---|
| P0 | E3 供应商协同 MVP | E1 商品/库存模型稳定 + E4 数据隔离 | 可在 E1 后半程并行 |
| P1 | E2 售后最小闭环 | E1 订单模型稳定 | 与 E3 并行 |
| P1 | E8 客服工单一期 | E1 订单 + E2 售后基础跑通 | E2 完成后启动 |

目标：Sprint 5 结束时，供应商可提报商品、可发货；用户可发起售后；客服可接单处理。

### 第三波：增长闭环（Sprint 5~7）

| 优先级 | Epic | 启动条件 | 并行关系 |
|---|---|---|---|
| P1 | E5 一级分销站内主链路 | E1 OrderPaid 事件 + 价格快照 | E5/E6 可并行，需共用快照源 |
| P1 | E6 积分闭环 MVP | E1 订单 + 价格引擎 | E5/E6 可并行 |
| P2 | E7 结算与对账可读 | E1/E2/E3/E5/E6 计费事件 | 在 E5/E6 后半程启动 |

目标：Sprint 7 结束时，分销佣金可计提/冲正，积分可发放/抵现/回冲，供应商账单可生成可查看。

**关键约束**：E5 与 E6 必须在启动前约定**计佣基数与积分基数取自同一价格快照**，否则分摊逻辑冲突。

### 第四波：渠道 + AI（Sprint 6~8，可并行）

| 优先级 | Epic | 启动条件 | 并行关系 |
|---|---|---|---|
| P2 | E9 渠道店铺 + 同步 | E1 商品 + E3 库存 + E4 权限 | 可在第二波后期启动 |
| P2 | E11 AI模型管理中心一期 | E4 权限基础 | 可在第一波后期并行低优启动 |

目标：Sprint 8 结束时，外部渠道可授权、可同步商品/库存；AI 模型可注册、可通过统一接口调用。

## 资源有限时的裁剪策略

如果资源不足以按上述四波全部推进，按以下优先级裁剪：

### 绝不裁剪（平台无法上线）

- E4 权限与审计底座
- E1 交易主链路
- E3 供应商协同 MVP

### 可延后但必须做（影响服务闭环）

- E2 售后最小闭环 → 可简化为"仅退款"，不做换货
- E8 客服工单一期 → 可简化为"工单记录"，不做实时会话

### 可延后（增长侧）

- E5 一级分销 → 可推后一个迭代，但表结构和事件预留不能省
- E6 积分闭环 → 同上
- E7 结算与对账 → 可只做流水落库，不做账单 UI

### 可独立延后（不影响主链路）

- E9 渠道同步 → 最多延后两个迭代
- E10 账号治理台 → 治理台 UI 可延后，但接口预留随 E4 一起做
- E11 AI模型管理 → 可整体推迟到增长闭环后

## 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| E4 底座延期 | 所有业务 Epic 阻塞 | E4 投入最资深后端，Sprint 1 即启动，Sprint 2 必须交付 |
| E1 支付回调联调困难 | 交易闭环延期 | 提前准备支付沙箱，Mock 回调可先跑通 E2E |
| E5/E6 快照源不统一 | 佣金与积分分摊冲突 | 启动 E5/E6 前召开一次"价格快照口径对齐会" |
| E3 供应商数据隔离不严格 | 数据泄漏风险 | E3 DoD 必须包含"跨 supplier_id 403"测试用例 |
| E9 外部平台 API 变更 | 同步失败 | 每个平台准备沙箱店铺/桩，回归测试覆盖 |

## 当前阶段结论

- 一期排序遵循"底座→交易→供给→服务→增长→渠道+AI"的递进逻辑
- E4（权限与审计底座）是全局阻塞项，必须最先完成
- E1（交易主链路）是业务阻塞项，完成后才能展开供应商、售后、分销、积分
- E5/E6 可并行但必须共享价格快照口径
- E9/E11 可独立延后，不影响平台最小可用
- 资源不足时，保底是 E4 + E1 + E3，这三个跑通即可形成"平台可交易、供应商可供货"的最小闭环
