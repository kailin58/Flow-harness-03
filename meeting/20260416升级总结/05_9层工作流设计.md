# 工作流升级：6层 → 9层

> ⚠️ **历史文档说明**：本文档记录的是早期"6层→9层"的设计过程。
> 当前架构已进一步升级为 **11层**（新增 L10 版本发布层、L11 进化实验层）。
> 请以 [00_升级总结总览.md](./00_升级总结总览.md) 中的 11 层架构为准，本文档仅作为设计演进历史参考，**不代表当前实施口径**。

---

**更新原因**：
- 原6层在多VP并行场景下缺少协作管理能力
- 权限管理与安全策略职责不同，需独立成层
- 预留入口/出口扩展点应对未来不可预料的需求

---

## 完整9层设计

```
┌──────────────────────────────────────────────────────────────┐
│  L1  入口扩展层   Gateway / 请求归一化 / 未来插件预留          │
│  L2  权限管理层   RBAC / 租户隔离 / 操作授权                  │
├──────────────────────────────────────────────────────────────┤
│  L3  任务编排层   分析 / 拆解 / 分配VP                        │
│  L4  协作编排层   Task State Machine / VP协调 / 冲突检测       │
│  L5  安全策略层   文件/命令准入 / 合规检查 / Deny-by-default   │
│  L6  执行监控层   ExecutionMonitor / AutoRetry                │
├──────────────────────────────────────────────────────────────┤
│  L7  Inspector层  目标对齐 / 完整性 / 质量 / 风险检查          │
│  L8  质量门禁层   QualityGate / 通过 or 打回                  │
│  L9  反馈闭环层   ReviewLoop / 复盘 / 固化 / 出口扩展          │
└──────────────────────────────────────────────────────────────┘
```

---

## 各层详解

### L1 — 入口扩展层（新增，预留）
```
职责：任务进入系统的第一道门
现在：pass-through（直接放行，空插件链）
未来可插入：
  ├── API认证（当REST API上线时）
  ├── 频率限制（防滥用）
  ├── 请求格式归一化（兼容不同输入格式）
  ├── A/B测试路由（不同用户走不同流程）
  └── 任何无法预料的入口逻辑

实现方式：插件钩子链
  LAYER1_HOOKS = []  // 现在为空，注册即生效
```

### L2 — 权限管理层（新增，独立）
```
职责：Who can do What（谁能做什么）
与L5安全策略的区别：
  L2 权限层：身份 + 角色 + 授权（RBAC）
             "这个用户/租户有没有权限发起这类任务"
  L5 安全层：操作合规 + 运行时安全
             "这个操作是否符合系统安全策略"

现在：pass-through（单用户场景直接放行）
未来实现：
  ├── 角色权限控制（RBAC）
  │   CEO/VP/总监各层的操作权限边界
  ├── 租户隔离
  │   SaaS场景下租户A不能访问租户B的数据
  ├── 操作授权
  │   敏感操作（支付/删除/部署）需要额外审批
  └── API Key管理
      用户的BYOK Key权限范围控制
```

### L3 — 任务编排层（原L1）
```
职责：CEO的核心调度逻辑
  step1_analyze   分析任务类型/目标/风险
  step2_decompose 拆解为子任务
  step3_assign    分配给对应VP（不再直接到总监）

变化：step3_assign路由目标从"总监"改为"VP"
      VP再负责分配给自己的总监
```

### L4 — 协作编排层（新增）
```
职责：多Agent并行执行时的协调管理
核心组件：
  Task State Machine
    PENDING → DELIBERATING → PLANNING → EXECUTING
    → INSPECTING → REVIEWING → DONE/FAILED

  VP协调器
    多VP并行时的进度同步
    跨VP通信请求的路由（CEO主持）

  冲突检测器
    发现Agent输出冲突时介入
    触发商议（DeliberationEngine）

  死锁检测器
    某状态停留超过阈值 → 自动告警

  看板数据收集
    收集所有Agent状态供看板展示
```

### L5 — 安全策略层（原L2）
```
职责：运行时操作安全（不变）
  文件/命令访问控制（Deny-by-default）
  包安装合规检查（ComplianceChecker）
  敏感操作拦截
```

### L6 — 执行监控层（原L3）
```
职责：执行过程监控与容错（不变）
  ExecutionMonitor 全程监控
  AutoRetry 智能重试（指数退避）
  超时检测
```

### L7 — Inspector检查层（原L4）
```
职责：输出质量检查（不变）
  目标对齐检查
  完整性验证
  质量评分
  风险扫描
  安全漏洞检测
```

### L8 — 质量门禁层（原L5）
```
职责：通过/拦截决策（不变）
  QualityGate 评估
  通过 → 进入L9
  未通过 → 打回L6重新执行
```

### L9 — 反馈闭环层（原L6 + 出口扩展）
```
职责：复盘 + 优化 + 出口处理
  step6_review 复盘分析
  优化建议写入KnowledgeBase
  经验固化

出口扩展（现在为空，预留）：
  ├── Webhook通知（任务完成回调）
  ├── 结果格式转换（适配不同消费方）
  ├── 归档/备份
  └── 计费结算触发
```

## 与原6层的对应关系

| 原6层 | 新9层 | 变化说明 |
|---|---|---|
| — | **L1 入口扩展层** | 新增，预留扩展 |
| — | **L2 权限管理层** | 新增，独立权限 |
| L1 任务编排层 | L3 任务编排层 | 路由改为VP |
| — | **L4 协作编排层** | 新增，状态机+协调 |
| L2 安全策略层 | L5 安全策略层 | 不变 |
| L3 执行监控层 | L6 执行监控层 | 不变 |
| L4 Inspector层 | L7 Inspector层 | 不变 |
| L5 质量门禁层 | L8 质量门禁层 | 不变 |
| L6 反馈闭环层 | L9 反馈闭环层 | 合并出口扩展 |

**原有6层逻辑100%保留，3个新层插入不破坏现有实现。**

## 实施策略

```
阶段1（P0）：
  L1/L2 空插件链占位（pass-through，零副作用）
  L4 基础实现（Task State Machine核心）
  L3 step3_assign 路由改为VP

阶段2（P2）：
  L2 权限层完整实现（RBAC + 多租户）
  L4 完整实现（VP协调 + 冲突检测 + 死锁告警）
  L1/L9 插件注册机制完善

阶段3（未来）：
  L1 接入认证/限流
  L9 接入Webhook/计费结算
```
