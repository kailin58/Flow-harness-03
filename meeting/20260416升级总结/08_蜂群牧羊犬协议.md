**状态**：设计稿，待实施

## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。

## 一、概述

蜂群协作模式是 deep_agent 执行树在大规模并行任务下的扩展执行模式。

**核心思想**：
- 调用方只下达「任务目标 + 期望结果」，不预先拆解步骤
- 蜂群围绕单一任务主题收敛，不在同一蜂群内并行混入多个主题
- 蜂群内部自行探索、拆解、分配任务块
- 施工只有工蜂，其他角色不进入执行位
- 牧羊犬专职监控是否按目标图纸执行，主动纠偏，不参与执行
- 猫头鹰只负责最终验收的否决与反馈，不负责整改与调度
- 企鹅只投影事实状态，不加工成决策意见
- 总监作为蜂王，只管目标、边界、验收契约和规则，不微观管理每只工蜂

**适用场景**：
- 任务边界模糊、范围大、步骤不可预知（如：扫描整个代码库）
- 适合大规模并行处理的任务（如：批量数据处理、全量分析）
- 不适合步骤严格有序、依赖链固定的任务（那类任务走传统编排）

**与原有规则的关系**：
- 组织层级不变（CEO → VP → 总监 → 经理 → deep_agent）
- COMM_RULES 不变（横向通信仍 DENY）
- 11层工作流不变，新增 L4-SWARM-COORDINATOR 子层承接
- 蜂群通信通过黑板（KnowledgeBase.writeShared）间接实现，不违反 COMM_RULES

## 二、角色定义

### 总监（蜂王）

**核心职责**：

| 职责维度 | 说明 |
|---------|------|
| 承接上级目标 | 接收上级下达的目标图纸，不得擅自改目标 |
| 设定边界 | 明确本轮任务边界、约束、终止条件 |
| 下发验收契约 | 在任务启动前明确最终交付的验收条件，供猫头鹰执行 |
| 调度蜂群 | 决定蜂群参数、波次、重做和是否升级上报 |

**蜂王在蜂群模式下不做的事**：
- ❌ 不预先拆解任务步骤
- ❌ 不主动分配具体任务块给工蜂
- ❌ 不实时监控每只工蜂（这是牧羊犬的工作）
- ❌ 不擅自修改上级目标图纸
- ❌ 不跳过猫头鹰直接宣布完成

---

### 经理（蜂队长）

| 职责 | 说明 |
|------|------|
| 孵化工蜂 | 按总监指令孵化初始批次工蜂，并同步孵化对应数量的牧羊犬 |
| 孵化牧羊犬 | 每 15 个工蜂配 1 只牧羊犬，同批孵化 |
| 处理增殖申请 | 接收工蜂的增殖请求，评估后决定是否孵化更多工蜂 |
| 处理牧羊犬上报 | 接收牧羊犬的异常报告，在权限范围内处理；超出权限上报总监 |

---

### 工蜂 deep_agent（执行者，唯一施工者）

| 属性 | 说明 |
|------|------|
| 标识 | `{ role: "worker", swarm_id: "...", squad_id: "..." }` |
| 任务来源 | 从黑板 `task_pool.*` 自取任务块，不由经理直接分配 |
| 感知方式 | 读黑板（任务池 + 纠偏信号），不直接与其他工蜂通信 |
| 产出 | 写执行结果到 `findings.*`，写进度到 `task_pool.*`（标记任务块状态） |
| 增殖申请 | 发现任务量超预期时，向经理申请增派工蜂（不可自行孵化） |
| 生命周期 | 任务型，任务块完成且任务池无新任务时可销毁 |

**工蜂边界**：
- ✅ 施工只有工蜂
- ❌ 不修改目标图纸
- ❌ 不承担监理职责
- ❌ 不承担最终验收职责

---

### 牧羊犬 deep_agent（监理者，非执行者）

| 属性 | 说明 |
|------|------|
| 标识 | `{ role: "sheepdog", swarm_id: "...", monitor_scope: "squad_A", task_id: "..." }` |
| 核心职责 | 持续读整块黑板，监督是否按目标图纸执行；发现偏移时先提醒，持续偏移才上报 |
| 能力档案 | L3-SKILL-ASSEMBLY 注入「监控类 Skill」，不含执行类 Skill |
| 配比 | 每 15 个工蜂配 1 只牧羊犬 |
| 生命周期 | 与蜂群同周期，蜂群收敛后最后销毁（确认无遗漏后销毁） |

**牧羊犬边界**：
- ✅ 像建筑监理一样监督是否按目标图纸施工
- ✅ 发现目标偏移、边界偏移、重复施工、卡死等异常后提醒/记录/上报
- ✅ 若工蜂收到提醒后自行修正，则不再上报
- ✅ 仅当工蜂未执行修正、持续偏离时，才触发上报
- ❌ 不修改目标图纸
- ❌ 不参与施工
- ❌ 不负责最终验收

---

### 猫头鹰 deep_agent（验收者，非调度者）

| 属性 | 说明 |
|------|------|
| 标识 | `{ role: "owl", swarm_id: "...", acceptance_scope: "final_delivery" }` |
| 核心职责 | 依据蜂王预先下发的目标图纸，在不偏离目标的前提下补充验收标准，并对最终交付结果做通过/否决判断 |
| 输入依据 | 目标图纸 + 冻结后的验收标准 + 最终交付结果 |
| 输出 | `acceptance.*` 中记录通过/否决结果、不通过项与验收报告，反馈给蜂王 |
| 生命周期 | 在蜂群收敛阶段激活，完成验收后销毁 |

**猫头鹰边界**：
- ✅ 通过即算验收
- ✅ 可补充具体验收标准，但不得偏离目标图纸
- ✅ 补充的验收标准必须经蜂王认可后方可生效
- ✅ 验收标准一旦达成共识即一次性冻结，施工过程中不得随意变更
- ✅ 不通过时只提问题，反馈给蜂王
- ✅ 若发现结果超出预期、契约不足或偏差无法裁定，报告蜂王；蜂王不能决策则升级上报
- ✅ 必须严格执行已冻结验收标准，不在验收阶段质疑蜂王图纸
- ✅ 若验收偏差过大，必须定位到上一步施工问题与对应施工清单
- ✅ 若最终不达标或施工超出上限，必须在验收前后撰写验收报告
- ❌ 不负责整改
- ❌ 不负责调度
- ❌ 不直接指挥工蜂

---

### 企鹅 deep_agent（实时投影者，非决策者）

| 属性 | 说明 |
|------|------|
| 标识 | `{ role: "penguin", swarm_id: "...", task_id: "...", projection_scope: "task_local" }` |
| 核心职责 | 将当前任务内的事实状态持续、原样、共享地投影出来，并收集全过程施工清单形成经验链 |
| 输入来源 | `task_pool.*` / `findings.*` / `handoff.*` / `correction.*` / `acceptance.*` / `swarm_status.*` |
| 输出 | 写 `projection.*`，用于共享展示当前进度、状态变化、验收状态，并归档施工清单 |
| 生命周期 | 每条任务内独立存在，随该任务关闭而销毁 |

**企鹅边界**：
- ✅ 每条任务仅对应一个独立企鹅
- ✅ 只投影该任务内的事实状态，不跨任务共享
- ✅ 只投影事实状态
- ✅ 持续共享谁在做什么、做到哪一步、当前是否卡住、验收是否通过
- ✅ 收集每一步施工清单，形成该任务的完整经验链
- ❌ 不加工成决策意见
- ❌ 不参与调度
- ❌ 不参与施工、监理或验收判定

## 三、黑板设计（信息素系统）

黑板基于 `KnowledgeBase.writeShared / readShared`，按命名空间隔离。

### 命名空间结构

```
blackboard
  ├── blueprint.*        目标图纸、验收标准与冻结状态
  │     { goal, boundaries, constraints, acceptance_draft, acceptance_frozen, source }
  │     写入权：蜂王
  │     追加写入权：猫头鹰（仅 acceptance_draft）
  │     冻结权：蜂王
  │     读取权：工蜂、牧羊犬、猫头鹰、企鹅
  │
  ├── task_pool.*        任务块池
  │     { id, area, status, claimed_by, created_at, updated_at }
  │     status: open | in_progress | done | blocked
  │     写入权：工蜂（认领/完成）、探索工蜂（新增任务块）
  │     读取权：所有工蜂、牧羊犬、企鹅
  │
  ├── findings.*         执行结果与发现
  │     { task_id, agent_id, result, timestamp }
  │     写入权：工蜂
  │     读取权：牧羊犬、猫头鹰、总监、VP（汇总时）、企鹅
  │
  ├── handoff.*          施工清单与交接单
  │     { task_id, step_id, worker_id, completed_work, outputs, next_step, risks, status, timestamp }
  │     写入权：工蜂
  │     读取权：下一只工蜂、牧羊犬、猫头鹰、企鹅、蜂王
  │
  ├── correction.*       牧羊犬纠偏信号
  │     { type, target_area, signal, expires_at }
  │     type: redirect | realign | dedup | alert
  │     写入权：牧羊犬
  │     读取权：所有工蜂（每次认领任务前必读）、企鹅
  │
  ├── acceptance.*       猫头鹰验收结果
  │     { result, failed_items, report_precheck, report_postcheck, accepted_at, owl_id }
  │     result: pass | reject
  │     写入权：猫头鹰
  │     读取权：蜂王、企鹅、VP（汇总时）
  │
  ├── projection.*       企鹅实时事实投影（任务级）
  │     { task_id, phase, active_workers, alerts, acceptance_state, updated_at }
  │     写入权：企鹅
  │     读取权：蜂王、经理、上级、人工看板（限当前任务）
  │
  └── swarm_status.*     蜂群全局状态（只读汇总）
        { total_tasks, completed, in_progress, blocked, coverage_pct }
        写入权：L4-SWARM-COORDINATOR（系统写入）
        读取权：总监、VP、牧羊犬、企鹅
```

### 跨域规则

```
同一 VP 内，不同总监的蜂群黑板默认隔离
  → 总监A的 findings.* 对总监B不可见
  → 跨总监共享需经 VP 授权，写入 VP 级共享空间

跨 VP 黑板读取 → DENY（与 COMM_RULES 一致，需走 VP↔VP 通道）
```

## 四、任务自拆解机制

调用方只下达：

```json
{
  "goal": "扫描整个 src/ 目录，找出所有安全漏洞",
  "expected_result": "结构化漏洞报告（按严重等级分类）",
  "constraints": {
    "max_agents": 200,
    "timeout_sec": 3600,
    "coverage_target": 1.0
  }
}
```

**蜂群自拆解流程**：

```
蜂王收到任务
  ↓
蜂王写入目标图纸到 blueprint.*
  ↓
牧羊犬与猫头鹰同步读取同一份目标图纸
  ↓
猫头鹰补充验收标准到 blueprint.acceptance_draft
  ↓
蜂王确认验收标准不偏离目标图纸 → 冻结为 blueprint.acceptance_frozen
  ↓
工蜂确认「此为验收标准」后，先自行拆解任务、评估风险、自查完整性并撰写执行细则
  ↓
执行细则展示到企鹅节点，状态 = 待命（尚未施工）
  ↓
牧羊犬核对执行细则是否严格符合目标图纸
  → 核图通过：允许开工
  → 核图退回：提醒工蜂修正，持续偏离才上报蜂王
  ↓
蜂王写入蜂群参数到 swarm_status.config
  ↓
经理孵化「探索波工蜂」（首批，小规模，约 5-10 只）+ 牧羊犬
  ↓
探索波工蜂：
  读取 blueprint.goal / blueprint.boundaries → 扫描任务边界（目录结构/文件列表/规模估算）
  → 把发现的可工作块写入 task_pool.*
    { id: "T001", area: "src/auth/login.js", status: "open" }
    { id: "T002", area: "src/auth/token.js", status: "open" }
    ...
  ↓
任务池出现任务块后，经理孵化「执行波工蜂」（大规模）
  ↓
主工蜂：
  自主决定施工队规模，并为每只工蜂分配明确的执行顺序与操作指令
  ↓
执行波工蜂：
  每次认领任务前：
    1. 读 correction.* 确认无纠偏信号
    2. 读取上一只工蜂的 handoff.* 施工清单
    3. 从 task_pool 取 status=open 的任务块，标记 in_progress
    4. 执行 → 写结果到 findings.*
    5. 写本步 handoff.*，说明已完成内容、交付物、下一步与风险
    6. 施工清单同步提供给企鹅收集经验链
    7. 前一只工蜂完成并写下施工清单后，自动触发下一只工蜂协同作业
    8. 执行中若发现新子任务 → 写入 task_pool.*（持续扩展）
  ↓
终止条件达成（覆盖率/质量/时限）→ 蜂王收敛蜂群
```

## 五、牧羊犬监控的五类异常

### 牧羊犬统一动作规则

```
牧羊犬接到工蜂任务时，同步读取 blueprint.*
施工阶段的监工权仅属于牧羊犬；猫头鹰可同步查看施工清单，但无权介入阻止施工

若所有操作均未偏离目标图纸：
  → 不提醒、不上报

若首次发现偏离：
  → 仅写 correction.* 提醒工蜂修正
  → 不立即上报

若工蜂自行发现并纠正偏差：
  → 记录已修正，结束本轮监控
  → 不上报

若工蜂收到提醒后仍未修正，且持续偏离：
  → 触发上报（经理或蜂王）

若上一只工蜂已写施工清单：
  → 由牧羊犬核对该施工清单是否仍符合图纸
  → 猫头鹰可见但不得接管监理
```

### 异常1：聚集异常（Clustering）

```
检测条件：blackboard.task_pool 中同一区域 in_progress 工蜂数 > 3
处理：
  写 correction.redirect:
    { type: "redirect", target_area: "src/auth/", signal: "已饱和，新工蜂避开", expires_at: now+300s }
  工蜂下次认领任务时读到信号 → 自动选其他任务块
上报条件：持续 60s 无改善，且工蜂未按提醒修正 → 上报给经理
```

### 异常2：覆盖盲区（Gap Detection）

```
检测条件：task_pool 中 status=open 且 created_at > 300s 无人认领
处理：
  先写 correction.alert 提醒蜂群注意该盲区
  若仍长期无人认领，再上报给经理："任务块 T_xxx 长期未被认领，可能需要特殊能力"
  经理决定：孵化具有对应能力的工蜂 或 标记该任务块 blocked
上报条件：提醒后仍无改善 → 上报（需要经理决策）
```

### 异常3：方向偏移（Drift Detection）

```
检测条件：findings.* 中最新产出与 goal 语义偏差（通过关键词/模式检测）
处理：
  写 correction.realign:
    { type: "realign", signal: "目标是安全漏洞，请重新对齐", goal_reminder: "..." }
  观察工蜂是否自行修正
  若仍持续偏移 → 上报给总监：说明偏移情况，请总监决策是否重置波次
上报条件：提醒后持续偏移（不立即上报）
```

### 异常4：卡死工蜂（Stuck Worker）

```
检测条件：某工蜂 claimed_by 任务块后，120s 内无任何黑板写入
处理：
  写 correction.alert 提醒目标工蜂恢复响应
  若提醒窗口内仍无写入 → 上报给经理：工蜂 agent_xxx 疑似卡死
  经理触发：L6-SYS 超时检测 → 标记该工蜂失活
  → 对应任务块 status 重置为 open，归还任务池
上报条件：提醒窗口内仍无任何写入 → 直接上报经理
```

### 异常5：重复劳动（Duplicate Detection）

```
检测条件：同一任务块被多个工蜂标记 in_progress（并发冲突）
处理：
  写 correction.dedup:
    { type: "dedup", task_id: "T_xxx", keep: "agent_A", release: "agent_B" }
  agent_B 读到信号 → 释放任务块 → 重新认领其他任务块
上报条件：高频重复（> 5次/分钟）且提醒后仍反复发生 → 上报总监，可能是任务池写入逻辑有问题
```

## 六、牧羊犬权限边界（写死）

```
牧羊犬可以：
  ✅ 读取整块黑板（所属蜂群的全部命名空间）
  ✅ 写 correction.* 纠偏信号到黑板
  ✅ 核对每一步施工清单是否符合目标图纸
  ✅ 向经理上报异常（触发孵化/终止工蜂）
  ✅ 向总监上报超出经理权限的问题（方向偏移/系统性异常）

牧羊犬不可以：
  ❌ 直接命令工蜂（只能提醒，不能直接下命令）
  ❌ 自己孵化工蜂（只有经理可以）
  ❌ 修改任务目标（只有总监可以）
  ❌ 直接修改工蜂的任务块认领状态（只能写 dedup 建议）
  ❌ 读取其他蜂群（跨 swarm_id）的黑板内容
  ❌ 将施工监工权转交给猫头鹰
```

## 七、负向效应防护机制

### 防护1：Contrarian 工蜂（逆向探索者）

```
规则：蜂群中固定 10% 的工蜂标记为 contrarian
行为：忽略 correction.redirect 信号，专门探索低活跃区域
目的：防止全员跟随错误信号，保证低信号区仍有覆盖
配置：由总监在蜂群参数中设定 contrarian_ratio（默认 0.1）
```

### 防护2：信号衰减（Signal Decay）

```
所有 correction.* 信号有 expires_at 时间戳
超过过期时间的信号自动失效（工蜂读到过期信号时忽略）
默认有效期：300 秒（可由总监调整）
目的：防止历史错误信号持续误导新工蜂
```

### 防护3：总监覆盖权（Herd Override）

```
总监可写入 swarm_status.override 强制指令
  { type: "force_disperse", reason: "检测到全员聚集错误区域" }
所有工蜂在下次任务认领时读到 override → 服从覆盖指令
这是总监对蜂群的最终控制权，优先于所有其他信号
```

### 防护4：牧羊犬漏检追责

```
若整个施工过程中牧羊犬均未发现偏差，而猫头鹰最终验收不通过：
  → 视为牧羊犬漏检
  → 猫头鹰在验收报告中标注对应漏检段与施工清单
  → 触发监理失职处置（记录、降权、复训或更换）
```

## 八、蜂群规模上限（写死，防止失控）

```
单个经理同时管理的 deep_agent 数量上限：20（工蜂 + 牧羊犬合计）
单次任务蜂群 deep_agent 总数上限：200
蜂群孵化深度上限：5 层（经理 → deep_agent 内部递归不超过 5 层）
牧羊犬数量上限：蜂群总工蜂数 / 15，向上取整

超出任何一条：
  L4-SWARM-COORDINATOR 拒绝增殖请求
  → 写 swarm_status.cap_reached = true
  → 牧羊犬上报总监
  → 总监决定：收敛蜂群 或 触发 Human-in-the-Loop
```

## 九、完整执行流程

```
① 任务下达
   CEO / VP → 总监：{ goal, expected_result, constraints }
   总监：不拆解，直接设定目标图纸、验收标准冻结流程、蜂群参数 + 终止条件 + 牧羊犬规则

② 蜂群启动
   总监 → 经理："启动蜂群，目标 = goal"
   经理 → 孵化探索波工蜂（5-10只）+ 第1只牧羊犬 + 该任务独立企鹅

③ 任务池构建（自拆解阶段）
   探索波工蜂读 goal → 扫描边界 → 写任务块到 task_pool
   工蜂先拆解、评估风险、自查完整性并写执行细则
   企鹅展示待命态
   牧羊犬核图通过后方可施工

④ 执行波启动
   task_pool 出现任务块 → 经理孵化执行波工蜂（批量）+ 更多牧羊犬
   主工蜂组建施工队并分配顺序
   工蜂从 task_pool 自取任务 → 执行 → 写 findings + handoff
   下一只工蜂基于施工清单继续执行
   企鹅同步收集施工清单形成经验链
   工蜂执行中发现新任务 → 写入 task_pool（任务池动态扩展）

⑤ 持续监控与纠偏
   牧羊犬：持续监控五类异常 → 先写 correction.* 提醒
            若工蜂未修正且持续偏离 → 再上报经理/总监
   猫头鹰：可见施工清单，但不介入施工监理
   总监：读 swarm_status → 判断是否需要波次调整或强制覆盖

⑥ 蜂群收敛
   终止条件满足（任意一条）：
     a. task_pool 全部 done 且无新任务块写入 60s
     b. coverage_pct 达到 constraints.coverage_target
     c. 超出 constraints.timeout_sec
     d. 蜂王主动发出 swarm_status.terminate 信号
   → 蜂王通知经理停止孵化 → 工蜂自然消耗任务池 → 销毁
   → 最后一只牧羊犬确认 task_pool 无遗漏 → 销毁

⑦ 最终验收
   猫头鹰读取 blueprint.acceptance_frozen + findings.*
   → 先写验收前报告（检查范围、标准、上限项）
   → 通过：写 acceptance.pass + 验收后报告，蜂群任务视为验收完成
   → 不通过：写 acceptance.reject + failed_items + 验收后报告，定位到上一步施工问题并反馈给蜂王决定是否重做

⑧ 结果汇总
   蜂王读 findings.* + acceptance.*
   → 按 expected_result 格式生成输出
   → L7 Inspector 质检 → L8 门禁 → 结果上报 VP
```

|---------|--------|---------|

| 牧羊犬上报处理 | L4-SWARM-COORDINATOR | 接收牧羊犬上报，决定升级给总监或经理 |
| 施工过程经验收集 | L6-SYS / 看板 | 企鹅同步收集 `handoff.*`，形成单任务经验链 |

| 蜂群结果门禁 | L8 质量门禁层 | 蜂群整体输出通过质量评分 |

## 十一、L4-SWARM-COORDINATOR 子层定义

```
归属：L4 协作编排层的新增子层

职责（五个组件）：

① Wave Scheduler（波次调度器）
   管理探索波 → 执行波的时序
   触发：task_pool 出现足够任务块时，自动通知经理孵化执行波
   参数：首波数量、扩展批次、最大并发

② Blackboard Task Manager（任务池管理器）
   维护 task_pool 的并发锁（防多个工蜂同时认领同一任务块）
   管理任务块的状态流转：open → in_progress → done / blocked
   清理：超时未完成的 in_progress 任务块重置为 open

③ Spawn Arbiter（增殖请求仲裁器）
   接收工蜂的增殖申请
   检查是否超出规模上限（max_agents）
   批准 → 通知经理孵化；拒绝 → 写入拒绝原因

④ Sheepdog Report Handler（牧羊犬上报处理器）
   接收牧羊犬上报
   判断处理层级：经理可处理 or 需升级总监
   触发相应处理：孵化工蜂 / 终止工蜂 / 通知总监

⑤ Convergence Judge（终止条件评估器）
   持续计算 swarm_status.coverage_pct 等指标
   终止条件满足 → 触发收敛流程
   写入 swarm_status.* 供总监和牧羊犬读取
```

## 十二、蜂群模式 vs 传统编排模式的选择规则

| 任务特征 | 推荐模式 |
|---------|---------|
| 步骤固定、依赖链明确 | 传统编排（L3 正常拆解分配） |
| 范围大、边界模糊、步骤不可预知 | 蜂群模式 |
| 需要大规模并行扫描/处理 | 蜂群模式 |
| 对结果一致性要求极高、不允许工蜂自主决策 | 传统编排 |
| 探索性任务（不知道会发现什么） | 蜂群模式 |

```
选择判断在 L3 任务编排层执行：
  if task.type in ["scan", "explore", "batch_process", "discovery"] 
     and task.scope == "large"
     and task.steps == "unknown":
       → 蜂群路由模式
  else:
       → 传统编排模式
```
