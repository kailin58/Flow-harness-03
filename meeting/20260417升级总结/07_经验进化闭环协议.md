# 经验进化闭环协议

**补充日期**：2026-04-14  
**性质**：缺口7补全——定义"经验沉淀→实验触发→上线"的完整闭环，含数据结构、触发条件、评审机制  
**前置文档**：[00_升级总结总览.md](./00_升级总结总览.md)（数据层 S1-S4 定义）、[06_HumanInTheLoop对话判定协议.md](./06_HumanInTheLoop对话判定协议.md)（correction_trace 数据来源）

## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。
- 治理模型统一采用“规则写死、参数可配、边界写死”：
  - 写死规则：流程门禁、接管链路、审计要求、发布与回滚约束。
  - 可配置参数：SLA、复评阈值、紧急策略、试点范围等运营参数。
  - 边界写死：仅允许边界内收紧，不允许突破平台红线；配置变更必须版本化发布并写审计。

## 一、协议总览

### 为什么需要专门的闭环协议

11层工作流已定义 L9（复盘）、L11（进化实验）、L10（发布）的独立职责，但三者之间的**数据流路径和触发机制**是断开的：

```
现有状态：
  L9 → "经验固化"（写哪里？不清楚）
  L11 → "自优化沙盒"（什么时候触发？不清楚）
  L8 → "通过/打回"（进化评审怎么做？不清楚）

本协议目标：
  打通 L9 → S2 → L11 → L8 → L10 → S4 的完整数据通路
```

### 三段式闭环

```
第一段：经验沉淀（任务完成 → L9 → S2）
第二段：实验触发（S2 阈值判断 → L11 沙盒实验）
第三段：提案评审（L11 → L8 → L10 → S4）
```

## 二、第一段：经验沉淀（L9 → S2）

### 2.1 触发时机

每次任务进入 **DONE** 或 **FAILED** 终态后，L9 反馈闭环层自动生成经验条目。无论成功失败均生成，失败的经验同样是 L11 的重要训练材料。

### 2.2 数据来源整合

L9 生成 experience_entry 需要从以下来源收集数据：

| 字段 | 数据来源 |
|------|------|
| `task_type` | S1 上下文库（任务元数据） |
| `success` | L4 状态机（终态判定） |
| `failure_layer` | S3 审计库（异常记录 + 状态流转记录） |
| `correction_trace` | S3 审计库（查询本任务的 escalation 记录，取 escalation_id 列表） |
| `final_result` | S2 制品库（最后一条有效制品摘要） |
| `confidence_delta` | L7 Inspector 层反馈（本次任务的置信度变化量） |

### 2.3 experience_entry 数据结构

```json
{
  "record_type": "experience",
  "task_id": "T-889",
  "task_type": "code-generation",
  "success": true,
  "failure_layer": null,
  "correction_trace": ["ESC-20260414-0023"],
  "final_result": "完成了用户注册模块的前端实现，包含表单校验和错误处理",
  "confidence_delta": 0.05,
  "created_at": "2026-04-14T10:30:00Z"
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `record_type` | string | ✅ | 固定为 `experience`，用于 S2 子类型过滤与生命周期管理 |
| `task_id` | string | ✅ | 关联的任务 ID，用于追溯 |
| `task_type` | string | ✅ | 任务类型，L11 按此分组聚合同类经验 |
| `success` | bool | ✅ | 本次任务是否最终成功 |
| `failure_layer` | string \| null | ✅ | 失败发生在哪一层（如 `L6`、`L7`），成功时为 null |
| `correction_trace` | string[] | ✅ | 人工介入的 escalation_id 列表（可为空数组），S3 中可查全量修正记录 |
| `final_result` | string | ✅ | 最终正确输出的摘要（最长 200 tokens），L11 准确度参考 |
| `confidence_delta` | float | ✅ | 本次任务使系统置信度提升（正值）或下降（负值）的量 |
| `created_at` | ISO8601 | ✅ | 经验条目生成时间 |

### 2.4 写入 S2 的约束

```
写入时：
  record_type = "experience"   ← 必须声明，否则 S2 无法识别子类型
  不可覆盖写，只能追加           ← 与 S2 通用约束一致
  绑定 task_id + L9 层标识      ← 与 S2 通用约束一致

生命周期：
  type=experience → 永久保留   ← 区别于 type=artifact 的N天保留
  归档权限：仅存储管理员可归档，归档后只读

correction_trace 字段处理：
  L9 查询 S3，筛选条件：task_id = 当前任务 AND record_type = escalation
  取所有 escalation_id 写入列表
  不复制修正详情，只存引用
  → S3 永远是修正记录的唯一完整来源
```

## 三、第二段：实验触发（S2 → L11）

### 3.1 触发时机

**每次 L9 写入 experience_entry 后**，L9 立即执行阈值判断（不是定时批处理，而是任务结束即判断）。

### 3.2 触发条件

满足以下**任意一个**条件即触发 L11：

| 条件 | 参数名（S4 规则库） | 默认值 | 说明 |
|------|------|--------|------|
| 同类任务历史失败率 ≥ 阈值 | `evo_trigger.failure_rate_threshold` | 0.20 | 查 S2 同 task_type 的历史条目，计算 success=false 的比例 |
| 同类任务人工修正记录累计 ≥ 阈值 | `evo_trigger.correction_count_threshold` | 10 | 查 S2 同 task_type 的历史条目，累加 correction_trace 非空条目数 |
| 管理员手动触发 | `evo_trigger.manual` | — | 无需阈值，直接进入 L11 |

**防误判保护：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `evo_trigger.min_sample_size` | 20 | 同类任务样本量不足时不触发，避免少量样本引发误判 |

### 3.3 触发流程

```
L9 写入 experience_entry 完成
  │
  ├─ 查询 S4：读取 evo_trigger 系列阈值参数
  │
  ├─ 查询 S2：同 task_type 的全部 experience 条目
  │   ├─ 计算失败率
  │   └─ 统计 correction_trace 非空条目数
  │
  ├─ 样本量 < min_sample_size → 不触发，结束
  │
  ├─ 失败率 ≥ threshold OR 修正数 ≥ threshold
  │   → 写 S3 审计记录（标记 L11_TRIGGERED，记录触发原因和当前统计值）
  │   → 调用 L11，传入 task_type 和触发原因
  │
  └─ 未达阈值 → 结束本轮，等待下次经验积累
```

## 四、第三段：提案评审（L11 → L8 → L10 → S4）

### 4.1 L11：沙盒实验与提案生成

L11 被触发后，执行以下步骤：

```
L11 进化实验层
  │
  ├─ Step 1：从 S2 拉取 task_type 全部 experience 条目
  │
  ├─ Step 2：分析经验条目，识别失败模式
  │   ├─ 高频失败层（failure_layer 统计）
  │   ├─ 需要人工介入的场景模式（correction_trace 非空条目分析）
  │   └─ 置信度下降模式（confidence_delta 负值聚类）
  │
  ├─ Step 3：生成 Proposal（三类之一）
  │   ├─ Prompt 变更：修改系统提示或上下文模板
  │   ├─ 路由规则变更：调整 L3 任务分配策略或模型选择规则
  │   └─ 工具调用顺序变更：调整 L6 的工具执行序列
  │   注：Proposal 是配置/策略变更，不是代码变更
  │
  ├─ Step 4：沙盒验证
  │   ├─ 用历史经验条目中的 task_id 对应任务重跑
  │   ├─ 对比新策略 vs 旧策略的成功率/置信度变化
  │   └─ 生成验证报告（沙盒指标：成功率变化 / 置信度变化 / 平均耗时变化）
  │
  └─ Step 5：调用 L8 进化评审模式，传入 Proposal + 经验样本 + 沙盒指标
```

**Proposal 数据结构：**

```json
{
  "proposal_id": "PROP-20260414-0001",
  "task_type": "code-generation",
  "trigger_reason": "failure_rate=0.23 (threshold=0.20), sample_size=35",
  "proposal_type": "prompt_change",
  "description": "在 code-generation 类任务的系统提示中增加错误处理示例模板",
  "diff": {
    "before": "...",
    "after": "..."
  },
  "sandbox_result": {
    "success_rate_before": 0.77,
    "success_rate_after": 0.91,
    "confidence_delta_avg_before": -0.03,
    "confidence_delta_avg_after": 0.04,
    "sample_count": 35
  },
  "created_at": "2026-04-14T11:00:00Z"
}
```

### 4.2 L8：进化评审模式

L8 在此场景下工作在**进化评审模式**，与日常的生产门禁模式不同：

| 维度 | 生产门禁模式 | 进化评审模式 |
|------|------|------|
| 输入 | 单个任务输出 | Proposal + 历史经验条目 + 沙盒指标 |
| 评估问题 | "这个输出达标吗？" | "新策略比旧策略好吗？" |
| 评估方式 | 单点质检 | 统计对比评分 |
| 输出 | 通过 / 打回 | 推荐采用（附评分） / 否决（附原因） |
| 最终决定权 | L8 自身 | L10 人工审批（L8 只提建议） |

**L8 进化评审评分维度：**

| 维度 | 权重 | 说明 |
|------|------|------|
| 成功率提升 | 40% | 新策略样本成功率 vs 旧策略成功率差值 |
| 置信度改善 | 30% | 平均 confidence_delta 从负转正的幅度 |
| 人工介入减少 | 20% | correction_trace 非空条目比例下降幅度 |
| 策略风险评估 | 10% | 变更影响范围和可回滚风险的综合评分 |

评分 ≥ 0.7 → L8 推荐采用  
评分 0.5 ~ 0.7 → L8 建议有条件采用（附具体条件）  
评分 < 0.5 → L8 否决，反馈否决原因给 L11

### 4.3 L10：审批与发布

L8 评审通过（推荐采用）后，进入 L10 发布链路：

```
L10 版本发布层
  │
  ├─ 生成候选版本（Candidate）
  │   └─ 绑定 proposal_id + L8 评审报告
  │
  ├─ 人工审批
  │   ├─ 审批通过 → 进入灰度
  │   └─ 审批拒绝 → 写 S3（PROPOSAL_REJECTED），通知 L11
  │
  ├─ 灰度上线（部分流量使用新策略）
  │   ├─ 观察期间新策略指标正常 → 全量
  │   └─ 指标异常 → 回滚，写 S3（ROLLBACK），通知 L11
  │
  └─ 全量上线
      └─ 写入 S4：更新对应规则的版本号
```

### 4.4 S4 规则版本更新

全量发布后，L10 将进化结果同步写入 S4 规则库：

```json
{
  "rule_id": "RULE-code-generation-prompt-v3",
  "version": "3.0",
  "previous_version": "2.0",
  "source_proposal_id": "PROP-20260414-0001",
  "effective_date": "2026-04-14T14:00:00Z",
  "change_summary": "code-generation 类任务 Prompt 模板更新，增加错误处理示例"
}
```

## 五、与现有协议的交叉引用

### 与 Human-in-the-Loop 协议（06文档）的关系

06文档的 correction_trace 数据是本协议的**上游输入**：

```
06协议（Human-in-the-Loop）
  CLOSED 时 → 写 S3：完整意图累加器快照（绑定 escalation_id）
                           ↓
07协议（经验进化闭环）
  L9 生成 experience_entry 时 → 查 S3，获取 escalation_id 列表
  correction_trace = [escalation_id_1, escalation_id_2, ...]
```

**关键原则**：experience_entry 的 correction_trace 只存 escalation_id，不复制修正详情。需要查看具体修正内容时，通过 escalation_id 反查 S3。

### 人工不可以做的（进化闭环边界）

| 禁止操作 | 原因 | 约束来源 |
|----------|------|----------|
| 直接修改 L11 的进化结果（Proposal） | 进化结果必须经 L10 发布链路，直改等于架空 L10 门禁 | L11 设计约束（同 06文档） |
| 跳过 L8 评审直接提交 L10 | L8 的对比评分是 L10 审批决策的依据，跳过则审批无据 | L8 设计约束 |
| 直接覆盖 S4 规则 | 规则变更必须经 L10 发布流程 | S4 设计约束 |
| 删除 S2 中的 experience 条目 | 经验条目永久保留，是 L11 统计基础；删除会导致触发条件失准 | S2 设计约束 |

## 六、完整审计轨迹（S3 写入点）

| 时机 | 写入内容 | 不可跳过 |
|------|----------|----------|
| L9 判定触发 L11 时 | 触发事件（task_type、触发原因、当前统计值） | ✅ |
| L11 生成 Proposal 时 | Proposal 摘要（绑定 proposal_id） | ✅ |
| L11 沙盒验证完成时 | 沙盒结果报告（绑定 proposal_id） | ✅ |
| L8 评审完成时 | 评审结论（评分 + 推荐/否决）（绑定 proposal_id） | ✅ |
| L10 人工审批时 | 审批决定（通过/拒绝 + 审批人）（绑定 proposal_id） | ✅ |
| L10 灰度回滚时 | 回滚事件（原因 + 回滚时间） | ✅ |
| L10 全量上线时 | 发布完成记录（绑定 proposal_id + S4 规则版本号） | ✅ |

## 七、阶段状态机（Proposal 生命周期）

```
TRIGGERED          ← L9 判定阈值满足，调用 L11
    │
    ▼
ANALYZING          ← L11 分析经验条目，识别失败模式
    │
    ▼
SANDBOXING         ← L11 生成 Proposal 并沙盒验证
    │
    ├─ 沙盒指标不达预期 → REJECTED_BY_SANDBOX（写 S3，反馈给 L11）
    │
    ▼
REVIEWING          ← L8 进化评审模式（对比评分）
    │
    ├─ 评分 < 0.5 → REJECTED_BY_L8（写 S3，反馈给 L11）
    │
    ▼
APPROVING          ← L10 人工审批等待
    │
    ├─ 审批拒绝 → REJECTED_BY_HUMAN（写 S3）
    │
    ▼
CANARY             ← 灰度上线，观察期
    │
    ├─ 指标异常 → ROLLED_BACK（写 S3，触发 L11 重新分析）
    │
    ▼
RELEASED           ← 全量上线，S4 规则版本更新
    │
    ▼
CLOSED（终态，S3 写入发布闭环记录）
```

## 八、参数配置全览（S4 规则库存储）

| 参数路径 | 默认值 | 可配置 | 说明 |
|------|--------|--------|------|
| `evo_trigger.failure_rate_threshold` | 0.20 | ✅ | 失败率触发阈值 |
| `evo_trigger.correction_count_threshold` | 10 | ✅ | 人工修正数触发阈值 |
| `evo_trigger.min_sample_size` | 20 | ✅ | 最小样本量保护 |
| `evo_l8.score_threshold_recommend` | 0.70 | ✅ | L8 评分高于此值推荐采用 |
| `evo_l8.score_threshold_conditional` | 0.50 | ✅ | L8 评分低于此值直接否决 |
| `evo_l10.canary_traffic_pct` | 0.10 | ✅ | 灰度阶段流量比例（10%） |
| `evo_l10.canary_observation_hours` | 24 | ✅ | 灰度观察时长（小时） |

## 九、与现有架构的交叉引用

| 模块 | 关系 |
|------|------|
| L9 反馈闭环层 | 第一段：生成 experience_entry，写入 S2，判定 L11 触发条件 |
| L11 进化实验层 | 第二段：拉取 S2 经验条目，生成 Proposal，沙盒验证 |
| L8 质量门禁层 | 第三段：进化评审模式，对比新旧策略评分 |
| L10 版本发布层 | 第三段：人工审批，灰度，全量，更新 S4 |
| S2 制品库 | experience_entry 写入目标（record_type=experience，永久保留） |
| S3 审计库 | 全链路写入点；correction_trace 的数据来源（L9 从此读取 escalation_id） |
| S4 规则库 | 触发阈值参数存储；L10 全量后写入规则版本号 |
| L7 Inspector 层 | confidence_delta 字段数据来源 |
