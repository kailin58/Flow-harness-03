## 核心决策一览

### 组织架构（横向）
- Agent层级：3层 → 4层（新增 VP 副总层）
- VP 层由平台统一治理（共11个VP位，不允许企业新增VP）
- 每个VP默认5个总监（平台标准骨架）
- 企业可在既有VP下增加总监（受权限与配额约束）
- 总监下子Agent数量为 N（按任务负载弹性扩缩）
- 组织能力树写死：**VP → 多总监 → 多子Deep Agent → 多Skills**
- 治理约束写死：Skills 调用受 L2/L5 约束；子Deep Agent输出必须经 L7/L8；进化只在 L11、上线只经 L10
- COMM_RULES 从"基于Agent ID"改为"基于level"

### 工作流（纵向）★ 本次更新
- 工作流从 **6层升级为11层**
- 新增 Layer 1（入口扩展）、Layer 2（权限管理）、Layer 4（协作编排）、Layer 10（版本发布）、Layer 11（进化实验）
- 权限管理**独立成层**（Layer 2），不与安全策略混用
- 详见：[05_9层工作流设计.md](./05_9层工作流设计.md)

### 模型
- 三种来源：平台内置 / BYOK / 本地模型
- 不引入重框架，使用官方SDK
- 配置走 `.flowharness/models.json` + 环境变量引用Key
- 模型分配/调度归属 **L3 任务编排层**（路由、降级、成本与SLA权衡）
- L6 仅负责执行期监控与反馈，L4 仅负责多Agent协作冲突协调（不承载模型分配主责）

### 协作
- 补充 Task State Machine（最缺的一环）
- 添加 Agent 看板（终端版先做，Web版后做）

### 长期
- 六边形架构（Ports & Adapters）
- MCP 协议先接，A2A 占位
- 平台化方向：框架 → 平台 → 生态

## 11层工作流全景

```
┌─────────────────────────────────────────────────────┐
│  L1  入口扩展层   【预留插件】网关/请求归一化         │
│  L2  权限管理层   【独立】RBAC/租户/操作授权          │ ← 新增
├─────────────────────────────────────────────────────┤
│  L3  任务编排层   分析/拆解/分配VP（原L1）            │
│  L4  协作编排层   状态机/VP协调/冲突检测              │ ← 新增
│  L5  安全策略层   准入/合规检查（原L2）               │
│  L6  执行监控层   Monitor/AutoRetry（原L3）           │
├─────────────────────────────────────────────────────┤
│  L7  Inspector层  质检/审查（原L4）                  │
│  L8  质量门禁层   通过/拦截（原L5）                   │
│  L9  反馈闭环层   复盘/优化（原L6）                   │
│  L10 版本发布层   候选/审批/灰度/回滚                 │ ← 新增
│  L11 进化实验层   沙盒自优化/实验评测/隔离            │ ← 新增
└─────────────────────────────────────────────────────┘
```

**两维度说明：**
```
横向（组织层级）：CEO → VP（平台固定11位）→ 总监（企业可扩）→ 子Agent（N）
纵向（工作流）  ：入口 → 权限 → 编排 → 协作 → 安全
                  → 执行 → 质检 → 门禁 → 闭环 → 发布 → 进化（11层）
两个维度正交，互不冲突
```

## 子层规划（系统预设不污染）

目标：用户Agent可进化，系统预设保持稳定且不被自动污染。

### 层级结构（含子层）

```
L1 入口扩展层
L2 权限管理层
  ├── L2-SYS   （系统预设身份与权限域）
  └── L2-USER  （用户Agent身份与权限域）
L3 任务编排层
L4 协作编排层
L5 安全策略层
L6 执行监控层
  ├── L6-SYS   （系统执行监控与告警）
  └── L6-USER  （用户执行监控与告警）
L7 Inspector层
L8 质量门禁层
L9 反馈闭环层
L10 版本发布层
  ├── L10-SYS-RELEASE   （系统预设发布通道）
  └── L10-USER-RELEASE  （用户Agent发布通道）
L11 进化实验层
  └── L11-USER-EVO      （用户自优化沙盒）
```

### 子层优先级（先规划）

1) **L2 权限管理层（第一优先）**
- 规划子层：`L2-SYS` / `L2-USER`
- 作用：先切清身份与作用域边界（谁能做什么）。

2) **L11 进化实验层（第二优先）**
- 规划子层：`L11-USER-EVO`
- 作用：仅用户侧进入自优化沙盒，系统预设默认不进化。

3) **L10 版本发布层（第三优先）**
- 规划子层：`L10-SYS-RELEASE` / `L10-USER-RELEASE`
- 作用：统一发布门禁，任何进化结果先过评测+审批再发布。

4) **L6 执行监控层（第四优先）**
- 规划子层：`L6-SYS` / `L6-USER`
- 作用：运行态隔离监控、限流、告警与审计分治。

### 核心原则（先写死）
- 进化只在用户隔离域发生，不允许直接写入系统预设域。
- 系统预设只接受发布层合入，不接受实验层直连。
- 任何发布必须可追踪、可审批、可回滚。

## 企业级能力分层归属（标准口径）

- **L1 入口扩展层**：注册/登录入口、请求归一化、租户识别、限流防刷。
- **L2 权限管理层**：角色体系、指挥权分配、租户与组织隔离、审批权限。
- **L3 任务编排层**：任务拆解、Agent分配、模型路由与调度策略。
- **L4 协作编排层**：多指挥者协调、跨Agent通信、冲突仲裁、状态机推进。
- **L5 安全策略层**：数据访问策略执行、合规检查、敏感操作拦截（deny-by-default）。
- **L6 执行监控层**：运行监控、告警、重试、熔断、分域监控（SYS/USER）。
- **L7 Inspector层**：目标对齐检查、完整性验证、质量评分、风险扫描。
- **L8 质量门禁层**：通过/打回决策，不达标结果不得进入发布链路。
- **L9 反馈闭环层**：复盘沉淀、经验回写、优化建议管理。
- **L10 版本发布层**：候选版本、审批、灰度发布、正式发布、回滚。
- **L11 进化实验层**：用户自优化沙盒、实验评测、进化结果提案（不可直达生产）。

## Deep Agent 分层归属与治理链

定位：Deep Agent 是执行能力体，不是总框架；总框架仍由11层治理链路承载。

- **主归属层：L3 任务编排层**
  - 负责 Deep Agent 的分配、路由、降级和成本/SLA权衡。
- **协同层：L4 协作编排层**
  - 负责多 Deep Agent 并行协作、冲突仲裁和状态推进。
- **运行层：L6 执行监控层**
  - 负责运行健康、时延、成本、失败重试与熔断反馈。
- **治理链：L7 -> L8 -> L10 -> L11**
  - L7 质量检查
  - L8 门禁决策
  - L10 发布与回滚
  - L11 进化实验（仅用户隔离域）

执行红线：
- Deep Agent 不得绕过 L2/L5 权限与安全策略。
- Deep Agent 输出不得绕过 L7/L8 直接进入生产。
- 进化结果不得直写生产，必须经 L10 发布链路。

## Skills 生命周期与分层约束

定位：Skills 为平行能力插件，由 Agent 挂载与调用，不构成新的组织层级。

生命周期：
- 注册（Registry）-> 沙盒验证（L11）-> 质量门禁（L8）-> 发布（L10）-> 运行监控（L6）-> 回滚/下线

分层约束：
- **L3**：Skills 路由与调用策略（由编排统一调度）。
- **L2**：Skills 调用权限控制（角色/租户/作用域）。
- **L5**：Skills 运行安全策略（命令/文件/API 白名单）。
- **L8/L10**：Skills 发布门禁、版本发布与回滚。
- **L11**：用户新增或变更 Skills 的实验与验证域。

治理要求：
- 用户可新增 Skills，但仅在 USER 域生效，不得污染 SYS 预设域。
- Skills 必须版本化、可审计、可回滚。
- 高风险 Skills 默认拒绝，需显式审批放行。

## 长期维护边界（执行准则）

为保证平台可持续演进且不失控，长期维护遵循以下五个边界：

1) **组织边界**
- VP 层平台固定治理（企业不可新增VP）。
- 企业可在既有VP下扩总监；总监下子Agent按N弹性扩缩。

2) **分层边界**
- L2/L5/L8/L10/L11 为核心治理链路，不可绕过。
- 任意能力扩展不得破坏分层职责边界。

3) **扩展边界**
- 用户可新增 Skills，但仅在 USER 域生效。
- Skills 调用必须受 L2 权限与 L5 安全策略约束。

4) **质量边界**
- 新能力默认先在 L11 沙盒验证，再经 L8/L10 门禁发布。
- 任何上线能力必须支持版本化与可回滚。

5) **审计边界**
- 全链路行为必须可追踪、可回放、可追责。
- 高风险动作必须有审批记录与执行留痕。

## 层级补全清单（原有/需更新/新增）

范围：仅针对“层级治理”继续完善项，不涉及代码实现。

### 1) 责任矩阵落表（Owner / 输入 / 输出 / SLO / 失败处理）

**原有：**
- 已有11层职责主描述（层级结构与职责边界已明确）。
- 已有部分输入/输出描述，但分散在不同文档中。

**需要更新：**
- 将每层职责统一落为标准矩阵字段：`Owner / 输入 / 输出 / SLO / 失败处理`。
- 补齐 `L10/L11` 的SLO与失败处理口径（当前偏原则描述）。

**新增：**
- 新增《11层责任矩阵总表》（一页可审阅版本）。
- 新增 Owner 默认归属规则（平台侧/租户侧/自动化流程）。

### 2) 跨层调用白名单（允许关系 + 禁止隐式穿透）

**原有：**
- 已有“核心治理链路不可绕过”原则（L2/L5/L8/L10/L11）。
- 已有“模型分配在L3，L4/L6协同”的职责边界共识。

**需要更新：**
- 将原则转成“可执行调用矩阵”：明确各层可调用路径与调用方向。
- 明确禁止路径（如实验层直达生产、编排层绕过门禁等）。

**新增：**
- 新增《跨层调用白名单表》（Allowed / Denied / Needs-Approval 三态）。
- 新增“隐式穿透告警规则”（出现未授权跨层即记录并告警）。

### 3) 变更分类机制（仅编号变化 / 职责变化 / 新增能力）

**原有：**
- 已有“仅改编号不改职责”的沟通共识。
- 已有部分新旧层映射描述，但缺少统一标记规范。

**需要更新：**
- 在总览中固定“变更分类说明”章节，后续改动统一打标签。
- 对关键层级调整补齐分类标记，避免评审误解为重构。

**新增：**
- 新增三类标准标签：
  - `仅编号变化`
  - `职责变化`
  - `新增能力`
- 新增《变更记录模板》：影响层、分类、原因、风险、回滚点。

## Skills 治理补全（关键优先）

### 必补项（写入治理标准）

1) **Skill Registry**
- 必填字段：`id / version / owner / scope / risk_level`。

2) **Skill 门禁流程**
- 标准链路：开发 -> 沙盒验证（L11）-> 质量门禁（L8）-> 发布（L10）。

3) **Skill 安全策略**
- 文件/命令/API 权限最小化。
- 默认拒绝，按白名单显式放行。

4) **Skill 兼容策略**
- 每个 Skill 必须声明依赖的 Agent 类型与最低平台版本。

5) **Skill 回滚机制**
- 每次升级必须支持快速回退到上个稳定版本。

### 状态标记（原有/需更新/新增）

**原有：**
- 已有 Skills 生命周期与分层约束（L2/L3/L5/L8/L10/L11）描述。
- 已有“用户可新增 Skills，但不可污染 SYS 域”的治理原则。

**需要更新：**
- 将 Registry 字段从描述性规则升级为强制校验项。
- 将门禁链路从原则升级为发布前必经流程。
- 将兼容策略与回滚要求纳入发布准入条件。

**新增：**
- 新增《Skill 注册规范（字段与校验）》。
- 新增《Skill 发布门禁清单》与《Skill 回滚SOP》。
