## VP层设计

### 写死（系统内置，所有用户都有）
```
数字VP（Digital VP）
  ├── Explore Agent    总监1  代码探索
  ├── Plan Agent       总监2  规划架构
  ├── General Agent    总监3  执行实施
  ├── Inspector Agent  总监4  质检审查
  └── Research Agent   总监5  研究查资料
```
**定位**：多行业通用的技术基础底座，任何业务都需要这5个能力。

### 占位（用户自建，按业务需要激活）

| VP | 定位 | 预设总监方向 |
|---|---|---|
| 智能VP | AI/ML能力 | 数据分析、模型训练、知识图谱、AI研究 |
| 通用力VP | 共享执行服务 | 进程管理、调度、系统集成、监控 |
| 生产VP | 上线运维 | 部署、CI/CD、配置、故障处理 |
| 销售运营VP | 市场增长 | CRM、营销、客户支持、收入分析 |
| 产品研发VP | 产品设计 | 需求、原型、路线图、用户研究 |
| 业务支持VP | 内部职能 | HR、法务、财务、合规 |
| 内容VP | 内容品牌 | 内容创作、审核、SEO、社媒 |

## 通信规则（6层终态版，与 AGENTS.md 一致）

### 直接允许
```
CEO      ↔  副总（VP）   （上下级直接沟通）
副总     ↔  自己的总监   （上下级直接沟通）
总监     ↔  自己的经理   （上下级直接沟通）
经理     ↔  自己的子Agent （上下级直接沟通）
子Agent  ↔  自己的副Deep Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
副总 ↔ 副总                        → CEO 强制主持
同VP下总监 ↔ 总监                   → 所属副总强制主持
同总监下经理 ↔ 经理                 → 所属总监强制主持
同经理下子Agent ↔ 子Agent            → 所属经理强制主持
同子Agent下副Deep Agent ↔ 副Deep Agent → 所属子Agent强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信         → DENY，必须上报自己的副总，走 副总↔副总 通道
任何跳级通信             → DENY（CEO 不直达总监/经理/子Agent；副总不直达经理/子Agent；总监不得跳过经理直连子Agent）
副Deep Agent 主动上报    → DENY，只能被动响应所属子Agent，不可主动越级
```

### 跨VP通信正确链路
```
IT总监A 需要和 运营总监B 沟通：
  IT总监A → 上报 IT副总
               └→ IT副总 ↔ 运营副总（CEO三方主持）
                              └→ 运营副总 传达 运营总监B
```
**设计原则**：副总是本域对外的唯一出口，不可被绕过（否则副总架空）

## 关键代码改动

### COMM_RULES：从"基于Agent ID" → "基于level"

```javascript
// 现在（错误方向）：点名道姓，加VP要改规则
match: ctx => ctx.fromId === 'supervisor' && ctx.toId === 'explore'

// 改后（正确）：基于层级，加任何VP自动生效
```
