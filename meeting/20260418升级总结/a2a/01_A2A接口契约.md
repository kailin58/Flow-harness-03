# A2A 接口契约

**性质**：`Protocol Port` 下 `A2A Adapter` 的可落地接口规范  
**归属**：`VP11（External）` 第三方 Agent 接入与 A2A 协议管理  
**上位依赖**：[`../04_长期架构.md`](../04_长期架构.md)、[`../03_Agent协作保障.md`](../03_Agent协作保障.md)、[`../06_HumanInTheLoop对话判定协议.md`](../06_HumanInTheLoop对话判定协议.md)、[`../19_层级扩展与跨层通讯协议.md`](../19_层级扩展与跨层通讯协议.md)、[`../../../AGENTS.md`](../../../AGENTS.md)

---

## 一、目标

在不改动内核组织层与治理链的前提下，为 Flow Harness 增加对外部 Agent 系统的标准化协作能力。

本契约解决：

- 外部 Agent 能力发现
- 外部 Agent 任务提交
- 长任务状态跟踪与流式回传
- 人工补充输入与超时接管
- 结果产物回收与审计留痕

本契约不解决：

- 内部 Agent 之间的直接通信替换
- 工具调用协议替代（工具仍优先走 `MCP / Tool Port`）
- 绕过 `security.permissions`（L6）/`security.policy`（L14）/`quality.gate`（L18）/`release.publish`（L22）（核心治理链）的快速直通

---

## 二、架构定位

```text
外部 Agent 系统
    ↕
L1.2 gateway.ingress.a2a（A2A Adapter / Protocol Port）
    ↕
L2 external.system_mgmt（外部系统管理层：注册 / 健康 / 降级）
    ↕
Flow Harness 内核（CEO / VP / 工作流 / 治理链）
```

定位规则：

1. A2A 仅负责“外部 Agent 协作”。
2. MCP 负责“外部工具与数据源接入”。
3. 内部 Agent 间协作仍走既有 `COMM_RULES` 与组织层级。
4. 外部协议差异仅允许停留在 `A2A Adapter` 内。

## 四、接口定义

### 4.1 能力发现 `discoverAgent`

作用：读取外部 Agent Card 并转换成 `ExternalAgentProfile`。

最小输入：

- `external_agent_id` 或 `agent_card_url`

最小输出：

- `ExternalAgentProfile`

### 4.2 提交任务 `sendTask`

作用：向外部 Agent 发起一次任务。

最小输入：

- `ExternalTaskRequest`

最小输出：

- `external_task_id`
- `accepted`
- `initial_status`

### 4.3 流式执行 `streamTask`

作用：处理长任务的持续状态更新与分块产物回传。

最小事件：

- `status_changed`
- `artifact_chunk`
- `input_required`
- `completed`
- `failed`

### 4.4 查询任务 `getTask`

作用：按 `external_task_id` 拉取当前状态与最终结果。

### 4.5 取消任务 `cancelTask`

作用：在超时、人工终止、上游撤销时取消外部任务。

## 五、状态映射（外部 -> 内部）

| 外部状态 | 桥接状态 | 内部建议状态 |
|---|---|---|
| `submitted` | `EXTERNAL_DISPATCHING` | `PENDING` / `DELIBERATING` |
| `working` | `EXTERNAL_WORKING` | `EXECUTING` |
| `input-required` | `EXTERNAL_INPUT_REQUIRED` | `HUMAN_PENDING` / HITL |
| `completed` | `EXTERNAL_RETURNED` | `INSPECTING` 前置态 |
| `failed` | `EXTERNAL_FAILED` | `NEEDS_FIX` / `FAILED` |
| `cancelled` | `EXTERNAL_CANCELLED` | `FAILED` / 已终止 |

规则：

1. 外部 `completed` 不等于内部可直接通过。
2. 外部返回结果必须重新进入内部检查与门禁。
3. 需要人工补充输入时，统一转入 HITL，而不是由 Adapter 自行决策。

## 六、安全与治理

### 6.0 外部接入硬规则

A2A 接入属于外部不可信边界，必须额外满足以下硬规则：

1. 外部 Agent 仅可接收我方**受控摘要 / 受控引用 / 脱敏副本**并返回结果，禁止主动读取我方文件、仓库、`S1-S4`、配置、凭据与内网目录。
2. A2A 通信必须全程加密；至少使用 `TLS`，高敏场景建议 `mTLS`，并对令牌、会话、回调地址与证据链接做受保护传输。
3. 外部 Agent 返回内容、附件、链接与流式分片均视为潜在污染源，必须做格式校验、白名单过滤、风险评级与必要的人工复核。
4. 外部 Agent 结果不得直接形成生产结论，必须重新进入 `security.permissions`（L6）/`security.policy`（L14）/`quality.gate`（L18）/`release.publish`（L22）（核心治理链）。

### 6.1 外发前检查

外发前至少检查：

- 当前租户 / 作用域是否允许调用该外部 Agent
- 是否命中 VP11 白名单或合作方清单
- 上下文是否包含敏感数据
- 是否需要脱敏
- 是否需要人工批准后才能外发

### 6.2 外发边界

禁止：

- 未经授权发送完整内部记忆库
- 未经脱敏发送敏感原文
- 给予外部 Agent 我方文件系统、仓库目录、数据库、对象存储与共享盘的直接读取能力
- 绕过 `security.permissions`（L6）/`security.policy`（L14）/`quality.gate`（L18）/`release.publish`（L22）直接使用外部结果作生产结论

允许：

- 发送压缩上下文摘要
- 发送已授权的附件引用或脱敏副本
- 发送结构化任务请求与约束
- 在受控网络与加密链路上回收结果并进入内部审查

## 七、审计事件（S3）

所有 A2A 关键动作均需异步写入 `S3`，并遵守 `AGENTS.md` 的缺口8写入约定。

建议事件类型：

- `A2A_AGENT_DISCOVERED`
- `A2A_TASK_SUBMITTED`
- `A2A_TASK_STATUS_CHANGED`
- `A2A_TASK_INPUT_REQUIRED`
- `A2A_TASK_COMPLETED`
- `A2A_TASK_FAILED`
- `A2A_TASK_CANCELLED`

每条事件最少包含：

- `trace_id`
- `task_id`
- `external_agent_id`
- `external_task_id`
- `event_type`
- `event_time`
- `tenant_scope`
- `evidence_ref`

## 八、与现有模块的关系

### 8.1 与 `VP11`

- `VP11` 负责接入治理、合作方管理、能力注册与路由策略。
- `VP11` 不是协议报文解析器；协议细节应留在 `A2A Adapter`。

### 8.2 与 `MCP`

- `MCP`：工具 / 数据源接入
- `A2A`：外部 Agent 接入

两者互补，不互相替代。

### 8.3 与 `HITL`

以下情况直接转入 `HITL`：

- 外部 Agent 请求补充输入
- 外部任务长时间无进展
- 外部任务失败且风险高
- 外部结果命中高风险判定

## 九、升级路径

### Phase 1：文档闭环

- 固化本接口契约
- 在 `04_长期架构.md` 与 `00_升级总结总览.md` 建立入口链接

### Phase 2：Mock Adapter

- 建立本地 `A2A Adapter` mock
- 跑通提交、查询、流式、取消、失败、人工补充输入

### Phase 3：单场景试点

- 只接 1 个外部 Agent
- 只试 1~2 个低风险场景

### Phase 4：多供应商扩展

- Agent Card 注册表
- 多外部 Agent 路由
- 策略化 fallback

---

## 十、结论

本方案的核心不是“先占位”，而是：

1. 先定义接口契约；
2. 把易变部分收敛到 `A2A Adapter`；
3. 保持内核、治理链和组织层稳定；
4. 以后协议升级、供应商替换、能力扩展都不需要推翻现有架构。
