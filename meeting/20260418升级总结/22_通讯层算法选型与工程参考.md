## 二、消息路由架构

### 2.1 学术最新进展

#### 论文1：Semantic Router DSL（2026，arXiv 2603.27299）

**核心思想**：用非图灵完备的声明式策略语言编写路由规则，编译器生成可验证的决策节点。

**关键能力**：
- 保证「穷举路由」——任意消息必命中且仅命中一条规则，无静默丢失
- 冲突自由分支——编译期即发现策略冲突，不等到运行时
- 单点策略变更，传播至所有层

**与项目对照**：`comm-router.js` 的 `COMM_RULES` 数组 + `Object.freeze` 模式已是声明式规则表的实现雏形，`layer-router.js` 可直接沿用此架构（`LAYER_ROUTES` 数组 + 启动时 `selfCheck()`）。

#### 论文2：MasRouter（2025，arXiv 2502.11133）

**核心思想**：多 Agent 路由——级联控制网络，依次决策「协作模式 → 角色分配 → LLM 选型」。

**数据**：overhead 降低最高 52.07%。

**与项目对照**：路由决策三步串行（§9.7 无模型原则兼容：协作模式用规则引擎，模型选型在 L3/L9 内执行）。

#### 论文3：Federation of Agents（2025，arXiv 2504.08623）

**核心思想**：VCV 语义路由 + HNSW 索引，MQTT pub-sub 解耦，比单模型基线快 13×。

**与项目对照**：仅在 Agent 数量 > 200 时才经济，**当前阶段过早引入**。

### 2.2 选型结论（与项目匹配）

```
阶段一（当前）：声明式规则数组（LAYER_ROUTES）+ 内存 Map 路由
               → 模式对齐 comm-router.js，零新依赖
               → 启动时 selfCheck() 验证规则完整性

阶段二：实现完整 layer-router.js（§15.2 DoD）
       → 加入拓扑版本热更新、旁路白名单、反向通道

阶段三（可选）：VP 间协同量大时接入轻量 pub-sub
阶段四（未来）：Agent 数 > 200 时评估 VCV 语义路由
```

## 三、限流算法

### 3.1 五种算法全对比

| 算法 | 内存 | 突发支持 | 精度 | CPU | 最适场景 |
|---|---|---|---|---|---|
| **固定窗口计数器** | O(1) | 允许边界 2× 突发 | 近似 | 极低 | 简单 / 低流量 |
| **令牌桶（Token Bucket）** | O(1) | ✅ 允许合理突发 | 精确 | 低 | 突发流量 + 平均速率上限 |
| **漏桶（Leaky Bucket）** | O(1) | ❌ 不允许突发 | 精确 | 低 | 严格输出平滑 |
| **滑动窗口计数器** | O(1) | 平滑突发 | 近似精确 | 低 | 通用 API 限流（**推荐**） |
| **滑动窗口日志** | O(n) | ❌ | 精确 | 中 | 高价值 API / 审计追踪 |

来源：[Redis 限流指南](https://redis.io/learn/howtos/ratelimiting)、[BackendBytes 对比](https://backendbytes.com/articles/rate-limiter-algorithms)

### 3.2 项目落地建议（与实际代码对照）

**落地位置**：`src/layer-hooks.js` 中已有 `RATE_LIMITER_HOOK`（TODO 占位），优先在此实现，不另起炉灶。

```
L1 入口（RATE_LIMITER_HOOK）：
  → 滑动窗口计数器（内存 O(1)，无边界突发问题，无需外部依赖）

layer-router 反向/escalate 通道（内部，§9.3.1 线②）：
  → 漏桶（强制输出平滑，防止 escalate 风暴 A4.4）

子任务并发池（§9.3.1 线③）：
  → 计数器信号量（per-trace_id，++/-- 计数，最简实现）
```

### 3.3 成功案例

- **AWS API Gateway**：令牌桶（允许短期突发）
- **Nginx**：漏桶（`limit_req`），严格平滑输出
- **Cloudflare**：滑动窗口，边缘节点分布式
- **Stripe**：每客户端令牌桶 + 全局固定窗口双层叠加

### 3.4 实现参考（零依赖，直接落入 `RATE_LIMITER_HOOK`）

```javascript
// 滑动窗口计数器，零依赖，O(1) 内存，适合项目当前阶段
// 落地位置：src/layer-hooks.js → RATE_LIMITER_HOOK
class SlidingWindowCounter {
  constructor({ windowMs = 60_000, maxRequests = 100 } = {}) {
    this.windowMs = windowMs;
    this.maxRequests = maxRequests;
    this.buckets = new Map(); // key → { prev, curr, lastReset }
  }

  check(key) {
    const now = Date.now();
    let b = this.buckets.get(key);
    if (!b) {
      b = { prev: 0, curr: 0, lastReset: now };
      this.buckets.set(key, b);
    }
    const elapsed = now - b.lastReset;

    if (elapsed >= this.windowMs) {
      b.prev = elapsed < this.windowMs * 2 ? b.curr : 0;
      b.curr = 0;
      b.lastReset = now;
    }

    const weight = 1 - elapsed / this.windowMs;
    const estimated = b.prev * weight + b.curr;
    if (estimated >= this.maxRequests) return false;

    b.curr++;
    return true;
  }

  // 定期清理过期 key，防止 Map 无限增长
  cleanup() {
    const cutoff = Date.now() - this.windowMs * 3;
    for (const [k, b] of this.buckets) {
      if (b.lastReset < cutoff) this.buckets.delete(k);
    }
  }
}
```

### 4.1 学术最新进展

- 最优插入性能的并发优先级队列，专为「插入远多于删除」的工作负载设计

#### BBQ（CMU-SNAP，2024）
- 硬件 O(1) 优先级队列（Integer Priority Queueing），用于线速交换机
- 思想可借鉴：「有限优先级桶」（3档桶）即是 IPQ 思想的简化版

### 4.2 关键发现：Array.shift() 不是 O(1)

> **这是 22_ 文档 v1.0 的严重错误，必须修正。**

| 实现 | dequeue 复杂度 | 100K 操作耗时 |
|---|---|---|
| `Array.shift()` | **O(n)**（V8 未优化，#42449） | **1,152 ms** |
| `denque` 双端队列 | O(1) | **1.29 ms**（快 484×） |
| 小数组（< 1000 项）时 `Array.shift()` | 约 O(1)（V8 孔洞优化） | 可接受 |

来源：[denque benchmarks](https://docs.page/invertase/denque/benchmarks)、[Node.js #42449](https://github.com/nodejs/node/issues/42449)

**项目决策（两条路径，按需选择）**：

```
路径 A（零依赖，适合当前极早期阶段）：
  ► 使用 3 档 Array 队列
  ► 强制硬限制：每档最大 500 条（total ≤ 1500 条）
  ► 此范围内 Array.shift() 在 V8 孔洞优化下约为 O(1)
  ► 超出限制则走 §9.3.3 升级流程

路径 B（最小依赖，稍后引入）：
  ► npm install denque（零运行时依赖，纯 JS，CJS 兼容）
  ► denque 在 Node.js 生态中被 Bull、node-redis 等大量使用
  ► 解除队列大小硬限制，支持 10K+ 条无性能降级
```

### 4.5 实现参考（路径 A，零依赖）

```javascript
// 三档优先级队列（零依赖版，对齐 19_ 文档 §9.3.1 命名）
// 约束：每档硬上限 500 条，总计 ≤ 1500 条
// 升级触发：见 §8.2，任一触发时引入 denque 或外部 MQ

const MAX_PER_TIER = 500;

class ThreeTierQueue {
  constructor() {
    // 命名与 19_ §9.3.1 对齐：critical > high > normal
    this.queues = { critical: [], high: [], normal: [] };
    this._sizes = { critical: 0, high: 0, normal: 0 };
  }

  enqueue(item, priority = 'normal') {
    if (!this.queues[priority]) throw new Error(`无效优先级: ${priority}`);
    if (this._sizes[priority] >= MAX_PER_TIER) return false; // 返回 false 表示拒绝
    this.queues[priority].push(item);
    this._sizes[priority]++;
    return true;
  }

  dequeue() {
    for (const tier of ['critical', 'high', 'normal']) {
      if (this._sizes[tier] > 0) {
        this._sizes[tier]--;
        return this.queues[tier].shift(); // 硬上限 500，V8 孔洞优化生效，约 O(1)
      }
    }
    return null;
  }

  get size() {
    return this._sizes.critical + this._sizes.high + this._sizes.normal;
  }

  isFull(tier = 'normal') {
    return this._sizes[tier] >= MAX_PER_TIER;
  }
}
```

### 5.4 背压（Backpressure）水位策略

与 §9.3.1 联动：

```
队列水位 < 50%  → 正常接入
队列水位 50–80% → 降级：拒绝 normal 优先级消息，返回 429 + Retry-After
队列水位 > 80%  → 开熔断：拒绝 high + normal，仅通过 critical
队列水位 > 95%  → 全熔断 + 告警（P1）
```

### 6.2 Flow Harness 当前阶段实现（零依赖路径）

Node.js **没有**内置 LRU Map。有两条路径：

```
路径 A（零依赖，手写 LRU）：
  ► 用 Map + 双链表实现
  ► 维护成本高，仅适合 ≤ 1000 条的场景

路径 B（最小依赖，推荐）：
  ► npm install lru-cache（389M 周下载量，Node.js 生态最主流 LRU 实现）
  ► 零运行时依赖，TypeScript 定义内置，CJS + ESM 兼容
  ► isaacs 维护，被 npm cli 等核心工具直接依赖
```

**规范**（对应 §9.4.1）：

```javascript
// 路径 B 实现参考
const { LRUCache } = require('lru-cache');

const idempotencyStore = new LRUCache({
  max: 10_000,                  // 最多 10K 条记录
  ttl: 24 * 60 * 60 * 1000,    // 默认 24h TTL
  ttlAutopurge: false,          // 按需延迟清理，减少 CPU 峰值
});

// 键格式：{tenant_id}:{trace_id}:{handoff_id}
// 状态字段：'pending' | 'processing' | 'completed' | 'failed'
function dedup(key, executor) {
  const existing = idempotencyStore.get(key);
  if (existing?.status === 'completed') return existing.result;      // 直接返回
  if (existing?.status === 'processing') throw { code: 409 };        // 拒绝并发重复
  
  idempotencyStore.set(key, { status: 'processing' });
  return executor().then(result => {
    idempotencyStore.set(key, { status: 'completed', result });
    return result;
  }).catch(err => {
    idempotencyStore.set(key, { status: 'failed', err: err.message });
    throw err;
  });
}
```

## 七、加密传输

### 7.1 ⚠️ 关键纠正：Istio Ambient 不适合当前阶段

> **v1.0 文档的最大错误**：推荐了 Istio Ambient 作为 mTLS 方案。
>
> **真相**：项目当前是**单体 Node.js 服务，无 Kubernetes，无 Docker 基础设施**（package.json 实证）。Istio 是 K8s 生态组件，引入代价极大（运维、证书管理、控制平面），**完全不适合当前阶段**。

正确的分阶段加密路径：

| 阶段 | 场景 | 推荐方案 | 复杂度 |
|---|---|---|---|
| **当前（单体 Node.js）** | 同进程层间调用 | **无需加密**（Object.freeze + 只读视图） | 零 |
| **当前（跨进程）** | 同主机 IPC | Unix Socket + 文件权限位 | 极低 |
| **近期（多服务）** | 跨主机 HTTP/gRPC | **Node.js 内置 `tls` 模块**（TLS 1.3） | 低 |
| **规模化（微服务集群）** | 内部服务网格 | Istio Ambient（ztunnel，K8s 内）或 Nginx mTLS | 中高 |
| **外部边界** | VP11 外部 Agent | 强制 mTLS（AGENTS.md 写死） | 中 |

### 7.2 Istio Ambient 性能（供未来规模化阶段参考）

来源：[Istio 官方性能报告 2025](https://istio.io/latest/blog/2025/ambient-performance)

| 模式 | vCPU | 内存 | 延迟 |
|---|---|---|---|
| Istio Sidecar（mTLS） | 0.20 vCPU / 1000 RPS | 60 MB | +2–5 ms P99 |
| Istio Ambient（ztunnel） | 0.06 vCPU | 12 MB | +0.5–1 ms P99 |

未来 K8s 阶段，Ambient 模式是首选（性能最优）。**当前阶段勿引入**。

### 7.3 HMAC 签名性能（当前阶段参考）

来源：[HMAC-SHA256 对比研究](https://etasr.com/index.php/ETASR/article/view/14149)

- 信封签名场景（payload < 4 KB）：**HMAC-SHA256 单次 < 0.1 ms**，Node.js `crypto` 内置
- bcrypt → HMAC-SHA256：延迟从 946 ms 降至 < 1 ms

**Flow Harness 选型**：
```
信封完整性签名  → HMAC-SHA256（Node.js crypto 内置，零依赖）
内容哈希校验   → SHA-256（无密钥，更快）
同进程内部 hop → 跳过签名（Object.freeze 已保证只读）
```

### 7.4 密钥算法推荐（2025 标准，跨主机阶段适用）

来源：[密码算法决策框架 2025](https://axis-intelligence.com/how-to-choose-cryptographic-algorithms-guide/)

| 用途 | 推荐算法 | 弃用 |
|---|---|---|
| 对称加密 | AES-256-GCM | AES-CBC（无认证） |
| 非对称/密钥交换 | ECDH P-384 / X25519 | RSA-2048（可用但趋于弃用） |
| 签名 | Ed25519 / ECDSA P-256 | RSA-PKCS1v15 |
| 消息认证 | HMAC-SHA256 | MD5-HMAC |
| TLS 版本 | TLS 1.3（强制），1.2（兼容降级） | TLS 1.0 / 1.1 |

## 八、消息队列选型（内存 vs 外部）

### 8.1 关键纠正：Kafka 上下文澄清

> **v1.0 文档的认知错误**：将 CLAUDE.md 中的「Kafka（outbox）」理解为 flow-harness 已有 Kafka。
>
> **真相**：
> - CLAUDE.md 描述的是 **ACmodus 目标平台**的技术栈（Node.js/Express/MySQL/Kafka），不是 flow-harness 本身
> - flow-harness 的 `package.json` **没有 Kafka 依赖**
> - Kafka outbox 是 ACmodus 业务服务（订单/支付）的跨服务事件投递，与 `layer-router` 内部管道完全不同的关注点

**两个域的分工**：

```
Kafka (outbox) → ACmodus 业务服务间「订单创建 → 支付服务」事件投递
                 写后异步，跨服务，需持久化，属 L2 外部系统管理层

layer-router 内部队列 → flow-harness Agent 编排框架内「L6 → L7 → L8」层间流转
                        同进程内，速率 << 1K/s，内存队列足够
                        两者职责不同，不应混淆
```

### 8.2 队列选型矩阵

来源：[Message Queue 2025 决策指南](https://www.youngju.dev/blog/culture/2026-03-22-message-queue-kafka-rabbitmq-sqs-comparison-2025.en)

| 方案 | 持久化 | 重放 | 跨进程 | 吞吐 | 成本 | 适用阶段 |
|---|---|---|---|---|---|---|
| **内存队列（当前）** | ❌ | ❌ | ❌ | 极高 | 零 | layer-router 阶段一 |
| **SQLite 溢出持久化** | ✅ | ✅ | ❌ | 中 | 极低 | 进程重启恢复 |
| **Redis Streams** | ✅ | ✅ | ✅ | 高 | 低 | 跨进程 / 多实例 |
| **Kafka** | ✅ | ✅ | ✅ | 极高 | 中 | 业务事件 / 审计重放 |

**⚠️ 警示案例**：某团队为每秒 2 条消息部署 6-broker Kafka，月耗 $14,000——flow-harness layer-router 当前规模绝对无需 Kafka。

### 8.3 升级触发条件（精确，对应 §9.3.3）

```
内存队列 → SQLite 溢出层（不是升级，是追加容灾）：
  ① 进程需要在重启后恢复未完成 trace

内存 → Redis Streams：
  ① 进程内存队列持续 > 256 MB（5 min 均值）
  ② 消息积压 > 10 000 条
  ③ 进程重启丢失任务 > 3 次 / 周
  ④ 需要跨进程/多实例协调

Redis → Kafka：
  ① 需要 7 天以上消息重放（L14.1 评测集重建）
  ② 跨服务消费者 > 3 个订阅同一管道
  ③ 峰值 > 100K msg/s
```
