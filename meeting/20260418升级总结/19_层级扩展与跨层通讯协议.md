## 一、为什么需要这份文档

当前架构存在两套"层"的概念，文档对其通讯规则的覆盖严重不对称：

| 维度 | 层级体系 | 通讯规则覆盖程度 |
|---|---|---|
| **横向（组织层）** | CEO/VP/总监/经理/子Agent/副Deep（OL0-OL5） | ✅ 完整定义：直接允许/三方会话/DENY，基于level路由 |
| **纵向（工作流层）** | 2M + L1-L23（2个基础设施模块 + 23个主层 + 若干子层） | ❌ 严重缺失：无消息信封、无注册协议、无反向反馈协议 |

**具体缺口：**

1. 任务从 `capability.market`（L5）流到 `security.permissions`（L6）时，消息长什么样？**没有标准信封**。
2. 新插入 `external.system_mgmt`（L2）外部系统管理层，原来的 L1→旧L2 管道怎么知道要"经过"新层？**没有注册机制**。
3. `quality.gate`（L18）质量门禁打回后，消息怎么回到 `execution.monitor`（L15）执行监控层？**没有反向反馈协议**。
4. 层号从11层升到23层时，代码/配置/接口里的层号引用全部失效。**没有稳定标识符体系**。
5. 什么条件下允许跳过某层（如 `quality.reflection`（L17）反思层）？**没有旁路白名单**。

本文档系统解决以上五个缺口。

## 二、两套通讯维度的完整定义

### 2.1 横向：组织通讯（已定义，此处仅对齐口径）

组织层使用编号 `OL0`–`OL5`（Organization Level）以区别于工作流层号：

```
OL0  CEO（Supervisor）
OL1  VP（副总，11个）
OL2  总监（Director，各VP下若干）
OL3  经理（Manager，M01-M11命名空间，实例不限）
OL4  子Agent（SubAgent，N个）
OL5  副Deep Agent（按需孵化，ephemeral）
```

**通讯规则（写死）**：

| 类型 | 规则 |
|---|---|
| 相邻级直连 | `OLn ↔ OL(n+1)` 直接允许 |
| 同级通讯 | 必须由共同上级主持三方会话 |
| 跨级/跳级 | DENY，无论向上还是向下 |
| 跨VP横穿 | 必须逐级上报到VP，VP↔VP由CEO主持 |

实现入口：`src/comm-router.js`，COMM_RULES 基于 `fromLevel / toLevel`。

---

### 2.2 纵向：工作流层间通讯（本文档核心）

工作流层是**管道（Pipeline）**，不是组织。任务沿管道流动，不同于组织通讯的"谁指挥谁"。

```
两类通讯方向：
  ① 顺向流转：任务从 Ln 流向 L(n+1)   ← 正常执行管道
  ② 反向反馈：Lm → Lk（m > k）        ← 质检打回/反思/降级
  
禁止行为：
  ③ 任意层直接跳层流转（除白名单外）
  ④ 外部绕过管道直接注入任意层中间位置
```

## 三、层稳定标识符体系（解决层号漂移问题）

### 3.1 问题根因

每次插入新主层，所有层编号顺延 +N，导致代码/配置/文档中的层号引用全部失效（已发生两次：11→15层、15→16层）。

### 3.2 解决方案：layer_id 与 layer_order 分离

每个层有两个标识：

| 字段 | 类型 | 作用 | 是否稳定 |
|---|---|---|---|
| `layer_id` | string | 稳定业务标识，永不变更 | ✅ 永久稳定 |
| `layer_order` | number | 执行顺序，随拓扑变化而变 | ⚠️ 会随插层变化 |
| `layer_display` | string | 展示用层号（如 L4、L7） | ⚠️ 会随插层变化 |

**所有代码、配置、接口引用必须使用 `layer_id`，不得使用 `layer_order` 或 `layer_display`。**

## 四、标准层间消息信封（Layer Message Envelope）

任务在工作流层之间流转时，必须携带统一信封。每一层读取信封中自己关心的字段，处理后附加输出，传递给下一层。

### 4.1 信封结构

```json
{
  "envelope_version": "1.0",

  "routing": {
    "trace_id":       "全链路追踪ID（贯穿全部16层，不可变）",
    "session_id":     "会话ID",
    "task_id":        "任务ID",
    "from_layer_id":  "发出层的 layer_id",
    "to_layer_id":    "目标层的 layer_id",
    "direction":      "forward | backward",
    "hop_count":      "经过层数（从0递增）",
    "created_at":     "信封创建时间",
    "ttl":            "最大允许跳数（防死循环，默认32）"
  },

  "payload": {
    "task_type":      "任务类型标签",
    "goal":           "任务目标（L6注入，全程只读）",
    "risk_level":     "当前评估的风险等级",
    "content":        "本层处理后的核心内容（各层追加，不覆盖历史）",
    "intermediate_result_refs": ["S2制品库中的中间结果引用列表"],
    "tool_output_refs":         ["工具输出引用列表"]
  },

  "control": {
    "skip_layers":    ["允许跳过的 layer_id 列表（由白名单注入）"],
    "bypass_reason":  "旁路原因（若非空则本层跳过检查）",
    "force_stop":     false,
    "priority":       "normal | high | critical"
  },

  "feedback": {
    "rejection_from_layer_id":  "打回来源层 layer_id（反向流转时填写）",
    "rejection_reason":         "打回原因",
    "retry_count":              "当前重试次数",
    "max_retry":                "允许最大重试次数",
    "correction_suggestion":    "修正建议（L10/L11写入，L9读取）"
  },

  "audit": {
    "layer_trail": [
      {
        "layer_id":    "经过层的 layer_id",
        "agent_id":    "处理本层的 agent_id",
        "entered_at":  "进入时间",
        "exited_at":   "离开时间",
        "outcome":     "pass | skip | reject | error",
        "note":        "本层处理摘要"
      }
    ]
  }
}
```

### 4.2 各层对信封的读写权限（写死）

| 字段区 | 任意层可读 | 任意层可追加 | 只有特定层可写 |
|---|---|---|---|
| `routing.*` | ✅ | ❌ | 路由引擎写；各层只读 |
| `payload.goal` | ✅ | ❌ | 仅 `orchestration.task`（L12）首次写入 |
| `payload.content` | ✅ | ✅（追加不覆盖） | — |
| `payload.risk_level` | ✅ | ❌ | `tool.management`（L4）/ `knowledge.retrieval`（L8）/ `process.orchestration`（L10）/ `orchestration.task`（L12）可更新 |
| `control.skip_layers` | ✅ | ❌ | 仅路由引擎根据白名单注入 |
| `feedback.*` | ✅ | ❌ | 反向流转发起层写入 |
| `audit.layer_trail` | ✅ | ✅（本层追加一条） | 不可修改历史条目 |

## 五、层注册协议（新增层必须满足）

每个层（含新增层）在进入生产前，必须在 **S4 规则库** 完成注册，提供以下契约：

### 5.1 层注册字段

```json
{
  "layer_id":         "唯一业务标识（kebab-case，如 model.routing）",
  "layer_display":    "展示层号（新增时由 topology_version 分配）",
  "layer_name":       "可读名称",
  "version":          "层版本号（语义化版本）",
  "owner_vp":         "所属VP（如 VP02、VP01）",
  "prev_layer_id":    "前一层的 layer_id（定义在管道中的位置）",
  "next_layer_id":    "后一层的 layer_id",
  "is_mandatory":     "true | false（false=可旁路）",
  "bypass_conditions": [
    {
      "condition":   "触发旁路的条件描述",
      "requires_approval": true
    }
  ],
  "input_schema":     "接受的 payload 结构（JSON Schema）",
  "output_schema":    "产出的 payload 结构（JSON Schema）",
  "failure_action":   "retry | skip | escalate | fail",
  "slo_ms":           "本层处理时限（毫秒）",
  "audit_required":   "true（所有层强制审计）",
  "registered_at":    "注册时间",
  "registered_by":    "注册者 agent_id"
}
```

### 5.2 注册门禁（新增层发布前必须通过）

```
新层注册审批链：

开发者提交注册申请
    ↓
L12（quality.gate）：验证 input/output schema 合法性
                       验证 layer_id 全局唯一
                       验证 prev/next_layer_id 存在
    ↓
VP02（平台治理VP）审批：评估是否真正需要新主层
                        还是应挂子层 Lx.n
    ↓
L22（release.publish）：版本化发布，更新 S4 topology_version
    ↓
路由引擎热加载新 topology（不需要重启服务）
```

## 六、顺向流转协议（Forward Flow）

### 6.1 标准顺向流

```
任务进入 L(n)
  │
  ├─ 1. 读取信封，验证 to_layer_id === 自己的 layer_id
  ├─ 2. 检查 bypass_conditions：
  │       满足任一旁路条件 + 白名单允许 → 跳过本层，信封加一条 skip 记录
  │       不满足旁路 → 执行本层处理逻辑
  ├─ 3. 处理完成后：
  │       追加 payload.content（不覆盖已有内容）
  │       在 audit.layer_trail 追加本层记录（outcome = pass）
  │       将 to_layer_id 更新为下一层的 layer_id
  └─ 4. 投递到下一层
```

### 6.2 旁路白名单（bypass whitelist）

仅以下条件满足时，允许跳过对应层：

| 被跳过层 | layer_id | 允许旁路的条件 | 需要审批 |
|---|---|---|---|
| L2 外部系统管理层 | `external.system_mgmt` | 任务不涉及任何外部系统调用 | 否 |
| L8 知识检索层 | `knowledge.retrieval` | 任务无需上下文检索（如纯计算/格式转换） | 否 |
| L17 反思验证层 | `quality.reflection` | Inspector 结果置信度 ≥ 阈值（默认 0.95） | 否 |
| L21 基准评测层 | `eval.benchmark` | 非评测触发场景（非每次执行都经过） | 否 |
| L23 进化实验层 | `evolution.experiment` | 非进化触发场景（常规执行不经此层） | 否 |

**任何其他层不允许旁路**（`security.permissions`（L6）权限 / `security.policy`（L14）安全 / `quality.gate`（L18）门禁 / `release.publish`（L22）发布 四个核心治理层强制不可旁路）。

## 七、反向反馈协议（Backward Feedback）

### 7.1 已定义的反向通道（写死）

反向流转是允许的，但必须通过显式定义的通道，不允许任意层向任意早期层发送消息：

| 反向发起层 | layer_id | 反向目标层 | layer_id | 触发条件 | 信封 direction |
|---|---|---|---|---|---|
| L18 质量门禁 | `quality.gate` | L15 执行监控 | `execution.monitor` | 门禁打回，要求重新执行 | `backward` |
| L17 反思验证 | `quality.reflection` | L16 Inspector | `quality.inspector` | 修正建议已生成，要求再次校验 | `backward` |
| L16 Inspector | `quality.inspector` | L15 执行监控 | `execution.monitor` | 发现严重问题，要求重执行（不经门禁） | `backward` |
| L13 协作编排 | `orchestration.collab` | L12 任务编排 | `orchestration.task` | 多VP协调失败，要求重新路由 | `backward` |
| L6 权限管理 | `security.permissions` | L1 入口扩展 | `gateway.ingress` | 权限校验失败，拦截并返回入口 | `backward` |

### 7.2 反向流转执行规则

```
反向流转触发
  │
  ├─ 1. 发起层构造反向信封：
  │       direction = "backward"
  │       rejection_from_layer_id = 发起层 layer_id
  │       rejection_reason = 原因描述
  │       to_layer_id = 目标层 layer_id
  │       retry_count++
  │
  ├─ 2. 检查 retry_count ≤ max_retry：
  │       超限 → 不再反向流转，直接触发 escalate（Human-in-the-Loop）
  │
  ├─ 3. 目标层接收反向信封：
  │       读取 rejection_reason 和 correction_suggestion
  │       重新执行，产出新的 payload.content（追加在旧内容之后，加时间戳区分）
  │
  └─ 4. 重新顺向流转（从目标层开始向下）
         经过的每一层都必须重新处理（不允许因"之前经过了"而跳过）
         audit.layer_trail 保留完整历史（包括原来的路径和重走的路径）
```

## 八、新增层的完整操作流程

### 8.1 新增主层（Full Layer）

适用场景：功能完全不属于现有任何主层，且无法通过子层 `Lx.n` 承载。

```
Step 1：需求评审
  发起者提交"新增层提案"到 VP02（平台治理VP）
  必须说明：
    - 为什么无法放入子层？
    - 新层放在哪两层之间（prev_layer_id / next_layer_id）？
    - 是否强制（is_mandatory）？
    - 对哪些现有 layer_id 有影响？

Step 2：影响评估
  VP02 评估：
    - 现有代码/配置中引用的 layer_id 不受影响（因为用的是 layer_id 不是 layer_order）
    - 仅 layer_order 和 layer_display 会刷新
    - 更新 layer_topology_version

Step 3：完成注册（按 §5.1 填写注册字段）

Step 4：通过 L22（release.publish）发布流程上线

Step 5：路由引擎热加载新 topology（无需重启）
  路由引擎读取 S4 中最新 topology_version
  自动调整管道顺序

Step 6：更新文档（00_升级总结总览.md 的层号对照速查表）
```

**禁止插层（Prohibit Insertion）**：
- 不允许在 `security.permissions`（L6）之前插入任何层（权限必须第一道关）
- 不允许在 `security.policy`（L14）和 `execution.monitor`（L15）之间插入层（安全必须紧贴执行）
- 不允许在 `quality.gate`（L18）和 `notification.dispatch`（L19）之间插入层（门禁必须直连输出）

### 8.2 新增子层（Sub-Layer `Lx.n`）

适用场景：现有主层的职责需要拆分或扩展，但不构成独立领域。

```
Step 1：在主层内命名：Lx.(max_n + 1)
        子层编号永不复用

Step 2：注册子层（layer_id = parent_layer_id + "." + sub_name）
        例：security.permissions.biometric

Step 3：子层仅在主层内部运作，不参与层间顺序管道
        子层可以有自己的 input/output schema
        但子层的输出必须汇入主层的 output，再流向下一主层

Step 4：子层发布走主层的发布流程（L15），版本与主层绑定
```

## 九、通讯路由引擎设计要求

负责实现本协议的路由引擎必须满足：

### 9.1 功能要求

```
1. 加载 topology：
   - 启动时从 S4 加载 topology_version 和 layers 列表
   - 支持热加载（不重启服务，新版 topology 生效）
   - 版本回滚：可切换回历史 topology_version

2. 路由决策：
   - 根据信封的 to_layer_id 找到对应层处理器
   - 检查该层是否在 skip_layers 白名单中
   - 执行本层处理或跳过

3. 旁路处理：
   - 验证旁路条件是否匹配 bypass_conditions
   - 旁路只能由路由引擎执行，层处理器不可自行决定旁路

4. 反向流转：
   - 检查 direction === "backward" 时，验证反向通道白名单
   - 非白名单的反向流转直接拒绝并告警

5. 防循环：
   - 维护 hop_count，超过 ttl 时强制终止并 escalate
   - 检测相同 from→to 的循环（audit.layer_trail 分析）

6. 审计：
   - 每次层间消息流转自动写 S3（异步，不阻塞主流程）

7. 消息队列与背压（必须内置，详见 §9.3）：
   - 顺向主管道：每个 layer_id 对应独立队列，解耦上下层背压
   - 反向/escalate 通道：独立队列 + 速率限制，防打回风暴淹没正常流
   - 子任务并发池：单 trace_id 并发子管道数上限（可配）
   - 队列超容量时按 priority 截流入等待，不直接丢弃（保证 trace 完整）
   - 等待时间超 slo_ms 触发 escalate，不无限排队绕过 ttl
```

### 9.3 消息队列与背压策略（写死结构，参数可配）

> **必要性**：当前规范已在 A4.1/A4.2/A4.4 识别三类触发场景（消息放大、同步阻塞、反向风暴）；`control.priority` 信封字段与 `backward_rate_limit` 配置项均依赖路由引擎内置队列才能生效。无队列时这些防护措施形同虚设。

#### 9.3.1 三条队列线

```
线 ① 顺向主管道队列（Forward Queue）
───────────────────────────────────────
  - 粒度：per-layer_id（每层各一条队列）
  - 结构：优先级队列（priority = critical > high > normal）
  - 容量：platform 默认上限，company/department 可配置收紧
  - 超容量策略：
      critical   → 入队（不可丢）
      high/normal → 依序入等待队列；等待队列满则拒绝（返回 429 + 写审计）
  - 背压传播：消费速率 < 生产速率时，入口层（L1）限流，
              不允许背压穿透到上游调用方超时

线 ② 反向/escalate 通道队列（Backward Queue）
───────────────────────────────────────
  - 独立于顺向主管道，防止打回风暴影响正常流
  - 速率限制：全局 + 租户双层（backward_rate_limit / escalate_rate_limit，见 14.4）
  - 去重合并：同一 trace_id 对同一目标层的并发反向打回，
              合并为一次重执行（防 A2.6 振荡）
  - 超限策略：入等待队列（不丢弃，保证 trace 审计完整）；
              等待超 slo_ms 则上升为 escalate，通知对应 VP

线 ③ 子任务并发池（Sub-task Semaphore）
───────────────────────────────────────
  - 粒度：per-trace_id
  - 上限：单 trace_id 同时活跃子管道数（可配，platform 默认值偏紧）
  - 超限：新子任务入挂起队列；有空位时按 FIFO + priority 调度进主管道
  - 全部子任务完成或超时后，semaphore 自动释放，写审计
```

#### 9.3.2 队列与信封字段的绑定

| 信封字段 | 与队列的关系 |
|----------|-------------|
| `control.priority` | 队列调度依据；已有 normal/high/critical 三级 |
| `routing.ttl` / `hop_count` | 队列等待时间计入生命周期；排队期间 `hop_count` 不递增，但等待时长超 `slo_ms` 触发 escalate |
| `control.handoff_id` | 同一消息被队列多次投递时，处理器凭此去重（幂等） |
| `backward_rate_limit` (14.4) | 反向队列全局 + 租户速率上限 |
| `layer_msg_ttl_hops` (14.4) | 队列满后被拒绝的消息须携带此字段做回溯定位 |

#### 9.3.3 当前阶段推荐实现（轻量优先）

```
当前阶段（单体或少服务）：
  内存优先级队列（如 Node.js AsyncQueue / 优先堆）
  + 溢出写本地文件/SQLite（崩溃恢复）
  + 启动时扫描未完成条目（与 S3 审计「先写缓冲、再重试」同哲学）

何时升级到外部 MQ（Kafka / Redis Stream）：
  ① 消息须跨进程/跨机器持久化（进程崩溃后 trace 不可丢）
  ② 消费速率持续低于生产速率，内存队列不够 buffer
  ③ 需要 trace 回放/Replay 能力（L14.1 评测集重放）
  ④ 多租户流量完全隔离（per-tenant 独立 topic，SaaS 级别）
  满足任一条时提案评审（VP02 + L22（release.publish）发布），不提前引入重依赖
```

#### 9.3.4 DoD（最小验收）

- [ ] 构造 L9 处理器慢速用例，上游 L6–L8 不因背压崩溃。  
- [ ] 构造单 `trace_id` 子任务超限用例，超限任务进挂起队列而非被丢弃。  
- [ ] 构造反向风暴用例（批量 L12→L9），速率限制生效，正常顺向流不受影响。  
- [ ] 同一 `trace_id` 并发反向打回，合并为一次重执行（无双写）。  
- [ ] 队列满拒绝请求时，拒绝事件写 S3 审计，不静默丢弃。

### 9.2 实现位置

```
src/comm-router.js         ← 现有，负责组织横向通讯（OL0-OL5）
src/layer-router.js        ← 新增，负责工作流纵向通讯（L1-L16）
src/layer-hooks.js         ← 现有，负责各层钩子注入
```

- `comm-router.js` 处理"谁指挥谁"（org level）
- `layer-router.js` 处理"任务走哪条管道"（workflow layer）

### 9.5 加密传输策略（分层决策，写死边界，参数可配）

> **上位文档**：`AGENTS.md` 已写死「外部交互至少 TLS，高敏场景 mTLS」；本节专注**内部管道加密**（进程内 / 跨进程 / 跨主机 / 静态存储 / 字段级），与外部边界加密共同构成**完整加密覆盖**。  
> **核心原则**：加密按**暴露面层次决策**，不追求每条信封都加密（性能代价高、收益递减）；**加密不替代身份校验**（§14.2/§A3.1），两者独立叠加。

#### 9.5.1 四层加密决策矩阵（写死分类，强度可配）

| 传输场景 | 要求级别 | 说明 |
|----------|----------|------|
| **同进程内层间传递**（layer-router 内部） | **不加密（写死）** | 同一进程内存，无网络暴露面；依赖内存隔离与对象只读视图（§9.4.4）；加密无收益且有 CPU 损耗。 |
| **同主机跨进程**（如 IPC / Unix socket） | **可选：OS 通道加密或凭据校验** | 默认依赖 OS 进程隔离；多租户或高敏场景可启用 Unix socket 权限位 + 服务账号凭据；不强制 TLS（通道本地，证书管理成本高于收益）。 |
| **跨主机 / 跨服务**（HTTP/gRPC/消息队列） | **强制 TLS；高敏强制 mTLS（写死）** | 与 `AGENTS.md` 外部边界规则对齐；内部服务间不因「内网可信」豁免；`mTLS_required` 按租户或服务分级可配（见 9.5.3）。 |
| **外部边界**（外部 Agent / MCP / 第三方平台） | **强制 mTLS（写死）** | `AGENTS.md` 已写死，本节不重复；路由引擎视外部调用为不可信，结果重新入治理链。 |

#### 9.5.2 静态加密（数据落盘）

| 存储对象 | 要求 |
|----------|------|
| **S3 审计库**（`audit.layer_trail` 等） | 加密静态存储（平台默认 AES-256 或等价算法）；密钥走 KMS，不硬编码。 |
| **队列溢出持久化**（§9.3.3 本地文件/SQLite） | 落盘前加密（至少对含 PII 或敏感 payload 引用的条目）；进程重启恢复时先解密。 |
| **幂等去重窗口溢出**（LRU 淘汰落盘） | 若含 `handoff_id` + `tenant_id` 组合，落盘须加密；纯 hash 值可不加密。 |
| **S4 拓扑/注册表** | 读写走授权通道（见 §A5.1 缓解）；本地缓存副本不必加密（已为只读热缓存），但须完整性校验（hash 验证版本一致性）。 |

#### 9.5.3 可配置项（按 `platform → company → department → user` 链注入）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `internal_tls_mode` | `required`（跨主机）/ `off`（进程内） | 跨主机强制，进程内禁止（写死分类，只可收紧） |
| `mtls_required_services` | `[]`（平台默认仅外部边界）| 企业可扩大到内部高敏服务列表 |
| `payload_field_encryption` | `off` | 启用后对 `payload.content` 中标注为 PII 的字段做字段级加密（见 9.5.4） |
| `queue_persistence_encryption` | `on` | 队列落盘是否加密；可配但不可关闭平台默认 on |
| `cert_rotation_days` | `90` | TLS/mTLS 证书轮换周期（天），轮换经 L22（release.publish）发布，写审计 |
| `kms_provider` | `local`（内置）/ 可扩展 | 密钥管理提供商；轮换策略与 `envelope_signing_mode` 共用 KMS（§A5.3 缓解） |

#### 9.5.4 字段级加密（PII / 高敏内容，可选）

```
触发条件：
  payload_field_encryption = on
  AND 字段被标注为 pii_level >= 1（由 `orchestration.task`（L12）编排层在首次写入 goal/content 时标注）

加密粒度：
  仅对 payload.content 中明确标注的字段值加密
  routing.* / audit.layer_trail / control.* 不加密（元信息须可路由与审计）

密钥绑定：
  加密密钥绑定 tenant_id（租户专用密钥，不共享）
  密钥由 KMS 管理，路由引擎不持久化明文密钥

对接影响：
  各层处理器若需读取加密字段，须持有对应 tenant 解密权限（与 §A3.2 Confused deputy 缓解联动）
  L4.1 审计写入前须脱敏（而非写入密文引起合规歧义）

性能影响（写入 §9.4.3 CPU 热点）：
  字段级加密仅对标注字段，不全量加密；批量处理时可并行
  建议在 L9（执行层）边界加密，而非每个 hop 都加解密
```

#### 9.5.5 证书与密钥生命周期

```
TLS 证书：
  ① 颁发：平台内置 CA 或企业 PKI
  ② 轮换：cert_rotation_days（可配，默认 90 天），轮换经 L22（release.publish）发布流程
  ③ 撤销：CRL / OCSP；撤销事件写 S3 审计并触发 P1 告警
  ④ 泄漏应急：立即撤销 + 重颁发，路由引擎拒绝旧证书（热更新，不重启）

信封签名密钥（HMAC/asymmetric）：
  ① 存储：KMS（不落明文到代码/配置）
  ② 轮换：与 cert_rotation_days 独立周期，可配
  ③ 泄漏：切换为拒绝未签名信封模式 + 强制全量重签（见 §A5.3）
  ④ 多版本共存：轮换期内新旧密钥双验（grace period 可配）

字段级加密密钥（租户专用）：
  ① 绑定 tenant_id，KMS 隔离存储
  ② 租户注销后密钥必须销毁，对应数据进入不可解密状态（满足被遗忘权 §A6.2 联动）
  ③ 密钥访问日志写 S3，可审计
```

#### 9.5.6 与性能的平衡（不过度加密原则）

```
加密只在「确实有网络或存储暴露面」的边界发生：
  ✅ 跨主机 HTTP/gRPC → TLS（强制）
  ✅ 外部边界 → mTLS（强制）
  ✅ 落盘队列/审计 → AES 静态加密（强制）
  ⚙️  字段级 PII → 按需（可配）
  ❌ 进程内 hop 传递 → 不加密（写死禁止）

CPU 影响量化（结合 §9.4.3）：
  TLS 握手：连接复用（keep-alive）后边际成本极低
  HMAC：< 1 μs/条，可接受
  AES-256 字段加密：按字段粒度，非全量；建议批量处理
  mTLS 附加握手：仅在连接建立时，不影响单条消息传输延迟
```

#### 9.5.7 DoD（加密传输最小验收）

- [ ] 跨主机路由调用全部走 TLS（自动化扫描无明文 HTTP 路由）。  
- [ ] 外部边界调用走 mTLS（证书双向验证通过）。  
- [ ] 队列溢出落盘条目加密，重启恢复正常（解密→入队→处理流程验证）。  
- [ ] 证书轮换不中断进飞请求（滚动轮换验证）。  
- [ ] 字段级加密启用时，`audit.layer_trail` 中不出现对应明文值（脱敏验证）。  
- [ ] 进程内层间传递无序列化/加密操作（profile 热路径验证）。

### 9.6 前台安全快路径与最优资源占用

> **目标**：让 L1（`gateway.ingress`）到 L6（`security.permissions`）的**安全前台**在拦截非法请求时消耗最少资源；让合法请求以最低延迟进入主管道。  
> **核心思路**：**越早拒绝代价越小**；把校验分级、缓存可重用决策、连接级操作摊销到连接建立时；不把昂贵操作放在热路径上。  
> **与其他节关系**：§9.3 队列（背压）、§9.4 缓存（热数据）、§9.5 加密（TLS 摊销）均是本节优化的基础设施；本节聚焦**前台安全校验的资源最优策略**。

#### 9.6.1 四级渐进校验（Fail Fast 原则）

```
每条入站信封在进入主管道前，经过四级递进校验：
越早的级越轻量；通过上一级才进入下一级

级别 1 — 语法与格式（O(1)，<0.1 ms）
  ① envelope_version 在已知版本表中
  ② 必填字段存在（routing.trace_id / tenant_id / from_layer_id / to_layer_id）
  ③ payload 大小 ≤ layer_msg_max_bytes
  ④ hop_count < ttl
  → 失败：立即拒绝（400），不进队列，不写 trail（写拒绝计数器即可）

级别 2 — 限流与配额（O(1)，缓存命中时 <0.1 ms）
  ① 全局 QPS 令牌桶（内存）
  ② 租户级 QPS 令牌桶（per-tenant 分片内存）
  ③ 反向/escalate 专项速率（§9.3 线②）
  → 失败：返回 429 + Retry-After，写速率限制计数器，不进主队列

级别 3 — 身份与租户（缓存命中 O(1)，缓存未命中走 `security.permissions`（L6））
  ① 从认证通道（TLS/mTLS 证书 或 服务账号 token）提取调用方身份
  ② 校验信封 tenant_id 与认证身份匹配（§A3.1 缓解）
  ③ `security.permissions`（L6）权限决策结果缓存（TTL 可配）：命中则直通，未命中才走 L6 完整 RBAC
  → 失败：返回 401/403，写审计（异步），不进主管道

级别 4 — 业务规则预检（仅当级别 1-3 全通过后执行）
  ① 任务类型是否在当前租户允许范围内（`capability.market`（L5）能力标签白名单）
  ② risk_level 是否触发人工介入前置门禁
  ③ 子任务并发数是否超 semaphore 上限（§9.3 线③）
  → 失败：进等待队列（不直接拒绝），同时写 trail 草稿
  → 全部通过：信封进入主管道（顺向队列 §9.3 线①）
```

#### 9.6.2 权限决策缓存（前台最大杠杆）

```
缓存结构：
  key   = hash(tenant_id + principal + action_type + resource_scope)
  value = { decision: allow|deny, expires_at, policy_version }

命中条件：
  policy_version 与 L4 当前策略版本一致（策略更新时主动失效）
  expires_at 未到期

缓存参数（可配，14.4）：
  auth_cache_ttl_seconds     默认 60s，企业可配置 10~300s
  auth_cache_max_entries     per-tenant LRU 上限
  policy_version_check_mode  strict（每次校验版本）| lazy（TTL 到期再校验）

安全约束（写死）：
  ① 缓存只存 allow/deny 决策，不存权限详情（减少缓存体积与泄漏面）
  ② 权限撤销（如账号禁用）必须主动失效对应 key，不等 TTL 自然过期
  ③ 缓存条目与 tenant_id 严格绑定，不允许跨租户命中
```

#### 9.6.3 连接级摊销（TLS/mTLS 成本归零化）

```
原则：把每连接一次性成本（握手、证书验证）摊销到连接生命周期，
      不让单次请求承担连接建立成本。

实施：
  ① 连接复用（HTTP/2 multiplexing 或 gRPC stream）：
     TLS 握手只在首次建立时发生；后续请求复用已加密通道
  ② 连接池（per-service 或 per-tenant）：
     预建 N 条连接（N 可配），避免突发流量触发批量握手
  ③ mTLS 证书缓存：
     已验证的对端证书在会话内缓存，不重复做链路校验
  ④ 与队列协同：
     连接池满时请求进 §9.3 线①等待，不让 TLS 握手成为等待理由
```

#### 9.6.4 前台资源预算（每条请求的硬上限，写死结构，参数可配）

```
每条进入前台的信封分配如下资源预算：

┌─────────────────────────────────────────────────────┐
│  前台处理时间预算    ≤ preflight_slo_ms（可配，默认5ms） │
│  前台 CPU 配额      级别1+2+3(缓存命中) = <1ms        │
│                    级别3(L4完整校验)   = preflight_slo │
│  前台内存配额       信封大小上限（layer_msg_max_bytes）  │
│                    + 身份解析临时对象（限生命周期）      │
│  拒绝不计入管道资源预算                                 │
└─────────────────────────────────────────────────────┘

超出时间预算：
  级别1-3 超时 → 直接拒绝（503）+ 告警（不进队列）
  级别4 超时   → 进等待队列（不拒绝，保证合法请求不丢）

可配项（14.4 配置链）：
  preflight_slo_ms         前台最大处理时限
  auth_cache_ttl_seconds   权限决策缓存 TTL
  conn_pool_size           连接池大小（per-service）
  rate_limit_burst         令牌桶突发量
```

#### 9.6.5 拒绝响应轻量化（前台不做昂贵操作）

```
拒绝时只做：
  ① 返回标准错误码（400/401/403/429/503）
  ② 递增对应计数器（内存，异步 flush 到指标系统）
  ③ 异步写拒绝事件到 S3（不阻塞响应）

拒绝时禁止：
  ❌ 构造完整信封（payload 已无用）
  ❌ 同步写 audit.layer_trail（昂贵，且请求未入管道）
  ❌ 触发 escalate 到组织层（批量拒绝会引发 §A2.5 风暴）
  ❌ 调用 L5+ 任何下游层（请求已死，不消耗主管道资源）
```

#### 9.6.6 前台与后台安全校验的分工（写死）

| 校验类型 | 归属层级 | 前台执行时机 | 理由 |
|----------|----------|-------------|------|
| 格式/大小/必填 | 前台（§9.6.1 级别1） | 入队前 | 最轻量，最早拦截 |
| 限流/配额 | 前台（§9.6.1 级别2） | 入队前 | 内存操作，O(1) |
| 身份与租户 | 前台（§9.6.1 级别3）+ L4 | 缓存命中：入队前；未命中：进 L4 | 缓存摊销高频场景 |
| RBAC 细粒度权限 | L4（`security.permissions`） | 主管道内 | 重操作，按需执行 |
| 内容合规/安全策略 | L8（`security.policy`） | 主管道内 | 依赖 L5/L6 上下文 |
| 结果质量 | L10/L12 | 主管道内 | 需完整执行结果 |

> **原则**：前台只做「无需业务上下文即可判定」的轻量校验；需要上下文、知识或执行结果的校验不前移。

#### 9.6.7 DoD（前台安全最优资源占用最小验收）

- [ ] 级别 1+2 拒绝路径响应时间 p99 ≤ 1 ms（压测验证）。  
- [ ] 权限决策缓存命中率 ≥ 80%（正常流量下，指标可查）。  
- [ ] 批量非法请求（1000 QPS 攻击）不引发 escalate 风暴，组织侧通讯不受影响。  
- [ ] 连接池预建，突发流量下 TLS 握手不出现在请求热路径耗时 top10。  
- [ ] 前台拒绝不触发 L5+ 任何层（profile 或 trace 验证）。  
- [ ] 权限撤销后，缓存主动失效，不等 TTL（自动化验证：撤销→下一请求立即被拒）。

### 9.7 通讯层无模型原则（写死红线）

> **定位**：通讯层（`layer-router`、Envelope 路由、§9.3–§9.6 所有子节）是平台的**管道基础设施**，模型是**管道里运行的内容**——两者严格分离，不允许在通讯层内部调用任何 LLM 或外部 AI 模型（含 BYOK、本地模型、MCP 模型工具）。

#### 9.7.1 禁止原因（写死）

| 约束维度 | 通讯层硬要求 | 模型推理现实 | 结论 |
|----------|------------|------------|------|
| **延迟** | §9.6 前台 p99 ≤ 1 ms，`preflight_slo_ms` ≤ 5 ms | LLM 推理 100 ms～10 s+，差 2–4 个数量级 | **不可用** |
| **确定性** | 路由、旁路、安全决策必须可审计、可重放，结果确定 | 模型输出概率性，同输入可能不同输出 | **不可用** |
| **可用性** | 通讯层是所有层的基础设施，不可有单点外部依赖 | 外部模型不可用 → 整个管道瘫痪 | **不可用** |
| **成本** | 每个 hop（16层 × N 任务 × M 租户）都触发推理 | 调用量爆炸，成本不可控 | **不可用** |
| **安全性** | 身份校验、租户隔离、旁路判定必须规则确定 | 模型可被对抗性输入操纵，不能做安全门 | **不可用** |

#### 9.7.2 模型的正确归属层（非通讯层）

```
通讯层（layer-router / Envelope / §9.3-§9.6）：
  纯规则 + 配置：O(1) 查表、LRU 缓存、令牌桶、TLS
  ❌ 不调任何模型

模型调用只在以下业务层发生：

  L6  orchestration.task    → 任务理解、意图识别、子任务拆解
  L9  execution.monitor     → Agent 实际执行（LLM 完成业务任务）
  L10 quality.inspector     → 输出质量评估（规则优先，模型辅助）
  L11 quality.reflection    → 修正建议生成（Inspector 结果触发）
  L3  capability.market     → 能力匹配（规则优先，模型辅助）
  L5  knowledge.retrieval   → 向量检索（检索模型，非生成模型）
```

#### 9.7.3 边界情况处理（轻量 ML / 异常检测）

有时会有「用轻量分类器在入口识别恶意请求」的需求，处理规则如下：

```
轻量分类器（非 LLM，如规则引擎 / 小型分类模型）：
  ✅ 可用，但归属 L8（security.policy），不归通讯层
  ✅ 分类结果以「结构化标志位」写入信封 control 区
  ✅ 路由引擎读标志位做路由决策（仍是规则判断，不是模型判断）
  → 通讯层本身仍保持无模型原则

禁止：
  ❌ 分类器直接决定「允许/拒绝路由」（绕过通讯层规则引擎）
  ❌ 分类器调用外部 LLM（延迟不可接受）
  ❌ 分类器与路由引擎共享线程池或内存段（避免资源竞争）
```

#### 9.7.4 与外部模型的正确边界

```
外部模型（SaaS LLM / BYOK / 本地 Ollama）均属于「外部不可信边界」：

  通讯层视角：
    外部模型的调用发生在 L9 执行层
    结果经 L10/L11/L12 治理链校验后才可继续流转
    通讯层只负责把信封从 L9 路由到 L10，不感知模型存在

  禁止：
    ❌ 通讯层（layer-router）直接持有或调用外部模型 SDK
    ❌ 把模型响应直接写入信封 routing 或 control 区（只能写 payload）
    ❌ 因「模型正在推理」而暂停/延迟路由决策（路由决策与模型执行解耦）
```

#### 9.7.5 DoD（无模型原则最小验收）

- [ ] 静态分析：`layer-router.js` 及 §9.3–§9.6 相关文件中无任何 LLM SDK 引用（`openai`、`anthropic`、`@google/generative-ai` 等）。  
- [ ] 压测：通讯层全程处理（含前台校验 + 路由 + 队列投递）不触发任何外部 HTTP 请求（网络 mock 验证）。  
- [ ] 故障注入：外部模型服务不可用时，通讯层路由与队列功能完全正常（模型故障不传播到通讯层）。

### 9.4 缓存策略与内存 / CPU 优化

> **目标**：在保持正确性与审计完整性的前提下，让路由引擎对**热路径**高效，对**冷数据**不占用稳态内存；CPU 密集操作集中化并可量化。  
> **原则**：缓存只加速读取，**不替代权威来源**（S4 / 注册表）；写操作、安全校验、审计写入**不走缓存**。

#### 9.4.1 六类缓存点

| 缓存对象 | 读写热度 | 建议策略 | 失效条件 | 内存量级 |
|----------|----------|----------|----------|----------|
| **拓扑表**（layer_id → next/prev/handler） | 极热（每次路由必读） | 启动时全量加载到内存；热更新时原子替换（新版完整加载后 swap，旧版引用计数归零后释放） | `topology_version` 变更 | O(层数)，极小 |
| **旁路白名单**（bypass whitelist） | 热（每个顺向 hop 检查） | 随拓扑表同批加载；以 `Set<layer_id>` 存储，O(1) 查找 | 同拓扑版本 | O(可旁路层数)，极小 |
| **处理器注册表**（layer_id → handler fn） | 极热 | Map 内存缓存；注册变更经 L22（release.publish）发布后原子替换（与拓扑表同生命周期） | 注册表版本变更 | O(层数 × handler 引用)，小 |
| **幂等去重窗口**（handoff_id → 已处理标志） | 热（每个处理器入口） | 固定大小 LRU（按租户分片）；窗口大小可配（14.4）；超 TTL 自动淘汰 | handoff_id TTL 到期 | 可配上限，默认偏紧 |
| **速率限制计数器**（backward / escalate QPS） | 热（每次反向/escalate） | 内存滑动窗口（per-tenant）；跨进程时升级到 Redis；不持久化（重启后重置，以 slo 时间窗为单位） | 窗口滑过 | O(租户数 × 窗口槽数)，可控 |
| **信封 Schema 校验编译结果** | 中热（每条信封入队前） | 编译后的 validator 对象缓存（按 `envelope_version` 键）；新版本首次编译后缓存；旧版本 LRU 淘汰 | `envelope_version` 不再使用 | O(版本数)，小 |

#### 9.4.2 内存保障（写死上限，参数可配）

```
路由引擎总内存使用必须有明确上限，分三段：

① 热缓存段（拓扑 + 白名单 + 注册表）：
   - 写死平台上限（如 64 MB），不可通过配置突破
   - 启动时分配完成，运行中只做原子替换，不增长

② 运行时缓存段（幂等去重 + 速率限制计数器）：
   - 平台默认上限（如 128 MB），company 可配置收紧
   - 超上限时 LRU 淘汰最旧条目，并写告警（不崩溃）

③ 队列段（§9.3 三条队列线）：
   - 每条队列独立上限（可配），超限触发拒绝/等待，不无限增长
   - 内存队列溢出后写本地持久化（§9.3.3），不进入①②段
```

#### 9.4.3 CPU 热点识别与控制

| CPU 密集操作 | 发生位置 | 优化方向 |
|-------------|----------|----------|
| **信封序列化 / 反序列化** | 每个 hop 入队与出队 | 仅在**跨进程/持久化边界**做完整序列化；进程内传递只传对象引用（配合 §4.2 只读视图，不做深拷贝） |
| **JSON Schema 校验** | 信封入队前 | 使用预编译 validator（见 9.4.1）；仅校验**当前层关心的字段**，不全量校验完整信封 |
| **`audit.layer_trail` hash chaining**（A1.5 缓解） | 每个 hop 追加 trail | 批量写（若 SLA 允许）；hash 计算异步进行，与主流程解耦（与 S3 异步写同哲学）；单条 hash 计算为 O(1) |
| **HMAC / 签名**（可选，`envelope_signing_mode`） | 跨主机边界 | 仅在跨主机场景启用；进程内路由不开签名；密钥缓存在内存（轮换时原子替换） |
| **优先级队列插入**（§9.3） | 每条消息入队 | 优先级只有 3 级（critical/high/normal），退化为 3 个 FIFO 桶，O(1) 操作，不用堆排序 |
| **幂等去重查找** | 每个处理器入口 | LRU Map，O(1) 查找；分 tenant 分片降低锁竞争 |

#### 9.4.4 信封对象生命周期（防内存泄漏）

```
创建：路由引擎在管道入口（L1 或 VP 域内入口）创建信封实例
  │
  ├─ 进程内传递：只传引用，不复制；各层通过只读视图访问
  │
  ├─ 跨进程/持久化边界：序列化为 JSON/Binary，接收端反序列化为新实例
  │
  ├─ 队列暂存：队列持有引用；出队后引用交给处理器，队列释放
  │
  └─ 管道终态（DONE / FAILED / ttl 超限）：
       信封元数据（routing / audit.layer_trail）写 S3 归档
       payload 大字段只保留引用（指针化）
       信封对象释放（GC 可回收）

禁止：
  ❌ 处理器持有信封引用超出本层处理生命周期
  ❌ 日志模块直接序列化完整信封（可能含 PII，与 A6.1 缓解一致）
  ❌ 信封实例在多条 trace_id 管道间复用（跨租户污染风险，A3.4）
```

#### 9.4.5 DoD（缓存与资源优化最小验收）

- [ ] 路由引擎热缓存段内存在压测 1 小时后无增长（稳态）。  
- [ ] 拓扑热更新时，进飞请求不受影响（原子替换，零停机）。  
- [ ] 幂等去重 LRU 达到上限后，最旧条目被淘汰，告警触发，引擎不崩溃。  
- [ ] 进程内 hop 传递无完整序列化（profile 验证无大对象拷贝热路径）。  
- [ ] 信封对象在管道终态后可被 GC 回收（无循环引用或跨 trace 持有）。

### 9.8 基础设施模块定义（M1/M2，写死）

> **架构定位**：Flow Harness 工作流架构由「**16层工作流管道**」与「**2个基础设施模块**」共同构成。  
> 基础设施模块不参与管道层间消息流转，不占 L1–L16 槽位；但对管道稳定运行是**强依赖**，不可缺失、不可替换、不可降级。  
> 两个模块**完全独立**，职责不交叉，不共享可写状态（写死）。

```
架构全景（写死）：

┌─────────────────────────────────────────────────────┐
│  M1  src/pipeline-control.js（控制面 / 管道大脑）                    │
│  M1.1 全局任务账本 + M1.2 全局水位表 + M1.3 拓扑仲裁器               │
│  俯瞰全局，不参与流转，不被管道流量影响                               │
└──────────────────┬────────────────────────────────────────────────────┘
                   │ 感知状态 / 限速信号（单向，写死）
┌──────────────────▼────────────────────────────────────────────────────┐
│  M2  src/layer-router.js（数据面 / 管道神经）                         │
│  层间消息投递唯一通道（§11.7 写死）                                   │
│  M2.1顺向队列 + M2.2反向通道 + M2.3旁路 + M2.4防循环 + M2.5限流熔断  │
└──────────────────┬────────────────────────────────────────────────────┘
                   │ 消息投递（每跳必经）
┌──────────────────▼────────────────────────────────────────────────────┐
│  L1 → L2 → … → L23（2M + 23L 工作流管道层）                         │
│  任务在此流转，消息不流经 M1/M2 本体                                  │
└────────────────────────────────────────────────────────────────────────┘
```

#### 9.8.1 M1：控制面模块（管道大脑）

| 字段 | 值 |
|---|---|
| 模块标识 | `M1` |
| 代码入口 | `src/pipeline-control.js` |
| 性质 | 基础设施模块（不是管道层，消息不流经此模块） |
| 可替换 | ❌ 否（写死，平台强依赖） |
| 可旁路 | ❌ 否（M2 投递前必须查询 M1 水位表与任务账本） |

**三项核心职责（写死）**：

```
① 全局任务账本（Global Task Registry）
   - 所有活跃 trace_id 的注册（进入 L1 时）与注销（管道终态时）
   - 记录每条 trace 当前所在层（layer_id）与状态
   - 统计单 trace_id 并发子管道数（配合 §9.3 线③上限）
   - 对外只读暴露给 CEO（OL0）和 VP（OL1）及看板（Dashboard）
   - 看板是账本的只读可视化消费者，不参与管道流转，不占 L 槽位

② 全局水位表（Global Watermark）
   - 每层（layer_id）队列水位实时快照（由 M2 上报）
   - 水位超阈值时向 M2 发出限速信号，M2 在 L6 出口执行降速
   - 不直接干预层处理器逻辑（只通过 M2 间接生效）

③ 拓扑仲裁器（Topology Arbitrator）
   - 热更新 topology 时，判断进飞 trace_id 的钉选策略
   - 确保同一 trace_id 全程走同一 topology_version
   - 新 topology 只对新 trace_id 生效，不中断进飞请求
   - 与 §14.2 topology_pinning 配置项联动
```

**禁止行为（写死）**：

```
❌ M1 不参与任何层的消息投递
❌ M1 不修改信封内容（只读信封元数据）
❌ M1 不直接调用任何层处理器函数
❌ M1 不被任何管道层直接调用
❌ M1 不持有 LLM 或外部模型依赖（与 §9.7 一致）
```

#### 9.8.2 M2：数据面模块（管道神经）

| 字段 | 值 |
|---|---|
| 模块标识 | `M2` |
| 代码入口 | `src/layer-router.js` |
| 性质 | 基础设施模块（所有层间消息必须经此投递） |
| 可替换 | ❌ 否（§11.7 写死，处理器不得直接互调） |
| 可旁路 | ❌ 否（任何层间流转绕过 M2 视同绕过治理链） |

**核心职责**：层间消息全部流转（顺向 + 反向 + 旁路 + 防循环 + 限流熔断），详见 §9.1–§9.7。

**与 M1 的协作（写死）**：

```
M2 每次投递消息时：
  → 通知 M1 更新 trace_id 当前所在层（异步，不阻塞投递）
  → 读取 M1 水位表，判断目标层是否限速
  → 读取 M1 任务账本，判断 trace_id 是否超并发上限

M1 发出限速信号时：
  → 只能通过 M2 的限速接口生效
  → 不直接干预层处理器逻辑

M1 ↔ M2 之间通信：
  → 独立内部通道，不经过 L1–L16 管道
  → 不产生层间消息信封（不走 §4.1 信封结构）
```

#### 9.8.3 M1/M2 的可扩展性（写死边界，能力可增）

```
写死（不可变）：
  - M1/M2 模块标识与代码入口（pipeline-control.js / layer-router.js）
  - M1 三项核心职责的语义（账本 / 水位 / 仲裁）
  - M2 作为唯一层间投递通道的地位
  - M1/M2 完全独立、不共享可写状态

可扩展（可增加能力，不可削减现有）：
  - M1 账本可增加新字段（如 cost_budget、SLA_deadline）
  - M1 水位表可增加新监控维度（如 CPU 水位、内存水位）
  - M2 可增加新的路由策略插件（须经 L22（release.publish）发布）
  - M1/M2 之间的协作接口可扩展（不可删除现有接口）
```

#### 9.8.4 DoD（M1/M2 基础设施模块最小验收）

- [ ] M1 全局任务账本：任意时刻可查询所有活跃 trace_id 及其当前层与状态。  
- [ ] M1 水位表：L9 队列水位 > 80% 时，L6 出口自动降速（M2 执行），不等 L9 崩溃。  
- [ ] M1 拓扑仲裁：热更新 topology 时，进飞请求不分裂（全程旧图或全程新图，单测验证）。  
- [ ] M2 唯一通道：代码扫描无层处理器直接调用其他层处理器函数（§11.7 验收）。  
- [ ] M1/M2 状态独立：两个模块无共享可写变量（单测验证）。  
- [ ] M1/M2 故障隔离：M1 重启期间 M2 仍可正常投递（降级为无限速模式），管道不中断。

## 十、横向与纵向通讯的交叉点

唯一交叉点：**组织层的 Agent 如何触发工作流层处理**。

```
交叉规则（写死）：

CEO（OL0）           → 触发 layer-router 从 L1 开始的完整管道
VP（OL1）            → 触发 layer-router 从 L12（orchestration.task 任务编排）开始的域内管道
总监（OL2）          → 不直接触发 layer-router；通过 VP 的域内管道接收任务
经理/子Agent（OL3/4）→ 只操作 L15（execution.monitor 执行监控层）内的执行单元，不触发新管道
副Deep Agent（OL5）  → 只操作 L9 内的原子执行，不触发任何管道

管道内部禁止绕过 L6（写死）：
  任何工作流层不得跳过 orchestration.task（L6）直接向 execution.monitor（L9）注入新任务；
  `orchestration.task`（L12）是管道内调度决策的唯一关卡，拆解 / 分配 / 路由决策必须经此层产出；
  组织层 Agent 如需触发执行，必须走上方"交叉规则"入口（CEO→`gateway.ingress`（L1）或 VP→`orchestration.task`（L12）），
  不得通过内部接口直连 `execution.monitor`（L15）处理器。

反方向（工作流层 → 组织层）：
  任何层触发 escalate → 通知到 CEO（OL0）处理
  `quality.gate`（L18）门禁打回 → 通知到对应 VP（OL1）
  `execution.monitor`（L15）执行异常 → 通知到对应经理（OL3）
```

## 十一、治理约束（写死）

1. **layer_id 不可删除**：层下线后状态变为 `retired`，layer_id 永久保留，不可复用。
2. **四个核心治理层不可旁路**：`security.permissions`（L4）/ `security.policy`（L8）/ `quality.gate`（L12）/ `release.publish`（L15）。
3. **反向通道不可扩展（除非审批）**：除 §7.1 白名单之外的反向流转路径，新增必须经 VP02 审批 + L22（release.publish）发布。
4. **信封 audit.layer_trail 不可篡改**：只能追加，不能修改历史记录，永久保留在 S3。
5. **topology 变更必须版本化**：每次层增删改，`topology_version` 自增，变更记录写 S3。
6. **代码/配置禁止直接引用 layer_order 或 layer_display**：必须使用 `layer_id`。
7. **处理器禁止直接跨层调用**：各层处理器不得直接调用其他层的处理器函数；所有层间消息流转（含顺向与反向）必须经 `layer-router.js` 投递。§14.3 列明的路由能力不得由处理器分散实现，本条在代码层与规范层同等强制；违反本条视同绕过治理链，参照核心治理层旁路处理。

## 十四、通讯安全加固与企业级配置基线

**目标**：在不大改 16 层主拓扑的前提下，让纵向层间通讯**默认更安全、行为更可预期**；**红线写死**、**强度与配额可配置**、**配置可审计**，并与 `security.permissions` / `security.policy` / `quality.gate` / `release.publish` 治理链对齐。

### 14.1 多层隔离（边界写死）

| 隔离维 | 要求 |
|--------|------|
| **组织 vs 工作流** | 横向 `comm-router` 与纵向 `layer-router` **不得共享可写状态**；组织消息不得直接改写信封 `payload` 中已由纵向管道冻结的字段（如 `goal` 只读区），变更须走规定的 backward/escalate 路径。 |
| **租户 / 企业** | 每封层间消息 **必须**携带 `tenant_id`（及建议携带 `company_id`、`department_id`）；路由引擎在投递前 **必须**校验处理器注册范围与信封一致，**禁止**跨租户路由（deny-by-default）。 |
| **环境** | 建议信封携带 `environment`（如 `prod` / `stage` / `sandbox`）；**禁止**用生产凭据驱动 `sandbox` 管道（凭据与拓扑配置分域）。 |
| **外部边界** | 外部结果进入信封前，须先经适配器脱敏与类型校验，再进入 `security.permissions` / `security.policy` 等治理链（与 `AGENTS.md` / `04_长期架构.md` 一致）。 |

### 14.2 信封强制字段与尺寸（写死最小集，其余可扩展）

在第四章信封之上，**企业级部署**须至少落实：

- **`routing.tenant_id`**（必填）、**`routing.principal`** 或等价主体标识（谁触发本段管道）。  
- **`envelope_version` 与 `topology_version` 钉选**：单次管道执行开始时应钉住 `topology_version`，热更新拓扑时**不得**让同一 `trace_id` 半段旧图、半段新图（要么请求级冻结，要么显式中断并重试）。  
- **体积上限**：`payload` 全量序列化大小上限（平台默认 + 企业可收紧）；超长内容**只传引用**（`intermediate_result_refs` 等），与「指针化审计」一致。  
- **幂等键**：建议每层处理出口带 `handoff_id`（`trace_id` + 单调 `hop_seq`），审计与重试去重依赖该键。

### 14.3 路由引擎集中强制（写死）

以下能力**只能**落在纵向路由引擎实现，**不得**由各层处理器自行实现等价逻辑：

1. **校验 `from_layer_id` / `to_layer_id`** 与当前拓扑边是否允许（含顺向默认边）。  
2. **解析并执行 `skip_layers`**：仅当与白名单及企业配置版本一致时生效；处理器**不得**自声明跳过 `security.permissions` / `security.policy` / `quality.gate` / `release.publish`。  
3. **反向路径**：仅允许文档 **第七章** 已列白名单边；新增边仍须 **第十一章** 审批与发布。  
4. **`hop_count` / `ttl` 与循环检测**：超限必须终止并 **escalate**，禁止静默丢弃。  
5. **可选完整性校验**：企业可配置对信封做 **HMAC 或签名**（密钥走密钥管理，轮换经 L15）；未启用时须在风险评估中备案。

### 14.4 企业级可配置项（参数可配，须版本化审计）

下列项通过 **`platform → company → department → user`** 注入，**仅能收紧不能突破平台红线**；变更写入审计（与 `00_` 全局配置原则一致）：

| 配置项 | 说明 |
|--------|------|
| `layer_msg_max_bytes` | 单层间传递信封最大字节数 |
| `layer_msg_ttl_hops` | 默认 `ttl`，可按租户调低 |
| `backward_rate_limit` / `escalate_rate_limit` | 反向与升级事件每租户 QPS / 日上限，防风暴 |
| `envelope_signing_mode` | `off` / `hmac` / `signature` 等分级 |
| `topology_pinning` | 请求级钉选拓扑版本策略（强制推荐 `on` 于生产） |
| `redaction_profile` | 日志与审计导出时的脱敏配置标识 |

### 14.5 可观测与应急（建议默认开启）

- **指标**：每层处理耗时、反向次数、旁路次数、`ttl` 触发次数、跨租户拒绝次数。  
- **告警**：连续循环嫌疑、`escalate` 突增、信封签名校验失败率超阈。  
- **应急**：支持按租户或全局 **冻结纵向管道**（仅允许 backward 至安全层或人工接管），配置经 L22（release.publish）发布。

### 14.6 与「最稳」落地的关系

**最稳组合**：**`layer_id` + 钉选 `topology_version` + 路由引擎单点强制 + 租户必填 + 信封体积与 ttl 双限 + 审计 handoff 幂等**；企业差异主要靠 **第十四章 14.4 配置表** 调节，避免为每个客户 fork 一套管道代码。

## 十五、通讯优化落地路线（分阶段，推荐顺序）

> **定位**：在不大改业务层语义的前提下，把「规范（第四～十一章）+ 安全基线（第十四章）」**按投入产出比**落到代码与运维；阶段间可发布、可回滚。  
> **原则**：先对齐语义与追踪，再单点纵向路由，再配置与观测，最后纵深加固；**禁止**在未完成阶段一、二前先行复杂签名或每层长文档分叉。

### 阶段一：语义与追踪对齐（短周期、低风险）

| # | 动作 | DoD（最小验收） |
|---|------|------------------|
| 1.1 | **区分两套信封**：Layer Message Envelope（本文第四章）与跨进程/IPC 任务包（如 `task-serializer`）在类型名、文档与代码注释中**不得混用同一未限定称谓**。 | Code review 可一眼区分；新代码检索 `envelope` 有明确前缀或类型。 |
| 1.2 | **全链路 `trace_id`**：凡跨模块/跨进程传递纵向任务上下文，须携带与 `12_`/可观测规范一致的 `trace_id`（及建议 `session_id`）。 | 任意一条抽样 trace 可从入口追到 L9 出口无断链。 |
| 1.3 | **实现—规范对照表**：维护「谁创建信封、谁只读、谁禁止改 `goal`/旁路」一页表（可放在本仓库 `docs/` 或 `14_实施基线映射表` 延伸行）。 | 路由引擎责任人、首接入口文件路径可查。 |
| 1.4 | **无模型原则静态核查**（§9.7）：确认 `layer-router` 及通讯相关文件无任何 LLM SDK 引用；明确模型调用归属层（L6/L9/L10/L11）写入对照表。 | §9.7.5 DoD 三项通过；对照表可查。 |

### 阶段二：薄纵向路由引擎（核心杠杆）

| # | 动作 | DoD（最小验收） |
|---|------|------------------|
| 2.1 | 实现 **`layer-router`（薄版）**：仅 **顺向** 下一跳；从 S4 或等价配置加载 `topology_version` + `layer_id` 边表；**钉选**请求级拓扑（第十四章 14.2）。 | 新插主层只需改拓扑配置 + 注册 handler，业务调用点不硬编码 `L7` 等展示号。 |

| 2.3 | **`hop_count` / `ttl` 与循环终止**：超限 **escalate**，禁止静默丢信。 | 构造死循环用例必触发终止与告警。 |
| 2.4 | **处理器注册表 deny-by-default**：未注册的 `layer_id` 不得执行；注册变更版本化并审计。 | 无隐式默认 handler。 |
| 2.5 | **旁路/反向第二阶段再接**：首版可先不支持或仅只读告警；**四治理层**（第十一章）在任何情况下不得被处理器自跳过（与 14.3 一致）。 | 代码检索无业务层自写 `skip_layers` 生效逻辑。 |
| 2.6 | **内置三条队列线（§9.3）**：顺向主管道队列、反向/escalate 通道队列、子任务并发池须与 `layer-router` 同期交付；队列 DoD 见 §9.3.4。 | §9.3.4 五项验收全部通过。 |
| 2.7 | **前台安全快路径（§9.6）**：四级渐进校验（格式→限流→身份→业务预检）、权限决策缓存、连接池；前台拒绝不触发 L5+ 或 escalate。 | §9.6.7 DoD 六项全部通过。 |

| # | 动作 | DoD（最小验收） |

| 3.3 | **可选 HMAC/签名**：仅在跨主机或多实例边界启用；密钥轮换走 L15 与密钥管理规范（§9.4.3）。 | `envelope_signing_mode=off` 时风险评估已备案。 |
| 3.4 | **内存与 CPU 资源上限落配置**：三段内存上限（§9.4.2）、队列上限（§9.3.1）、去重 LRU 大小均通过 14.4 配置链管理，压测验证稳态无增长。 | 压测 1 小时路由引擎内存稳态；§9.4.5 DoD 全部通过。 |

### 阶段四：纵深加固（按暴露面选做）

| # | 动作 | DoD（最小验收） |
|---|------|------------------|
| 4.1 | **`intermediate_result_refs` 等引用白名单**：scheme、租户前缀、对象 ACL 校验，防 SSRF/跨租户读。 | 安全用例集通过。 |
| 4.2 | **重放与幂等**：`handoff_id` + 处理端去重窗口与副作用幂等策略。 | 重复投递不产生双写。 |
| 4.3 | **拓扑热更竞态**：单测覆盖「请求中拓扑切换」仅影响后续请求或显式中断策略。 | 与 14.2 钉选策略无矛盾。 |
| 4.4 | **加密传输完整验收**（§9.5）：跨主机 TLS 扫描、mTLS 双向验证、证书滚动轮换、字段级 PII 不落明文 trail、进程内无加密热路径。 | §9.5.7 DoD 六项全部通过。 |

### 15.5 刻意后置（避免过早优化）

- 为 16 个主层**各写独立长篇**实施文档（维护成本高于收益，枢纽仍以 `00_` 为准）。  
- 在 **阶段二完成前** 上全量多签、全量多机密轮换（运维成本高、调试难）。  
- 在 **双信封未区分（阶段一未完成）** 前扩展 Envelope 字段，易形成第三套语义。

### 15.6 版本说明

本路线随 `19_` 文档版本迭代；重大阶段变更须在 `00_升级总结总览.md` 演进记录中记一笔，并在 `topology_version` 或路由引擎配置变更中可审计。

## 附录A：通讯层威胁建模与风险清单（系统性）

> **用途**：供安全评审、实施验收与红队参考。  
> **分层**：按「信封与契约 / 路由引擎 / 身份与授权 / 流量与可用性 / 供应链与配置 / 数据与隐私 / 实现落差」七类组织，每条含风险级别与建议缓解方向。  
> **与文档关系**：第十四章是「已写死的基线」，本附录是「基线落地后仍需持续关注的潜在面」，两者互补不重复。

### A.1 信封与契约类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A1.1 | **双信封语义混用**：`task-serializer` IPC 包与 Layer Message Envelope 如长期未区分，`trace_id`/`tenant_id` 只在一套里存在，另一套成为绕过主体校验的**侧信道**。 | 高 | 阶段一 1.1：强制前缀/类型名区分；静态审查禁止两者共用未限定字段名。 |
| A1.2 | **`payload.content` 无界追加**：每层只追加不覆盖，长链路/反复重试后 payload 可无限膨胀，造成**序列化/传输/日志存储放大攻击**。 | 中 | 信封追加前检查当前累积大小；超 `layer_msg_max_bytes` 触发截断或分段策略（配置项，见 14.4）。 |
| A1.3 | **`goal` 字段竞态写**：`orchestration.task`（L6）被规定为唯一写入方，但若有多条 VP 域并发触发管道，两段「首次写入」可能产生**目标字段争抢或覆盖**。 | 中 | `goal` 写入须带 CAS（compare-and-set）或一次性写标志；后续写视为违规并审计告警。 |
| A1.4 | **`hop_count` 不单调/可伪造**：若信封可被处理器修改，恶意或有缺陷的处理器可将 `hop_count` 清零，绕过 `ttl` 防循环机制。 | 高 | 路由引擎单方维护并递增 `hop_count`，处理器只读；信封提交给路由引擎时校验 `hop_count` 单调性（不可减）。 |
| A1.5 | **`audit.layer_trail` 可本地伪造**：规范要求只追加，但若实现层未对 trail 做完整性保护，攻击者/缺陷代码可能重写历史条目，造成**取证失真**。 | 高 | trail 追加时对已有条目做 hash chaining（每条包含上一条摘要）；S3 写入后不可删改（与审计缺口 8 原则一致）。 |
| A1.6 | **`envelope_version` 不校验**：路由引擎若不检查信封版本号，旧格式信封可能被新版处理器以错误方式解析，或新字段被旧处理器静默丢弃（**静默降级**）。 | 中 | 路由引擎校验 `envelope_version` 与已知版本列表；不匹配时拒绝并审计，不做静默降级。 |

### A.2 路由引擎类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A2.1 | **路由引擎未实现/不全量覆盖**（实现落差）：`layer-router` 未落地时，各模块直连下一层；任何模块可自写 `skip_layers` 生效逻辑，第十四章集中强制**形同虚设**。 | 极高 | 阶段二优先交付；过渡期内明确「临时直连许可清单 + 禁止自写旁路」的代码约定，纳入 code review checklist。 |
| A2.2 | **处理器注册竞态**：动态注册若无分布式锁或版本 CAS，两个并发注册可能**产生同一 `layer_id` 的双 handler**，路由引擎不确定性选择一个。 | 中 | 注册操作须幂等且版本化；S4 写注册时使用乐观锁或 append-only log，路由引擎以「最新版本」为准。 |
| A2.3 | **`bypass_conditions` 条件本身可被注入**：条件描述若含动态求值（如 DSL 表达式），攻击者可构造满足旁路条件的输入，**绕过强制层**。 | 高 | `bypass_conditions` 只支持固定枚举条件 + 参数（不允许自由文本求值）；条件类型白名单在 S4 注册表中写死。 |
| A2.4 | **拓扑热更竞态（请求级）**：热加载新 topology 时，进飞中的 `trace_id` 若未钉选，**同一链路半段走旧图半段走新图**，可导致层间边不存在（路由失败）或跳过/重过某治理层。 | 高 | 请求开始时强制钉选 `topology_version`（写入信封 routing 区，只读）；路由引擎拒绝以不同版本继续同一 `trace_id`。 |
| A2.5 | **`escalate` 风暴**：底层异常（如 L9 大量超时）若每条都产生 escalate 消息，**组织侧（CEO/VP）通讯通道被淹没**，掩盖根因告警；同时触发大量 HITL 人工接管需求。 | 中 | 路由引擎对 escalate 做**去重聚合 + 速率限制**（见 14.4 `escalate_rate_limit`）；同一根因在时间窗内只上送一次，其余排队等待。 |
| A2.6 | **反向多边组合振荡**：多条合法反向边同时触发（如 L12→L9 + L10→L9 并发），若未在目标层做状态协调，产生**双重重执行竞态**，结果不确定。 | 中 | 同一 `trace_id` 的目标层重执行须在状态机层面加锁（幂等令牌 `handoff_id`）；并发反向打回合并为一次重执行。 |

### A.3 身份与授权类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A3.1 | **`tenant_id` 仅靠信封自报，无签名验证**：内网其他服务或有缺陷的处理器可**构造任意 `tenant_id`**，冒充高权限租户。 | 极高 | 跨主机/跨进程边界强制 TLS + 服务账号凭据校验，高敏启用 mTLS（**§9.5.1**）；路由引擎不信任信封里的 `tenant_id`，须与认证通道中的调用方身份一致。 |
| A3.2 | **Confused deputy**：处理器用**平台高权限服务账号**代理用户请求访问 S1–S4，未把信封里的 `principal` 下传给资源 ACL 检查，造成**权限升级**。 | 高 | 每层 handler 在访问存储/工具时，须以**信封 `principal`** 而非服务自身账号作为鉴权主体；平台服务账号不允许代理访问用户租户的核心数据域。 |
| A3.3 | **层注册表被非授权写入**：S4 注册表若 ACL 配置错误，任意组件可注册或覆盖 handler；攻击者注册**恶意 handler** 替换合法处理器。 | 极高 | S4 注册表写操作须双因素（VP02 + L22（release.publish）发布链）；路由引擎只从「已发布版本」读处理器，不接受运行时直接注入（与第十一章一致）。 |
| A3.4 | **`principal` 跨租户污染**：多租户共享进程时，上一请求的 `principal` 未在信封生命周期结束时清除，被下一请求继承（**线程/上下文泄漏**）。 | 高 | 信封须与请求生命周期严格绑定（不复用实例）；路由引擎在创建每条新管道时强制新建信封，禁止复用跨租户信封对象。 |

### A.4 流量与可用性类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A4.1 | **消息放大**：一条入站请求经 L6 拆解为 N 个子任务，每个子任务各走完整 L1–L16 管道，总信封数 = N × 16；若 N 无上限，**背压打穿下游**。 | 高 | L6 拆解须有子任务上限（可配）；路由引擎子任务并发池（**§9.3 线③**）限制单 `trace_id` 活跃子管道数；超限入挂起队列不丢弃。 |
| A4.2 | **同步阻塞管道**：若某层处理逻辑同步等待外部 IO（如 LLM 推理），整条管道阻塞；高并发时**线程/协程耗尽**导致入口拒绝新请求。 | 中 | 涉及外部 IO 的层须异步执行 + 超时（`slo_ms`）；路由引擎顺向主管道队列（**§9.3 线①**）解耦上下层背压，入口层限流不穿透。 |
| A4.3 | **`ttl` 截断业务死锁**：`ttl` 过短时，长合法链路被强制终止；对业务表现为**假超时失败**，产生不可回滚的副作用（已执行一半）。 | 中 | `ttl` 默认值基于实测最长合法链路设置（可配，见 14.4）；触发前先记录当前状态并安全收口，不立即硬终止；队列等待期间 `hop_count` 不递增（**§9.3.2**）。 |
| A4.4 | **反向风暴**：外部攻击或大批量失败任务同时触发 L12→L9 反向，超过重试池承载，产生**重试雪崩**。 | 中 | 路由引擎反向/escalate 通道队列（**§9.3 线②**）实施全局 + 租户双层速率限制；超限入等待队列不丢弃；同 `trace_id` 并发反向合并为一次重执行（防振荡）。 |

### A.5 供应链与配置类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A5.1 | **拓扑配置被篡改**：S4 中的 `topology_version` + 边表若被篡改，可**增加恶意层/删除治理层/修改合法反向边**；路由引擎热加载后即生效，无人工介入。 | 极高 | 拓扑配置变更须 VP02 审批 + L22（release.publish）发布 + 变更内容哈希验证；路由引擎只加载校验通过的版本；加载失败回退上一已知良好版本。 |
| A5.2 | **`bypass_conditions` 配置条目被恶意扩大**：若有权修改 S4 的内部用户扩大旁路条件范围，治理层被合法地绕过而不触发任何告警。 | 高 | bypass 条件变更须独立审批轨（VP02 + 独立安全审核）；每次变更写 S3 并触发告警；核心治理层的旁路条目禁止通过普通 P0 变更流程。 |
| A5.3 | **HMAC/签名密钥泄漏或长期不轮换**：密钥一旦泄漏，攻击者可**伪造任意合法信封**通过签名校验。 | 高 | 密钥走 KMS（**§9.5.5**）；轮换周期可配（`cert_rotation_days`），经 L22（release.publish）发布；泄漏后切换为拒绝未签名模式 + 强制全量重签；多版本 grace period 内新旧密钥双验。 |
| A5.4 | **配置链注入**（`platform→company→department→user`）：若 user 级配置可突破 company 级上限（实现时校验链断），攻击者注册低权限账户后**把 `ttl` 改到极大、把体积上限改到极大**绕过保护。 | 高 | 每级配置覆盖时须和上级取 min（更严格的那个）；配置链求值逻辑单点实现，不允许各层自行解释配置继承。 |

### A.6 数据与隐私类

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A6.1 | **`payload.content` 含敏感字段跨层明文流转**：各层追加内容，若某层写入了密钥、PII、用户隐私，后续所有层均可读；日志与 trail 也会落入 S3。 | 高 | 各层处理器在写 `payload.content` 前须经脱敏 hook（与 `redaction_profile` 对齐）；敏感字段只存引用（对象键），不存明文值。 |
| A6.2 | **`audit.layer_trail` 大量 PII 落 S3**：`note` 字段若含用户输入摘要，长期运行后 S3 成为**未经 GDPR 等合规处理的 PII 库**。 | 中 | trail `note` 字段须按 `redaction_profile` 脱敏后再写 S3；提供按 `trace_id` 删除/匿名化接口（满足被遗忘权）。 |
| A6.3 | **`intermediate_result_refs` 跨租户可读**：若引用存储后端（对象存储/S2）的 bucket/prefix ACL 未按租户隔离，A 租户的引用指向 B 租户的对象。 | 极高 | 引用生成时强制前缀带 `tenant_id`；引用解析时路由引擎或存储适配器校验调用方租户与前缀一致，不匹配直接拒绝。 |
| A6.4 | **`rejection_reason` / `correction_suggestion` 含模型推断内容泄漏**：反向信封中若含模型对用户输入的分析结论，可能**把一个租户的输入语义暴露给另一租户的处理器**（共享进程场景）。 | 中 | 反向信封同样需要租户绑定校验；共享进程时反向信封的处理须在隔离上下文内执行。 |

### A.7 实现落差类（设计已写但实现未跟上）

| # | 风险描述 | 级别 | 建议缓解 |
|---|----------|------|----------|
| A7.1 | **`layer-router` 未落地（核心）**：所有集中强制（旁路、反向、ttl、租户）均依赖此入口；未落地前所有 A 类风险的防御措施实际为零。 | 极高 | 阶段二最高优先；过渡期通过 lint rule / code review checklist 阻断自造旁路代码合入。 |
| A7.2 | **`trace_id` 断链**：某模块未透传 `trace_id`，导致链路中间断链；排障时无法重建完整执行路径，也无法完整审计。 | 高 | 阶段一 1.2；可用 integration test + trace sampling 验收「任意抽样链路无断链」。 |
| A7.3 | **处理器对信封字段只读约定未强制**：规范写明处理器不可修改 `routing.*` / `goal` / `skip_layers`，但若无运行时防护（如 Object.freeze 或 Proxy），**仍可静默修改**。 | 中 | 路由引擎在交付信封给处理器前做深度冻结（或传递只读视图）；处理器只能调用指定的 append API。 |
| A7.4 | **子层内部通讯未受信封约束**：规范说子层在主层内部运作，但若子层直接相互调用（不经主层出口），**逃脱 `layer_trail` 审计与租户校验**。 | 中 | 子层间通讯须通过主层协调（同一上下文对象），不允许子层直接触发另一主层的处理器；子层内部调用须同样携带 `trace_id` 并追加 trail 子条目。 |
