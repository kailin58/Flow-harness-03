## 一、P0推进原则（写死）

1. 先跑通最小链路，再扩展能力覆盖。
2. 每个任务必须有“完成定义（DoD）”和验收证据。
3. 关键链路优先：会话中间层 -> 编排策略 -> 门禁发布 -> 回写监控。
4. 任何上线动作均遵守 `L8/L10` 门禁与回滚要求。

## 三、任务分解与验收标准

### P0-1 实施基线映射表

**目标**：把规范字段和接口映射到具体模块、责任与状态。  
**最小输出**：

- 会话字段映射（`session_id`、`strategy`、`risk_level` 等）
- 状态码映射（会话状态 + 执行状态）
- 错误码映射（`E_*` 到处理动作）
- 接口映射（后端接口 -> 前端调用点）

**DoD**：

1. 映射表覆盖 `12` 文档所有“必填字段 + 最小接口”。
2. 每行包含：负责人、模块位置、实现状态、截止时间。
3. 映射表已评审通过并可作为开发输入。

### P0-2 最小链路端到端打通

**目标**：验证核心闭环可运行。  
**最小链路**：

`目标输入 -> 后端生成编排 -> 前端呈现 -> 人工确认节点 -> 质检 -> 外发 -> 回写(L6/L10/L9)`

**DoD**：

1. 链路可重复执行（至少 2 次）。
2. 每步有状态回写与审计事件。
3. 外发失败可触发重试/熔断/人工接管。
4. 有完整执行证据（日志、状态流、审计ID）。

### P0-3 会话中间层接口定版

**目标**：冻结 V1 接口，避免前后端反复改。  
**最小接口集**：

- `createSession` / `getSession` / `updateSession`
- `pauseSession` / `resumeSession` / `settleSession`
- `generatePlanByGoal` / `evaluateStrategy` / `runABC`
- `requestHumanConfirm` / `submitHumanDecision`
- `publishArtifacts` / `rollbackPublish`

**DoD**：

1. 每个接口有请求/响应样例。
2. 明确幂等规则、错误码、重试语义。
3. 对外发布 `v1` 版本号与兼容窗口。

### P0-4 默认阈值参数表

**目标**：把“原则”变成“可执行参数”。  
**最小参数**：

- ABC 触发阈值
- 会话超时阈值
- 重试次数与退避策略
- 门禁评分阈值
- 回滚触发条件

**DoD**：

1. 参数均有默认值、上下限、适用范围。
2. 参数调整有审批链与审计记录。
3. 参数表与 `11` 文档策略一致。

### P0-5 最小监控看板上线

**目标**：具备运行可观测性。  
**最小指标**：

- 会话成功率 / 失败率 / 超时率
- ABC 触发率 / 择优收益率
- 外发成功率 / 重试率 / 回滚率
- 人工接管率

**DoD**：

1. 面板可按租户、策略、会话类型筛选。
2. 告警规则可触发通知（至少 1 条链路）。
3. 指标口径与回写口径一致（`L6/L10/L9`）。

### P0-6 运行手册（Runbook）

**目标**：异常场景可按手册处理，不依赖口头经验。  
**最小章节**：

- 外发失败处理
- 死信队列处理
- 人工接管流程
- 回滚流程
- 发布冻结与恢复

**DoD**：

1. 每个异常有“触发条件 -> 处理步骤 -> 验证步骤”。
2. 覆盖至少 3 个高频故障场景。
3. 手册与当前策略版本绑定并可追溯。

### P0-7 历史业务未收口项清零

**目标**：清理“之前已写规则但未形成闭环”的遗留项，防止虚假完成。  
**最小输出**：

- 未收口项台账（来源文档、问题描述、责任层级、当前状态）
- 指标回补（财务指标、ROI、Token 投入产出、业务结果）
- 收口证据（审计ID、看板截图、会议结论、发布/回滚记录）

**DoD**：

1. 未收口项全部绑定：`负责人 + 截止时间 + 风险等级 + 证据链接`。
2. 每项均有“处理动作 -> 结果状态 -> 复盘结论”闭环记录。
3. 对 CEO/VP/总监/经理四层形成月度经营复盘口径并固定到看板。
4. 无“口头完成、无证据关闭”的条目残留。

### P0-8 性能优化与代码驱动提效

**目标**：把“系统性能优化”与“代码自动化提效”变成可核算收益。  
**最小输出**：

- 关键链路性能基线（优化前/后）：时延、吞吐、失败率、资源消耗
- 自动化改造清单：人工步骤替代项、脚本/服务化落地点、回滚方案
- 经营收益清单：人时节省、Token 成本节省、净收益改善

**DoD**：

1. 至少完成 1 条关键链路性能优化并给出对比证据。
2. 至少完成 1 项“人工 -> 代码驱动”改造并稳定运行。
3. 每项改造都绑定 ROI 说明与审计证据，不接受“仅技术完成”。
4. 优化项同步进入 CEO/VP/总监/经理经营复盘看板。

### P0-9 全链路业务编排样板落地

**目标**：验证平台具备“任务输入 -> 业务执行 -> 持续优化”的全流程自主能力。  
**最小输出**：

- 样板场景：如“开服装店”完整链路配置与执行记录
- 链路产物：调研报告、品牌参考、商品文案、发布记录、运营复盘
- 优化记录：至少 1 轮 A/B/C 试跑与择优结论

**DoD**：

1. 跑通“调研 -> 方案 -> 生产 -> 发布 -> 监控 -> 优化”闭环。
2. 每环节具备状态回写、审计ID、异常处理记录。
3. 至少产出 1 份经营结果报告（投入、产出、ROI、Token 净收益）。
4. 样板可复跑，且支持人工确认节点与回滚。

### P0-10 规则学习与支付人工干预落地

**目标**：建立“先学习、后演练、再执行”的硬流程，并落实支付人工干预与经营策略闭环。  
**最小输出**：

- 学习包机制：`learning_pack` 字段、生成规则、准入校验
- 演练机制：沙盘演练报告模板、通过阈值、失败拦截规则
- 支付审批链：扣费/打款/退款人工确认节点与审计记录
- 策略机制：定价、售后、客服策略模板与回写指标

**DoD**：

1. 新任务未通过学习与演练时，系统自动阻断执行。
2. 支付链路全部具备人工审批记录与回滚路径。
3. 定价、售后、客服策略均可配置、可灰度、可回滚。
4. 经营看板新增策略效果指标（转化、退货率、满意度、ROI）。

### P0-11 电脑操作与软件自动化能力落地

**目标**：让系统具备可治理的电脑操作与软件使用能力，替代高重复人工流程。  
**最小输出**：

- 端侧操作能力：窗口控制、文件处理、浏览器操作、步骤回放
- 软件适配能力：办公软件、运营后台、表单系统的自动化模板
- 风险门禁：高风险动作人工确认、白名单应用控制、异常熔断与回滚
- 审计体系：操作日志、截图证据、审计ID与责任归属

**DoD**：

1. 至少 1 条真实业务链路由“人工操作”改为“电脑+软件自动化执行”。
2. 高风险动作默认人工确认，未经审批不得自动放行。
3. 自动化流程具备失败重试、回滚与断点续跑能力。
4. 全过程可审计并绑定 `agent_id/task_id/session_id`。

### P0-12 单任务分工与全局复用机制落地

**目标**：落实“一个 Agent 单一任务，分工协作执行，全局能力可复用”。  
**最小输出**：

- 任务主责机制：原子任务唯一主责字段与冲突拦截规则
- 协作机制：主责/协作/验收角色定义与状态流转
- 复用机制：能力包注册、检索、版本约束、作用域挂载
- 审计机制：复用记录与“不复用原因”记录模板

**DoD**：

1. 编排层可强制校验“一个原子任务仅一个主责 Agent”。
2. 跨任务协作链路清晰，且无跳级分派与跳级回传。
3. 新任务执行前完成复用检索，并保留复用或不复用证据。
4. 复用能力调用全链路可追溯（`skill_id/version/scope_level`）。

### P0-13 网页识别能力建设

**目标**：让系统在网页自动化场景具备“可识别、可校验、可防误操作”的能力。  
**最小输出**：

- 页面识别能力：结构区域、元素语义、关键字段、页面状态识别
- 动作校验能力：动作前目标确认、动作后回执校验、异常页拦截
- 风险控制能力：低置信度人工确认、风控页降级、重试与熔断策略
- 观测能力：识别准确率、误操作率、阻断率、人工接管率看板

**DoD**：

1. 至少 1 条网页业务链路接入网页识别能力并稳定运行。
2. 关键动作（发布/支付/删除）全部具备“前确认+后校验”证据。
3. 页面变更或异常时可自动降级并触发人工接管。
4. 识别结果全链路可审计并绑定 `task_id/session_id/审计ID`。

### P0-14 学院能力评测闭环落地

**目标**：将关键能力建设（经验沉淀/创新/盈利/市场机会识别/自主学习）纳入学院统一评测与发布链。  
**最小输出**：

- 学院能力课纲：五类能力对应课程、等级、训练样本要求
- 评测门槛：通过阈值、失败回炉规则、补考机制
- 发布前校验：能力升级需先学院评测通过再灰度发布
- 回流机制：生产表现回流学院，更新课程包与评测题库

**DoD**：

1. 五类能力均建立学院课纲并绑定负责人。
2. 能力发布前存在可执行的学院评测门禁，未通过自动阻断发布。
3. 至少完成 1 轮“学院训练 -> 评测 -> 灰度 -> 复盘回流”闭环。
4. 评测结果与经营结果可关联追踪（ROI、净收益、成功率变化）。

### P0-15 配置文件与端口治理落地

**目标**：建立可审计、可回滚、可分层的配置与端口治理体系。  
**最小输出**：

- 配置分层方案：`platform/company/department/user` 作用域与覆盖顺序
- 端口注册表：`Protocol/Tool/LLM` 三类端口的ID、归属、风险、状态
- 发布门禁：高风险端口双签、灰度发布、失败自动降级
- 回滚机制：配置版本回滚、端口版本回滚、证据与审计绑定

**DoD**：

1. 至少 1 条端口配置链路实现版本化发布与一键回滚。
2. 高风险端口变更具备双签与灰度记录，未审批不得发布。
3. 端口异常可触发降级与人工接管，不中断主流程。
4. 配置与端口变更全链路可追溯（版本号、审批人、审计ID）。

### P0-16 第三方通讯软件接入落地

**目标**：将外部通讯（IM/邮件/短信/Webhook）纳入统一治理链，实现可发送、可追踪、可降级。  
**最小输出**：

- 通道注册：`channel_id/channel_type/provider/credential_ref`
- 消息发送：模板化发送、异步投递、回执回写
- 风险门禁：高风险消息审批后发送，未审批自动阻断
- 异常策略：限流、重试、降级、人工接管
- 首批渠道：微信（企业微信）、飞书、钉钉三通道适配
- 双轨方案：`API主路径 + 桌面安装镜像备路径`（登录沟通受控）

**DoD**：

1. 微信（企业微信）、飞书、钉钉三通道均完成接入并稳定发送。
2. 消息任务具备状态追踪（`PENDING/SENT/FAILED/RETRYING/DEGRADED`）。
3. 高风险消息发送必须有审批记录与审计ID。
4. 发送失败可自动重试并在超阈值时降级到人工接管。
5. 三通道均完成“双轨方案演练”并有切换证据（API -> 桌面镜像 -> API回切）。

### P0-17 前端多端统一能力落地

**目标**：建设 PC/H5/小程序/平板/APP 五端一致的多状态前端能力。  
**最小输出**：

- 多端矩阵：各端职责边界、能力分层、可执行动作清单
- 一致性机制：统一状态枚举、统一错误码、统一审计追踪
- 权限分层：高风险动作端侧限制（默认仅 PC/APP）
- 联调报告：五端状态联动与异常链路演示证据
- 动物角色映射：角色卡片与真实 `agent_id/level/status` 绑定规则
- 学习虚拟币排行榜：总榜/学习榜/进步榜/贡献榜与审计回溯

**DoD**：

1. 五端均可展示主状态/子状态/门禁状态/异常状态。
2. 五端状态口径一致，且与后端事件流一致。
3. 高风险动作在非授权端自动拦截并给出申请路径。
4. 至少完成 1 轮五端联调并产出审计证据。
5. 动物角色展示状态与后端真实状态一致，并有抽检记录。
6. 学习虚拟币排行榜可按周期与层级筛选，并具备加减币审计证据。

### P0-18 前端性能与设备基线优化

**目标**：建立前端设备配置基线与性能优化闭环，保证多端稳定可用。  
**最小输出**：

- 设备基线表：开发最低档/联调推荐档/生产执行档
- 渲染优化项：分层加载、虚拟滚动、增量更新、动效降级
- 监控面板：TTI、状态刷新延迟、卡顿率、内存峰值
- 对比报告：每次优化的前后数据与收益说明

**DoD**：

1. 至少完成 1 条关键页面的性能基线采集与优化对比。
2. 多端（PC/H5/小程序/平板/APP）具备可用性验收记录。
3. 低性能设备能稳定完成状态查看、审批、回执追踪。
4. 性能数据与优化证据可审计、可复盘。

### P0-19 分层拓扑配置化改造

**目标**：将“11层分工写死”升级为“拓扑配置化可扩展”，支持后续新增层级/角色而不改核心代码。  
**最小输出**：

- 拓扑配置结构：`layers / roles / routing / constraints / topology_version`
- 兼容策略：无配置时使用默认基线，有配置时动态加载
- 治理流程：拓扑变更审批、灰度生效、回滚机制
- 审计要求：每次变更绑定审计ID与影响范围

**DoD**：

1. 新增一个层级可通过配置生效，不改核心编排代码。
2. 角色挂载与任务路由由配置驱动，支持版本回滚。
3. 非法拓扑（孤儿节点/越权调用路径）可被校验并阻断发布。
4. 变更链路可追溯（`topology_version + 审批记录 + 审计ID`）。
5. 每层支持新增子层级且数量不设固定上限，扩展后仍满足治理校验。

### P0-20 VP拓扑扩展配置化改造

**目标**：将“11个VP写死”升级为“VP基线11、数量可扩展”的配置化治理模式。  
**最小输出**：

- VP配置结构：`vp_registry / vp_profiles / vp_routing / vp_constraints`
- 扩展流程：新增VP申请、审批、生效、回滚
- 兼容策略：保留VP01~VP11默认基线，新增VP不影响存量链路
- 审计要求：新增/停用VP均绑定审计ID与影响评估

**DoD**：

1. 新增一个VP可通过配置发布生效，不改核心编排代码。
2. VP级路由与能力边界由配置驱动，支持版本回滚。
3. 非法VP配置（重复ID/越权路由/跨域污染）可被校验并阻断发布。
4. 变更链路可追溯（`topology_version + vp_registry_version + 审批记录 + 审计ID`）。
