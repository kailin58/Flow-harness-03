## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。
- 治理模型统一采用“规则写死、参数可配、边界写死”：
  - 写死规则：流程门禁、接管链路、审计要求、发布与回滚约束。
  - 可配置参数：SLA、复评阈值、紧急策略、试点范围等运营参数。
  - 边界写死：仅允许边界内收紧，不允许突破平台红线；配置变更必须版本化发布并写审计。
- 接口原则：所有接口不做占位；设计可做全局通用与扩展增强，但接口定义本身必须可直接落地，不保留仅供后补的占位接口。
- 全局配置原则：凡可由配置承载的真实业务值，一律配置化管理，能配置就不写死。
- 落地口径：正文默认只固化字段名、规则、约束、示例值与生效顺序；真实实值通过 `platform -> company -> department -> user` 配置链注入并审计。

## 一、协议总览

### 为什么需要专门的闭环协议

2M+23L 工作流已定义 `feedback.loop`（L20）、`evolution.experiment`（L23）、`quality.gate`（L18）、`release.publish`（L22）的独立职责，但四者之间的**数据流路径和触发机制**是断开的：

```
现有状态：
  feedback.loop（L20）→ "经验固化"（写哪里？不清楚）
  evolution.experiment（L23）→ "自优化沙盒"（什么时候触发？不清楚）
  quality.gate（L18）→ "通过/打回"（进化评审怎么做？不清楚）
  release.publish（L22）→ "审批/灰度/全量"（如何承接提案？不清楚）

本协议目标：
  打通 feedback.loop（L20）→ S2 → evolution.experiment（L23）→ quality.gate（L18）→ release.publish（L22）→ S4 的完整数据通路
```

### 三段式闭环

```
第一段：经验沉淀（任务完成 → `feedback.loop`（L20）→ S2）
第二段：实验触发（S2 阈值判断 → `evolution.experiment`（L23）沙盒实验）
第三段：提案评审（`evolution.experiment`（L23）→ `quality.gate`（L18）→ `release.publish`（L22）→ S4）
```

## 二、第一段：经验沉淀（`feedback.loop`（L20）→ S2）

### 2.1 触发时机

每次任务进入 **DONE** 或 **FAILED** 终态后，`feedback.loop`（L20）反馈闭环层自动生成经验条目。无论成功失败均生成，失败的经验同样是 `evolution.experiment`（L23）的重要训练材料。

### 2.2 数据来源整合

`feedback.loop`（L20）生成 experience_entry 需要从以下来源收集数据：

| 字段 | 数据来源 |
|------|------|
| `task_type` | S1 上下文库（任务元数据） |
| `success` | `orchestration.collab`（L13）状态机（终态判定） |
| `failure_layer` | S3 审计库（异常记录 + 状态流转记录） |
| `correction_trace` | S3 审计库（查询本任务的 escalation 记录，取 escalation_id 列表） |
| `final_result` | S2 制品库（最后一条有效制品摘要） |
| `confidence_delta` | `quality.inspector`（L16）Inspector 层反馈（本次任务的置信度变化量） |

### 2.3 experience_entry 数据结构

```json
{
  "record_type": "experience",
  "task_id": "T-889",
  "task_type": "code-generation",
  "success": true,
  "failure_layer": null,
  "error_category": null,
  "decision_path": [
    {"step": 1, "layer_id": "orchestration.task", "layer_display": "L12", "decision": "识别为代码生成任务", "reason": "用户输入含明确函数需求"},
    {"step": 2, "layer_id": "quality.gate", "layer_display": "L18", "decision": "通过质量门禁", "reason": "格式/逻辑/完整性校验均通过"}
  ],
  "correction_trace": ["ESC-20260414-0023"],
  "final_result": "完成了用户注册模块的前端实现，包含表单校验和错误处理",
  "confidence_delta": 0.05,
  "created_at": "2026-04-14T10:30:00Z"
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `record_type` | string | ✅ | 固定为 `experience`，用于 S2 子类型过滤与生命周期管理 |
| `task_id` | string | ✅ | 关联的任务 ID，用于追溯 |
| `task_type` | string | ✅ | 任务类型，`evolution.experiment`（L23）按此分组聚合同类经验 |
| `success` | bool | ✅ | 本次任务是否最终成功 |
| `failure_layer` | string \| null | ✅ | 失败发生在哪一层，使用 `layer_id`（如 `execution.monitor`、`quality.inspector`），成功时为 null |
| `error_category` | string \| null | ⭕ | 错误类别标签（如 `format_error` / `logic_error` / `rule_violation` / `tool_timeout` / `permission_denied`）；成功时为 null；`quality.inspector`（L16）写入 |
| `decision_path` | object[] | ⭕ | 决策路径（`{step, layer_id, layer_display, decision, reason}`；关键分叉节点记录，供 `evolution.experiment`（L23）聚合同类决策模式）|
| `correction_trace` | string[] | ✅ | 人工介入的 escalation_id 列表（可为空数组），S3 中可查全量修正记录 |
| `final_result` | string | ✅ | 最终正确输出的摘要（最长 200 tokens），`evolution.experiment`（L23）准确度参考 |
| `confidence_delta` | float | ✅ | 本次任务使系统置信度提升（正值）或下降（负值）的量 |
| `created_at` | ISO8601 | ✅ | 经验条目生成时间 |

### 2.4 写入 S2 的约束

```
写入时：
  record_type = "experience"   ← 必须声明，否则 S2 无法识别子类型
  不可覆盖写，只能追加           ← 与 S2 通用约束一致
  绑定 task_id + `feedback.loop`（L20）层标识      ← 与 S2 通用约束一致

生命周期：
  type=experience → 永久保留   ← 区别于 type=artifact 的N天保留
  归档权限：仅存储管理员可归档，归档后只读

correction_trace 字段处理：
  `feedback.loop`（L20）查询 S3，筛选条件：task_id = 当前任务 AND record_type = escalation
  取所有 escalation_id 写入列表
  不复制修正详情，只存引用
  → S3 永远是修正记录的唯一完整来源
```

## 三、第二段：实验触发（S2 → `evolution.experiment`（L23））

### 3.1 触发时机

**每次 `feedback.loop`（L20）写入 experience_entry 后**，`feedback.loop`（L20）立即执行阈值判断（不是定时批处理，而是任务结束即判断）。

### 3.2 触发条件

满足以下**任意一个**条件即触发 `evolution.experiment`（L23）：

| 条件 | 参数名（S4 规则库） | 默认值 | 说明 |
|------|------|--------|------|
| 同类任务历史失败率 ≥ 阈值 | `evo_trigger.failure_rate_threshold` | 0.20 | 查 S2 同 task_type 的历史条目，计算 success=false 的比例 |
| 同类任务人工修正记录累计 ≥ 阈值 | `evo_trigger.correction_count_threshold` | 10 | 查 S2 同 task_type 的历史条目，累加 correction_trace 非空条目数 |
| 管理员手动触发 | `evo_trigger.manual` | — | 无需阈值，直接进入 `evolution.experiment`（L23） |

**防误判保护：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `evo_trigger.min_sample_size` | 20 | 同类任务样本量不足时不触发，避免少量样本引发误判 |

### 3.3 触发流程

```
feedback.loop（L20）写入 experience_entry 完成
  │
  ├─ 查询 S4：读取 evo_trigger 系列阈值参数
  │
  ├─ 查询 S2：同 task_type 的全部 experience 条目
  │   ├─ 计算失败率
  │   └─ 统计 correction_trace 非空条目数
  │
  ├─ 样本量 < min_sample_size → 不触发，结束
  │
  ├─ 失败率 ≥ threshold OR 修正数 ≥ threshold
  │   → 写 S3 审计记录（标记 EVOLUTION_EXPERIMENT_TRIGGERED，记录触发原因和当前统计值）
  │   → 调用 evolution.experiment（L23），传入 task_type 和触发原因
  │
  └─ 未达阈值 → 结束本轮，等待下次经验积累
```

## 四、第三段：提案评审（`evolution.experiment`（L23）→ `quality.gate`（L18）→ `release.publish`（L22）→ S4）

### 4.1 `evolution.experiment`（L23）：沙盒实验与提案生成

`evolution.experiment`（L23）被触发后，执行以下步骤：

```
evolution.experiment（L23）进化实验层
  │
  ├─ Step 1：从 S2 拉取 task_type 全部 experience 条目
  │
  ├─ Step 2：分析经验条目，识别失败模式
  │   ├─ 高频失败层（failure_layer 统计）
  │   ├─ 需要人工介入的场景模式（correction_trace 非空条目分析）
  │   └─ 置信度下降模式（confidence_delta 负值聚类）
  │
  ├─ Step 3：生成 Proposal（七类之一）
  │   ├─ Prompt 变更：修改系统提示或上下文模板
  │   ├─ 路由规则变更：调整 orchestration.task（L12）任务分配策略或 model.management（L3）模型选择规则
  │   ├─ 工具调用顺序变更：调整 execution.monitor（L15）的工具执行序列
  │   ├─ 检索策略变更：调整 knowledge.retrieval（L8）的检索范围/排序/召回参数
  │   ├─ 缓存机制变更：调整缓存命中策略或缓存失效规则
  │   ├─ 阈值配置变更：调整触发阈值（如复评率/失败率/置信度下限，参数可配）
  │   └─ 规则优先级变更：调整 security.policy（L14）的业务规则执行优先级
  │   注：Proposal 是配置/策略变更，不是代码变更；所有变更须经 evolution.experiment（L23）沙盒验证后才可进入 quality.gate（L18）评审与 release.publish（L22）发布
  │
  ├─ Step 4：沙盒验证
  │   ├─ 用历史经验条目中的 task_id 对应任务重跑
  │   ├─ 对比新策略 vs 旧策略的成功率/置信度变化
  │   └─ 生成验证报告（沙盒指标：成功率变化 / 置信度变化 / 平均耗时变化）
  │
  └─ Step 5：调用 quality.gate（L18）进化评审模式，传入 Proposal + 经验样本 + 沙盒指标
```

**`proposal_type` 枚举（七类自优化入口）：**

| 值 | 含义 | 对应层 |
|---|---|---|
| `prompt_change` | Prompt模板变更 | `knowledge.management`（L7）Prompt管理 / `orchestration.task`（L12）上下文模板 |
| `routing_change` | Agent路由策略变更 | `orchestration.task`（L12）任务分配策略 / `model.management`（L3）模型路由 |
| `tool_order_change` | 工具调用顺序变更 | `execution.monitor`（L15）工具执行序列 |
| `retrieval_change` | 检索策略变更 | `knowledge.retrieval`（L8）检索策略 |
| `cache_change` | 缓存机制变更 | `execution.monitor`（L15）缓存/执行策略 |
| `threshold_change` | 阈值配置变更 | S4 规则库参数，受 `security.policy`（L14）治理 |
| `rule_priority_change` | 规则优先级变更 | `security.policy`（L14）业务规则优先级 |

**Proposal 数据结构：**

```json
{
  "proposal_id": "PROP-20260414-0001",
  "task_type": "code-generation",
  "trigger_reason": "failure_rate=0.23 (threshold=0.20), sample_size=35",
  "proposal_type": "prompt_change",
  "description": "在 code-generation 类任务的系统提示中增加错误处理示例模板",
  "diff": {
    "before": "...",
    "after": "..."
  },
  "sandbox_result": {
    "success_rate_before": 0.77,
    "success_rate_after": 0.91,
    "confidence_delta_avg_before": -0.03,
    "confidence_delta_avg_after": 0.04,
    "sample_count": 35
  },
  "created_at": "2026-04-14T11:00:00Z"
}
```

### 4.2 `quality.gate`（L18）：进化评审模式

`quality.gate`（L18）在此场景下工作在**进化评审模式**，与日常的生产门禁模式不同：

| 维度 | 生产门禁模式 | 进化评审模式 |
|------|------|------|
| 输入 | 单个任务输出 | Proposal + 历史经验条目 + 沙盒指标 |
| 评估问题 | "这个输出达标吗？" | "新策略比旧策略好吗？" |
| 评估方式 | 单点质检 | 统计对比评分 |
| 输出 | 通过 / 打回 | 推荐采用（附评分） / 否决（附原因） |
| 最终决定权 | `quality.gate`（L18）自身 | `release.publish`（L22）人工审批（`quality.gate`（L18）只提建议） |

**`quality.gate`（L18）进化评审评分维度：**

| 维度 | 权重 | 说明 |
|------|------|------|
| 成功率提升 | 40% | 新策略样本成功率 vs 旧策略成功率差值 |
| 置信度改善 | 30% | 平均 confidence_delta 从负转正的幅度 |
| 人工介入减少 | 20% | correction_trace 非空条目比例下降幅度 |
| 策略风险评估 | 10% | 变更影响范围和可回滚风险的综合评分 |

评分 ≥ 0.7 → `quality.gate`（L18）推荐采用<br>
评分 0.5 ~ 0.7 → `quality.gate`（L18）建议有条件采用（附具体条件）<br>
评分 < 0.5 → `quality.gate`（L18）否决，反馈否决原因给 `evolution.experiment`（L23）

### 4.3 `release.publish`（L22）：审批与发布

`quality.gate`（L18）评审通过（推荐采用）后，进入 `release.publish`（L22）发布链路：

```
release.publish（L22）版本发布层
  │
  ├─ 生成候选版本（Candidate）
  │   └─ 绑定 proposal_id + quality.gate（L18）评审报告
  │
  ├─ 人工审批
  │   ├─ 审批通过 → 进入灰度
  │   └─ 审批拒绝 → 写 S3（PROPOSAL_REJECTED），通知 evolution.experiment（L23）
  │
  ├─ 灰度上线（部分流量使用新策略）
  │   ├─ 观察期间新策略指标正常 → 全量
  │   └─ 指标异常 → 回滚，写 S3（ROLLBACK），通知 evolution.experiment（L23）
  │
  └─ 全量上线
      └─ 写入 S4：更新对应规则的版本号
```

### 4.4 S4 规则版本更新

全量发布后，`release.publish`（L22）将进化结果同步写入 S4 规则库：

```json
{
  "rule_id": "RULE-code-generation-prompt-v3",
  "version": "3.0",
  "previous_version": "2.0",
  "source_proposal_id": "PROP-20260414-0001",
  "effective_date": "2026-04-14T14:00:00Z",
  "change_summary": "code-generation 类任务 Prompt 模板更新，增加错误处理示例"
}
```

## 五、与现有协议的交叉引用

### 与 Human-in-the-Loop 协议（06文档）的关系

06文档的 correction_trace 数据是本协议的**上游输入**：

```
06协议（Human-in-the-Loop）
  CLOSED 时 → 写 S3：完整意图累加器快照（绑定 escalation_id）
                           ↓
07协议（经验进化闭环）
  feedback.loop（L20）生成 experience_entry 时 → 查 S3，获取 escalation_id 列表
  correction_trace = [escalation_id_1, escalation_id_2, ...]
```

**关键原则**：experience_entry 的 correction_trace 只存 escalation_id，不复制修正详情。需要查看具体修正内容时，通过 escalation_id 反查 S3。

### 人工不可以做的（进化闭环边界）

| 禁止操作 | 原因 | 约束来源 |
|----------|------|----------|
| 直接修改 `evolution.experiment`（L23）的进化结果（Proposal） | 进化结果必须经 `release.publish`（L22）发布链路，直改等于架空发布门禁 | `evolution.experiment`（L23）设计约束（同 06文档） |
| 跳过 `quality.gate`（L18）评审直接提交 `release.publish`（L22） | `quality.gate`（L18）的对比评分是 `release.publish`（L22）审批决策的依据，跳过则审批无据 | `quality.gate`（L18）设计约束 |
| 直接覆盖 S4 规则 | 规则变更必须经 `release.publish`（L22）发布流程 | S4 设计约束 |
| 删除 S2 中的 experience 条目 | 经验条目永久保留，是 `evolution.experiment`（L23）统计基础；删除会导致触发条件失准 | S2 设计约束 |

## 六、完整审计轨迹（S3 写入点）

| 时机 | 写入内容 | 不可跳过 |
|------|----------|----------|
| `feedback.loop`（L20）判定触发 `evolution.experiment`（L23）时 | 触发事件（task_type、触发原因、当前统计值） | ✅ |
| `evolution.experiment`（L23）生成 Proposal 时 | Proposal 摘要（绑定 proposal_id） | ✅ |
| `evolution.experiment`（L23）沙盒验证完成时 | 沙盒结果报告（绑定 proposal_id） | ✅ |
| `quality.gate`（L18）评审完成时 | 评审结论（评分 + 推荐/否决）（绑定 proposal_id） | ✅ |
| `release.publish`（L22）人工审批时 | 审批决定（通过/拒绝 + 审批人）（绑定 proposal_id） | ✅ |
| `release.publish`（L22）灰度回滚时 | 回滚事件（原因 + 回滚时间） | ✅ |
| `release.publish`（L22）全量上线时 | 发布完成记录（绑定 proposal_id + S4 规则版本号） | ✅ |

## 七、阶段状态机（Proposal 生命周期）

```
TRIGGERED          ← feedback.loop（L20）判定阈值满足，调用 evolution.experiment（L23）
    │
    ▼
ANALYZING          ← evolution.experiment（L23）分析经验条目，识别失败模式
    │
    ▼
SANDBOXING         ← evolution.experiment（L23）生成 Proposal 并沙盒验证
    │
    ├─ 沙盒指标不达预期 → REJECTED_BY_SANDBOX（写 S3，反馈给 evolution.experiment（L23））
    │
    ▼
REVIEWING          ← quality.gate（L18）进化评审模式（对比评分）
    │
    ├─ 评分 < 0.5 → REJECTED_BY_QUALITY_GATE（写 S3，反馈给 evolution.experiment（L23））
    │
    ▼
APPROVING          ← release.publish（L22）人工审批等待
    │
    ├─ 审批拒绝 → REJECTED_BY_HUMAN（写 S3）
    │
    ▼
CANARY             ← 灰度上线，观察期
    │
    ├─ 指标异常 → ROLLED_BACK（写 S3，触发 evolution.experiment（L23）重新分析）
    │
    ▼
RELEASED           ← 全量上线，S4 规则版本更新
    │
    ▼
CLOSED（终态，S3 写入发布闭环记录）
```

## 九、与现有架构的交叉引用

| 模块 | 关系 |
|------|------|
| `feedback.loop`（L20）反馈闭环层 | 第一段：生成 experience_entry，写入 S2，判定 `evolution.experiment`（L23）触发条件 |
| `evolution.experiment`（L23）进化实验层 | 第二段：拉取 S2 经验条目，生成 Proposal，沙盒验证 |
| `quality.gate`（L18）质量门禁层 | 第三段：进化评审模式，对比新旧策略评分 |
| `release.publish`（L22）版本发布层 | 第三段：人工审批，灰度，全量，更新 S4 |
| S2 制品库 | experience_entry 写入目标（record_type=experience，永久保留） |
| S3 审计库 | 全链路写入点；correction_trace 的数据来源（`feedback.loop`（L20）从此读取 escalation_id） |
| S4 规则库 | 触发阈值参数存储；`release.publish`（L22）全量后写入规则版本号 |
| `quality.inspector`（L16）Inspector 层 | confidence_delta 字段数据来源 |
