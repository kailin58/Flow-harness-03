# 主题一：3层 → 4层 → 5层 → 6层架构升级

---

> 📌 架构速查 → [2M+23L 完整层级（含子层）](./00_升级总结总览.md#2m--23l-架构速查v230-权威版)
> `layer_id` 永久稳定；`Lxx` 仅为显示编号。核心治理链路（不可绕过）：`security.permissions`(L6) / `security.policy`(L14) / `quality.gate`(L18) / `release.publish`(L22)

## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。
- 治理模型统一采用“规则写死、参数可配、边界写死”：
  - 写死规则：流程门禁、接管链路、审计要求、发布与回滚约束。
  - 可配置参数：SLA、复评阈值、紧急策略、试点范围等运营参数。
  - 边界写死：仅允许边界内收紧，不允许突破平台红线；配置变更必须版本化发布并写审计。
- 接口原则：所有接口不做占位；设计可做全局通用与扩展增强，但接口定义本身必须可直接落地，不保留仅供后补的占位接口。
- 全局配置原则：凡可由配置承载的真实业务值，一律配置化管理，能配置就不写死。
- 落地口径：正文默认只固化字段名、规则、约束、示例值与生效顺序；真实实值通过 `platform -> company -> department -> user` 配置链注入并审计。

## 层级变化

```
旧（3层）          新（4层）           演进（5层）                    当前组织终态（6层）
─────────          ─────────────────── ─────────────────────────── ─────────────────────────────
L0: CEO            L0: CEO             L0: CEO                         L0: CEO
L1: 5总监          L1: 副总（VP层）    L1: 副总（VP层）                 L1: 副总（VP层）
L2: 子Agent        L2: 总监            L2: 总监                         L2: 总监
                   L3: 子Agent         L3: 子Agent                     L3: 经理（M01–M11，实例不限）← 新增
                                       L4: 副Deep Agent                 L4: 子Agent
                                                                        L5: 副Deep Agent（仅挂在 L4 子Agent 下）
```

### L3 经理层 定位

| 属性 | 说明 |
|---|---|
| **挂载位置** | 每个 **L2 总监** 之下；**不得**由总监跳过经理直连子 Agent |
| **命名空间** | **M01–M11** 与 **VP01–VP11** 一一对应；平台不允许第 12 条经理命名空间 |
| **实例数量** | 每条命名空间下 **0..∞** 个经理 Agent（按需创建），不设人数上限 |
| **下一级** | **仅**管理 **L4 子Agent**；**不得**挂载或直连副 Deep |

### L5 副Deep Agent 定位

| 属性 | 说明 |
|---|---|
| **挂载位置** | 每个 **L4 子Agent** 下，按需挂载 |
| **角色** | 深度执行单元，负责单一原子任务的极深递归处理 |
| **与 L4 区别** | L4 子Agent 负责任务拆解与协调；L5 副Deep Agent 只做单点深挖，不再向下分发 |
| **触发条件** | 子Agent 判断某子任务复杂度超阈值时，主动孵化一个副Deep Agent 处理 |
| **生命周期** | 任务型（ephemeral）：任务完成即销毁，不长驻 |

## CEO 预处理管道与路由机制

### 两条入口路线

**路线1：CEO 自动路由（默认）**

```
用户输入（自然语言）
    ↓
识别 user_id → L6 security.permissions（权限管理层）查询该用户的VP访问范围
    ↓
[节点1] 需求精确化
[节点2] 任务拆解
    ↓
CEO 路由决策（Step1-5 能力标签匹配）
    ↓
路由到匹配VP → VP → 总监 → 经理 → 子Agent
```

**路线2：用户显式指定（@ 或 \ 前缀）**

```
用户输入 + @VP名 或 \VP指令
    ↓
L6 security.permissions（权限管理层）校验：该用户是否有权访问此VP
  ├─ 无权限 → 拦截并提示用户
  └─ 有权限 ↓
[节点1] 需求精确化（仍然执行）
[节点2] 任务拆解（仍然执行）
    ↓
跳过 CEO 路由决策（目标VP已明确）
    ↓
直接路由到指定VP → VP → 总监 → 经理 → 子Agent
```

> **路线2 完整预检流程（方案C：转交VP前预检，不匹配直接提示用户）**
>
> ```
> 用户输入 @VP名 或 \VP指令
>     ↓
> 预检1：L6 security.permissions（权限管理层）校验（该用户是否有权访问此VP？）
>   ├─ 无权限 → 拦截，返回：
>   │    "你无权访问 @{VP名}
>   │     你当前可访问的VP（共N个）：
>   │       · VP编号  VP名称
>   │       · ...
>   │     请重新指定 或 直接输入任务由系统自动路由"
>   └─ 有权限 ↓
> 预检2：能力匹配校验（该VP的能力标签是否覆盖本次任务类型？）
>   ├─ 不匹配 → 拦截，返回：
>   │    "@{VP名} 不支持「{任务类型}」类任务
>   │     该VP支持的能力范围：{能力标签列表}
>   │     推荐改用：@{匹配度最高的VP名}
>   │     或直接输入任务由系统自动路由"
>   └─ 匹配 ↓
> [节点1] 需求精确化
> [节点2] 任务拆解
>     ↓
> 直接路由到指定VP → VP → 总监 → 经理 → 子Agent
> ```
>
> **两类提示的数据来源：**
> - 用户可访问VP列表 → L6 security.permissions（权限管理层，按 user_id 查询）
> - VP能力标签 → S4 规则库（VP自己注册的能力范围）
> - 推荐VP → 复用 Step2 能力标签匹配逻辑

### CEO 路由决策树（Step1-5，仅路线1执行）

```
Step1：读任务元数据
  → 任务类型标签（TaskType）
  → 需要能力标签（RequiredCapabilities）
  → 风险等级（RiskLevel）

Step2：匹配能力标签 → VP列表
  能力标签注册在 S4 规则库
  每个VP声明自己覆盖哪些能力

Step3：
  单VP命中 → 直接分配
  多VP命中 → 按"主责VP"原则选最高匹配度1个，其余作协同VP
  零VP命中 → 路由到 VP01（数字技术VP），走通用兜底路径

Step4：风险等级 ≥ 高 → 路由前先经 L6 security.permissions（权限管理层）确认CEO有授权

Step5：分配完成 → 写入 S1上下文库（任务分配记录）

核心原则：CEO 不做"猜"，只做"匹配"
  能力标签由VP自己声明并注册到S4
  新增VP只需更新注册表，不需要改CEO逻辑
```

### CEO 三种工作模式

```
模式A：接收输入模式（主动）
  触发：用户发起新对话/任务
  动作：驱动预处理管道 → 执行路由决策（路线1）或权限校验（路线2）

模式B：全局监听模式（被动）
  触发：任务执行中
  动作：只读取 L13 orchestration.collab 看板数据，不介入VP/总监内部决策
  CEO知道：每个VP的当前状态、整体进度、是否有风险

模式C：介入模式（触发式）
  触发条件（满足任意一个才介入）：
    ① VP之间结论冲突，需要CEO仲裁
    ② L13 orchestration.collab（L13.1 状态机）触发 escalate（死锁/超时）
    ③ 用户发起新一轮对话（目标可能改变）
    ④ 执行中发现高风险操作，风险等级升级
  动作：重新介入决策，可调整路由或终止任务
```

### 预处理管道节点定义

每个会话/任务默认挂载两个节点，顺序固定：

```
[节点1] 需求精确化
  输入：用户模糊描述
  输出：结构化、精确的任务意图
  Sub-functions：
    · 意图识别
    · 歧义消解
    · 上下文补全（读 S1 历史）
    · 约束条件提取

[节点2] 任务拆解
  输入：精确的任务意图
  输出：可分配的子任务列表（含能力标签）
  Sub-functions：
    · 子任务生成
    · 依赖关系识别
    · 能力标签标注（RequiredCapabilities）
    · 优先级排序

[节点N] 平台管理员可添加更多节点...
```

**节点标准字段：**

```
node_id         唯一标识
name            节点名称
order           执行顺序（管道内位置）
status          active / disabled
sub_functions[]
  ├── func_id
  ├── func_name
  ├── enabled     true / false（可单独开关）
  └── config{}    可配置参数（如：歧义消解置信度阈值）
input_schema    节点接收的数据格式
output_schema   节点输出的数据格式
on_failure      skip / retry / escalate
```

## 通信规则（6层版）

### 直接允许
```
CEO      ↔  副总           （上下级直接沟通）
副总     ↔  自己的总监      （上下级直接沟通）
总监     ↔  自己的经理      （上下级直接沟通）
经理     ↔  自己的子Agent   （上下级直接沟通）
子Agent  ↔  自己的副Deep Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
副总 ↔ 副总                        → CEO 强制主持
同VP下总监 ↔ 总监                   → 所属副总强制主持
同总监下经理 ↔ 经理                 → 所属总监强制主持
同经理下子Agent ↔ 子Agent            → 所属经理强制主持
同子Agent下副Deep Agent ↔ 副Deep Agent → 所属子Agent强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信         → DENY，必须上报自己的副总，走 副总↔副总 通道
任何跳级通信             → DENY（CEO 不直达总监/经理/子Agent；副总不直达经理/子Agent；总监不得跳过经理直连子Agent）
副Deep Agent 主动上报    → DENY，只能被动响应上级子Agent，不可主动越级
```

### 跨VP通信正确链路
```
IT总监A 需要和 运营总监B 沟通：
  IT总监A → 上报 IT VP
               └→ IT VP ↔ 运营VP（CEO三方主持）
                              └→ 运营VP 传达 运营总监B
```
**设计原则**：VP是本域对外的唯一出口，不可被绕过（否则VP架空）

## CEO 路由逻辑（补全缺口2）

### 设计原则

> CEO 不做"猜"，只做"匹配"。
> 路由依据来自 VP 自己声明并注册到 S4 规则库的能力标签，CEO 按注册表路由。
> 加新 VP 时只需更新注册表，不需要改 CEO 逻辑。

**核心定性**：CEO 路由 = **规则优先 + 兜底降级**，不是"智能推断"。

---

### CEO 路由决策树（5步）

```
Step 1：读任务元数据
        → 任务类型标签（TaskType）
        → 需要能力标签（RequiredCapabilities）
        → 风险等级（RiskLevel）

Step 2：匹配能力标签 → VP 候选列表
        查询 S4 规则库中各 VP 的能力标签声明
        找到所有 capabilities 包含 RequiredCapabilities 的 VP

Step 3：路由决策
        ┌ 单VP命中  → 直接分配给该 VP
        ├ 多VP命中  → 按"主责VP"原则选最高匹配度的1个
        │             其余作协同VP（通过 CEO 三方主持通道协作）
        └ 零VP命中  → 兜底路由到 VP01（数字技术VP）走通用兜底路径

Step 4：风险门控
        RiskLevel ≥ 高 → 路由前先经 L6 security.permissions（权限管理层）确认 CEO 有授权
        授权通过 → 继续执行分配
        授权拒绝 → 任务挂起，写 S3 审计库，通知请求方

Step 5：分配完成
        → 写入 S1 上下文库（任务ID / 分配的VP / 协同VP列表 / 路由依据 / 时间戳）
```

---

### 多VP命中时：主责VP vs 协同VP 执行模型

**匹配度量化规则**（用于 Step 3 选主责VP）：

| 匹配情况 | 匹配度得分 |
|---|---|
| `capabilities` 完全包含 RequiredCapabilities | 100分 |
| `primary_tasks` 包含当前 TaskType | +20分加权 |
| `max_concurrent` 未满 | 可选，否则降分 |
| `fallback_priority` 越小 | 同分时优先 |

得分最高者 = **主责VP**（串行接收完整任务指令，负责任务交付）
其余命中VP = **协同VP**（按需触发，只接收需要跨域支援的子任务，不负责整体交付）

```
执行模型：
  主责VP串行推进主任务
       │
       ├─ 遇到需要协同VP能力时 → CEO 发起三方会话 → 协同VP处理子任务
       │                                               └→ 结果返回主责VP
       └─ 主责VP整合所有结果 → 上报CEO
```

**关键约束**：协同VP不接受主责VP直接指挥，必须通过 CEO 三方主持通道，保持层级规则一致。

### 路由失败与降级策略（完整版）

| 失败场景 | 触发条件 | 降级行为 |
|---|---|---|
| **零VP命中** | S4中无VP的capabilities匹配 | 路由到VP01（数字技术VP）走通用兜底路径 |
| **命中VP已满载** | 目标VP的`max_concurrent`已达上限 | 按`fallback_priority`顺序找下一个候选VP；全部满载则进入排队队列 |
| **VP不可用（离线/故障）** | VP健康检查失败 | 跳过该VP，按匹配度找次优VP；写S3异常记录；通知运维告警 |
| **VP01兜底也不可用** | 所有VP均不可用 | 任务挂起，写S3审计库，返回用户"系统繁忙"，等待VP恢复后自动重试 |
| **L6 security.permissions 授权拒绝（高风险）** | CEO无权限路由该高风险任务 | 任务挂起，写S3审计库，通知有权限的管理员审批，审批通过后重新触发路由 |
| **S4规则库不可读** | 能力标签注册表查询失败 | 停止路由，不做兜底猜测，返回错误，写S3审计库 |

**降级优先顺序**：
```
命中VP可用 → 命中VP满载找次优 → 零命中走VP01兜底 → VP01不可用进排队 → 全部不可用任务挂起
```

**不允许的降级行为**：
- CEO不允许在S4不可读时"猜测"路由给任何VP（防止路由逻辑退化为黑盒）
- 降级路径本身必须写S3审计，保证可追溯

### VP 能力标签注册规范

每个 VP 在 **S4 规则库** 中声明自己的能力标签，CEO 路由完全依赖此注册表。

| 字段 | 说明 |
|---|---|
| `vp_id` | VP 唯一标识 |
| `capabilities` | 该VP覆盖的能力标签列表（如 `["code", "deploy", "debug"]`） |
| `primary_tasks` | 该VP的主责任务类型列表 |
| `fallback_priority` | 兜底优先级（数字越小越优先作为兜底VP）|
| `max_concurrent` | 最大并发任务数（超出则路由给下一优先VP）|

**更新规则**：
- 能力标签变更必须走 L22 release.publish（版本发布层）流程，不允许直接覆盖生产注册表
- 每次变更自动触发 S3 审计记录

---

### CEO 全局感知与不干预原则

CEO 需要把握全局、了解发生了什么，并做出整体决策，但**不应过度干预**下层执行。

| 行为 | CEO应做 | CEO不应做 |
|---|---|---|
| 任务接收 | 读取任务元数据、判断路由 | 直接分配给总监、经理或子Agent（跳级） |
| 跨VP协调 | 主持 VP ↔ VP 三方会话 | 替VP做决策、直接指挥VP下的总监 |
| 风险管控 | 高风险任务确认授权 | 绕过 L6 security.permissions（权限管理层）自行放行 |
| 全局感知 | 读取 S1 上下文库了解全局状态 | 实时介入每个VP的执行细节 |
| 异常处理 | 收到VP上报后决策是否重新路由 | 主动轮询经理/子Agent执行细节 |

---

### 完整对话执行流程（含两个预设节点）

整个流程描述：CEO 通过用户界面直接与用户聊天交互，判断任务应传给哪个 VP；VP 接到指令后将任务分配给对应总监；总监将任务分派给下属 **经理**；经理再分派给 **子 Agent** 执行；子 Agent 在需要时孵化 **副 Deep** 处理原子任务。

```
用户（UI聊天界面）
        ↓ 输入任务/指令
┌──────────────────────────────────────────────────────┐
│  预设节点 A（可配置）                                  │
│  功能示例：任务裁剪 / 意图标准化 / 元数据注入           │
│  位置：CEO 接收用户输入之前                             │
└──────────────────────────────────────────────────────┘
        ↓
   CEO（L0）
   ├─ 全局感知：读 S1 上下文库，了解当前任务链状态
   ├─ 执行路由决策树（5步）
   └─ 分配给目标 VP，写 S1 分配记录
        ↓
┌──────────────────────────────────────────────────────┐
│  预设节点 B（可配置）                                  │
│  功能示例：任务优先级排序 / 并发控制 / SLA预检          │
│  位置：CEO 路由完成、VP 接收任务之前                    │
└──────────────────────────────────────────────────────┘
        ↓
   副总 VP（L1）
   └─ 将任务拆解分配给下属总监
        ↓
   总监（L2）
   └─ 将子任务分派给下属经理
        ↓
   经理（L3）
   └─ 分派给子Agent执行
        ↓
   子Agent（L4）
   └─ 必要时孵化 副Deep Agent（L5）处理原子任务
```

**两个预设节点说明：**

| 节点 | 位置 | 默认行为 | 可配置功能方向 |
|---|---|---|---|
| **节点A** | 用户输入 → CEO 之间 | 透传（不处理） | 任务裁剪、意图标准化、任务模板匹配、元数据注入 |
| **节点B** | CEO路由完成 → VP接收之间 | 透传（不处理） | 优先级排序、并发限流、SLA预检、风险二次确认 |

- 每个节点内部功能拆分方式类似 **Skill（技能）** 概念：节点由多个可组合的 Skill 组成，可按业务需要启用/禁用单个 Skill
- 节点之间相互独立，节点A的输出即节点B的输入上游
- 节点配置变更走 L22 release.publish（版本发布层）流程，不允许运行时热修改

## Skills 与 VP 归属（补全缺口3）

### 核心设计原则

> Skills 按**调用者归属**分层，而非按功能归属。
> 每个 Agent 完全管理自己的 Skills，彼此隔离。
> 通用 Skills 可发布到 Skills 平台供共享，但调用权限仍受归属层级约束。

---

### 三级归属模型

#### Level 1：平台级 Skill（Platform Skill）

| 属性 | 说明 |
|---|---|
| **调用范围** | 任意 VP、任意总监、任意经理、任意子 Agent（含副 Deep）均可调用（仍受 L6 security.permissions（权限管理层）作用域约束） |
| **注册与发布责任** | VP02（平台治理VP）负责注册与发布 |
| **典型示例** | 文件读写、代码执行、HTTP请求、日志写入 |
| **隔离要求** | 调用时仍受 L6 security.permissions（权限管理层）**四级主体**作用域约束，**跨企业（`tenant_id`）** 调用默认拒绝 |

#### Level 2：VP级 Skill（VP Skill）

| 属性 | 说明 |
|---|---|
| **调用范围** | 仅该 VP 及其下属总监、经理与子 Agent 可调用 |
| **注册与发布责任** | 对应 VP 自己负责注册与发布 |
| **典型示例** | CRM数据读取（归销售增长VP）、模型调用封装（归智能VP） |
| **隔离要求** | 其他 VP 无法调用，跨VP使用须通过平台级重新发布 |

#### Level 3：总监级 Skill（Director Skill）

| 属性 | 说明 |
|---|---|
| **调用范围** | 仅该总监及其下属经理与子 Agent 可调用 |
| **注册与发布责任** | 总监负责注册，所属 VP 审批后发布 |
| **典型示例** | 特定数据库的查询模板、专属格式化工具 |
| **隔离要求** | 同VP下其他总监无法调用；总监离线则其 Skill 暂停可用 |

---

### Agent 级完全隔离

**每个 Agent 独立管理自己的 Skills 集合，彼此完全隔离。**

```
CEO（L0）          → 不挂载 Skill，只做路由与感知
副总 VP（L1）      → 挂载 VP级 Skill
总监（L2）         → 挂载 总监级 Skill + 可调用 VP级 / 平台级
经理（L3）         → 可调用上级开放的 Skill；是否挂载私有 Skill 由实现决定（默认轻量调度）
子Agent（L4）      → 挂载 Agent私有 Skill + 可调用上级开放的 Skill
副Deep Agent（L5） → 挂载 Agent私有 Skill + 只可调用所属子Agent开放的 Skill
```

**隔离规则：**
- 同级 Agent 之间的 Skill 默认不可见、不可调用
- 下级无法调用上级未显式开放的 Skill
- 同一经理下的两个子Agent，互相无法直接调用对方的 Skill（必须通过经理协调）

---

### Skills 平台（共享发布中心）

通用 Skill 可发布到 **Skills 平台**，避免重复开发。发布后变为 Platform Skill 或 VP Skill，调用权限随归属层级走。

#### Skills 平台注册字段

| 字段 | 说明 |
|---|---|
| `skill_id` | 全局唯一标识 |
| `version` | 语义化版本号 |
| `owner_type` | 归属类型：`platform` / `vp` / `director` |
| `owner_id` | 归属主体 ID（VP ID 或 总监 ID） |
| `published_by` | 发布者 Agent ID |
| `scope` | 可调用范围：`global` / `vp:{vp_id}` / `director:{dir_id}` |
| `dependencies` | 依赖的其他 Skill ID 列表 |
| `risk_level` | 低 / 中 / 高 / 极高 |
| `call_count` | 累计被调用次数（平台统计） |
| `active_agents` | 当前挂载此 Skill 的 Agent 数量 |
| `last_published_at` | 最近发布时间 |

#### Skills 使用频率统计（运营数据）

| 统计维度 | 说明 |
|---|---|
| 每个 Agent 挂载了多少个 Skill | 监控 Skill 堆积与冗余 |
| 每个 Skill 被调用次数 | 判断 Skill 存活价值 |
| 每个 Skill 被多少 Agent 使用 | 高复用 Skill 优先升为 Platform Skill |
| 发布者维度 | 谁发布了多少 Skill，发布质量如何 |
| 僵尸 Skill 检测 | 90天零调用自动标记，等待下线审批 |

统计数据写入 **S3 审计库**，技能调用记录由 L14 security.policy / L15 execution.monitor 写入（现有机制已覆盖）。

---

### Skills 依赖管理

每个 Skill 在注册时声明 `dependencies[]`，发布时由 **L18 quality.gate（质量门禁层）**执行依赖树检查。

**检查规则：**
- 禁止循环依赖（A依赖B，B依赖A → 拒绝发布）
- 依赖的 Skill 必须已发布且版本稳定
- 依赖的 Skill 归属层级不得高于当前 Skill（总监级 Skill 不得依赖另一个总监级 Skill，除非同属一个总监）

---

### 用户 / 企业 Skills 隔离模型

所有 Skill 归属明确区分 `SYS域` 和 `USER域`，两者完全隔离：

```
SYS域（系统预设）
  ├── Platform Skill（VP02管理，平台内置）
  └── 各VP / 总监的系统预设 Skill

USER域（用户/企业自建）
  ├── 企业级 Skill（企业管理员发布，仅本企业可用）
  │     └── 企业VP级 / 企业总监级 按同等三级归属模型管理
  └── 用户级 Skill（个人用户发布，仅本人 Agent 可用）
```

**隔离约束：**

| 行为 | 是否允许 |
|---|---|
| 用户 Skill 调用同企业其他用户的 Skill | ❌ 默认禁止 |
| 企业 Skill 读取其他企业的 Skill | ❌ 完全禁止 |
| 用户 Skill 申请升级为企业级 Skill | ✅ 需企业管理员审批 + L22 release.publish（版本发布层）流程 |
| 企业 Skill 申请升级为 Platform Skill | ✅ 需平台审核 + VP02 审批 + L22 release.publish（版本发布层）流程 |
| SYS域 Skill 被 USER域 调用 | ✅ Platform Skill 可调用；VP/总监级需显式授权 |
| USER域 Skill 写入 SYS域 | ❌ 绝对禁止 |

---

### Skills 治理链路总结

```
开发者写 Skill
    ↓
L23 evolution.experiment（进化实验层）（功能验证 + 安全隔离测试）
    ↓
L18 quality.gate（质量门禁）（依赖树检查 + 风险评级 + 兼容性检查）
    ↓
L22 release.publish（版本发布层）（版本化 + 审批 + 灰度 + 正式发布）
    ↓
Skills 平台注册表（S4 规则库）写入
    ↓
Agent 挂载调用（受 L6 security.permissions（权限管理层）+ L14 security.policy（安全策略层）约束）
    ↓
S3 审计库记录调用行为（频率统计 + 追溯）
```
