# 17_第三方通讯软件接入规范

**性质**：外部通讯接入规范（V1）  
**目标**：在不破坏治理链的前提下，安全接入第三方通讯软件（企业IM/邮件/短信/机器人）  
**归属端口**：`Protocol Port`（外部协议）+ `Tool Port`（发送工具）

---

## 一、接入范围与目标

支持的外部通讯类型（示例）：

- 企业即时通讯（如企业微信、微信,钉钉、飞书、Slack、Teams）
- 邮件通道（SMTP/API网关）
- 短信/语音通知通道
- 第三方机器人Webhook通知

核心目标：

1. 通知可达（状态、审批、异常、回滚）
2. 回执可追（发送结果、失败原因、重试轨迹）
3. 治理不绕过（外部发送仍受 `L4/L8/L12/L15` 约束；layer_id：`security.permissions`/`security.policy`/`quality.gate`/`release.publish`）

---

## 一点一、首批接入清单（写死）

首批通讯软件固定为：

1. 微信（企业微信）
2. 个人微信
3. 飞书
4. 钉钉

统一要求：

- 统一通道抽象：`channel_type=im`
- 统一消息任务模型与回执模型
- 统一审批门禁与审计追踪（`trace_id`）
- 统一失败重试与降级策略

---

## 一点二、双轨接入方案（写死）

第三方通讯接入仅允许两条标准路径：

### 方案A：API接入（优先）

- 方式：调用官方开放API / 官方Webhook能力
- 优点：稳定、可观测、可追踪、易治理
- 适用：有官方开放平台能力、可获取应用凭据、支持回执查询

### 方案B：桌面安装镜像接入（受控）

- 方式：在受控电脑安装官方客户端，通过镜像执行实现登录与沟通
- 约束：必须在隔离执行环境中运行，不得直接开放宿主机权限
- 适用：短期无可用API、或API能力不足以覆盖关键场景

### 方案B定位说明（写死）

方案B属于“接管软件执行逻辑”，本质为受控RPA/桌面自动化，不等同于原生API接入。

接管边界：

1. 仅接管“通讯发送与回执获取”相关动作，不得扩展到无关系统操作。
2. 仅在白名单软件与白名单账号下执行，不得复用个人账号或越权登录。
3. 仅在审批通过的任务中启用，不得默认全量开启。

禁止动作：

1. 禁止读取本地非授权文件、系统凭据、浏览器保存密码。
2. 禁止绕过人工审批执行高风险消息。
3. 禁止把桌面接管结果直接写为最终业务结论，必须回写治理链复核（经 L10 Inspector + L12 质量门禁）。

退出条件：

1. API路径恢复可用且稳定后，必须执行“B -> A”回切。
2. 桌面接管故障率、人工介入率连续超阈值时，必须暂停B并转人工通道。
3. 任一安全审计不通过时，立即冻结B通道并复盘。

### 选择规则（写死）

1. 默认优先方案A（API接入）；仅在A不可行时使用方案B。
2. 方案B必须补充人工值守与风险预案，且通过专项审批后方可启用。
3. 方案B产生的操作轨迹必须全量留痕（截图/日志/审计ID）。
4. 一旦API能力可用，必须评估并迁移回方案A。

---

## 二、接入原则（写死）

1. 所有第三方通讯接入视为外部不可信边界，默认最小权限。
2. 仅允许“单向发送 + 回执回传”，禁止外部组件主动读取内部数据。
3. 凭据必须通过 `credential_ref` 引用，不得明文写入配置或日志。
4. 高风险消息（支付、发布、删除、权限变更）必须走审批链后发送。
5. 发送失败不得阻断主业务流程，必须走异步重试与降级通道。

---

## 三、最小接入模型

### 3.1 通讯通道对象

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `channel_id` | string | ✅ | 通讯通道唯一ID |
| `channel_type` | enum | ✅ | `im` / `email` / `sms` / `webhook` |
| `provider` | string | ✅ | 服务商标识 |
| `credential_ref` | string | ✅ | 凭据引用 |
| `status` | enum | ✅ | `draft` / `active` / `frozen` / `retired` |
| `risk_level` | enum | ✅ | 风险等级 |
| `retry_policy` | object | ⭕ | 重试策略 |
| `rate_limit` | object | ⭕ | 限流策略 |

### 3.3 渠道适配最小能力（微信/飞书/钉钉）

| 渠道 | 最小能力要求 |
|---|---|
| 微信（企业微信） | 应用机器人发送、部门/用户定向发送、回执回写 |
| 飞书 | 机器人Webhook发送、会话定向发送、消息ID回执 |
| 钉钉 | 机器人发送、群/用户定向发送、发送结果回执 |

适配说明：

1. 渠道差异通过 Adapter 吸收，不暴露到业务编排层。
2. 上层仅使用统一接口，不写死厂商SDK细节。
3. 每个渠道需同时定义 API 路径与桌面镜像路径的可用性状态（`api_ready` / `desktop_ready`）。

### 3.2 消息任务对象

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `message_task_id` | string | ✅ | 消息任务ID |
| `channel_id` | string | ✅ | 通道ID |
| `message_type` | enum | ✅ | `notice` / `approval` / `alert` / `report` |
| `template_id` | string | ⭕ | 模板ID |
| `payload_ref` | string | ✅ | 消息体引用 |
| `trace_id` | string | ✅ | 审计追踪ID |
| `delivery_status` | enum | ✅ | `PENDING` / `SENT` / `FAILED` / `RETRYING` / `DEGRADED` |
| `ack_ref` | string | ⭕ | 回执引用 |

---

## 四、标准流程（写死）

1. 创建消息任务 -> 校验消息风险等级与审批要求
2. 通过门禁后异步发送（不阻塞主流程）
3. 接收回执并回写 `delivery_status`
4. 失败进入重试，超过阈值触发降级与人工接管
5. 所有动作写审计事件并可在前端看板追踪

## 六、错误码（最小集）

| 错误码 | 说明 | 默认动作 |
|---|---|---|
| `E_COMM_CHANNEL_NOT_FOUND` | 通道不存在 | 终止并告警 |
| `E_COMM_CREDENTIAL_MISSING` | 凭据缺失 | 阻断发送并审计 |
| `E_COMM_PERMISSION_DENIED` | 权限不足 | 拒绝并审计 |
| `E_COMM_RATE_LIMITED` | 命中限流 | 延迟重试 |
| `E_COMM_DELIVERY_FAILED` | 发送失败 | 重试/降级 |
| `E_COMM_ACK_TIMEOUT` | 回执超时 | 转人工接管 |
| `E_COMM_API_UNAVAILABLE` | API不可用 | 评估降级至桌面镜像 |
| `E_COMM_DESKTOP_SESSION_INVALID` | 桌面会话失效 | 重新登录并人工确认 |

---

## 七、验收标准（DoD）

1. 首批完成微信（企业微信）、飞书、钉钉三通道接入并稳定发送。
2. 支持消息状态全链路追踪（含回执、重试、降级）。
3. 高风险消息具备审批门禁，未审批无法发送。
4. 失败场景具备异步重试与人工接管机制。
5. 前端看板可查看发送状态与审计证据。
6. 三通道均完成“双轨预案”（API主路径 + 桌面镜像备路径）并完成演练。
