# 20_消费互联网平台接入规范

**性质**：外部系统接入规范（T9 消费互联网平台适配器）  
**目标**：规范接入消费级互联网平台（资讯/社交/电商），填补 T1-T8 无法覆盖的缺口  
**归属端口**：`Tool Port`（主）+ `Protocol Port`（社交平台 API）  
**上位依赖**：[`./19_层级扩展与跨层通讯协议.md`](./19_层级扩展与跨层通讯协议.md)、[`./17_第三方通讯软件接入规范.md`](./17_第三方通讯软件接入规范.md)、[`./00_升级总结总览.md`](./00_升级总结总览.md)

---

## 一、为什么需要 T9

现有 T1-T8 工具适配器的覆盖边界如下：

| 适配器 | 覆盖类型 | 不覆盖 |
|---|---|---|
| T4 企业系统 | ERP/CRM/OA 等**企业内部**系统 | 淘宝/京东/小红书等**消费级**平台 |
| T5 API 服务 | 通用 REST/GraphQL 调用 | 需登录态、账号绑定的平台操作 |
| T6 搜索引擎 | 关键词查询 | 内容订阅/主动推送/平台内发帖 |
| T8 自动化平台 | Zapier/n8n 等 RPA 编排 | 需账号身份的平台原生操作 |

**消费互联网平台的独特性：**

1. **账号绑定**：必须以特定账号身份登录并保持 Session
2. **操作频率限制**：平台有反自动化机制，超频会触发封号风险
3. **接口高度不稳定**：平台 UI 和 API 频繁变更，适配器需有版本管理
4. **内容合规要求**：发布操作受平台内容政策约束
5. **数据隐私敏感**：用户数据和账号凭据安全要求高于企业内部系统

## 二、T9 适配器分类与覆盖范围

T9 下分三个子类，每个子类有独立接入方案和风险等级：

### T9.1 内容资讯平台（ContentPlatformAdapter）

**覆盖平台（示例）**：今日头条、36kr、虎嗅、B站、知乎、微信公众号文章

**核心操作**：

| 操作 | 方向 | 说明 |
|---|---|---|
| 内容订阅 | 读取 | 关注特定账号/话题，获取最新内容 |
| 内容抓取 | 读取 | 按关键词/URL 抓取文章/视频摘要 |
| 内容监控 | 读取 | 定时轮询，检测新内容并推送 |
| 内容归档 | 写入S2 | 将内容存入 S2 制品库作为知识资产 |

**接入方式优先级**：
```
A. 平台开放 API（RSS/OpenAPI）← 优先
B. 网页抓取（§5.5 网页识别引擎 + L9 浏览器）← 次选
C. 不涉及写入操作，最大风险等级：中
```

---

### T9.2 社交媒体平台（SocialMediaAdapter）

**覆盖平台（示例）**：微博、小红书、抖音、Twitter/X、Instagram

**核心操作**：

| 操作 | 方向 | 风险等级 | 需审批 |
|---|---|---|---|
| 内容发布（图文/视频） | 写入 | 高 | ✅ |
| 定时发布 | 写入 | 高 | ✅ |
| 评论/互动 | 写入 | 中 | ⭕（批量时必须） |
| 粉丝/数据分析 | 读取 | 低 | 否 |
| 私信发送 | 写入 | 高 | ✅ |
| 账号状态监控 | 读取 | 低 | 否 |

**账号登录态管理（关键约束，写死）**：

```
1. 每个平台账号对应一个独立 session_profile，存 S4 规则库
2. Session 不得共享（账号A的session不得被账号B的任务使用）
3. Session 过期后必须人工重新授权，不得自动替换凭据
4. 同一账号并发操作上限 = 1（串行，防封号）
5. 操作间隔：写入操作必须携带 min_interval_ms（参数可配，默认 3000ms）
```

**多平台统一接口抽象**：

```json
{
  "publish_content": {
    "platform":     "weibo | xiaohongshu | douyin | twitter",
    "account_id":   "账号ID（绑定到 session_profile）",
    "content_type": "text | image | video | mixed",
    "text":         "正文内容",
    "media_refs":   ["S2 制品库中的媒体文件引用"],
    "scheduled_at": "定时发布时间（可选）",
    "tags":         ["话题标签"],
    "trace_id":     "审计追踪ID"
  }
}
```

**平台差异吸收**：各平台的字段限制、媒体要求、API 差异由 Adapter 内部吸收，不暴露到编排层。

---

### T9.3 电商平台（EcommercePlatformAdapter）

**覆盖平台（示例）**：淘宝/天猫、京东、拼多多、抖音电商、快手小店

> **与 T4 的区别**：T4 企业系统适配器覆盖**企业内部**的 ERP/CRM（如 SAP/Salesforce），T9.3 覆盖**消费级电商平台**的店铺运营操作。两者不重叠。

**核心操作**：

| 操作类别 | 具体操作 | 风险等级 | 需审批 |
|---|---|---|---|
| 商品管理 | 上架/下架/改价/改库存/改标题描述 | 高 | ✅ |
| 订单处理 | 确认发货/取消订单/退款审批 | 极高 | ✅ 人工双签 |
| 营销活动 | 报名活动/设置优惠/推广投放 | 高 | ✅ |
| 数据分析 | 销量/流量/转化率/竞品分析 | 低 | 否 |
| 客服工具 | 发送模板消息/自动回复 | 中 | ⭕ |
| 内容发布 | 主图/详情页/视频上传 | 高 | ✅ |

**多平台操作统一接口（写死）**：

```json
{
  "product_operation": {
    "platform":    "taobao | jd | pinduoduo | douyin_shop",
    "account_id":  "店铺账号ID",
    "operation":   "publish | update_price | update_stock | takedown",
    "product_id":  "商品ID（平台侧）",
    "payload":     "操作内容",
    "trace_id":    "审计追踪ID",
    "idempotency_key": "幂等键（防重复提交）"
  }
}
```

**电商操作硬规则（写死）**：

1. 所有价格变更必须附带**前后值对比**并写 S3 审计
2. 批量改价（>10 个 SKU）必须人工二次确认
3. 订单退款操作必须人工双签，不允许 Agent 自动执行
4. 库存操作必须使用幂等键，防止网络重试导致重复扣减
5. 高风险操作（下架、清仓、批量推广）必须经 L12 门禁 + L15 发布链

## 三、接入方式（三类统一规则）

### 3.1 双轨接入方案（与 17_通讯文档保持一致）

**方案 A：平台开放 API（优先）**
- 通过平台官方开放平台 API（OAuth2.0 + 官方 SDK）
- 适用：平台有开放 API 且覆盖目标操作

**方案 B：受控浏览器/桌面执行（备用）**
- 通过 L9 执行监控层的 smart-browser.js + §5.5 网页识别
- 适用：目标操作无 API，或 API 限制严重
- 约束：必须在白名单域名内执行，操作全程截图留痕

### 3.2 方案选择规则（写死）

```
1. 优先方案A；仅当A不可行时启用B
2. 方案B须经专项审批（L15 发布流程）
3. 方案B产生的所有操作截图/日志写 S3 审计（异步）
4. 一旦API能力可用，30天内必须评估并迁移回方案A
```

## 四、T9 适配器注册字段（继承工具注册表通用字段，新增）

在 `00_升级总结总览.md` 工具注册表基础上，T9 类适配器新增以下字段：

| 字段 | 说明 |
|---|---|
| `platform_type` | `content_platform` / `social_media` / `ecommerce_platform` |
| `platform_id` | 平台标识（如 `taobao`、`xiaohongshu`） |
| `account_binding_required` | 是否需要账号绑定（T9 全部为 true） |
| `session_profile_id` | 绑定的 session_profile（S4 存储） |
| `max_concurrent_ops` | 同一账号最大并发操作数（默认 1） |
| `min_op_interval_ms` | 相邻操作最小间隔（防封号，参数可配） |
| `write_ops_require_approval` | 写入操作是否强制审批（默认 true） |
| `anti_detection_profile` | 反检测策略配置（鼠标移动/延迟随机化等，参数可配） |

## 五、层归属与治理链

### 5.1 各操作在 16 层中的归属

```
T9 操作流程（以社交媒体内容发布为例）：

用户指令
  ↓
L4（security.permissions）     权限校验：有无该平台账号的操作授权
  ↓
L6（orchestration.task）       任务编排：拆解发布子任务，注入内容合规规则
  ↓
L8（security.policy）          安全策略：内容敏感度检测、域名白名单验证
  ↓
L9（execution.monitor）        执行：调用 T9.2 Adapter，发布内容
                                    smart-browser.js / site-strategy.js
  ↓
L10（quality.inspector）       检查：发布结果验证、回执确认
  ↓
L12（quality.gate）            门禁：确认发布成功，若失败打回 L9 重试
  ↓
L13（feedback.loop）           反馈：发布结果写 S2，经验沉淀
```

### 5.2 各子类最高风险等级与必经治理层

| 子类 | 最高风险 | 必经治理层（layer_id） |
|---|---|---|
| T9.1 内容资讯（只读） | 中 | `security.permissions` / `security.policy` |
| T9.2 社交媒体（写入） | 高 | `security.permissions` / `security.policy` / `quality.gate` / `release.publish` |
| T9.3 电商平台（写入） | 极高 | 全部四个核心治理层 + 人工双签 |

## 六、账号安全约束（写死，不可配置化降低）

1. **凭据隔离**：账号密码/Token 必须存 S4 规则库的 `credential_ref`，不得明文写入任何配置、日志、消息体。
2. **Session 隔离**：不同租户的账号 Session 完全隔离，不得共享或迁移。
3. **最小权限**：只申请完成目标任务所需的最小平台权限范围，不申请超出范围的 scope。
4. **操作留痕**：所有写入操作必须保留：操作时间、操作内容快照、操作结果、审计ID，永久存 S3。
5. **封号应急**：任一账号触发平台封号/风控告警 → 立即冻结该账号所有 session_profile → 人工介入 → 写 S3 事件。
6. **多账号隔离**：同一平台的不同账号（如多个淘宝店铺）各自独立 session_profile，不交叉执行。

## 八、与 17_第三方通讯软件接入规范 的关系

| 维度 | 17_通讯 | 20_消费互联网（本文） |
|---|---|---|
| 核心场景 | 发通知/审批消息/报告 | 内容发布/运营操作/数据采集 |
| 信息流向 | 系统 → 人（通知） | 系统 ↔ 平台（双向操作） |
| 账号性质 | 机器人账号（企业应用） | 业务操作账号（店铺/媒体号） |
| 主归属层 | L1.3 外部平台互操作子层 | L9 执行监控层（Tool Port） |
| 最高风险 | 高（高风险消息需审批） | 极高（电商写入需人工双签） |

**使用原则**：
- 在社交/电商平台**发消息通知**（如订单提醒）→ 走 `17_` 通讯规范
- 在社交/电商平台**执行业务操作**（如发帖、改价）→ 走本文档 `20_` 规范
