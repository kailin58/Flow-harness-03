# 15_全链路业务编排能力规范

**性质**：业务编排能力规范（目标驱动）  
**目标**：平台具备“从任务输入到持续运营优化”的全链路自主执行能力  
**适用范围**：电商、内容运营、门店运营、跨平台投放等可流程化业务

---

## 一、目标能力定义

当用户发起任务（例如“帮我开一家服装店：某网站拉货，某平台开店”）时，系统必须可完成以下闭环：

1. 目标澄清与约束识别
2. 调研与市场分析
3. 品牌与选品策略生成
4. 文案与素材生产
5. 商品发布与渠道分发
6. 运营监控与效果复盘
7. 持续学习与策略优化

---

## 一点一、虚实边界原则（写死）

1. 虚拟层仅指 Agent 的呈现形态与交互空间（角色、场景、看板叙事）。
2. 业务层必须是真实业务：真实目标、真实流程、真实数据、真实结果。
3. 虚拟呈现不得替代真实业务状态，不得覆盖真实治理链与审计链。
4. 一切决策、发布、审批、回滚、经营结论以后端真实状态与审计证据为准。

---

## 一点二、1+1+1 架构原则（写死）

明确采用 `1+1+1` 结构：

1. `1` 前端服务（展示与交互层）
2. `1` 后端服务（业务编排与治理层）
3. `1` 数据库服务（独立数据层）

硬约束：

1. 前端不得直连数据库，所有读写必须通过后端API。
2. 三层必须具备故障隔离能力，单层故障不得直接拖垮全链路。
3. 三层必须具备降级策略（如只读模式、人工接管、消息延迟重试）。
4. 逻辑分层固定为 `1+1+1`，物理部署可单机（开发/演示）或分离（生产推荐）。

---

## 二、全链路标准流程（写死）

### 2.0 学习前置层（先学习再执行）

- 任何新任务先进入“规则学习与编排演练”，不得直接跳执行。
- 学习对象：平台规则、目标平台政策、类目规范、历史案例、失败样本。
- 学习输出：`learning_pack`（规则摘要、禁用动作、必审节点、风险清单）。
- 准入门槛：未产出 `learning_pack` 且未通过演练，不得进入发布执行层。

### 2.1 任务接入层

- 输入：业务目标、预算范围、时间窗口、平台与渠道约束
- 输出：结构化目标卡（`goal_id`、`constraints`、`success_criteria`）

### 2.2 调研分析层

- 自动收集：品类趋势、竞品价格带、爆款特征、平台规则
- 产出：调研报告（机会点、风险点、建议优先级）

### 2.3 方案编排层

- 自动生成执行方案：品牌定位、选品策略、上新节奏、投放节奏
- 输出可执行编排图：支持人工确认节点与片段拼接
- 必须先完成“沙盘演练”：按平台规则模拟一轮完整执行并产出演练报告

### 2.4 内容生产层

- 生成：商品标题、卖点、详情文案、素材脚本（图/视频/音频）
- 规则：必须过质量门禁与合规检查后进入发布队列
- 视觉识别：素材入库前必须完成“清晰度/合规/品牌一致性/风险元素”识别

### 2.5 发布执行层

- 对接目标平台：商品创建、信息填充、价格库存同步、发布动作
- 异常处理：失败重试、熔断、人工接管、回滚
- 支付相关动作（扣费、打款、退款）默认人工干预，不允许全自动放行
- 电脑与软件操作：支持在受控边界内执行浏览器、运营后台、办公软件自动化流程
- 网页识别：执行前后自动识别页面结构、关键字段与状态码，避免误点/误填/误发布

### 2.5.1 经营策略层（定价 / 售后 / 客服）

- 定价策略：按成本、竞品、毛利目标生成价格建议与调整区间
- 售后策略：退换货规则、时效承诺、补偿策略、风险升级路径
- 客服策略：FAQ 自动应答 + 人工接管分流 + 异常工单升级
- 所有策略调整必须留痕并回写经营结果（转化、退货率、满意度、ROI）

### 2.6 运营优化层

- 监控指标：曝光、点击、转化、退货率、ROI、Token 成本收益
- 自动优化：标题/素材/价格策略 A/B/C 试跑与择优
- 视觉优化：基于素材表现与视觉识别结果自动生成替换建议（主图、细节图、短视频封面）

### 2.7 经验沉淀层

- 结果回写：策略版本、适用条件、失败原因、收益提升幅度
- 能力进化：仅在隔离域实验，通过门禁后发布

## 三、角色分工（CEO -> VP -> 总监 -> 经理）

- `CEO`：下达经营目标与预算边界，审核全局 ROI 与止损策略
- `VP`：拆解域级目标并组织跨团队资源协同
- `总监`：形成可执行业务方案并把控质量/成本平衡
- `经理`：推进迭代执行、状态回写、异常清零与收口
- `子Agent/副Deep`：执行具体作业并提供证据回传

### 3.1 分工协作硬规则（写死）

1. 单 Agent 单一任务：每个 Agent 同时只承接一个原子任务，不混合承担多个主责。
2. 主责唯一：每个原子任务只允许一个主责 Agent，其余 Agent 仅作为协作或验收角色。
3. 逐级协作：任务必须按“总监 -> 经理 -> 子Agent（-> 副Deep）”下发与回传，不得跳级。
4. 全局复用优先：编排前先检索可复用能力包，命中则复用，不重复造轮子。
5. 复用留痕：所有复用能力必须记录 `skill_id/version/scope_level` 与调用证据。

---

## 四、最小能力清单（必须具备）

1. 目标理解与约束抽取
2. 外部调研与结构化报告生成
3. 多步骤流程编排（支持人工确认）
4. 多模态内容生成与质检
5. 多平台发布适配与失败回滚
6. 视觉识别能力（图像/OCR/视频关键帧/合规识别/质量识别）
7. 支付人工干预机制（扣费/打款/退款三类动作）
8. 定价策略引擎（建议价、区间、调价触发条件）
9. 售后与客服编排能力（自动分流 + 人工升级）
10. 指标监控与经营归因（含 Token 投入产出）
11. 自主学习与策略版本化优化
12. 电脑操作能力（桌面流程执行、文件与窗口控制、操作回放）
13. 软件使用能力（浏览器/办公软件/运营后台自动化执行）
14. 网页识别能力（DOM语义识别、页面状态识别、关键字段抽取、异常页面识别）

## 四点一、视觉识别系统能力建设（写死）

### 4.1 能力范围

- 商品图识别：类目、属性、风格、材质、颜色
- 质检识别：清晰度、遮挡、构图、重复图、低质图
- 合规识别：违禁元素、侵权标识、风险文案截图
- 场景识别：模特/场景匹配度、品类一致性、平台风格一致性

### 4.2 执行链路

1. 素材进入系统后先做视觉预检并生成结构化标签。
2. 识别结果进入编排引擎，影响选图、文案、发布与定价策略。
3. 低置信度命中自动转人工复核；高风险命中自动阻断发布。
4. 识别结果与发布效果联动回写，形成持续学习样本。

### 4.3 指标与经营口径

- 识别准确率、召回率、误报率、漏报率
- 素材通过率、人工复核率、风险拦截率
- 视觉优化带来的点击率提升、转化率提升、退货率变化、ROI 改善

## 五、治理与安全约束（写死）

1. 所有外部接入遵守“单向输入 + 强隔离 + 加密通信”。
2. 外部结果不得直接入生产结论，必须走 `L2/L5/L8/L10` 治理链。
3. 业务实值配置化管理，正文不写死真实地址、密钥、账户凭据。
4. 自动化执行必须可审计、可回滚、可人工接管。
5. 支付链路默认人工确认，除非 `L10` 发布专项豁免且有审计证据。
6. 定价、售后、客服策略变更必须经过演练验证后再灰度放量。
7. 软件自动化执行必须在白名单应用内运行，越界调用默认拒绝。
8. 网页自动化执行必须经过“页面识别 -> 目标确认 -> 动作执行 -> 回执校验”四步闭环。

---

## 六、验收标准（DoD）

满足以下条件，方可判定“具备全链路业务编排能力”：

1. 至少跑通 1 条完整业务链路（从目标输入到发布与复盘）。
2. 每个环节有状态回写、审计ID、证据链接。
3. 至少完成 1 轮 A/B/C 策略优化并产出择优结论。
4. 提交经营结果：投入、产出、ROI、Token 净收益变化。
5. 支付动作具备人工干预记录与审批证据。
6. 定价、售后、客服三类策略均完成“学习 -> 演练 -> 执行 -> 复盘”闭环。
7. 视觉识别链路跑通并产出量化结果（准确率/风险拦截率/业务提升指标）。
8. 电脑操作与软件使用链路具备稳定执行、失败回滚与审计证据。
9. 任务执行满足“单 Agent 单一任务 + 主责唯一 + 能力复用留痕”三项约束。
10. 网页识别链路具备页面状态识别准确率与关键动作一致性校验证据。

---

## 七、最小验证与学院协同（写死）

### 7.1 投入策略：先最小验证，再逐步加大投入

1. 所有新能力先走最小验证（MVP）链路，不得直接全量投入。
2. MVP 通过后按“灰度 -> 小流量 -> 多场景 -> 全量”四阶段扩展。
3. 每阶段必须提交投入产出报告（人力、Token、工具成本、收益变化）。
4. 未达收益门槛或风险超阈值时，必须暂停扩容并回到优化阶段。

### 7.2 关键能力建设目标

必须持续建设并量化以下能力：

- 经验沉淀能力：形成可复用案例、规则包、失败样本库
- 创新能力：提出新策略并通过沙盒/A-B-C 验证
- 盈利能力：持续提升 ROI 与净收益率
- 市场机会识别能力：自动发现新品类、新渠道、新增长窗口
- 自主学习能力：从执行结果中学习并优化策略版本

### 7.3 学院协同要求（必走）

上述五类能力均需纳入学院体系进行标准化训练与发布，执行要求：

1. 在学院建立对应能力课纲、评测标准与毕业门槛。
2. 训练数据与案例回流到学院资产库，形成可复用课程包。
3. 能力升级必须先在学院评测通过，再进入生产灰度。
4. 学院未收录/未评测通过的能力，不得宣称“平台已具备”。

### 7.4 阶段验收口径

- 阶段一（最小验证）：跑通 1 条链路并完成收益可行性证明
- 阶段二（小规模放量）：至少 3 个场景稳定运行并达成正向 ROI
- 阶段三（规模化）：跨渠道复用，学院评测通过率达标，形成标准能力包
