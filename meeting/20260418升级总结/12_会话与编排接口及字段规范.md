# 12_会话与编排接口及字段规范

**性质**：会话中间层落地规范（术语 + 字段 + 状态/错误码 + 最小接口 + 权限矩阵）  
**适用范围**：会话中间层、后端编排引擎、前端流程呈现层  
**上位依赖**：[`./11_会话治理规范.md`](./11_会话治理规范.md)、[`./10_Agent基础能力与分层裁剪规范.md`](./10_Agent基础能力与分层裁剪规范.md)、[`../../AGENTS.md`](../../AGENTS.md)

---

## 一、术语表（统一口径）

- **会话（Session）**：围绕一个目标持续推进的上下文容器。
- **会话中间层（Session Orchestrator）**：统一管理会话状态、策略路由、治理钩子的后端组件。
- **编排策略（Strategy）**：会话推进模式，如 `linear`、`manager_worker`、`handoff`、`swarm`、`external`。
- **流程片段（Segment）**：可复用、可版本化的流程子链路。
- **修订版本（Revision）**：同一会话下的输出迭代版本。
- **发布版产物（Publish Artifact）**：用于外发或对客交付的结果。
- **可编辑版产物（Editable Artifact）**：可继续编辑完善的结果源。

## 三、状态码标准

### 3.1 会话状态码

| 状态码 | 含义 |
|---|---|
| `CREATED` | 会话已创建 |
| `ACTIVE` | 会话进行中 |
| `PAUSED` | 会话暂停等待 |
| `RESUMED` | 会话恢复执行 |
| `SETTLED` | 会话收口完成 |
| `ARCHIVED` | 会话归档只读 |

### 3.2 编排执行状态码

| 状态码 | 含义 |
|---|---|
| `PLAN_READY` | 编排方案已生成 |
| `RUNNING` | 执行中 |
| `HUMAN_PENDING` | 等待人工确认 |
| `QUALITY_PENDING` | 等待质量门禁 |
| `PUBLISH_PENDING` | 等待发布决策 |
| `DONE` | 执行完成 |
| `FAILED` | 执行失败 |

## 四、错误码标准（最小集）

| 错误码 | 说明 | 默认动作 |
|---|---|---|
| `E_SESSION_NOT_FOUND` | 会话不存在 | 终止并告警 |
| `E_SESSION_STATE_INVALID` | 状态不允许当前操作 | 拒绝并回写 |
| `E_PERMISSION_DENIED` | 权限不足 | 拒绝并审计 |
| `E_SCOPE_VIOLATION` | 作用域越界 | 拒绝并告警 |
| `E_STRATEGY_INVALID` | 策略不合法 | 降级到默认策略 |
| `E_SEGMENT_VERSION_MISSING` | 片段版本缺失 | 拒绝拼接 |
| `E_HUMAN_CONFIRM_REQUIRED` | 缺少人工确认 | 转 `HUMAN_PENDING` |
| `E_GATE_REJECTED` | 门禁拒绝 | 回退或终止 |
| `E_PUBLISH_FAILED` | 外发失败 | 重试/熔断/死信 |
| `E_ROLLBACK_FAILED` | 回滚失败 | 升级人工接管 |
| `E_LEARNING_NOT_READY` | 学习包未就绪 | 阻断执行并回写 |
| `E_REHEARSAL_REJECTED` | 演练未通过 | 阻断执行并转人工 |
| `E_PRIMARY_AGENT_CONFLICT` | 主责Agent冲突 | 拒绝分派并告警 |
| `E_PAYMENT_APPROVAL_MISSING` | 缺少支付审批 | 转 `HUMAN_PENDING` |
| `E_SOFTWARE_SCOPE_VIOLATION` | 软件白名单越界 | 拒绝执行并审计 |
| `E_WEB_PAGE_STATE_INVALID` | 页面状态不满足执行条件 | 降级/拦截/人工接管 |
| `E_ACADEMY_ASSESSMENT_REQUIRED` | 缺少学院评测 | 阻断发布并转评测 |
| `E_ACADEMY_GATE_BLOCKED` | 学院门禁未通过 | 阻断发布并回写原因 |

## 八、变更规则

1. 字段、状态码、错误码变更需版本化。
2. 任何接口变更需提供兼容窗口与迁移说明。
3. 权限矩阵变更需同步更新 `10` 与 `11` 文档。

## 九、前端多状态展示与交互规范（V1）

### 9.1 状态分层模型（前端统一口径）

前端必须采用“主状态 + 子状态 + 门禁状态 + 异常状态”四层模型，不得仅展示单一状态。

- 主状态：`CREATED / ACTIVE / PAUSED / RESUMED / SETTLED / ARCHIVED`
- 子状态：`PLAN_READY / RUNNING / HUMAN_PENDING / QUALITY_PENDING / PUBLISH_PENDING / DONE / FAILED`
- 门禁状态：`LEARNING_PENDING / REHEARSAL_PENDING / ACADEMY_BLOCKED / PAYMENT_PENDING / WEB_CHECK_PENDING`
- 异常状态：`RETRYING / DEGRADED / ESCALATED / ROLLBACKING`

### 9.2 页面最小组件清单

1. 状态总览条：展示主状态、子状态、风险等级、最近更新时间。
2. 流程时间线：展示每一步状态变更、触发人/Agent、证据链接、审计ID。
3. 人工接管面板：支付审批、门禁处理、异常处置、恢复执行入口。
4. 自动化执行面板：电脑/软件/网页执行状态、前后校验结果、失败重试进度。
5. 经营与评测面板：ROI、Token 成本、成功率、学院评测结果与门禁状态。

### 9.3 状态驱动交互规则（写死）

1. 前端状态仅由后端事件驱动更新，不得本地推断关键状态。
2. 任一 `PENDING/BLOCKED` 状态必须显示“阻塞原因 + 下一步动作 + 负责人”。
3. 高风险动作（发布/支付/删除）必须二次确认，且必须校验目标对象一致性。
4. 进入 `FAILED/DEGRADED/ESCALATED` 时，必须自动展示回滚或人工接管入口。
5. 任一状态变更必须可追溯到 `trace_id` 与对应审计事件。

### 9.4 最小接口映射（前端必接）

- 会话与状态：`getSession`、`updateSession`
- 编排与准入：`evaluateStrategy`、`validateLearningGate`、`validateAcademyGate`
- 人工确认：`requestHumanConfirm`、`submitHumanDecision`、`requestPaymentApproval`、`submitPaymentApproval`
- 自动化与网页：`runDesktopFlow`、`runSoftwareFlow`、`recognizeWebPage`、`confirmWebTarget`、`verifyWebActionReceipt`
- 发布与回滚：`publishArtifacts`、`rollbackPublish`、`getPublishStatus`
- 评测与经营关联：`submitAcademyAssessment`、`getAcademyAssessmentResult`、`linkProductionMetrics`

### 9.5 前端验收最小标准（DoD）

1. 可同时展示四层状态（主/子/门禁/异常）并正确联动。
2. 至少覆盖 1 条完整链路：`ACTIVE -> RUNNING -> HUMAN_PENDING -> DONE -> SETTLED`。
3. 至少覆盖 1 条异常链路：`RUNNING -> FAILED -> RETRYING/ESCALATED -> ROLLBACKING`。
4. 每次状态变更都可在前端查看审计ID与证据链接。

## 十、配置文件与端口治理规范（V1）

### 10.1 配置分层与文件边界（写死）

配置必须按 `platform -> company -> department -> user` 分层，不得混写。

- `platform`：平台默认与硬边界配置（门禁、审计、阈值上限）
- `company`：企业级业务策略配置（渠道、预算、运营模板）
- `department`：部门级执行策略（流程片段、角色分工、SLA）
- `user`：个人偏好与低风险参数（展示偏好、提醒阈值）

配置文件规则：

1. 配置与代码分离，禁止把真实业务实值硬编码在流程代码。
2. 配置变更必须版本化（`config_version`）并有发布记录。
3. 配置加载必须带租户与作用域键，禁止跨租户读写。
4. 配置发布必须可回滚，且回滚不依赖手工改库。

### 10.2 端口注册与命名规范（写死）

本项目端口仅允许归属到三类：

- `Protocol Port`：外部协议协作（A2A / 外部Agent）
- `Tool Port`：工具与MCP接入
- `LLM Port`：模型调用与推理接入

端口命名规范（最小）：

- `port_id`：全局唯一，如 `tool.web-recognition.v1`
- `port_type`：`protocol` / `tool` / `llm`
- `owner_vp`：所属VP（如 `VP11` / `VP01`）
- `risk_level`：`low` / `medium` / `high` / `critical`
- `status`：`draft` / `active` / `frozen` / `retired`

### 10.3 端口配置最小字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `port_id` | string | ✅ | 端口唯一ID |
| `port_type` | enum | ✅ | 端口类型 |
| `config_version` | string | ✅ | 配置版本号 |
| `owner_scope` | enum | ✅ | 配置作用域 |
| `owner_id` | string | ✅ | 归属主体 |
| `allowed_domains` | string[] | ⭕ | 白名单域名（网页/外部端口） |
| `credential_ref` | string | ⭕ | 凭据引用（不存明文） |
| `rate_limit` | object | ⭕ | 限流策略 |
| `timeout_ms` | number | ⭕ | 超时阈值 |
| `retry_policy` | object | ⭕ | 重试与退避策略 |
| `rollback_version` | string | ⭕ | 回滚目标版本 |

### 10.4 配置与端口发布门禁

1. 端口或配置变更必须经过评审与审计，禁止直接热改生产。
2. 高风险端口（支付、发布、删除）必须双签与灰度验证。
3. 端口不可用或越界时必须降级到安全路径并触发人工接管。
4. 端口变更需同步更新前端可见状态与阻塞原因文案。

### 10.5 监控指标（配置/端口）

- 配置加载成功率、配置回滚成功率
- 端口调用成功率、超时率、熔断率、降级率
- 端口越界拦截率、白名单命中率
- 配置版本漂移告警数、未审计变更拦截数
