## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。
- 治理模型统一采用“规则写死、参数可配、边界写死”：
  - 写死规则：流程门禁、接管链路、审计要求、发布与回滚约束。
  - 可配置参数：SLA、复评阈值、紧急策略、试点范围等运营参数。
  - 边界写死：仅允许边界内收紧，不允许突破平台红线；配置变更必须版本化发布并写审计。
- 接口原则：所有接口不做占位；设计可做全局通用与扩展增强，但接口定义本身必须可直接落地，不保留仅供后补的占位接口。
- 全局配置原则：凡可由配置承载的真实业务值，一律配置化管理，能配置就不写死。
- 落地口径：正文默认只固化字段名、规则、约束、示例值与生效顺序；真实实值通过 `platform -> company -> department -> user` 配置链注入并审计。

## 框架选择

### 结论：官方SDK + 纯Node.js

```
运行时：      Node.js（保持，不迁移）
LLM调用：     官方SDK（最稳定，官方维护）
              openai / @anthropic-ai/sdk / @google/generative-ai
本地模型：    直接 fetch 调用（Ollama API简单，不需要SDK）
不引入：      LangChain / AutoGen / CrewAI（避免框架绑定）
```

### 为什么不用 LangChain
```
❌ 版本更新频繁，破坏性变更多
❌ 太重，大量用不到的功能
❌ 框架本身也在试错，不够稳定
❌ 会限制我们自研架构的灵活性

官方SDK优势：
✅ 跟模型同步更新
✅ 极轻量，只做一件事
✅ 官方支持，问题有保障
✅ 换模型只换SDK，不改架构
```

## 六边形架构（Ports & Adapters）

```
              外部LLM（Claude/GPT/Gemini/本地）
                    ↕  LLM Port
外部协议     ←→  [ Adapter 层 ]  ←→   内核
(MCP/A2A)         ↕  Tool Port         CEO
                  ↕  Storage Port    + VP层
              外部存储/工具            + 通信规则
              (DB/文件/API)           + 编排逻辑
```

### 5个端口

| 端口 | 作用 | 现在 | 未来 |
|---|---|---|---|
| LLM Port | 模型调用 | MockAdapter | Claude/GPT/Gemini/Local Adapter |
| Protocol Port | 外部Agent通信 | 内存消息 | MCP Adapter / A2A Adapter |
| Storage Port | 数据持久化 | MemoryAdapter | File / SQLite / CloudDB |
| Tool Port | 外部工具调用 | 无 | Browser / File / API / MCP工具 |
| Gateway Port | 用户接入 | CLI | REST API / WebSocket / SDK |

### 外部接入组件层级归属（写死）

| 外部组件 | 端口归属 | 组织归属 | 工作流约束 | 数据权限 |
|---|---|---|---|---|
|| 外部 Agent / scale / 第三方协作服务 | `Protocol Port` | `VP11（External）` | 结果回流后必须经 `security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22）治理链 | 仅接收受控摘要/受控引用；禁止直读我方文件与存储 |
|| 外部模型（SaaS / BYOK / Local URL） | `LLM Port` | 模型管理域 | 由 `model.management`（L3）统一注册、路由、成本与健康治理；不得绕过 `security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22） | 不得获得仓库、文件系统、`S1-S4` 直读权 |
|| MCP 接口 / 外部工具 | `Tool Port` | 工具接入治理域 | 由 `tool.management`（L4）统一注册、风险分级、白名单与健康治理；不得绕过 `security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22） | 默认只读或按白名单开放；禁止直读未授权本地文件 |

### 核心原则
```
内核（6层组织规则 + 2M+23L纵向路由规则）永远不直接接触外部
所有外部依赖通过 Adapter 隔离
外部变化（模型升级/协议迭代/存储切换）只影响 Adapter
内核永远稳定
```

### 外部接入安全原则（写死）

```text
1. 所有外部接入均视为不可信边界
2. 交互方式统一为“单向输入 + 结果回传”，不允许外部组件主动读取我方系统资源
3. 外部交互必须使用加密通信（TLS；高敏场景建议 mTLS）
4. 外部结果不得直入生产结论，必须重新进入内部治理链
5. 外部输入与返回值均按潜在污染源处理，必须做格式校验、白名单过滤与风险评级
```

## 协议支持

### MCP（Model Context Protocol）— 先做
```
Anthropic 推出，Cursor 已在使用
作用：
  ← 让外部工具接入我们的 Agent
  → 我们的 Agent 调用外部 MCP 工具

直接受益：Research Agent 可以调用任意 MCP 工具
          （浏览器、数据库、文件系统等）
约束：MCP 仅可通过 Tool Port 的受控 Adapter 接入；外部 MCP Server 不得获得我方仓库、文件系统与 S1-S4 的主动读取能力
```

### A2A（Agent-to-Agent）— 接口契约先行
```
Google 推出，2025年发布
作用：跨系统 Agent 通信标准
现状：协议已具备落地条件，但项目内先收敛为接口契约
现在：通过 `A2A Adapter` 接入；协议变化仅影响 Adapter，不影响内核
文档入口：见 `./a2a/01_A2A接口契约.md`
```

## 防外部变化策略

### 外部变化类型
```
LLM模型：   GPT-4 → GPT-5 / Claude升级 / 本地模型崛起
协议标准：  MCP进化 / A2A推广 / 未来新标准
工具平台：  浏览器API变 / OS接口变 / 云服务变
合规要求：  AI治理法规 / 数据安全要求
```

### 应对原则
```
变化快的 → 全部放 Adapter 层（外面）
不变的   → 放内核（6层组织规则 / 2M+23L纵向路由规则 / 核心治理链）

版本化：每个 Adapter 支持多版本
        旧版本保留（老用户不受影响）
        新版本并行（新功能逐步迁移）
```

## 不重复造轮子原则

每次设计新功能前，Research Agent 执行：
```
1. web_search  "xxx pattern existing frameworks 2026"
2. doc_lookup  相关官方文档
3. 输出对比表：现有方案 vs 我们的需求 gap
4. 结论：借用 / 改造 / 原创
5. 写入 KnowledgeBase（external命名空间）
   └→ Plan Agent 读取后再做方案设计
```

### 已调研结论
```
AutoGen 嵌套GroupChat  → 可借鉴三方会话机制实现
MetaGPT 发布订阅      → KnowledgeBase已参考（已实现）
CrewAI manager        → 参考VP层任务委托模式

我们原创的部分：
  ✦ 6层组织层级（OL0-OL5）+ 跳级DENY规则
  ✦ 跨VP必须逐级上报的升级链
  ✦ 横向 `comm-router` + 纵向 `layer-router` 分离驱动通信路由
  ✦ VP 作为跨域出口的唯一性设计
```
