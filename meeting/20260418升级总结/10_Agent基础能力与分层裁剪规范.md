## 一、目标

建立统一的 Agent 能力基线，避免以下问题：

1. 各层 Agent 能力定义不一致，导致职责漂移。
2. 能力与权限边界混淆，导致越权执行。
3. 只列“会做什么”，未定义“谁可做、何时做、做到什么程度”。
4. 经验沉淀与自优化缺少治理约束，带来污染与失控风险。

### 1.0 可工程化基线（写死，所有 Agent 必须满足）

**核心原则：把 Agent 从"黑盒助手"变成"可工程化模块"。**

| 基线项 | 含义 | 验收标准 |
|---|---|---|
| 明确输入 | 每次执行有明确定义的入参结构 | 有 schema 定义，无隐式依赖 |
| 明确输出 | 每次执行有明确定义的出参结构 | 有 schema 定义，结果可被下游消费 |
| 可测试 | 能在隔离环境中独立验证 | 有单测/集成测试用例，可重复执行 |
| 可回滚 | 执行结果可撤销或还原至前一状态 | 有明确回滚 SOP，变更版本化 |
| 可审计 | 执行全过程可追溯 | 写入 trace_id + agent_id + 时间戳，存 S3 |
| 可人工接管 | 任意节点可暂停并移交人工处理 | HITL 接管链路已定义，超时有升级动作 |

> 来源：`meeting/升级方向.md` §三.第一阶段.2（不在此重述，只写验收标准）

---

### 1.1 本次更新（11层分工配置化）

为支持后续持续升级，本规范从“层级/分工写死”升级为“拓扑配置化 + 治理规则固化”：

1. **可配置项**：层级数量、层级命名、角色挂载、路由映射、协作关系、扩展节点。
2. **固化项**：治理红线、审批门禁、审计要求、发布与回滚约束。
3. **升级原则**：新增层级或新增角色优先通过配置发布，不以改源码作为前置条件。
4. **扩展口径**：每层均允许继续增加扩展节点，子层级数量不设上限。
5. **VP口径**：平台级VP数量按配置可扩展，11个为默认基线，平台管理员可通过配置审批新增；企业用户不可自行新增VP（仅可在平台预设VP范围内激活或停用可选VP）。经理命名空间随VP编号同步扩展，始终保持与VP一一对应。

## 二、能力分层模型

本规范将能力分为三层：

1. **通用基础能力**：默认基础能力池（所有 Agent 初始化可见）。
2. **平台级增强能力**：平台公共引擎能力（由治理链统一约束）。
3. **分层裁剪与授权能力**：按组织层级与作用域进行启用、限权、封禁。

## 三、智能体身份与能力画像（规则固化，实例配置化）

每个智能体必须拥有独立身份与独立能力画像，禁止“匿名智能体”与“共享无归属能力”。

### 3.1 独立身份

每个智能体在注册时必须具备最小身份元数据：

- `agent_id`（全局唯一）
- `agent_name`
- `level`（CEO / VP / 总监 / 经理 / 子Agent / 副Deep）
- `owner_scope`（platform / company / department / user）
- `owner_id`（归属主体）
- `parent_id`（上级智能体；CEO可为空）
- `status`（active / frozen / retired）
- `created_at` / `updated_at`

约束：

1. `agent_id` 全局唯一且不可复用。
2. 任何能力调用必须可追溯到明确 `agent_id`。
3. 停用或冻结智能体后，其能力自动停止对外可调用。

### 3.2 独立能力画像

每个智能体必须维护独立能力画像（Capability Profile），至少包含：

- 默认必备能力清单
- 可挂载能力清单
- 显式禁止能力清单
- 能力版本号与生效时间
- 能力风险等级
- 能力调用边界（作用域 / 数据域 / 工具域）

约束：

1. 能力必须“挂在智能体身份上”，禁止脱离身份单独漂移。
2. 同名能力在不同智能体上可有不同版本与边界。
3. 低层智能体不得继承上层未显式下放的能力。

### 3.3 身份与能力的一致性校验

能力执行前至少校验：

- 调用者身份是否合法且处于 `active`
- 调用能力是否在其能力画像中被授权
- 调用能力是否与其层级职责一致
- 调用目标是否跨越作用域边界

任一校验失败 -> 默认拒绝执行并写审计事件。

## 五、平台级增强能力（新增治理固化项）

### 5.1 经验沉淀引擎（平台公共能力）

**定位**：经验闭环能力，不归属于单个 Agent 私有技能。  
**目标**：将执行过程、失败样例、修正轨迹与复盘结论沉淀为可复用资产。

**最小能力集合**：

- 经验采集
- 经验清洗
- 经验分类
- 经验归档
- 经验索引
- 经验检索
- 经验回放
- 经验复用
- 经验淘汰/归档策略
- 经验证据绑定
- 经验质量评估

### 5.2 自优化治理体系（平台公共能力）

**定位**：进化治理能力，不等于自由学习能力。  
**目标**：保证“可进化”与“可治理”同时成立，防止污染与失控。

**最小能力集合**：

- 优化提案生成
- 提案分类
- 沙盒验证
- A/B 对比
- 质量门禁
- 审批发布
- 灰度验证
- 回滚机制
- 风险隔离
- 污染防护
- 版本留痕
- 进化审计

**硬规则**：

1. 优化只能在隔离域进行，不得直写生产。
2. 优化结果必须经门禁与发布链。
3. 任意优化均需可回滚、可审计、可追溯。

### 5.3 视觉识别能力引擎（平台公共能力）

**定位**：面向图像/视频的结构化识别能力，不归属单个 Agent 私有实现。  
**目标**：为“选品、质检、素材审查、违规识别、页面理解”提供统一视觉底座。

**最小能力集合**：

- 图像目标识别（类目、款式、属性、Logo、文本）
- 图像质量检测（清晰度、噪点、分辨率、遮挡、构图）
- 图像合规检测（敏感内容、侵权风险、平台禁限售元素）
- 商品信息抽取（颜色、材质、场景、卖点标签）
- 页面视觉理解（版式区域识别、关键按钮/字段识别）
- 视频关键帧抽取与镜头段落识别
- OCR 文字识别与字段结构化
- 识别结果置信度评估与低置信度人工复核触发

**治理约束（固化）**：

1. 视觉识别输出默认视为“候选结论”，不得直接越过门禁入生产。
2. 低置信度或高风险命中必须触发人工复核节点。
3. 视觉模型版本、阈值、规则变更必须版本化并可回滚。
4. 视觉数据处理遵守租户隔离与最小必要原则，不得跨租户复用原始素材。

### 5.4 电脑操作与软件使用能力引擎（平台公共能力）

**定位**：统一的端侧操作与软件执行能力，为业务自动化提供稳定执行底座。  
**目标**：让 Agent 能在受控边界内完成“人机操作替代”，降低人工重复操作成本。

**最小能力集合**：

- 桌面基础操作（打开/关闭应用、窗口切换、截图、文件读写触发）
- 浏览器操作（导航、表单填写、页面检索、下载上传）
- 办公软件操作（表格、文档、演示、邮件模板化处理）
- 运营后台操作（商品录入、库存更新、订单查询、工单流转）
- RPA 流程编排（多软件串联、失败重试、断点续跑）
- 操作回放与证据留痕（操作日志、截图、步骤审计ID）

**治理约束（固化）**：

1. 软件操作默认最小权限运行，禁止未授权本地目录与凭据读取。
2. 高风险动作（删除、支付、批量发布）必须人工确认或双签。
3. 自动化脚本与流程版本化管理，变更必须可回滚。
4. 操作过程必须全链路可审计，并可追溯到 `agent_id/task_id/session_id`。

### 5.5 网页识别能力引擎（平台公共能力）

**定位**：针对网页界面与DOM语义的识别能力底座。  
**目标**：支持对电商平台、运营后台、表单系统的页面结构理解与稳定自动化。

**最小能力集合**：

- 页面结构识别（布局区域、导航、列表、详情、弹窗）
- 元素语义识别（按钮、输入框、下拉、表格、富文本）
- 关键信息抽取（价格、库存、SKU、订单状态、规则提示）
- 页面状态识别（加载中、成功、失败、权限不足、风控拦截）
- 反脆弱定位（元素变更后的回退定位与多策略匹配）
- 网页OCR补偿（图片文本、画布文本的兜底识别）

**治理约束（固化）**：

1. 网页识别结果必须经过置信度校验，低置信度动作需人工确认。
2. 页面结构变更触发自动降级与告警，不得盲目连续重试。
3. 网页自动化仅允许在授权域名白名单内执行。
4. 关键动作（发布、支付、删除）必须二次校验页面状态与目标对象一致性。

## 六、分层裁剪矩阵（组织层级）

### 6.1 CEO（L0）

**主责能力**：

- 目标澄清、范围识别、完成标准识别
- 全局路由、冲突协调、风险识别
- 全局状态感知、收口整合、复盘
- 多 Agent 协作编排管理（跨 VP/总监/经理的协同编排与冲突仲裁）
- 协作拓扑与并发策略管理（并发上限、分组策略、升级路径、收口责任链）
- 盈亏平衡评估（投入产出、成本收益、资源配比）
- 预算与配额治理（按阶段/VP/任务类型设上限与分配）
- ROI 优先级排序（高收益、低风险、短闭环优先）
- 止损与退出机制（低收益高消耗任务触发降级/暂停/终止）
- 资源重配能力（跨 VP 动态调配人力、算力、模型档位、工具额度）
- 经营风险预警（成本超阈值、收益不达标、效率异常告警）
- 目标-预算一致性校验（目标变更时同步校验预算与周期匹配）
- 经营复盘闭环（投入产出结果回写策略，驱动下一轮优化）
- 多租户公平策略（防止单租户/单部门长期过量占用资源）
- 规则约束、权限边界识别

**默认限制**：

- 不承担细粒度重执行
- 不直接承担底层工具作业
- 不直接替代财务记账与核算系统；仅做治理层面的经营平衡决策

### 6.2 VP（L1）

**主责能力**：

- 领域拆解、路由、协作编排
- 风险识别、结果验收、状态回写
- 证据链组织、复盘

**默认限制**：

- 不承担子任务细粒度重执行
- 不绕过经理层直连子Agent执行

### 6.3 总监（L2）

**主责能力**：

- 任务拆解、流程规划、委派收口
- 搜索整合、逻辑校验、结果验收
- 风险识别、异常处理、复盘

**可扩展能力**：

- 代码能力 / 多模态理解 / 数据分析（按域授权）

### 6.4 经理（L3）

**主责能力**：

- 排程分包、依赖管理、进度同步
- 冲突协调、状态回写、异常上报
- 收口整合

### 6.5 子Agent（L4）

**主责能力**：

- 执行作业（检索、工具、代码、多模态）
- 自检校验、状态回写、证据链组织
- 异常处理与协作

### 6.6 副Deep Agent（L5）

**主责能力**：

- 单点深挖（深度检索、深度分析、深度校验）

**默认限制**：

- 不承担全局路由与跨域协调
- 不承担发布审批与治理决策

### 6.7 NG助理分层配置（VP / 总监 / 经理）

定位：`NG助理` 为分层辅助能力，不替代该层主体职责。

**VP（L1）- NG战略助理（必配）**：

- 跨域协同建议、冲突预警、升级路径建议
- 预算/配额/ROI 辅助分析与风险提示
- 关键节点证据清单校验与治理链提醒

**总监（L2）- NG编排助理（必配）**：

- 任务拆解建议、执行路径建议、资源调度建议
- 门禁前置检查（输入完整性、依赖完整性、风险项）
- 交付物一致性检查与收口清单生成

**经理（L3）- NG执行助理（默认配，可按规模裁剪）**：

- 排程建议、依赖跟踪、阻塞项聚合
- 状态回写草案、异常分级、上报建议
- 证据链整理与联调清单维护

**通用边界（固化）**：

1. `NG助理` 仅提供“建议/校验/提醒”，不得替代 VP/总监/经理做最终审批与发布决策。
2. `NG助理` 不得跨层越权调用下级执行单元，不得绕过“总监 -> 经理 -> 子Agent”链路。
3. 所有 `NG助理` 调用必须进入核心治理链路：`security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22），并审计留痕。
4. `NG助理` 能力包按 `platform -> company -> department -> user` 作用域挂载，不得跨租户污染。

### 6.8 全层级经营指标责任（CEO / VP / 总监 / 经理）

原则：`CEO/VP/总监/经理` 均需背经营结果，不仅背任务完成率。

**统一核心指标（固化）**：

- 财务指标：预算执行率、成本偏差率、边际成本变化
- 投产比指标：`ROI = 收益 / 投入成本`（按周期核算）
- Token 经营指标：单位 Token 收益、Token 成本、净收益率
- 业务结果指标：交付完成率、验收通过率、复用率、返工率

**分层责任口径（固化）**：

- `CEO（L0）`：对全局损益、总 ROI、全局 Token 净收益负责；负责止损阈值与资源重配。
- `VP（L1）`：对本域损益、域内 ROI、域内 Token 经营效率负责；负责跨团队资源取舍与达成。
- `总监（L2）`：对项目包 ROI、执行成本、质量-成本平衡负责；负责方案层面的提效降本。
- `经理（L3）`：对迭代级投入产出、Token 消耗效率、按期交付负责；负责日/周节奏纠偏与阻塞消减。

**核算与归因规则（固化）**：

1. 成本核算至少覆盖：人力、模型 Token、工具调用、外部服务费用。
2. Token 盈利归因必须按 `task_id + session_id + strategy + model_id` 维度记录。
3. 收益核算可按“直接收益 + 间接收益估算”分层呈现，但口径需版本化并可审计。
4. 不允许仅汇报“完成了什么”，必须同步汇报“投入多少、产出多少、净收益如何”。

**收口机制（固化）**：

1. 所有“已写未收口”业务项必须进入清零清单，绑定负责人、截止时间、证据链接。
2. 周收口：经理与总监完成周度偏差分析与纠偏动作；VP完成域内周复盘。
3. 月收口：CEO完成跨域损益与 ROI 复盘，输出下月预算与配额调整。
4. 未收口项不得在汇报中标记“完成”，不得进入“稳定运行”声明。

### 6.9 经营目标逐级下达与回传（治理固化）

原则：经营指标必须“自上而下分解、逐级承接负责、自下而上证据回传”。

**下达链路（固化）**：

1. `CEO -> VP`：下达全局预算、域级 ROI 目标、Token 经营上限与盈利目标。
2. `VP -> 总监`：拆解为项目包目标（阶段成本、阶段收益、质量门槛、交付里程碑）。
3. `总监 -> 经理`：落实到迭代任务（周目标、日节奏、投入产出、风险阈值）。
4. `经理 -> 子Agent`：仅下达执行目标与约束，不下放治理审批权。

**回传链路（固化）**：

1. `子Agent -> 经理`：回传执行证据、Token 消耗、异常与阻塞。
2. `经理 -> 总监`：回传迭代达成率、偏差原因、纠偏动作与预计 ROI 影响。
3. `总监 -> VP`：回传项目包收益实现、成本偏差、质量与风险变化。
4. `VP -> CEO`：回传域级损益、域级 ROI、Token 净收益与下周期资源建议。

**禁止项（固化）**：

1. 禁止跳级下达与跳级汇报（除 `L10` 专项应急授权）。
2. 禁止“只报进度不报经营指标”，必须同步上报投入、产出、净收益。
3. 禁止无证据关单；所有回传必须绑定 `task_id/session_id/审计ID`。
4. 禁止以局部完成替代整体收口，必须以层级目标达成为准。

### 6.10 性能优先与代码驱动增益（治理固化）

原则：系统设计优先追求“高性能 + 高质量 + 自动化代码驱动”，将性能提升直接转化为经营收益。

**技术与质量目标（固化）**：

1. 关键链路优先优化：会话中间层、编排决策、门禁校验、外发回写。
2. 代码质量优先：可读、可测、可观测、可回滚，不以短期堆功能替代工程质量。
3. 自动化优先：能由代码驱动完成的环节，不依赖人工重复操作。
4. 先进技术采用原则：先小范围验证收益，再分阶段放量，不盲目全量切换。

**经营转化口径（固化）**：

1. 每次性能优化必须给出“优化前/后”对比：时延、吞吐、失败率、资源消耗。
2. 每次代码自动化改造必须给出“节省量”与“收益量”：
   - 节省量：人时节省、Token 节省、调用成本节省。
   - 收益量：交付提速、成功率提升、净收益改善。
3. 坚持“节省即盈利”核算口径：每减少可计量成本即计入利润改善项。
4. 优化项若无量化收益与证据，不得标记为“经营贡献已完成”。

**验收要求（固化）**：

1. 必须附性能基线与压测/回放证据链接。
2. 必须附代码质量证据（测试结果、质量门禁、回滚预案）。
3. 必须附经营结果证据（ROI 影响说明、Token 成本变化、净收益变化）。

## 八、治理与安全约束

### 8.1 四级技能隔离（平台 / 公司 / 部门 / 用户）

技能与能力资产统一按四级作用域管理：

- `platform`：平台级公共能力与治理能力（仅平台治理流程可变更）
- `company`：公司级能力模板与企业规则（仅公司治理角色可变更）
- `department`：部门级能力包与业务策略（仅部门授权角色可变更）
- `user`：用户级个人技能与偏好能力（仅用户或其授权代理可变更）

隔离规则：

1. 默认“向下可见、横向隔离、向上不可见”。
2. 低域（`user/department`）能力不得默认上行污染 `company/platform`。
3. 跨域共享必须走授权流程，且保留审计记录、版本号与回滚点。
4. 任何跨租户能力引用默认拒绝，除非有显式白名单与审批证据。

### 8.2 技能生命周期隔离

技能在“注册 -> 验证 -> 发布 -> 调用 -> 复盘 -> 迭代”全生命周期必须绑定作用域键：

- 最小隔离键：`tenant_id + scope_level + owner_id + skill_id + version`
- 发布隔离：不同作用域的同名技能必须使用独立版本线
- 调用隔离：调用时必须同时校验调用者层级、作用域与目标技能域
- 回写隔离：经验沉淀和优化结果不得跨域直接覆盖原技能

### 8.3 通用治理约束

1. 能力可见不等于能力可用，实际可用受 `security.permissions`（L6）权限管理层和作用域约束。
2. 外部相关能力默认受限，按白名单与风险等级放开。
3. 高风险能力必须走 `quality.gate`（L18）/ `release.publish`（L22）审批链。
4. 任何跨租户、跨域调用默认拒绝。
5. 能力调用全过程需可审计、可追溯。

## 十一、分工协作与全局复用（规则固化，拓扑可扩展）

### 11.1 单 Agent 单一任务原则

1. 每个 Agent 在同一执行阶段只承担一个明确任务单元（One Agent One Task）。
2. 一个任务单元只能有一个主责 Agent，不允许多 Agent 对同一任务主责重叠。
3. 涉及多步骤目标时必须拆分为多个原子任务，由编排层串联，不得由单 Agent 混做。

### 11.2 协作分工原则

1. 统一采用“主责 Agent + 协作 Agent + 验收 Agent”三角色协作模型。
2. 主责 Agent 仅对本任务单元结果负责，不承担跨单元总收口。
3. 跨单元收口由上级编排角色负责（经理/总监/VP），不得下沉给执行 Agent。

### 11.3 全局适配与复用原则

1. 可复用能力必须沉淀为标准能力包，不以临时脚本散落在单任务中。
2. 复用能力包必须声明适用边界、输入输出契约、版本号与回滚点。
3. 复用能力遵循 `platform -> company -> department -> user` 作用域挂载，不得跨租户污染。
4. 新任务优先复用已有能力包，确需新增时必须给出“不复用原因”和复用评估记录。

### 11.4 分层拓扑配置规范（新增）

为避免“11层分工写死”，分层治理采用拓扑配置驱动：

1. 每次运行加载 `layer_topology`（含 `layers / roles / routing / constraints`）。
2. `layers` 支持新增层级，且使用 `order` 决定执行顺序，不依赖固定编号总数。
3. `roles` 支持按 `parent_id` 挂载，允许在既有治理边界内扩展角色树。
4. `routing` 支持 `task_type -> role_id` 与 `capability -> role_id` 双路由，未命中走配置化兜底。
5. `constraints` 固化跨层调用红线（禁止跳级、禁止绕过 `security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22）；不得绕过 `quality.inspector`（L16）检查职责）。
6. 新增层级必须先经 `release.publish`（L22）发布链审批，未经发布不得在生产生效。
7. 每一层的子层级/子节点不设固定上限，按负载与治理策略弹性扩展。
