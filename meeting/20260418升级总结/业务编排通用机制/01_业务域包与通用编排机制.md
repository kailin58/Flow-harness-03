# 业务域包与通用编排机制

**性质**：平台级业务编排补充规范  
**目标**：在不为具体行业新增主层级的前提下，满足不同业务的建模、编排、执行、审计与复盘需求。  
**适用范围**：服装、电商、餐饮、教育、企业管理、本地生活、跨境业务、内容运营等可流程化业务。  

---

## 一、核心结论

平台需要满足具体业务，但不应把具体业务写死进主工作流层级。

统一口径：

```text
主层级负责治理职责
业务域包负责业务知识
编排引擎负责生成任务图
Adapter 负责外部平台差异
业务中心负责业务对象与状态
数据层负责事实、证据与审计
```

因此：

```text
换业务时，切换或新增 Business Domain Pack。
不因某个行业、某个平台、某类商品而新增主层。
```

---

## 二、为什么不把具体业务写进主层级

主层级应承接稳定的平台治理职责，例如：

- 入口接入
- 外部系统管理
- 能力发现
- 权限管理
- 知识检索
- 任务编排
- 协作推进
- 安全策略
- 执行监控
- 结果检查
- 质量门禁
- 反馈闭环
- 评测
- 发布
- 实验

具体业务，例如服装、餐饮、教育、电商、管理软件，不是新的治理职责，而是业务域。

若把具体业务写成主层，会导致：

- 主层按行业无限膨胀
- 平台治理链和业务菜单混在一起
- 换业务时需要改主架构
- 一层一事原则被破坏
- 服装能跑但餐饮、教育、企业管理难以复用

所以平台主层不承接行业字段、行业对象和行业流程细节。

---

## 三、基本原则

### 3.1 一层一事

每个主层只管理一种稳定职责，不管理具体业务类型。

示例：

```text
L2 外部系统管理层：
只管外部系统是谁、凭据在哪里、连接是否健康。
不管商品、订单、课程、菜品、客户、合同等业务对象。
```

### 3.2 业务域可插拔

垂直业务通过业务域包接入平台。

```text
服装业务 = apparel_retail domain pack
餐饮业务 = restaurant_ops domain pack
教育业务 = education_service domain pack
企业管理 = enterprise_ops domain pack
```

### 3.3 编排和业务分离

编排层生成任务图，不直接持有行业细节。

```text
L12 orchestration.task（任务编排层）：
负责把业务目标拆成可执行任务图。

Business Domain Pack：
负责告诉 L12 该业务有哪些对象、流程、规则、指标和风险点。
```

### 3.4 治理链不被业务绕过

任何业务都必须进入统一治理链。

```text
权限、安全、执行、检查、门禁、审计、复盘、发布、实验
```

不允许业务域包直接绕过平台治理。

---

## 四、业务域包 Business Domain Pack

### 4.1 定义

Business Domain Pack 是某个业务领域的可加载知识与流程包。

它不改变主工作流层级，只为编排层提供业务上下文。

### 4.2 最小结构

```text
BusinessDomainPack
├── domain_id
├── domain_name
├── business_objects
├── workflow_templates
├── rule_packs
├── external_system_profiles
├── adapter_requirements
├── risk_policies
├── metrics
├── human_review_points
├── acceptance_criteria
└── sample_tasks
```

### 4.3 字段说明

| 字段 | 说明 |
|---|---|
| `domain_id` | 业务域稳定标识，例如 `apparel_retail`、`restaurant_ops` |
| `domain_name` | 业务域名称 |
| `business_objects` | 该业务涉及的对象模型 |
| `workflow_templates` | 常见流程模板 |
| `rule_packs` | 业务规则、平台规则、禁用动作、合规边界 |
| `external_system_profiles` | 该业务常用外部系统类型 |
| `adapter_requirements` | 需要哪些 Adapter 能力 |
| `risk_policies` | 高风险动作、审批要求、阻断条件 |
| `metrics` | 业务指标与复盘口径 |
| `human_review_points` | 必须人工确认的节点 |
| `acceptance_criteria` | 业务完成的验收标准 |
| `sample_tasks` | 样板任务，用于演练、评测和回归 |

## 五、业务蓝图 Business Blueprint

### 5.1 定义

Business Blueprint 是针对一次具体业务目标生成的执行蓝图。

它来自：

```text
用户目标 + Business Domain Pack + 当前约束 + 外部系统状态
```

### 5.2 最小结构

```text
BusinessBlueprint
├── blueprint_id
├── domain_id
├── goal
├── constraints
├── stages
├── deliverables
├── required_capabilities
├── external_dependencies
├── risk_points
├── human_checkpoints
└── success_metrics
```

### 5.3 职责边界

Business Blueprint 只描述本次业务如何落地，不承担执行。

执行必须继续转成 Task Graph，并进入治理链。

---

## 六、任务图 Task Graph

### 6.1 定义

Task Graph 是 Agent、工具、外部系统可以执行的任务结构。

它由 L12 orchestration.task（任务编排层）基于 Business Blueprint 生成。

### 6.2 最小结构

```text
TaskGraph
├── task_graph_id
├── nodes
├── edges
├── owner_agent
├── input_contract
├── output_contract
├── tool_requirements
├── approval_required
├── risk_level
└── rollback_policy
```

### 6.3 节点最小字段

```text
TaskNode
├── node_id
├── node_type
├── owner
├── inputs
├── outputs
├── dependencies
├── external_system_refs
├── required_adapters
├── risk_level
├── approval_required
├── retry_policy
├── rollback_policy
└── evidence_required
```

---

## 七、执行状态 Execution State

### 7.1 定义

Execution State 描述业务任务执行过程中的真实状态。

它不能只存在于对话里，必须可审计、可回放、可恢复。

### 7.2 最小结构

```text
ExecutionState
├── execution_id
├── domain_id
├── blueprint_id
├── task_graph_id
├── current_stage
├── node_status
├── external_call_status
├── approval_status
├── error_state
├── retry_count
├── audit_refs
├── evidence_refs
└── metric_snapshots
```

## 八、通用业务编排流程

```text
用户提出业务目标
        │
        ▼
识别业务域
        │
        ▼
加载 Business Domain Pack
        │
        ▼
生成 Business Blueprint
        │
        ▼
生成 Task Graph
        │
        ▼
进入统一治理链
        │
        ▼
执行、检查、门禁、审计
        │
        ▼
结果回写、复盘、评测、实验、发布
```

### 8.1 有业务域包时

```text
1. 识别 domain_id
2. 加载已发布 Business Domain Pack
3. 检查外部系统依赖
4. 匹配能力与 Adapter
5. 生成 Business Blueprint
6. 生成 Task Graph
7. 进入统一治理链执行
8. 回写结果与复盘指标
```

### 8.2 没有业务域包时

```text
1. 进入业务学习流程
2. 采集业务规则、对象、流程、风险、指标
3. 生成临时 Business Domain Pack 草案
4. 进行沙盘演练
5. 人工确认关键规则与高风险节点
6. 通过门禁后作为候选业务包
7. 经发布层发布后，后续任务方可复用
```

没有业务域包时，不允许直接进入高风险生产执行。

## 九、与现有层级的关系

业务域包机制不新增主层，而是通过已有层的职责分工落地。

| 业务域包 / 机制概念 | 对应层 | layer_id | 职责说明 |
|---|---|---|---|
| `external_system_profiles` | L2 外部系统管理层 | `external.system_mgmt` | 外部系统注册 / 认证凭据 / 健康监控 / 降级策略 |
| `adapter_requirements` | L2 外部系统管理层 | `external.system_mgmt` | 外部系统连接与接入注册 |
| `workflow_templates` / `rule_packs` | L7 知识管理层 | `knowledge.management` | 域包写入与版本治理；规则存入知识库 |
| 域包运行时加载 | L8 知识检索层 | `knowledge.retrieval` | 运行时检索域包业务规则与流程模板 |
| Business Blueprint / Task Graph 生成 | L12 任务编排层 | `orchestration.task` | 需求理解 / 任务拆解 / 路由分配 |
| 多 Agent / 多 VP 协调 | L13 协作编排层 | `orchestration.collab` | 多VP并发 / 进度同步 / 冲突仲裁 |
| `risk_policies` / `human_review_points` | L14 安全策略层 | `security.policy` | 安全合规 / 风险熔断 / 高风险审批 ★ 核心治理 |
| Task Graph 实际执行 | L15 执行监控层 | `execution.monitor` | 工具调用 / 执行监控 / 自动重试 |
| `acceptance_criteria` 校验 | L16/L17/L18 质量保障区 | `quality.inspector` / `quality.reflection` / `quality.gate` | 检查 / 反思 / 门禁 ★ |
| `metrics` / 复盘 | L20 反馈闭环层 | `feedback.loop` | 复盘分析 / 优化建议 / 经验固化 |
| 域包版本发布 | L22 版本发布层 | `release.publish` | 候选审批 / 灰度回滚 ★ 核心治理 |
| 沙盘演练 / 新域包验证 | L23 进化实验层 | `evolution.experiment` | 隔离沙盒实验 / 评测验证 |

**核心治理链路（写死，不可旁路）**：`security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22）。

任何业务域包的流程，无论哪个行业，都必须经过以上四层治理，不允许绕过。

## 十、外部系统与业务域的关系

外部系统类型进入 L2 注册表，不进入主层级。

```text
external_system_type:
  news_content_platform
  social_platform
  ecommerce_platform
  enterprise_system
  search_provider
  communication_provider
  generic_api_provider
  automation_provider
```

业务域包引用这些外部系统类型，而不是直接修改主层。

示例：

```text
BusinessDomainPack
  └── external_system_profiles
        ├── ecommerce_platform
        ├── communication_provider
        ├── enterprise_system
        └── search_provider
```

## 十一、业务中心与平台层级的关系

业务中心负责具体业务对象与状态机。

平台层级不直接持有行业对象字段。

通用业务中心类型可包括：

- 商品中心
- 内容中心
- 用户中心
- 订单中心
- 库存中心
- 售后中心
- 客服中心
- 供应商中心
- 财务结算中心
- 渠道中心
- 经营分析中心

这些中心是否启用，由 Business Domain Pack 声明。

平台主层只负责编排、治理、执行和复盘。

---

## 十二、DoD：业务编排能力最低验收标准

平台具备通用业务编排能力，至少需要满足：

1. 能识别业务域并绑定 `domain_id`。
2. 能加载已发布的 Business Domain Pack。
3. 能在没有业务域包时进入业务学习与建包流程。
4. 能抽取用户目标、约束、时间、预算、风险边界。
5. 能生成 Business Blueprint。
6. 能生成 Task Graph。
7. 每个任务节点具备输入、输出、负责人、依赖、工具、风险等级。
8. 能标记人工确认节点和高风险审批节点。
9. 能检查外部系统是否已注册、凭据是否有效、连接是否健康。
10. 能把任务交给统一治理链执行。
11. 能对失败任务进行重试、降级、熔断或人工接管。
12. 能记录审计、证据、状态、指标快照。
13. 能输出业务结果复盘。
14. 能把稳定流程、规则、策略沉淀为可发布版本。
15. 换业务时不修改主层级，只新增或切换业务域包。

## 十三、文档落位建议

本文件只定义通用机制。

具体业务样板不写入主层级文档，可单独维护为业务域包，例如：

```text
business-domain-packs/
  apparel_retail/
    domain.md
    workflows.md
    risks.md
    metrics.md
    adapters.md

  restaurant_ops/
    domain.md
    workflows.md
    risks.md
    metrics.md
    adapters.md
```

主架构文档只引用业务域包机制，不展开行业字段。

## 十四、最终口径

```text
平台必须满足业务，但业务不进入主层级。
业务通过 Business Domain Pack 进入平台。
L12 orchestration.task 将业务域包转为业务蓝图和任务图。
核心治理链路（写死，不可旁路）：L6 security.permissions / L14 security.policy / L18 quality.gate / L22 release.publish。
L15 负责执行；L16/L17/L18 负责检查与门禁；L20 负责复盘；L22 负责发布；L23 负责实验。
换业务时切换业务域包，不新增主层。
```
