# 业务域包与通用编排机制

**性质**：平台级业务编排补充规范  
**目标**：在不为具体行业新增主层级的前提下，满足不同业务的建模、编排、执行、审计与复盘需求。  
**适用范围**：服装、电商、餐饮、教育、企业管理、本地生活、跨境业务、内容运营等可流程化业务。  

**topology_version**：v23.0（纵向工作流为 **2M + 23L**；组织为 **6 层 OL0–OL5**）。

## 坐标系：6 层组织（横向）与 2M+23L（纵向）

**演进口径（保留）**：组织侧为 **3 层 → 4 层 → 5 层 → 6 层** 终态（OL0–OL5）；工作流侧为 **6 层 → … → 23 层（v23.0，2M+23L）**，全景与 `layer_id` 以上方 **架构速查** 与 `19` 为准。

**读法**：横向解决「谁指挥谁」；纵向解决「任务走哪条管道」。业务域包**不新增主工作流层**，只在既有 **L1–L23（及 2M）** 上提供业务知识与编排输入。

## 核心结论与原则

平台需要满足具体业务，但不应把具体业务写死进主工作流层级。

```text
主层级负责治理职责
业务域包负责业务知识
编排引擎负责生成任务图
Adapter 负责外部平台差异
业务中心负责业务对象与状态
数据层负责事实、证据与审计
```

```text
换业务时，切换或新增 Business Domain Pack。
不因某个行业、某个平台、某类商品而新增主层。
```

主层级应承接稳定的平台治理职责，例如：入口接入、外部系统管理、能力发现、权限管理、知识检索、任务编排、协作推进、安全策略、执行监控、结果检查、质量门禁、反馈闭环、评测、发布、实验。具体业务（服装、餐饮、教育、电商、管理软件等）是**业务域**，不是新的治理职责。

若把具体业务写成主层：主层按行业无限膨胀；治理链与业务菜单混淆；换业务需改主架构；一层一事被破坏；难以跨行业复用。故平台主层不承接行业字段、行业对象和行业流程细节。

**原则（与下表一致处不重复展开）**

- **一层一事**：每个主层只管理一种稳定职责，不管理具体业务类型（例：L2 只管外部系统是谁、凭据与健康，不管订单/课程/菜品等业务对象）。  
- **业务域可插拔**：垂直业务通过 Business Domain Pack 接入（如 `apparel_retail`、`restaurant_ops`、`education_service`、`enterprise_ops`）。  
- **编排与业务分离**：`orchestration.task`（L12）生成任务图；域包向 L12 提供对象、流程、规则、指标与风险点。  
- **治理链不被业务绕过**：任何业务须进入统一治理链；不允许业务域包绕开平台治理。

## 域包在纵向链（2M+23L）上的落点

业务域包机制**不新增主层**，经下列 `layer_id` 分工落地（职责以本表为入口，细节见 `00` / `19`）。

| 域包 / 机制概念 | 层 | layer_id | 职责说明（收敛） |
|---|---|---|---|
| `external_system_profiles`、`adapter_requirements` | L2 | `external.system_mgmt` | 外部系统画像、注册、凭据、健康、接入与 Adapter 需求 |
| `workflow_templates`、`rule_packs` | L7 | `knowledge.management` | 域包写入与版本治理；规则入库 |
| 域包运行时加载 | L8 | `knowledge.retrieval` | 运行时检索域包规则与流程模板 |
| Business Blueprint / Task Graph 生成 | L12 | `orchestration.task` | 目标理解、拆解、路由与任务图生成 |
| 多 Agent / 多 VP 协调 | L13 | `orchestration.collab` | 多 VP 并发、进度同步、冲突仲裁 |
| `risk_policies`、`human_review_points` | L14 | `security.policy` | 合规、风险熔断、高风险审批（核心治理） |
| Task Graph 执行 | L15 | `execution.monitor` | 工具调用、执行监控、自动重试 |
| `acceptance_criteria` 校验 | L16–L18 | `quality.inspector` / `quality.reflection` / `quality.gate` | 检查、反思、门禁（核心治理含 L18） |
| `metrics`、复盘 | L20 | `feedback.loop` | 复盘、优化建议、经验固化 |
| 域包版本发布 | L22 | `release.publish` | 候选审批、灰度回滚（核心治理） |
| 沙盘 / 新域包验证 | L23 | `evolution.experiment` | 隔离沙盒、评测验证 |

**核心治理链路（写死，不可旁路）**：`security.permissions`（L6） / `security.policy`（L14） / `quality.gate`（L18） / `release.publish`（L22）。任何业务域包流程必须经过以上四层，不得绕过。

## 通用业务编排流程

```text
用户提出业务目标
        │
        ▼
识别业务域
        │
        ▼
加载 Business Domain Pack
        │
        ▼
生成 Business Blueprint
        │
        ▼
生成 Task Graph
        │
        ▼
进入统一治理链
        │
        ▼
执行、检查、门禁、审计
        │
        ▼
结果回写、复盘、评测、实验、发布
```

**有已发布域包时**：识别 `domain_id` → 加载已发布 Business Domain Pack → 检查外部系统依赖 → 匹配能力与 Adapter → 生成 Business Blueprint → 生成 Task Graph → 进入统一治理链执行 → 回写结果与复盘指标。

**无业务域包时**：进入业务学习流程 → 采集规则、对象、流程、风险、指标 → 生成临时域包草案 → 沙盘演练 → 人工确认关键规则与高风险节点 → 过门禁后为候选业务包 → 经发布层发布后，后续任务方可复用。

**约束（高风险）**：无业务域包时，**不允许**直接进入高风险生产执行。

## 最终口径

```text
平台必须满足业务，但业务不进入主层级。
业务通过 Business Domain Pack 进入平台。
L12 orchestration.task 将业务域包转为业务蓝图和任务图。
核心治理链路（写死，不可旁路）：L6 security.permissions / L14 security.policy / L18 quality.gate / L22 release.publish。
L15 执行；L16/L17/L18 检查与门禁；L20 复盘；L22 发布；L23 实验。
换业务时切换业务域包，不新增主层。
```

## 附录 A：对象结构（定义与最小字段）

### A.1 Business Domain Pack

Business Domain Pack 是某业务领域的可加载知识与流程包；**不改变**主工作流层级，只为编排层提供业务上下文。

```text
BusinessDomainPack
├── domain_id
├── domain_name
├── business_objects
├── workflow_templates
├── rule_packs
├── external_system_profiles
├── adapter_requirements
├── risk_policies
├── metrics
├── human_review_points
├── acceptance_criteria
└── sample_tasks
```

| 字段 | 说明 |
|---|---|
| `domain_id` | 业务域稳定标识，例如 `apparel_retail`、`restaurant_ops` |
| `domain_name` | 业务域名称 |
| `business_objects` | 该业务涉及的对象模型 |
| `workflow_templates` | 常见流程模板 |
| `rule_packs` | 业务规则、平台规则、禁用动作、合规边界 |
| `external_system_profiles` | 该业务常用外部系统类型 |
| `adapter_requirements` | 需要哪些 Adapter 能力 |
| `risk_policies` | 高风险动作、审批要求、阻断条件 |
| `metrics` | 业务指标与复盘口径 |
| `human_review_points` | 必须人工确认的节点 |
| `acceptance_criteria` | 业务完成的验收标准 |
| `sample_tasks` | 样板任务，用于演练、评测和回归 |

### A.2 Business Blueprint

针对一次具体业务目标的执行蓝图，来自：用户目标 + Business Domain Pack + 当前约束 + 外部系统状态。只描述本次如何落地，**不承担执行**；须再转为 Task Graph 并进入治理链。

```text
BusinessBlueprint
├── blueprint_id
├── domain_id
├── goal
├── constraints
├── stages
├── deliverables
├── required_capabilities
├── external_dependencies
├── risk_points
├── human_checkpoints
└── success_metrics
```

### A.3 Task Graph

Agent、工具、外部系统可执行的任务结构；由 L12 基于 Business Blueprint 生成。

```text
TaskGraph
├── task_graph_id
├── nodes
├── edges
├── owner_agent
├── input_contract
├── output_contract
├── tool_requirements
├── approval_required
├── risk_level
└── rollback_policy
```

```text
TaskNode
├── node_id
├── node_type
├── owner
├── inputs
├── outputs
├── dependencies
├── external_system_refs
├── required_adapters
├── risk_level
├── approval_required
├── retry_policy
├── rollback_policy
└── evidence_required
```

### A.4 Execution State

描述执行中的真实状态；须可审计、可回放、可恢复，不得只存在于对话中。

```text
ExecutionState
├── execution_id
├── domain_id
├── blueprint_id
├── task_graph_id
├── current_stage
├── node_status
├── external_call_status
├── approval_status
├── error_state
├── retry_count
├── audit_refs
├── evidence_refs
└── metric_snapshots
```

## 附录 B：外部系统类型与业务中心

外部系统类型进入 **L2** 注册表，不进入主层级「行业化」扩展。

```text
external_system_type:
  news_content_platform
  social_platform
  ecommerce_platform
  enterprise_system
  search_provider
  communication_provider
  generic_api_provider
  automation_provider
```

域包引用类型而非改主层，示例：

```text
BusinessDomainPack
  └── external_system_profiles
        ├── ecommerce_platform
        ├── communication_provider
        ├── enterprise_system
        └── search_provider
```

业务中心负责具体业务对象与状态机；平台主层不直接持有行业对象字段。通用业务中心类型可包括：商品中心、内容中心、用户中心、订单中心、库存中心、售后中心、客服中心、供应商中心、财务结算中心、渠道中心、经营分析中心等。**是否启用**由 Business Domain Pack 声明；平台主层负责编排、治理、执行与复盘。

## 附录 C：DoD 与文档落位

**DoD：业务编排能力最低验收标准**

1. 能识别业务域并绑定 `domain_id`。  
2. 能加载已发布的 Business Domain Pack。  
3. 能在没有业务域包时进入业务学习与建包流程。  
4. 能抽取用户目标、约束、时间、预算、风险边界。  
5. 能生成 Business Blueprint。  
6. 能生成 Task Graph。  
7. 每个任务节点具备输入、输出、负责人、依赖、工具、风险等级。  
8. 能标记人工确认节点和高风险审批节点。  
9. 能检查外部系统是否已注册、凭据是否有效、连接是否健康。  
10. 能把任务交给统一治理链执行。  
11. 能对失败任务进行重试、降级、熔断或人工接管。  
12. 能记录审计、证据、状态、指标快照。  
13. 能输出业务结果复盘。  
14. 能把稳定流程、规则、策略沉淀为可发布版本。  
15. 换业务时不修改主层级，只新增或切换业务域包。

**文档落位建议**：本文件只定义通用机制；具体行业样板放入独立业务域包目录，不写入主层级文档。

```text
business-domain-packs/
  apparel_retail/
    domain.md
    workflows.md
    risks.md
    metrics.md
    adapters.md

  restaurant_ops/
    domain.md
    workflows.md
    risks.md
    metrics.md
    adapters.md
```

主架构文档只引用业务域包机制，不展开行业字段。
