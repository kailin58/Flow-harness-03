# 16_前端多状态看板规范

**性质**：前端实现规范（V1）  
**目标**：建设可追踪、可接管、可回滚的多状态前端看板  
**适用范围**：会话中间层前端、编排执行前端、人工接管与经营复盘前端（PC/H5/小程序/平板/APP）  
**架构定位**：跨层观察者，不占 L 槽位，不参与管道消息流转  
**组织责任**：VP01 → 总监6 DigitalOps（数据看板与指标汇总 / 运营异常识别）

**数据来源（只读，不写入任何层）**：

| 来源 | 提供数据 | 实时性 |
|---|---|---|
| M1.1 全局任务账本 | 所有活跃 trace_id / 当前所在层 / 并发状态 | 实时 |
| M1.2 全局水位表 | 每层队列压力 / 系统健康指标 | 实时 |
| S3 审计库 | 历史审批状态 / 异常记录 / 操作留痕 | 追溯 |
| S1 上下文库 | 任务目标 / 中间结果 / 决策日志 | 任务级 |

**告警推送**：看板异步告警由 `notification.dispatch`（L19）推送给用户，看板本身为拉取型。

## 一、设计目标

前端必须支撑以下能力：

1. 多状态可视化（主状态/子状态/门禁状态/异常状态）
2. 关键动作可控（发布、支付、删除等高风险动作）
3. 人工接管可执行（审批、回滚、恢复、升级）
4. 证据链可追溯（trace_id、审计ID、证据链接）

## 二点一、多端矩阵与职责分层（写死）

### 2.1 终端矩阵

- `PC`：主控端（全量功能，治理与运维优先）
- `H5`：轻量访问端（查询、审批、确认、查看证据）
- `小程序`：高频操作端（消息触达、快捷审批、状态查看）
- `平板`：现场协同端（看板监控、流程接管、多角色会审）
- `APP`：移动主端（通知、审批、异常接管、关键动作确认）

### 2.2 多端能力分层

1. 全量能力（仅PC/平板）：
   - 全流程编排操作
   - 回滚与恢复控制
   - 高级诊断与证据审计
2. 标准能力（APP/H5）：
   - 状态查看
   - 人工审批
   - 异常接管与确认
3. 轻量能力（小程序）：
   - 状态提醒
   - 快速审批
   - 阻塞原因查看

### 2.3 一致性要求（写死）

1. 五端共享同一状态枚举与错误码口径。
2. 五端共享同一审计追踪键（`trace_id`）。
3. 高风险动作默认仅允许 `PC/APP` 执行，其他端只允许发起申请。
4. 多端状态同步延迟需可观测并告警。

## 二点二、动物角色可视化规范（写死）

前端允许使用“动物角色”作为 Agent 的视觉表达，但必须遵守“视觉层不改变业务真值”原则。

### 2.2.1 角色映射原则

1. 动物角色仅用于展示身份与状态，不得替代真实 `agent_id` 与层级字段。
2. 每个角色必须绑定真实元数据：`agent_id`、`level`、`status`、`risk_level`。
3. 同一角色可复用皮肤风格，但主责能力与权限以后端字段为准。

### 2.2.2 角色状态映射（最小）

- 工作中：高亮动画 + 当前任务标签
- 学习中：学习进度环 + 学院轨道标签
- 待命中：低亮待机态 + 可分配提示
- 异常中：风险色边框 + 接管入口
- 审批中：门禁图标 + 阻塞原因

### 2.2.3 禁止项

1. 禁止通过前端角色交互绕过审批、门禁、回滚流程。
2. 禁止出现“角色状态”与“后端真实状态”不一致。
3. 禁止使用角色动画掩盖失败/阻塞/降级状态。

### 2.2.4 验收要求

1. 角色卡片点击后可展开真实审计信息（`trace_id`、证据链接）。
2. 角色展示状态与后端状态一致率需可观测。
3. 异常角色必须支持一键跳转人工接管面板。

## 三、状态模型（前端统一口径）

### 3.1 主状态

`CREATED / ACTIVE / PAUSED / RESUMED / SETTLED / ARCHIVED`

### 3.2 子状态

`PLAN_READY / RUNNING / HUMAN_PENDING / QUALITY_PENDING / PUBLISH_PENDING / DONE / FAILED`

### 3.3 门禁状态

`LEARNING_PENDING / REHEARSAL_PENDING / ACADEMY_BLOCKED / PAYMENT_PENDING / WEB_CHECK_PENDING`

### 3.4 异常状态

`RETRYING / DEGRADED / ESCALATED / ROLLBACKING`

---

## 四、交互规则（写死）

1. 状态更新必须由后端事件驱动，前端不允许推断关键状态。
2. 任一 `PENDING/BLOCKED` 状态必须显示：
   - 阻塞原因
   - 下一步动作
   - 责任角色
3. 高风险动作必须二次确认，并校验动作对象一致性。
4. 进入异常状态后必须自动展示“回滚/人工接管”入口。
5. 所有状态变更都必须可追溯到 `trace_id`。

## 六点一、学习虚拟币排行榜（写死）

### 6.1 展示目标

展示“谁在学习、谁贡献了学习产出、谁获得了虚拟币奖励”，支持历史周期对比。

### 6.2 榜单维度（最小）

1. 总榜：按 `coin_balance` 排序
2. 学习榜：按 `learning_score` 排序
3. 进步榜：按周期增量 `coin_income_delta` 排序
4. 贡献榜：按“经验沉淀/复用贡献”得分排序

### 6.3 展示规则

1. 榜单必须可切换周期：日 / 周 / 月。
2. 榜单必须支持按组织层级筛选：VP / 总监 / 经理 / Agent。
3. 角色卡片展示“当前币值 + 本期增减 + 学习状态”。
4. 排行变化需展示来源说明（任务、学习、评测、惩罚/扣减）。

### 6.4 治理约束

1. 虚拟币仅作为内部激励与学习运营指标，不直接替代真实财务记账。
2. 任何加币/扣币必须带审计事件与规则ID。
3. 异常增减（突增、批量同值）必须触发风控告警与复核。
4. 榜单数据必须以后端结算结果为准，前端不得本地计算真值。

---

## 七、异常场景处理（最小）

1. 页面状态异常（`E_WEB_PAGE_STATE_INVALID`）
   - 自动降级
   - 显示人工接管入口
2. 支付审批缺失（`E_PAYMENT_APPROVAL_MISSING`）
   - 阻断发布
   - 跳转审批面板
3. 学院门禁阻断（`E_ACADEMY_GATE_BLOCKED`）
   - 禁止发布
   - 展示未通过原因与补救路径

---

## 八、验收标准（DoD）

1. 可同时展示四层状态并正确联动。
2. 跑通 1 条完整链路：
   `ACTIVE -> RUNNING -> HUMAN_PENDING -> DONE -> SETTLED`
3. 跑通 1 条异常链路：
   `RUNNING -> FAILED -> ESCALATED -> ROLLBACKING`
4. 高风险动作具备二次确认与回执校验。
5. 每次状态变更可查审计ID与证据链接。
6. PC/H5/小程序/平板/APP 五端状态口径一致并通过联调验收。
7. 动物角色展示层与后端真实状态一致，并通过审计抽检。
8. 学习虚拟币排行榜支持多维筛选、周期切换与审计追溯。

## 九、设备配置与性能优化规范（V1）

### 9.1 设备配置基线（建议）

#### 开发最低档（本地开发）

- CPU：4 核及以上
- 内存：16GB
- 磁盘：可用空间 50GB+（日志/截图/回放）
- 网络：稳定外网与低抖动

#### 联调推荐档（多端联调）

- CPU：8 核及以上
- 内存：32GB
- 磁盘：SSD，100GB+ 可用空间
- 显示：双屏优先（看板 + 调试）

#### 生产执行档（持续运行）

- CPU：按并发规模弹性扩容
- 内存：32GB+（视自动化并发上调）
- 磁盘：日志与证据留痕分盘存储
- 网络：独立带宽监控与故障告警

### 9.2 前端渲染优化策略（写死）

1. 首屏分层加载：先渲染状态总览，再加载图表/动效。
2. 列表与时间线使用虚拟滚动，避免全量渲染。
3. 角色与场景资源懒加载，优先加载可视区素材。
4. 高耗能动画在低性能设备自动降级（降帧/静态替代）。
5. 高频状态更新采用增量更新，禁止全量重绘。

### 9.3 监控指标（前端性能）

- 首屏可交互时间（TTI）
- 状态刷新延迟
- 页面卡顿率（长任务占比）
- 内存峰值与泄漏趋势
- 动画降级触发率

### 9.4 性能治理规则

1. 无性能基线数据，不得宣称“已优化”。
2. 每次优化必须给出“优化前/后”对比证据。
3. 低端设备体验不得低于最低可用标准（状态可见、审批可用、回执可追）。

## 十、AGENT看板页面设计（默认方案）

> 本节为默认设计方案：在无额外设计干预前，前端按本方案直接实施。

### 10.1 页面信息架构

页面采用 `Header + Left + Center + Right + Bottom` 五区结构：

1. Header（全局控制区）
   - 时间范围切换（日/周/月）
   - 组织过滤（VP/总监/经理）
   - 全局搜索（`agent_id` / 任务ID / 审计ID）
2. Left（Agent生态区）
   - 动物角色网格视图
   - 工作中/学习中/待命中/异常中筛选
   - 角色点击展开详情抽屉
3. Center（主看板区）
   - 状态总览卡（四层状态统计）
   - 实时执行时间线（推进、阻塞、回滚）
   - 任务热力图（组织 x 任务类型）
4. Right（治理接管区）
   - 待审批队列
   - 异常告警队列
   - 人工接管入口
5. Bottom（经营学习区）
   - ROI / Token / 成功率趋势
   - 学习虚拟币排行榜
   - 学院评测通过率与门禁拦截率

### 10.2 核心模块清单

- `AgentWorldPanel`
- `AgentStatusCounter`
- `ExecutionTimelineBoard`
- `GovernanceQueuePanel`
- `TakeoverActionPanel`
- `LearningCoinLeaderboardPanel`
- `OpsAndAcademyMetricsPanel`
- `AgentAuditDrawer`

### 10.3 默认视觉与交互

1. 风格：动物世界主题 + 工业信息看板混合。
2. 状态优先级：异常 > 门禁阻塞 > 审批中 > 工作中 > 学习中 > 待命中。
3. 颜色语义：
   - 正常：绿色
   - 学习：蓝色
   - 门禁/审批：黄色
   - 异常/回滚：红色
4. 角色动效用于提示，不得遮挡风险信息。

### 10.4 默认交互流程

1. 点击角色 -> 打开 `AgentAuditDrawer` -> 查看真实字段与审计链。
2. 点击异常 -> 跳转 `TakeoverActionPanel` -> 接管或回滚。
3. 点击排行 -> 查看加减币来源与规则ID -> 追溯任务证据。
4. 切换组织维度 -> 全区联动刷新，统一时间窗口。

### 10.5 默认上线范围（MVP）

第一阶段（必须）：

- 四层状态总览
- 动物角色列表与详情
- 待审批/异常接管队列
- 学习虚拟币排行榜
- 审计证据抽屉

第二阶段（增强）：

- 任务热力图
- 学院能力地图
- 场景皮肤切换
