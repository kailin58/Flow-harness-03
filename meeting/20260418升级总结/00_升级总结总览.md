## 平台目标

**终态定位：**
> 工作流引擎 + 数字员工系统 + 知识操作系统 + 智能协作网络

**当前阶段（第一阶段：2026—2028）：**
> 从"可用的 AI"走向"可控的 Agent 系统"

**阶段核心目标：**
> 用起来很聪明，结果稳定、过程可控、规模化接入业务。

**建设原则：**
> 不追求最复杂的多智能体架构，而是把基础能力打牢。

**第一阶段关键词：** 标准化 · 可控性 · 工具化 · 验证化

## 全局前置沟通规则（统一口径）

- 任何任务下发前，最少完成 `2` 轮目标沟通（目标、范围、完成标准三项对齐）。
- 任务理解必须配图（流程图/结构图/决策图至少一种）后再进入执行。
- 本规则当前作为流程与文档规范执行，系统侧“上线强制门禁”暂不启用。
- 治理模型统一采用“规则写死、参数可配、边界写死”：
  - 写死规则：流程门禁、接管链路、审计要求、发布与回滚约束。
  - 可配置参数：SLA、复评阈值、紧急策略、试点范围等运营参数。
  - 边界写死：仅允许边界内收紧，不允许突破平台红线；配置变更必须版本化发布并写审计。
- 接口原则：所有接口不做占位；设计可做全局通用与扩展增强，但接口定义本身必须可直接落地，不保留仅供后补的占位接口。
- 全局配置原则：凡可由配置承载的真实业务值，一律配置化管理，能配置就不写死。
- 落地口径：正文默认只固化字段名、规则、约束、示例值与生效顺序；真实实值通过 `platform -> company -> department -> user` 配置链注入并审计。
- 外部接入原则：所有外部接入的 Agent、scale/扩展服务、模型及 MCP 接口均为**单向输入 + 强隔离 + 加密通信**；外部组件不得直读我方文件、源码、`S1-S4`、配置与凭据，且返回结果必须重新进入我方治理链。
- **纵向层间通讯安全（企业级）**：信封必带隔离与主体字段；旁路、反向与跳层**仅允许**由纵向路由引擎按白名单执行；阈值、配额、签名策略等走 **`platform → company → department → user`** 配置链并审计。加固基线见 [19_层级扩展与跨层通讯协议.md](./19_层级扩展与跨层通讯协议.md) **第十四章**；**落地顺序与 DoD** 见同文档 **第十五章**。
- **平台默认（全域一致，无豁免不得偏离）**：凡人机分流、必审超时、对客进度/消息枚举，**一律**遵守 [09学院](./09学院.md) **`17.7.5` + `17.7.8`**（必审超时仅等待、三态 `IN_FLIGHT` / `HUMAN_PENDING` / `SETTLED`、免审可自动推进）；与预印本/内容分级并行的基线见 **`15.2` / `15.3`**。仅 **L10 已发布之专项豁免条款**可替代，且须可审计。

## 核心决策一览

### 组织架构（横向）
- Agent层级：3层 → 4层 → 5层 → **6层**（新增 VP 副总层、**经理层**、副Deep Agent 层；终态链：总监 → 经理 → 子Agent → 副Deep）
- VP 层由平台统一治理（11个为当前基线，可按配置扩展新增）
- VP01 数字技术VP 下设 6 个总监（Explore/Plan/General/Inspector/Research/DigitalOps）
- 其他VP预设总监方向按域规划，详见：[01_3层升4层架构.md](./01_3层升4层架构.md)
- 预设VP基线（11个）：VP01~VP11；可按配置激活、停用与新增扩展VP
- 企业可在既有VP下增加总监（受权限与配额约束）
- 总监下挂 **经理**（M01–M11 命名空间与 VP 对齐，经理实例数不限）；经理下子Agent数量为 N（按任务负载弹性扩缩）
- 组织能力树写死：**VP → 多总监 → 多经理 → 多子Agent →（按需）副Deep → 多Skills**
- 治理约束写死：Skills 调用受 `security.permissions`（L6）/ `security.policy`（L14）约束；子Deep Agent输出必须经 `quality.inspector`（L16）/ `quality.gate`（L18）；进化只在 `evolution.experiment`（L23）、上线只经 `release.publish`（L22）
- COMM_RULES 从"基于Agent ID"改为"基于level"

### 工作流（纵向）★ 本次更新
- 工作流演进：**6层 → 9层 → 11层 → 15层 → 16层 → 23层**，当前为 **23层（v23.0）**
- 基础设施模块：**M1** `src/pipeline-control.js`（控制面/管道大脑）/ **M2** `src/layer-router.js`（数据面/管道神经），不占 L 槽位
- 本轮新增7层：**L3 模型管理**（model.management）/ **L4 原子工具**（tool.management）/ **L7 知识管理**（knowledge.management）/ **L9 会话管理**（session.lifecycle）/ **L10 流程编排**（process.orchestration）/ **L11 流程智能**（process.intelligence）/ **L19 通知推送**（notification.dispatch）
- **layer_id 永久稳定，layer_order/layer_display 随拓扑变化，所有代码引用必须用 layer_id**
- **每层只管一个专项**：主层定义领域边界，多职责通过子层拆分承载
- **扩展原则**：现有主层内新增能力挂子层 `Lx.n`；全新领域才开新主层，编号顺延，需评审；禁止插层
- **子层命名规范**：`Lx.1` 起按职责递增，编号永不复用
- 详见：本文件"15层工作流全景（含子层）"章节

### 模型
- 三种来源：平台内置 / BYOK / 本地模型
- 不引入重框架，使用官方SDK
- 配置走 `.flowharness/models.json` + 环境变量引用Key
- 模型分配/调度归属 **`model.management`（L3）模型管理层**（路由、降级、成本与SLA权衡）
- `execution.monitor`（L15）仅负责执行期监控与反馈，`orchestration.collab`（L13）仅负责多Agent协作冲突协调（不承载模型分配主责）

### 协作
- 补充 Task State Machine（最缺的一环）
- 添加 Agent 看板（终端版先做，Web版后做）

### 学院（跨层能力）★ 本次补充
- 学院采用「**可跑顺、可闭环、可扩展**」方案，详见：[09学院.md](./09学院.md)
- 组织归属：**VP04（AgentOps）主责**课纲/测试/证书运营，**VP02（Governance）协管**合规、审计与发布红线
- 运行口径：**平台内核写死 + 变量版本化 + 自动化主导 + 人只验收结果**
- 课程与来源：默认对齐“VP 数量 = track 数量”；来源含论文/OER/专业官网/人工多模态投喂

## `security.permissions`（L6）与 `security.policy`（L14）职责区分（关键说明）

> 两层最容易混淆，必须区分清楚再实施，否则实现时职责必然交叉。

| 维度 | L6 `security.permissions` 权限管理层 | L14 `security.policy` 安全策略层 |
|------|-------------|-------------|
| 核心问题 | **Who can do What**（谁有权限做什么） | **How to do safely**（怎么做才合规） |
| 检查对象 | 用户身份 / 角色 / 租户归属 | 操作内容 / 文件路径 / 命令参数 |
| 典型场景 | 这个用户/租户有没有权限发起这类任务 | 这个文件读取操作是否在白名单内 |
| 实现机制 | RBAC / 租户隔离 / 审批授权 | Deny-by-Default / 白名单 / 合规规则 |
| 触发时机 | 任务进入编排前（`security.permissions` 先于 `orchestration.task`） | 每次工具调用/操作执行时（`security.policy` 伴随 `execution.monitor`） |

**记忆口诀**：`security.permissions` 管"人"，`security.policy` 管"事"。

## Skills 生命周期与分层约束

定位：Skills 为平行能力插件，由 Agent 挂载与调用，不构成新的组织层级。

生命周期：
- 注册（Registry）-> 沙盒验证（`evolution.experiment` L23）-> 质量门禁（`quality.gate` L18）-> 发布（`release.publish` L22）-> 运行监控（`execution.monitor` L15）-> 回滚/下线

分层约束：
- **`orchestration.task`（L12）**：Skills 路由与调用策略（由任务编排层统一调度）。
- **`security.permissions`（L6）**：Skills 调用权限控制（角色/租户/作用域）。
- **`security.policy`（L14）**：Skills 运行安全策略（命令/文件/API 白名单）。
- **`quality.gate`（L18）/ `release.publish`（L22）**：Skills 发布门禁、版本发布与回滚。
- **`evolution.experiment`（L23）**：用户新增或变更 Skills 的实验与验证域。

治理要求：
- 用户可新增 Skills，但仅在 USER 域生效，不得污染 SYS 预设域。
- Skills 必须版本化、可审计、可回滚。
- 高风险 Skills 默认拒绝，需显式审批放行。

### Skills VP 归属模型

Skills 按"调用者归属"分三级，不按功能分类：

```
平台级 Skill（Platform Skill）
  归属：VP02 平台治理VP 负责注册与发布
  调用权：任意VP、任意总监均可调用
  示例：文件读写、代码执行、HTTP请求

VP级 Skill（VP Skill）
  归属：对应VP自己负责注册与发布
  调用权：仅该VP及其下属总监可调用
  示例：CRM数据读取（归 VP07 销售增长VP）

总监级 Skill（Director Skill）
  归属：总监负责注册，所属VP审批发布
  调用权：仅该总监及其经理与子Agent可调用
  示例：特定数据库查询模板
```

**Skills 依赖管理：**
```
每个 Skill 声明 dependencies[]
发布时由 quality.gate（L18）质量门禁层做依赖树检查
禁止循环依赖
上游 Skill 升级时，自动通知所有依赖它的 Agent
```

```
escalation_id      唯一ID（全链路可追踪）

decision_options   提供给人工的选项列表（最少2个，最多5个）
timeout_sec        人工响应超时时间
on_timeout_action  超时后的自动处理（默认：最保守选项）
```

## 长期维护边界（执行准则）

为保证平台可持续演进且不失控，长期维护遵循以下六个边界：

1) **组织边界**
- VP 层平台固定治理（企业不可新增VP）。
- 企业可在既有VP下扩总监；总监下经理实例不限；经理下子Agent按N弹性扩缩。

2) **分层边界**
- **核心治理链路（写死，不可绕过）**：`security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22）。
- 任意能力扩展不得破坏分层职责边界。

3) **扩展边界**
- 用户可新增 Skills，但仅在 USER 域生效。
- Skills 调用必须受 `security.permissions` 权限层与 `security.policy` 安全策略层约束。

4) **质量边界**
- 新能力默认先在 `evolution.experiment`（L23）沙盒验证，再经 `quality.gate`（L18）/ `release.publish`（L22）门禁发布。
- 任何上线能力必须支持版本化与可回滚。

5) **审计边界**
- 全链路行为必须可追踪、可回放、可追责。
- 高风险动作必须有审批记录与执行留痕。

6) **拓扑演进边界**
- 23层是当前基线，不是最终上限；可通过配置新增层级与角色。
- 禁止在代码中写死层级总数、固定角色数量、固定路由映射。
- 拓扑变更必须版本化（`topology_version`）并通过 `release.publish`（L22）审批后生效。
- 每层允许扩展子层级，子层级数量不设上限（受治理规则与资源配额约束）。
- VP数量不设固定上限，11个仅为默认基线；新增VP必须走配置发布与治理审批。

### S3 分布式写入保障原则（缺口8）

> **补充日期**：2026-04-14
> **性质**：实现约定——"写入不可拒绝"在分布式场景下的具体保障机制
> **状态**：✅ 原则已定，实施时遵守
>
> **Agent 执行端摘要**：仓库根目录 [AGENTS.md](../../AGENTS.md) 已收录 **6 层** Agent 组织与本节三条原则的精简版，便于 Claude Code / Cursor / Codex 直接对齐；S3 细节与图示以本节为准。

**问题背景**："写入不可拒绝"是 S3 的核心约束，但在分布式场景下（存储服务暂时不可用、网络抖动、节点重启等），若处理不当，会导致主流程被 S3 写入阻塞、审计失败中断业务流程，或业务逻辑误依赖 S3 返回值产生错误控制流。

**三条保障原则（写入架构文档，实现时遵守）：**

**原则1：S3写入必须异步，不阻塞主流程**
```
主流程（L3/L4/L6/L8/L10 等任意层）
  │
  ├─ 执行业务逻辑（同步，主流程）
  │
  └─ 触发 S3 审计写入（异步，fire-and-forget）
       → 不等待写入结果，不阻塞下一步业务动作
```

**原则2：S3写入失败时的兜底链路**
```
S3 写入请求
  │
  ├─ 成功 → 正常归档
  │
  └─ 失败（S3不可用 / 超时 / 网络错误）
       │
       ├─ Step1: 写本地缓冲文件（Local Buffer）
       │          路径：<工作目录>/.audit_buffer/<timestamp>_<event_id>.json
       │          格式：与 S3 审计记录相同结构（保证后续可无损回写）
       │
       ├─ Step2: 定时重试（指数退避）
       │          1s → 2s → 4s → 8s → ... → 上限 N 次（默认10次）
       │          重试成功 → 将本地缓冲文件内容写入 S3 → 删除缓冲文件
       │
       └─ Step3: 重试超过 N 次 → 触发系统告警
                  告警级别：P1（高优先级）
                  告警内容：未同步的缓冲文件数量 + 最早未同步时间戳
                  ⚠️ 告警不触发业务流程中断，主流程继续执行
```

**原则3：S3 是"后置记录"，不是"前置控制"**
```
正确写法：业务决策先行 → 异步写 S3（后置记录）
错误写法：等待 S3 写入结果 → 基于结果决定下一步业务逻辑

任何层在写 S3 时，不可以：
  × 等待 S3 写入返回值再决定下一步业务逻辑
  × 把 S3 写入成功作为业务流程推进的前提条件
  × 因为 S3 不可用而拒绝继续执行主流程
```

**本地缓冲文件一致性保障（实现时遵守）：**
- 缓冲文件采用原子写入（先写临时文件再重命名），防止写入中途崩溃导致文件损坏。
- 系统启动时自动扫描缓冲目录，恢复未同步的历史记录。
- 缓冲文件本身不作为"已审计"凭证，S3 写入成功后方视为审计完成。

## 经验库 → 进化闭环（三段式路径）

### 第一段：经验沉淀（L13 → S2）

```
任务完成后，`feedback.loop`（L20）反馈闭环层生成「经验条目」写入 S2 制品库：
  task_type          任务类型
  success            true / false
  failure_layer      失败发生在哪一层（如有）
  correction_trace   人工修正记录（如有）
  final_result       最终正确输出
  confidence_delta   本次任务使系统置信度提升/下降值
```

### 第二段：实验触发（S2 → `evolution.experiment` L23）

```
触发条件（满足任意一个）：
  - 同类任务累计失败率超阈值（如 > 20%）
  - 人工修正记录累计超N条（如 > 10条）
  - 管理员手动触发实验

`evolution.experiment`（L23）进化实验层从 S2 拉取该类型经验条目
→ 生成优化提案（Proposal）

提案内容范围（低风险优先）：
  · Prompt 模板变更
  · 工具调用顺序调整
  · Agent 路由策略调整
  · 检索策略优化
  （不直接涉及核心代码）
```

### 第三段：提案发布（`evolution.experiment` L23 → `quality.gate` L18 → `release.publish` L22）

```
evolution.experiment（L23）在隔离沙盒中验证提案效果（新策略 vs 旧策略 A/B对比）
    ↓
quality.gate（L18）质量门禁层评估（新策略评分 > 旧策略才通过）
    ↓
release.publish（L22）版本发布层：审批 → 灰度上线 → 全量
    ↓
全量后同步更新 S4 规则库对应规则版本
```

**闭环保障原则：**
```
进化只在 evolution.experiment（L23）用户隔离沙盒发生
系统预设域不参与进化
任何进化结果不可直写生产，必须经 release.publish（L22）发布链路
```

## 工具接入层（Tool Integration Layer）

> **补充日期**：2026-04-13
> **性质**：架构扩展——Tool Port 的完整规划
> **核心命题**：未来最关键的能力不是"模型会不会答"，而是"系统能不能拿到正确上下文并执行正确动作"

---

### 定位：在六边形架构中的位置

工具接入层是**Tool Port 的具体实现集合**，是 Agent 与外部世界之间的标准化桥梁。

```
内核（CEO + VP + 编排逻辑）
        ↓ 只能通过 Tool Port 调用
   ┌────────────────────────┐
   │      Tool Port         │  ← 统一接口契约，内核永不直连外部
   └────────────┬───────────┘
                ↓
   ┌────────────────────────────────────────────────┐
   │              工具接入层（Tool Adapters）         │
   │  T1文件  T2知识库  T3数据库  T4企业系统          │
   │  T5 API  T6搜索   T7表格   T8自动化平台         │
   └────────────────────────────────────────────────┘
                ↓
        外部真实系统
```

**关键原则**：适配器变化不影响内核；内核升级不破坏适配器；工具故障不传导到 Agent 层。

### 8类工具适配器规划

按接入特性分为四组：

#### 第一组：本地资源类

**T1 文件系统适配器（FileSystemAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 读写本地/容器文件、目录遍历、文件监听、临时文件管理 |
| 核心风险 | 路径穿越攻击、意外删除、写入生产配置 |
| 权限策略 | 白名单目录制（Deny-by-Default），读/写/删 三级独立授权 |
| 失败处理 | 不存在返回空而非报错；写失败必须记录审计事件（S3可用则异步写入S3，S3不可用则写本地缓冲文件，后台重试回写；不可因S3不可用而阻塞主流程） |
| 与现有层关系 | `security.policy`（L14）安全策略层负责路径白名单校验；S2 制品库归档工具输出 |

---

#### 第二组：知识检索类

**T2 知识库适配器（KnowledgeBaseAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 向量检索、全文检索、文档片段召回、知识图谱查询 |
| 核心价值 | 为 Agent 提供领域专有上下文，解决模型训练数据时效问题 |
| 权限策略 | 按知识域隔离（租户A的知识库不可被租户B检索） |
| 质量机制 | 召回结果置信度低于阈值时，标记为"低置信参考"而非直接使用 |
| 与现有层关系 | 召回结果写入 S1 上下文库供全链路使用；`quality.inspector`（L16）Inspector 做来源核查 |

**T6 搜索引擎适配器（SearchAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 互联网实时搜索、内网搜索、代码库搜索、文档搜索 |
| 核心风险 | 搜索结果噪音、幻觉注入、敏感信息泄露给外部搜索引擎 |
| 权限策略 | 外网搜索需显式开启；内网搜索按角色隔离 |
| 失败处理 | 搜索超时返回空集，不阻断主流程；结果必须标注来源与时间戳 |
| 与现有层关系 | 结果写入 S1 上下文库；`security.policy`（L14）安全策略层过滤敏感查询词 |

#### 第三组：数据存取类

**T3 数据库适配器（DatabaseAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | 关系型DB查询/写入、NoSQL读写、时序数据库、向量数据库 |
| 核心风险 | SQL注入、意外全表删除、跨租户数据污染 |
| 权限策略 | 只读/读写/管理三级；DDL操作（建表/删表）默认禁止 |
| 操作约束 | 写操作必须在事务内；影响行数超过阈值需二次确认 |
| 与现有层关系 | 写操作触发 S3 审计记录；`quality.gate`（L18）门禁可设置数据变更审批规则 |

**T7 表格系统适配器（SpreadsheetAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | Excel/Google Sheets 读写、数据透视、公式计算、图表生成 |
| 典型场景 | 财务报表生成、数据分析输出、业务指标汇总 |
| 权限策略 | 按文件/Sheet/单元格区域三级授权 |
| 失败处理 | 格式异常时返回原始数据而非报错；写失败保留草稿 |
| 与现有层关系 | 输出文件写入 S2 制品库；版本变更触发 S3 修改历史 |

#### 第四组：外部系统类

**T4 企业系统适配器（EnterpriseAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | ERP、CRM、HRM、OA审批、工单系统、财务系统 |
| 核心特点 | 这类系统的操作往往**不可逆**（如财务记账、合同签署） |
| 权限策略 | 读取默认允许；写入/审批操作强制走 `quality.gate`（L18）门禁 + `release.publish`（L22）审批链 |
| 幂等要求 | 所有写操作必须携带幂等Key，防止网络重试导致重复提交 |
| 与现有层关系 | 操作结果写 S3 审计库；审批状态同步到 S3 审批状态链 |

**T5 API服务适配器（APIAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | REST API、GraphQL、gRPC、Webhook 接收、OAuth 授权流 |
| 核心风险 | 密钥泄露、频率超限、第三方服务不稳定传导到内部 |

**T8 自动化平台适配器（AutomationAdapter）**

| 项目 | 说明 |
|------|------|
| 覆盖能力 | RPA机器人调度、Zapier/n8n工作流触发、定时任务、消息队列 |
| 核心特点 | 触发后执行异步，结果需要回调或轮询获取 |
| 权限策略 | 触发权限与结果读取权限分离；高影响自动化需 `quality.gate`（L18）门禁审批 |
| 异步机制 | 触发时生成追踪ID写 S1 上下文库；回调结果写 S2 制品库 |
| 与现有层关系 | 异步状态变更触发 S3 审计；`orchestration.collab`（L13）协作编排层管理异步等待状态 |

### 工具注册表（Tool Registry）

所有工具适配器上线前必须在注册表中登记，注册表存储在 **S4 规则库**。

**必填字段：**

| 字段 | 说明 |
|------|------|
| `tool_id` | 全局唯一标识 |
| `category` | T1-T8 分类 |
| `risk_level` | 低/中/高/极高（影响调用审批策略） |
| `allowed_callers` | 允许调用的 Agent 层级（level）列表 |
| `operations` | 支持的操作列表及各操作的读/写/管理级别 |
| `rate_limit` | 频率限制（次/分钟） |
| `timeout_ms` | 超时阈值 |
| `fallback` | 超时或失败时的降级策略 |
| `idempotent` | 是否幂等（影响重试策略） |
| `audit_required` | 是否强制写 S3 审计（高风险操作必须为 true） |

### 工具调用标准链路

Agent 发起工具调用的完整路径：

```
Agent（任意层级）
  │
  ├─ 1. 查询工具注册表（S4）→ 确认工具存在且版本有效
  │
  ├─ 2. `security.permissions`（L6）权限检查 → 调用方 level 是否在 allowed_callers 中
  │
  ├─ 3. `security.policy`（L14）安全策略检查 → 操作是否在白名单，参数是否合规
  │
  ├─ 4. 风险等级判断
  │     低/中风险 → 直接执行
  │     高风险    → 需 `quality.gate`（L18）门禁审批后执行
  │     极高风险  → 需 `release.publish`（L22）发布链路人工审批
  │
  ├─ 5. Tool Port → 路由到对应 Adapter → 调用外部系统
  │
  ├─ 6. 结果处理
  │     成功 → 输出写 S2 制品库 → 返回 Agent
  │     失败 → 触发 `execution.monitor`（L15）重试/熔断逻辑（业务决策先行）
  │             + 异步写 S3 异常记录（后置记录，不阻塞L15决策）
  │
  └─ 7. 审计写入（risk_level=高时必须）→ 异步写 S3 审计库（不等待返回值）
```

---

### 工具治理规则

#### 分级审批策略

| 风险等级 | 典型操作 | 审批要求 |
|---------|---------|---------|
| 低 | 只读查询、搜索、知识库检索 | 直接执行，事后审计 |
| 中 | 文件写入、数据库写入、API调用 | `security.policy`（L14）合规检查通过即可执行 |
| 高 | 企业系统写入、批量数据变更 | `quality.gate`（L18）门禁审批，同步通知责任人 |
| 极高 | 财务操作、合同签署、生产DB删除 | `release.publish`（L22）人工审批，双人确认，留痕 |

#### 稳定性治理

| 机制 | 说明 | 管理层 |
|------|------|-------|
| 熔断器 | 连续失败超阈值暂停调用，自动恢复检测 | `execution.monitor`（L15） |
| 限流 | 按工具/租户/Agent 三级限流 | `security.permissions`（L6）权限管理层 |
| 超时隔离 | 工具超时不阻断主流程，降级返回 | `execution.monitor`（L15） |
| 重试策略 | 仅幂等操作允许自动重试，指数退避 | `execution.monitor`（L15） |
| 版本管理 | 适配器变更必须版本化，旧版本保留 | `release.publish`（L22）版本发布 |

### 工具接入层与三维度的关系汇总

| 维度 | 关系说明 |
|------|---------|
| **工作流层（纵向）** | `security.permissions`（L6）做权限校验；`security.policy`（L14）做安全检查；`execution.monitor`（L15）做熔断/重试；`quality.gate`（L18）做高风险审批；`release.publish`（L22）管适配器版本发布 |

### MCP 协议作为工具接入标准

**现状**：Tool Port 当前为内部定义接口。
**方向**：MCP（Model Context Protocol）作为外部工具接入的标准化协议优先落地。

```
MCP 协议的价值：
  外部系统只需实现 MCP Server → 自动接入所有 MCP-兼容的 Agent
  内部 Agent 只需调用 MCP Client → 无感知切换后端工具

接入路径：
  Phase 1（当前）：Tool Port 内部接口，Adapter 逐一手写
  Phase 2（中期）：MCP Adapter 统一接入，支持 MCP Server 热注册
  Phase 3（长期）：工具市场（任何人发布 MCP Server 即可被发现和使用）
```

## Human-in-the-Loop 对话判定协议（缺口5）

> **补充日期**：2026-04-14

### 补全背景

原有 escalate 骨架（见 03_Agent协作保障.md）写于架构早期，仅覆盖 `orchestration.collab`（L13）超时一个触发源，且只定义了"冻结→通知→等待"三步骨架，缺少：

- 触发来源的完整收口（`release.publish`（L22）、Tool Port 后加的两个触发源未覆盖）
- 对话判定机制（意图不清晰时如何多轮对话直至收敛）
- 长上下文处理方案（无限轮次对话如何保持 O(1) 上下文）
- 标准化报文结构（escalation 5字段协议）
- 人工操作权边界（人工可以做什么、不可以做什么）

---

### 核心设计结论

**意图漏斗协议**：每轮对话不保留原文，只提取结构化"意图 delta"并更新累加器。

```
上下文预算（每轮恒定，O(1)）：
  escalation_snapshot  ≈ 500 tokens  （固定）
+ intent_accumulator   ≈ 300 tokens  （有界结构体）
+ last_exchange        ≈ 200 tokens  （仅最后1轮原文）
─────────────────────────────────────
总计                   ≈ 1000 tokens （与轮次无关）
```

对话阶段状态机：
```
ESCALATED → PRESENTING → CLARIFYING（多轮）→ CONFIRMING → EXECUTING → CLOSED
                                  ↑                 ↓
                              （未清晰）       （人工修正）
                                  │
                          round >= max_rounds
                                  ↓
                         FORCE_CONFIRMING（强制 Y/N）
```


## 经验进化闭环协议（缺口7）

> **补充日期**：2026-04-14
> **性质**：数据层补全——定义"经验沉淀→实验触发→上线"的完整闭环路径
> **完整文档**：[07_经验进化闭环协议.md](./07_经验进化闭环协议.md)
> **状态**：✅ 文档已完成

---

### 补全背景

现有文档对各工作流层均有职责描述，但缺少以下三个关键路径（完整展开以 [07_经验进化闭环协议.md](./07_经验进化闭环协议.md) 为准）：

- `feedback.loop`（L20）将经验条目写入 S2 的具体路径（数据结构 + 写入流程）
- S2 → `evolution.experiment`（L23）的触发条件（失败率阈值、修正记录数等；阈值参数存 S4）
- `evolution.experiment`（L23）→ `quality.gate`（L18）→ `release.publish`（L22）→ S4 的评审与发布闭环

---

### 核心设计结论

**三个关键决策：**

1. **经验条目写 S2（record_type = experience）**：S2 内部按子类型区分保留策略，经验条目永久保留（**`evolution.experiment`（L23）**聚合同类经验时的读取来源）；S3 作为审计轨迹，不混入学习数据。
2. **correction_trace 存引用不存副本**：**`feedback.loop`（L20）**将本次任务涉及的 escalation_id 列表写入 `correction_trace` 字段，S3 保留完整修正数据的唯一来源。
3. **失败率与阈值判定**：**`feedback.loop`（L20）**在任务结束后聚合查询 S2 历史经验条目，从 S4 读取触发阈值；达阈值则触发 **`evolution.experiment`（L23）**（Proposal 生成纳入 **`process.intelligence`（L11）**，流程见 [07_经验进化闭环协议.md](./07_经验进化闭环协议.md)）。

**完整闭环路径：**
```
任务完成
  ↓
feedback.loop（L20）反馈闭环层
  ├─ 从 S3 查询本任务的 escalation 记录 → 获取 correction_trace（escalation_id 列表）
  ├─ 生成 experience_entry（6字段）
  ├─ 写入 S2（record_type=experience，永久保留）
  └─ 聚合查询 S2 同类历史经验条目，从 S4 读取触发阈值
        ├─ 失败率 > 阈值 → 触发 evolution.experiment（L23）
        ├─ 修正记录数 > 阈值 → 触发 evolution.experiment（L23）
        └─ 未达阈值 → 结束本轮

evolution.experiment（L23）进化实验层（被触发后）
  ├─ 从 S2 拉取该类型全部 experience 条目
  ├─ 生成 Proposal（Prompt变更 / 路由规则变更 / 工具调用顺序变更；与 `process.intelligence`（L11）协同，见 [07_经验进化闭环协议.md](./07_经验进化闭环协议.md)）
  └─ 沙盒验证 Proposal 效果
        ↓（仅沙盒完成后进入下栏；与 [07_经验进化闭环协议.md](./07_经验进化闭环协议.md)「四、第三段：提案评审」一致）

  └─ 新策略 vs 旧策略对比评分 → 推荐采用 or 否决
        ↓（进入发布层：`release.publish`（L22）；与 [07_经验进化闭环协议.md](./07_经验进化闭环协议.md)「四、第三段」一致）

release.publish（L22）版本发布层
  ├─ 人工审批 → 灰度上线 → 全量
  └─ 全量后写入 S4：更新对应规则的版本号
```

### 关键数据结构

**experience_entry（S2：由 `feedback.loop`（L20）写入；由 `evolution.experiment`（L23）读取）：**

```json
{

  "task_type": "任务类型（如 code-generation / research / review）",

  "failure_layer": null,
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `record_type` | string | 固定为 `experience`，用于 S2 子类型过滤 |
| `task_type` | string | 任务类型，`evolution.experiment`（L23）按此分组聚合 |
| `success` | bool | 本次任务是否最终成功 |
| `failure_layer` | string \| null | 失败发生在哪一层，成功时为 null |
| `correction_trace` | string[] | 人工介入的 escalation_id 列表，S3 中可查全量修正记录 |
| `final_result` | string | 最终正确输出的摘要，`evolution.experiment`（L23）训练准确度参考 |
| `confidence_delta` | float | 本次任务使系统置信度提升/下降多少（`quality.inspector`（L16）Inspector 反馈来源） |

## S3 分布式写入保障（缺口8）

> **补充日期**：2026-04-14
> **性质**：实现约定补全——"写入不可拒绝"原则的分布式场景保障机制
> **状态**：✅ 原则已定

---

### 补全背景

S3 审计库的核心约束"写入不可拒绝"早在数据层设计时已确立，但原有描述只给出了目标，缺少在分布式场景下的具体保障机制。此外，多处文档对 S3 写入的描述存在以下问题：

- **隐式同步语义**：将"写 S3"作为内联步骤写在流程箭头中，读者实现时容易写成阻塞调用。
- **循环依赖**：T1 适配器原描述"写失败必须写 S3 审计记录"——当 S3 本身不可用时，该句话逻辑上无法执行。
- **前置控制误用**：工具调用链路步骤6原描述"写 S3 → 触发 L6 重试/熔断"，使业务决策依赖于 S3 写入结果，违反 S3 后置记录定位。

---

### 三条核心原则

1. **S3写入必须异步**，不阻塞主流程（fire-and-forget）。
2. **S3写入失败时的兜底链路**：写本地缓冲文件 → 指数退避重试（上限N次）→ 超限触发P1告警（不中断业务）。
3. **S3是后置记录，不是前置控制**：任何层不得等待 S3 写入返回值来决定下一步业务逻辑。

详见本文档"S3 分布式写入保障原则（缺口8）"章节（位于 S3 审计库定义紧后方）。

## 跨层通讯空白补全（缺口19）

> **补充日期**：2026-04-18
> **性质**：架构空白补全——工作流纵向层间通讯协议（之前仅定义了组织横向通讯）
> **完整文档**：[19_层级扩展与跨层通讯协议.md](./19_层级扩展与跨层通讯协议.md)
> **状态**：✅ 文档已完成

### 补全背景

现有文档对"组织横向通讯"（CEO/VP/总监/经理/子Agent 之间谁指挥谁）定义完整，但对"工作流纵向层间通讯"（任务如何在 L1-L23 之间流转）存在系统性缺失：

| 缺失项 | 影响 |
|---|---|
| 无标准层间消息信封 | 各层自定义消息格式，无法统一追踪 |
| 无层注册协议 | 新增层无法声明"我接受什么/我产出什么" |
| 无旁路白名单 | 无法明确哪些层可跳过、何时可跳过 |
| 无反向反馈通道定义 | `quality.gate`→`execution.monitor` 打回、`quality.reflection`→`quality.inspector` 反思等路径未标准化 |
| 层号漂移无解 | 每次插层所有代码层号引用失效 |

### 核心设计结论

**1. layer_id 稳定标识符体系**：`layer_id`（如 `security.permissions`）永不变更，代码/配置/接口全部引用 `layer_id`，不再引用 `layer_order`（Lxx 展示编号）。

**2. 标准层间消息信封（Layer Message Envelope）**：统一的 JSON 结构，含 `routing`（路由信息）/ `payload`（内容）/ `control`（旁路控制）/ `feedback`（反向反馈）/ `audit`（全链路轨迹）五个区。

**3. 层注册协议**：每层必须在 S4 注册 `input_schema`、`output_schema`、`bypass_conditions`、`slo_ms` 等字段，新增层发布前必须完成注册。

**4. 旁路白名单（bypass whitelist）**：仅 `external.system_mgmt`（L2）/ `knowledge.retrieval`（L8）/ `quality.reflection`（L17）/ `eval.benchmark`（L21）/ `evolution.experiment`（L23）五层在特定条件下可旁路；`security.permissions`（L6）/ `security.policy`（L14）/ `quality.gate`（L18）/ `release.publish`（L22）四个核心治理层强制不可旁路。

**5. 反向反馈通道（写死）**：
```
quality.gate（L18）        → execution.monitor（L15）    门禁打回重执行
quality.reflection（L17）  → quality.inspector（L16）    修正后再校验
quality.inspector（L16）   → execution.monitor（L15）    严重问题直接重执行
orchestration.collab（L13）→ orchestration.task（L12）   协作失败重路由
security.permissions（L6） → gateway.ingress（L1）       权限失败返回入口
```

**6. 新增层操作流程**：提案 → VP02评估 → S4注册 → `release.publish`（L22）发布 → 路由引擎热加载 topology → 文档更新。

### 关键约束（新增，写死）

1. 代码/配置中禁止硬编码 `layer_order` 或 `layer_display`，只能引用 `layer_id`。
2. 四个核心治理层（`security.permissions` / `security.policy` / `quality.gate` / `release.publish`）不可旁路。
3. 反向反馈只能走 §7.1 白名单通道，任意层不得自行向任意早期层发送消息。
4. 新增主层必须先经 VP02 审批，禁止插层到 L4 之前、L8-L9 之间、L12-L13 之间。
