## 一、职责边界（写死）

| 归属 | 职责 |
|------|------|
| **L14.1（本子层）** | 将已结束任务**加工为可复跑、可版本管理的评测样本（benchmark case）**；维护样本状态机、采样与预算、去重策略；产出供 **L14.2** 执行的评测集清单与 schema 版本。 |
| **L14.2** | 按评测集对系统能力执行评测（调度、指标、统计显著性等），**不**承担「从原始任务组装样本」的主责。 |
| **L14.3** | 新旧策略对比与回归分析，**消费** L14.2 结果。 |
| **L13** | 复盘、优化建议、经验条目与长期记忆；可与 L14.1 **双向引用**（`experience_ref` / `benchmark_case_ref`），但**可复跑评测集资产**的权威写入归 L14.1。 |
| **L9** | 执行与监控、工具调用过程、重试与超时等**运行时事实**；只产出事件流，**不**定义 benchmark  schema。 |
| **L4.1（审计子层）** | 全链路留痕与审计存储；L14.1 **读取**经授权摘要或指针，**不得**把审计层当作评测集主库。 |
| **L16** | 沙盒实验与自优化提案；**读取**已晋升的评测集版本，**不**替代 L14.1 写入样本。 |

## 二、稳定 / 优质 / 经济（平台默认策略）

### 2.1 更稳定

- **异步非阻塞**：任务进入终态（成功/失败）后，由 L14.1 **异步**消费已闭合的 `trace_id` 数据管道组装样本；组装失败可重试、可告警，**不改变**该任务业务终态结论。
- **幂等**：同一 `trace_id` 对应**至多一条**终态 benchmark case（重复投递覆盖或 noop，由实现选定一种并写死）。
- **版本化**：评测样本须带 `benchmark_schema_version`；schema 升级时旧版本评测集仍可回放。
- **契约引用**：层间与配置引用使用 **`layer_id`**（见 `19_` 文），禁止以易变的 `layer_display` 作为唯一键。

### 2.2 更优（质量）

- **最小完备字段集**（与愿景对齐）：原始输入、中间轨迹引用、工具调用摘要、人工修改记录（若有）、最终参考答案或标注、错误原因标签（失败/部分成功场景）。
- **样本状态机**（建议枚举，可配置展示名）：`draft`（字段不齐）→ `ready`（机检通过）→ `promoted`（进入正式领域 benchmark，可选人工抽检）→ `archived`。
- **晋升门槛**：进入 `promoted` 前须满足 L14.1 定义的最小字段与脱敏/租户隔离检查；与 **L12** 门禁口径对齐的部分由平台规则表配置（规则写死、参数可配）。
- **分层建库**（参数可配）：如「核心黄金集 / 扩展集 / 失败专集」分桶；**黄金集**默认小且可强制人工复核比例。

### 2.3 更经济

- **采样与预算**：非全量升格；失败与高风险任务可提高采样权重；成功任务分层随机采样；按租户/全局配置**每日上限**（配置链：`platform → company → department → user`）。
- **存储指针化**：case 内对长轨迹与大 payload **仅存引用**（对象键、审计索引、摘要哈希），热数据保留索引，冷数据归档策略可配。
- **评测执行解耦**：L14.1 只负责**生成与晋升**评测集；**批量评测**由 L14.2 按调度触发（定时/发版前），避免每次任务结束即跑全量模型评测。

## 三、与相邻层数据流（顺向摘要）

```
任务终态（L9 闭合 + L4.1 审计可查）
    → L14.1 组装 draft case → 机检/去重/采样 → ready / promoted
    → L14.2 载入评测集执行评测 → L14.3 对比
    → L16 读取 promoted 版本做实验（不绕过 L12/L15）
```

反向反馈（质检打回、重跑）遵循 `19_` 文「反向反馈通道」与旁路白名单；L14.1 **不**因评测失败而改写历史任务事实，只生成新 case 版本或标注 `superseded_by`。
