## 一、项目概述

### 1.1 项目定位

RTK (Rust Token Killer) 是一个高性能 CLI 代理工具，通过智能过滤和压缩命令输出，将 LLM token 消耗降低 60-90%。项目采用 Rust 编写，单二进制部署，启动开销 <10ms。

### 1.2 核心价值主张

| 指标 | 数值 |
|------|------|
| Token节省率 | 60-90% |
| 启动开销 | 5-15ms |
| 二进制大小 | ~4.1MB |
| 内存占用 | 2-5MB |
| 支持命令数 | 100+ |
| 支持AI工具 | 10种 |

### 1.3 项目规模

```
源代码统计:
├── Rust源文件: 110+ 个
├── 命令模块: 42 个
├── 基础设施模块: 22 个
├── TOML过滤器: 60+ 个
├── 测试固件: 多个生态专用
└── CI/CD工作流: 4个
```

### 2.2 命令生命周期 (六阶段执行流)

```
Phase 1: PARSE
─────────────────────────────
$ rtk git log --oneline -5 -v

Clap Parser提取:
  • Command: Commands::Git
  • Args: ["log", "--oneline", "-5"]
  • Flags: verbose = 1

           ↓

Phase 2: ROUTE
─────────────────────────────
main.rs: match Commands::Git { args, .. }
  ↓
git::run(args, verbose)

           ↓

Phase 3: EXECUTE
─────────────────────────────
std::process::Command::new("git")
    .args(["log", "--oneline", "-5"])
    .output()

捕获输出:
  • stdout: 原始命令输出
  • stderr: 错误信息
  • exit_code: 退出码

           ↓

Phase 4: FILTER
─────────────────────────────
应用过滤策略:
  • 统计提取
  • 错误分组
  • 去重压缩
  • 结构简化

结果: Token减少60-90%

           ↓

Phase 5: PRINT
─────────────────────────────
if verbose > 0 {
    eprintln!("调试信息");
}
println!("{}", 过滤后输出);

           ↓

Phase 6: TRACK
─────────────────────────────
tracking::track(
    original_cmd: "git log --oneline -5",
    input: 原始输出,
    output: 过滤后输出
)

SQLite INSERT:
  • input_tokens: 125
  • output_tokens: 5
  • savings_pct: 96.0
```

## 三、过滤策略体系

### 3.1 十二种过滤策略

| 策略 | 技术 | 适用模块 | 节省率 |
|------|------|----------|--------|
| 1. 统计提取 | 计数/聚合，丢弃详情 | git status, git log, pnpm list | 90-99% |
| 2. 错误聚焦 | 仅显示stderr | runner (err模式) | 60-80% |
| 3. 模式分组 | 按规则/文件分组计数 | lint, tsc, grep | 80-90% |
| 4. 去重压缩 | 唯一行+计数 | log_cmd | 70-85% |
| 5. 结构提取 | 仅保留键和类型 | json_cmd | 80-95% |
| 6. 代码过滤 | 按级别过滤代码 | read, smart | 60-90% |
| 7. 失败聚焦 | 仅显示失败测试 | vitest, playwright, pytest | 94-99% |
| 8. 树压缩 | 目录树聚合 | ls, tree | 50-70% |
| 9. 进度过滤 | 移除ANSI进度条 | wget, pnpm install | 85-95% |
| 10. JSON/文本双模式 | 优先JSON API | ruff, pip | 80%+ |
| 11. 状态机解析 | 跟踪测试状态 | pytest | 90%+ |
| 12. NDJSON流式 | 逐行JSON解析 | go test | 90%+ |

### 3.2 代码过滤级别

```rust
// FilterLevel::None - 保留全部

    // Sum all items
    items.iter().map(|i| i.value).sum()

// FilterLevel::Minimal - 仅移除注释 (20-40%节省)
fn calculate_total(items: &[Item]) -> i32 {
    items.iter().map(|i| i.value).sum()
}

// FilterLevel::Aggressive - 移除注释+函数体 (60-90%节省)
fn calculate_total(items: &[Item]) -> i32 { ... }
```

## 五、Token追踪系统

### 5.1 SQLite架构

```sql
CREATE TABLE commands (
    id              INTEGER PRIMARY KEY,
    timestamp       TEXT NOT NULL,      -- RFC3339格式
    original_cmd    TEXT NOT NULL,      -- "git log --oneline -5"
    rtk_cmd         TEXT NOT NULL,      -- "rtk git log --oneline -5"
    input_tokens    INTEGER NOT NULL,   -- 125
    output_tokens   INTEGER NOT NULL,   -- 5
    saved_tokens    INTEGER NOT NULL,   -- 120
    savings_pct     REAL NOT NULL,      -- 96.0
    exec_time_ms    INTEGER DEFAULT 0   -- 执行时间(毫秒)
);
```

### 5.2 Token估算算法

```rust
fn estimate_tokens(text: &str) -> usize {
    (text.len() as f64 / 4.0).ceil() as usize
}

// 启发式: ~4字符 = 1 token (GPT风格分词)
```

### 5.3 数据保留策略

- **保留期**: 90天
- **自动清理**: 每次INSERT时执行DELETE
- **存储位置**: `~/.local/share/rtk/history.db`

## 六、设计原则与约束

### 6.1 核心设计原则

1. **单一职责**: 每个模块处理一种命令类型
2. **最小开销**: ~5-15ms代理开销
3. **退出码保留**: CI/CD可靠性
4. **故障安全**: 过滤失败时回退到原始输出
5. **透明性**: `-v`标志可查看原始输出

### 6.2 Rust开发约束

| 规则 | 原因 |
|------|------|
| **禁止async** | 避免5-10ms启动开销 |
| **禁止生产环境unwrap()** | 防止panic中断用户工作流 |
| **lazy_static! 正则** | 避免每次调用重编译 |
| **回退模式** | 过滤失败时执行原始命令 |
| **退出码传播** | 保持CI/CD正确性 |

### 6.3 错误处理模式

```rust
// ✅ 正确: 上下文链
let content = fs::read_to_string(path)
    .with_context(|| format!("Failed to read: {}", path.display()))?;

// ❌ 错误: 无上下文
let content = fs::read_to_string(path)?;
```

    let input = include_str!("../tests/fixtures/cmd_raw.txt");

## 十五、总结与建议

### 15.1 项目优势

1. **高效节省**: 60-90% token减少
