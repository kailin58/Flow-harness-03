## Git 分支与合并策略

```
main (保护分支，不直接推送)
 │
 ├── wave-1/a1-fix-simulate        ← 端1
 ├── wave-1/a2-fix-cli-exit        ← 端2
 ├── wave-1/b1-skill-loader        ← 端3
 │    ↓ 全部完成 + npm test 通过后合入 main
 │
 ├── wave-2/a3-evolution           ← 从最新 main 拉
 ├── wave-2/b2-skill-files         ← 从最新 main 拉
 ├── wave-2/c2-command-system      ← 从最新 main 拉
 │    ↓ 全部完成 + npm test 通过后合入 main
 │
 └── ... 以此类推
```

### 规则

1. **每个 Step 一个分支**：命名 `wave-N/step-id`
2. **Wave 内并行开发**：各端在各自分支工作，互不干扰
3. **Wave 完成后统一合入**：所有分支 PR 合入 main，有冲突时按冲突矩阵顺序解决
4. **下一 Wave 从最新 main 拉取**：确保每个 Wave 基于上一 Wave 的完整产出

## 端间同步机制

多端并行时需要明确的进度同步规则：

| 检查点 | 触发条件 | 动作 |
|--------|---------|------|
| Wave 门禁 | Wave N 全部步骤完成 | 人工确认：全部分支已合入 main + `npm test` 通过 → 通知各端可进入 Wave N+1 |
| 冲突协调 | 同一文件被多步骤修改 | 按冲突矩阵顺序合并，先完成的先合入 |
| 阻塞升级 | 某步骤失败且阻塞后续 | 优先修复该步骤，其他端可继续不依赖该步骤的工作 |

### 同步方式（推荐）

- **方式 A（轻量）**：共享文档/消息群，每完成一个 Step 发一条确认
- **方式 B（严格）**：通过 PR 状态判断，main 分支上的合并记录即进度记录

## 回滚策略

| 场景 | 回滚方式 |
|------|---------|
| 单个 Step 引入 bug | `git revert` 该 Step 的合并提交，修复后重新 PR |
| 整个 Wave 失败 | 回退到上一 Wave 的 main tag（每个 Wave 合入后打 tag：`wave-1-done`, `wave-2-done`...）|
| npm test 退化 | 不允许合入 main，在分支上修复后重新验证 |
