## 一、Flow Harness 完整架构分析

### 1.1 软件架构

#### 整体架构模式

Flow Harness 采用 **6 层分层架构 + 领域驱动设计**：

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: 任务编排层 (Task Orchestration)                     │
│   - Supervisor Agent (CEO)                                  │
│   - Task Analyzer / Decomposer / Dispatcher                 │
├─────────────────────────────────────────────────────────────┤
│ Layer 2: 安全策略层 (Security Policy)                        │
│   - Policy Checker (文件/命令策略)                           │
│   - Role Permission (6角色 × 22操作权限矩阵)                  │
│   - AGENTS.md Runtime Parser                                │
├─────────────────────────────────────────────────────────────┤
│ Layer 3: 执行监控层 (Execution Monitoring)                   │
│   - Execution Monitor (实时监控)                             │
│   - Deviation Detector (偏差检测)                            │
│   - Error Pattern Recognizer (错误模式识别)                   │
│   - Auto Retry (自动重试 + 断路器)                           │
│   - Self Healing (自愈引擎)                                  │
├─────────────────────────────────────────────────────────────┤
│ Layer 4: 检查层 (Inspection)                                │
│   - Inspector (6项深度检查)                                  │
│   - Goal Alignment / Spec Compliance                        │
│   - Semantic Correctness / Impact Analysis                  │
│   - Security Scan / AGENTS.md Compliance                    │
├─────────────────────────────────────────────────────────────┤
│ Layer 5: 质量门禁层 (Quality Gate)                           │
│   - Quality Gate (Lint/测试/安全扫描)                         │
│   - Human-in-the-Loop (人工审批)                             │
├─────────────────────────────────────────────────────────────┤
│ Layer 6: 反馈闭环层 (Feedback Loop)                          │
│   - Review Loop (6a回顾→6b优化→6c验证→6d固化)                 │
│   - Evolution Engine (自动进化 6 大能力)                      │
│   - Knowledge Base (知识库)                                  │
│   - Memory Store (四类记忆系统)                               │
└─────────────────────────────────────────────────────────────┘
```

## 二、ai-website-cloner-template 架构分析

### 2.1 软件架构

#### 整体架构模式

ai-website-cloner-template 采用 **多阶段流水线 + 并行构建** 模式：

```
┌─────────────────────────────────────────────────────────────┐
│              /clone-website Skill (入口)                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Phase 1: Reconnaissance (侦察)                             │
│    - 全页截图 (desktop/mobile)                              │
│    - 全局提取 (fonts/colors/favicons)                       │
│    - 强制交互扫描 (scroll/click/hover/responsive)           │
│    - 页面拓扑分析                                           │
│      ↓                                                      │
│  Phase 2: Foundation (基础) ──── [必须完成]                  │
│    - 更新 fonts/layout.tsx                                  │
│    - 更新 globals.css (design tokens)                       │
│    - 创建 TypeScript interfaces                             │
│    - 提取 SVG icons                                         │
│    - 下载全局资源                                           │
│    - 验证: npm run build                                    │
│      ↓                                                      │
│  Phase 3: Component Specs (组件规范)                         │
│    - 为每个区块写 spec 文件                                  │
│    - spec 包含: 截图、CSS 值、行为、内容、资源路径            │
│    - 存储到 docs/research/components/                       │
│      ↓                                                      │
│  Phase 4: Parallel Build (并行构建)                          │
│    - 为每个组件创建 Git Worktree                             │
│    - 并行派发 Builder Agent                                  │
│    - 每个 Agent 收到完整 spec inline                        │
│    - 验证: npx tsc --noEmit                                 │
│      ↓                                                      │
│  Phase 5: Assembly & QA (组装验证)                           │
│    - 合并所有 Worktree                                       │
│    - 组装页面                                                │
│    - 验证: npm run build                                    │
│    - 视觉对比验证                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 核心设计原则

| 原则 | 描述 |
|------|------|
| **完整性优先于速度** | 每个 Builder 必须收到所有需要的信息，不能猜测 |
| **小任务、完美结果** | 复杂度预算：spec 超过 150 行就拆分 |
| **真实内容、真实资源** | 使用目标站点的实际文本、图片、视频 |
| **基础先行** | Foundation 阶段必须完成才能进入并行构建 |
| **提取外观和行为** | 不仅提取 CSS，还要提取交互行为 |
| **Spec 文件即真理** | 每个 Builder 收到 spec 文件的完整内容 |
| **构建必须通过** | 每个 Builder 完成后必须 `tsc --noEmit` 通过 |

## 五、最终建议方案

### 5.1 核心结论

## ❌ 不建议直接集成 ai-website-cloner-template

### 5.2 决策理由

1. **定位不同**：Flow Harness 是通用编排系统，ai-website-cloner 是专用工具
2. **技术栈冲突严重**：直接集成会破坏 Flow Harness 的轻量级定位
3. **运行环境不兼容**：ai-website-cloner 需要 AI Agent + 浏览器环境
4. **用户群体不同**：Flow Harness 面向 AI Agent 开发者，网站克隆是边缘需求
5. **但设计模式优秀**：值得借鉴并行构建、流水线门控、Spec 沉淀等设计

### 5.3 推荐方案：借鉴模式 + 松耦合集成

```
┌─────────────────────────────────────────────────────────────┐
│              Flow Harness 核心 (保持不变)                    │
│                                                             │

│  │ SandboxManager  │ ← 已有 Git Worktree 隔离               │
│  └─────────────────┘                                        │

│  │ParallelExecutor │   │PipelineExecutor │ ← 借鉴新增        │
```

### 5.4 具体借鉴方案

#### 借鉴点 1: 并行执行优化 (ParallelExecutor)

**借鉴内容**: ai-website-cloner 的 Worktree 并行构建模式

**实现方式**: 新增独立模块，不修改现有代码

```javascript
// src/parallel-executor.js (新增)
class ParallelExecutor {
  // 借鉴: 多 Worktree 并行 + 结果合并
  async executeParallel(tasks, executor, context) {
    const worktrees = await this.createWorktrees(tasks);
    const results = await Promise.all(/* 并行执行 */);
    const mergeResult = await this.mergeWorktrees(worktrees);
    await this.cleanupWorktrees(worktrees);
    return { results, mergeResult };
  }
}
```

**配置开关**: `execution.parallel.enabled: false` (默认禁用)

#### 借鉴点 2: 流水线门控 (PipelineExecutor)

**借鉴内容**: ai-website-cloner 的多阶段质量门控

**实现方式**: 新增独立模块，提供可选执行模式

```javascript
// src/pipeline-executor.js (新增)
const PIPELINE_STAGES = {
  RECONNAISSANCE: { gates: ['info_complete'] },
  FOUNDATION: { gates: ['build_pass'] },
  BUILD: { gates: ['typecheck_pass'] },
  ASSEMBLY: { gates: ['final_build_pass'] }
};

class PipelineExecutor {
  async execute(context) {
    for (const stage of this.stages) {
      const result = await this.executeStage(stage, context);
      if (!result.passed && this.stopOnGateFailure) break;
    }
  }
}
```

**配置开关**: `execution.mode: closed_loop | pipeline`

#### 借鉴点 3: Spec 文件沉淀

**借鉴内容**: ai-website-cloner 的组件 Spec 文件设计

**实现方式**: 扩展 KnowledgeBase，新增方法

```javascript
// src/knowledge-base.js 扩展
class KnowledgeBase {
  // 新增方法
  writeSpec(specName, spec) { /* ... */ }
  readSpec(specName) { /* ... */ }
  listSpecs() { /* ... */ }
  calculateReusability(spec) { /* ... */ }
}
```

**配置开关**: `knowledge.specs.enabled: false`

#### 借鉴点 4: 外部技能注册

**借鉴内容**: ai-website-cloner 的 `/clone-website` 技能

**实现方式**: 注册为外部技能，不引入代码


```javascript
// 自动降级: 新功能失败时回退到原有逻辑
async handleTask_v2(task, context) {
  try {
    if (this.executionMode === 'pipeline') {
      return await this.handleTask_pipeline(task, context);
    }
    return await this.handleTask(task, context);
  } catch (error) {
    this.logger.warn('New mode failed, falling back');
    return await this.handleTask(task, context);
  }
}
```
