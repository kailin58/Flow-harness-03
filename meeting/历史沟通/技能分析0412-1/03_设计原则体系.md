# 设计原则体系深度分析

> 本文档深入分析 Impeccable 的七大设计领域原则

---

## 一、Typography (排版)

### 1.1 经典排版原则

#### 垂直节奏

行高应该是**所有**垂直间距的基础单位：

```
如果正文 text = 16px，line-height: 1.5 = 24px
→ 所有间距值应该是 24px 的倍数
→ 文本和空间共享数学基础，创建潜意识和谐
```

#### 模块化比例与层次

**常见错误**: 太多字体大小且太接近 (14px, 15px, 16px, 18px...) → 创建浑浊的层次

**正确做法**: 更少的大小，更多的对比

| 角色 | 典型比例 | 用例 |
|------|----------|------|
| xs | 0.75rem (12px) | 说明、法律文本 |
| sm | 0.875rem (14px) | 次要 UI、元数据 |
| base | 1rem (16px) | 正文 |
| lg | 1.25-1.5rem (20-24px) | 小标题、引导文本 |
| xl+ | 2-4rem (32-64px) | 标题、英雄文本 |

**流行比例**:
- 1.25 (大三度)
- 1.333 (纯四度)
- 1.5 (纯五度)

### 1.2 字体选择与配对

#### 选择流程

```
Step 1: 阅读简报，写下 3 个具体品牌词汇
        ❌ "现代"、"优雅"（死类别）
        ✅ "温暖、机械、有主见"
        ✅ "冷静、临床、谨慎"
        ✅ "手工制作、有点奇怪"

Step 2: 列出你通常会选择的前 3 种字体 → 拒绝它们
        它们很可能来自禁止列表

Step 3: 浏览字体目录，想象字体作为物理对象
        博物馆展品说明、手绘店铺招牌、1970s 大型机手册...

Step 4: 交叉检查结果
        如果最终选择符合你的反射模式，返回 Step 3
```

#### 禁止字体列表

```
Fraunces, Newsreader, Lora, Crimson, Crimson Pro, Crimson Text
Playfair Display, Cormorant, Cormorant Garamond
Syne, IBM Plex Mono, IBM Plex Sans, IBM Plex Serif
Space Mono, Space Grotesk
Inter, DM Sans, DM Serif Display, DM Serif Text
Outfit, Plus Jakarta Sans, Instrument Sans, Instrument Serif
Roboto, Open Sans, Lato, Montserrat, Arial, Helvetica
```

#### 配对原则

**非显而易见的真相**: 你通常不需要第二种字体。一个字体家族的多个粗细创建比两个竞争字体更清晰的层次。

当配对时，在多个轴上对比：
- Serif + Sans（结构对比）
- Geometric + Humanist（个性对比）
- Condensed display + Wide body（比例对比）

**永不配对**: 相似但不完全相同的字体（如两个几何无衬线）→ 创建没有清晰层次的视觉紧张

### 1.3 现代 Web 排版

#### 流体排版

```css
/* 营销/内容页面：标题使用 clamp */
h1 {
  font-size: clamp(2rem, 5vw + 1rem, 4rem);
}

/* 应用 UI/仪表板：固定 rem 刻度 */
.body-text {
  font-size: 1rem; /* 不使用流体 */
}
```

**规则**: 流体用于营销内容，固定用于应用 UI

#### OpenType 特性

```css
/* 表格数字用于数据对齐 */
.data-table { font-variant-numeric: tabular-nums; }

/* 正确分数 */
.recipe-amount { font-variant-numeric: diagonal-fractions; }

/* 小型大写用于缩写 */
abbr { font-variant-caps: all-small-caps; }

/* 代码中禁用连字 */
code { font-variant-ligatures: none; }

/* 启用字距调整 */
body { font-kerning: normal; }
```

### 1.4 可访问性考量

- **永不禁用缩放**: `user-scalable=no` 破坏可访问性
- **使用 rem/em**: 尊重用户浏览器设置
- **最小 16px 正文**: 更小会眼睛疲劳
- **足够触摸目标**: 文本链接需要创建 44px+ 点击目标的内边距或行高

## 二、Color & Contrast (颜色与对比)

### 2.1 色彩空间：使用 OKLCH

**停止使用 HSL**。OKLCH 是感知均匀的：

```css
/* OKLCH 语法 */
color: oklch(lightness chroma hue);
/* lightness: 0-100% */
/* chroma: ~0-0.4 */
/* hue: 0-360 */
```

**构建主色及其变体**:
- 保持 chroma+hue 大致恒定
- 变化 lightness
- **接近白色或黑色时减少 chroma**（高 chroma 在极端亮度看起来俗气）

### 2.2 功能调色板构建

#### 色调中性色

**纯灰已死**。零 chroma 的中性色在品牌色旁边感觉无生气。

```css
/* 添加微小 chroma (0.005-0.015)，色调向品牌色 */
--neutral-100: oklch(98% 0.01 250); /* 轻微品牌色调 */
--neutral-900: oklch(15% 0.01 250);
```

**避免陷阱**: 永远色调向暖橙色或冷蓝色。这两个是最懒惰的默认值。

#### 调色板结构

| 角色 | 目的 | 示例 |
|------|------|------|
| **Primary** | 品牌、CTA、关键操作 | 1 种颜色，3-5 种色调 |
| **Neutral** | 文本、背景、边框 | 9-11 色调刻度 |
| **Semantic** | 成功、错误、警告、信息 | 4 种颜色，各 2-3 色调 |
| **Surface** | 卡片、模态框、覆盖层 | 2-3 高度级别 |

**跳过次要/第三色**，除非真正需要。大多数应用用一种强调色工作良好。

### 2.3 60-30-10 规则

这是关于**视觉重量**，不是像素计数：

```
60%: 中性背景、留白、基础表面
30%: 次要颜色——文本、边框、非活动状态
10%: 强调——CTA、高亮、焦点状态
```

**常见错误**: 到处使用强调色因为它是"品牌色"。

**真相**: 强调色有效是因为它们稀少。过度使用杀死它们的力量。

### 2.4 对比与可访问性

#### WCAG 要求

| 内容类型 | AA 最低 | AAA 目标 |
|----------|---------|----------|
| 正文 | 4.5:1 | 7:1 |
| 大文本 (18px+ 或 14px 粗体) | 3:1 | 4.5:1 |
| UI 组件、图标 | 3:1 | 4.5:1 |
| 非必要装饰 | 无 | 无 |

**陷阱**: 占位符文本仍需要 4.5:1。那个浅灰占位符？通常失败 WCAG。

#### 危险颜色组合

- 浅灰文本在白色上（#1 可访问性失败）
- **灰色文本在任何彩色背景上**——灰色在颜色上看起来褪色和死寂
- 红色文本在绿色背景上（或反之）——8% 男性无法区分
- 蓝色文本在红色背景上（视觉振动）
- 黄色文本在白色上（几乎总是失败）
- 浅色细文本在图片上（不可预测的对比度）

### 2.5 主题：深色模式

**深色模式不是反转的浅色模式**。需要不同的设计决策：

| 浅色模式 | 深色模式 |
|----------|----------|
| 阴影创建深度 | 更亮的表面创建深度（无阴影） |
| 深色文本在浅色上 | 浅色文本在深色上（减少字体粗细） |
| 鲜艳强调 | 略微降低饱和度的强调 |
| 白色背景 | 永不纯黑——使用深灰 (oklch 12-18%) |

**深色模式深度**:
```css
/* 3 步表面刻度，更高高度更亮 */
--surface-1: oklch(15% 0.01 250);
--surface-2: oklch(20% 0.01 250);
--surface-3: oklch(25% 0.01 250);
```

**减少正文粗细**: 从 400 降到 350，因为浅色文本在深色上看起来更重。

## 三、Layout & Space (布局与空间)

### 3.1 间距系统

#### 使用 4pt 基础，不是 8pt

8pt 系统太粗糙——你经常需要 12px（在 8 和 16 之间）。

**刻度**: 4, 8, 12, 16, 24, 32, 48, 64, 96px

#### 语义命名 Token

```css
/* 正确：语义命名 */
--space-sm: 0.5rem;
--space-md: 1rem;
--space-lg: 1.5rem;

/* 错误：值命名 */
--spacing-8: 0.5rem;  /* 不要这样 */
```

**使用 `gap`** 而非 margin 用于兄弟间距——消除 margin 塌陷及其清理技巧。

### 3.2 网格系统

#### 自调整网格

```css
/* 无断点的响应式网格 */
grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
```

- 列至少 280px
- 每行尽可能多
- 剩余部分拉伸

### 3.3 视觉层次

#### 斜视测试

模糊你的眼睛（或截图并模糊）。你还能识别：
- 最重要的元素？
- 第二重要的？
- 清晰的分组？

如果一切模糊后看起来相同重量，你有层次问题。

#### 通过多维度层次

**不要仅依赖大小**。组合：

| 工具 | 强层次 | 弱层次 |
|------|--------|--------|
| **大小** | 3:1 比例或更多 | <2:1 比例 |
| **粗细** | 粗体 vs 常规 | 中等 vs 常规 |
| **颜色** | 高对比度 | 相似色调 |
| **位置** | 顶部/左侧（主要） | 底部/右侧 |
| **空间** | 被留白包围 | 拥挤 |

**最佳层次使用 2-3 维度**: 一个更大、更粗、且上方有更多空间的标题。

### 3.4 卡片不是必需的

卡片被过度使用。间距和对齐自然创建视觉分组。

**仅在以下情况使用卡片**:
- 内容真正独特且可操作
- 项目需要在网格中视觉比较
- 内容需要清晰的交互边界

**永不嵌套卡片中的卡片**——使用间距、排版和微妙分隔线。

### 3.5 容器查询

视口查询用于页面布局。**容器查询用于组件**：

```css
.card-container {
  container-type: inline-size;
}

@container (min-width: 400px) {
  .card {
    grid-template-columns: 120px 1fr;
  }
}
```

**为什么重要**: 侧边栏中的窄卡片保持紧凑，而主内容区域的相同卡片扩展——自动，无需视口技巧。

## 四、Motion Design (动效设计)

### 4.1 持续时间：100/300/500 规则

| 持续时间 | 用例 | 示例 |
|----------|------|------|
| **100-150ms** | 即时反馈 | 按钮按下、切换、颜色变化 |
| **200-300ms** | 状态变化 | 菜单打开、工具提示、悬停状态 |
| **300-500ms** | 布局变化 | 手风琴、模态框、抽屉 |
| **500-800ms** | 入场动画 | 页面加载、英雄揭示 |

**退出动画比入场快**——使用约 75% 的入场持续时间。

### 4.2 缓动：选择正确的曲线

**不要使用 `ease`**。它是妥协，很少最优。

| 曲线 | 用于 | CSS |
|------|------|-----|
| **ease-out** | 元素入场 | `cubic-bezier(0.16, 1, 0.3, 1)` |
| **ease-in** | 元素离场 | `cubic-bezier(0.7, 0, 0.84, 0)` |
| **ease-in-out** | 状态切换 | `cubic-bezier(0.65, 0, 0.35, 1)` |

**微交互的指数曲线**:

```css
/* Quart out - 平滑、精致（推荐默认） */
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);

/* Quint out - 略微更戏剧性 */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);

/* Expo out - 利落、自信 */
--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);
```

**避免弹跳和弹性曲线**。它们在 2015 年流行但现在感觉俗气业余。真实物体停止时不会弹跳——它们平滑减速。

### 4.3 只动画两个属性

**transform** 和 **opacity** —— 其他一切都导致布局重计算。

**高度动画**：使用 `grid-template-rows: 0fr → 1fr` 而非直接动画 `height`。

### 4.4 交错动画

```css
/* 使用 CSS 自定义属性实现更清晰的交错 */
animation-delay: calc(var(--i, 0) * 50ms);
/* HTML: style="--i: 0" */
```

**封顶总交错时间**——10 个项目 × 50ms = 500ms 总计。对于许多项目，减少每项延迟或封顶交错计数。

### 4.5 减少动画

**这不是可选的**。前庭障碍影响约 35% 的 40 岁以上成年人。

```css
/* 正常定义动画 */
.card {
  animation: slide-up 500ms ease-out;
}

/* 为减少动画提供替代 */
@media (prefers-reduced-motion: reduce) {
  .card {
    animation: fade-in 200ms ease-out;  /* 交叉淡入而非运动 */
  }
}
```

**保留什么**: 功能性动画如进度条、加载旋转器（减速）和焦点指示器仍应工作——只是没有空间移动。

## 五、Interaction Design (交互设计)

### 5.1 八种交互状态

每个交互元素需要设计这些状态：

| 状态 | 何时 | 视觉处理 |
|------|------|----------|
| **Default** | 静止时 | 基础样式 |
| **Hover** | 指针悬停（非触摸） | 微妙提升、颜色偏移 |
| **Focus** | 键盘/编程焦点 | 可见环 |
| **Active** | 被按下 | 按下、更深 |
| **Disabled** | 非交互 | 降低不透明度、无指针 |
| **Loading** | 处理中 | 旋转器、骨架屏 |
| **Error** | 无效状态 | 红色边框、图标、消息 |
| **Success** | 已完成 | 绿色勾选、确认 |

**常见遗漏**: 设计悬停但未设计焦点，或反之。它们是不同的。键盘用户永看不到悬停状态。

### 5.2 焦点环：做对

**永不 `outline: none` 无替代**。这是可访问性违规。

```css
/* 为鼠标/触摸隐藏焦点环 */
button:focus {
  outline: none;
}

/* 为键盘显示焦点环 */
button:focus-visible {
  outline: 2px solid var(--color-accent);
  outline-offset: 2px;
}
```

**焦点环设计**:
- 高对比度（与相邻颜色至少 3:1）
- 2-3px 厚
- 与元素偏移（不在内部）
- 所有交互元素一致

### 5.3 表单设计

**占位符不是标签**——它们在输入时消失。始终使用可见的 `<label>` 元素。

**在 blur 时验证**，而非每次按键（例外：密码强度）。

**错误放在字段下方**，使用 `aria-describedby` 连接。

### 5.4 加载状态

**乐观更新**: 立即显示成功，失败时回滚。用于低风险操作（点赞、关注），而非支付或破坏性操作。

**骨架屏 > 旋转器**——它们预览内容形状，感觉比通用旋转器更快。

### 5.5 模态框：Inert 方法

```html
<!-- 模态框打开时 -->
<main inert>
  <!-- 模态框后的内容无法聚焦或点击 -->
</main>
<dialog open>
  <h2>Modal Title</h2>
</dialog>
```

或使用原生 `<dialog>` 元素：
```javascript
const dialog = document.querySelector('dialog');
dialog.showModal();  // 打开，带焦点陷阱，Escape 关闭
```

### 5.6 Popover API

用于工具提示、下拉菜单和非模态覆盖层：

```html
<button popovertarget="menu">Open menu</button>
<div id="menu" popover>
  <button>Option 1</button>
  <button>Option 2</button>
</div>
```

**好处**: 轻量关闭（点击外部关闭）、正确堆叠、无 z-index 战争、默认可访问。

### 5.7 破坏性操作：撤销 > 确认

**撤销比确认对话框更好**——用户会心不在焉地点击确认。

- 立即从 UI 移除
- 显示撤销 toast
- toast 过期后实际删除

**仅在以下情况使用确认**: 真正不可逆的操作（账户删除）、高成本操作、批量操作。

## 六、Responsive Design (响应式设计)

### 6.1 移动优先

从移动的基础样式开始，使用 `min-width` 查询叠加复杂性。

桌面优先（`max-width`）意味着移动首先加载不必要的样式。

### 6.2 断点：内容驱动

不要追逐设备尺寸——让内容告诉你在哪里断。

1. 从窄开始
2. 拉伸直到设计断裂
3. 在那里添加断点

三个断点通常足够（640, 768, 1024px）。

使用 `clamp()` 实现无断点的流体值。

### 6.3 检测输入方式，不只是屏幕尺寸

**屏幕尺寸不告诉你输入方式。** 笔记本电脑配触摸屏、平板电脑配键盘：

```css
/* 精细指针（鼠标、触控板） */
@media (pointer: fine) {
  .button { padding: 8px 16px; }
}

/* 粗糙指针（触摸、触笔） */
@media (pointer: coarse) {
  .button { padding: 12px 20px; }  /* 更大的触摸目标 */
}

/* 设备支持悬停 */
@media (hover: hover) {
  .card:hover { transform: translateY(-2px); }
}
```

**关键**: 不要依赖悬停实现功能。触摸用户无法悬停。

### 6.4 安全区域：处理刘海

```css
body {
  padding-top: env(safe-area-inset-top);
  padding-bottom: env(safe-area-inset-bottom);
}

.footer {
  padding-bottom: max(1rem, env(safe-area-inset-bottom));
}
```

启用 viewport-fit:
```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

## 七、UX Writing (UX 文案)

### 7.1 按钮标签问题

**永不使用 "OK"、"Submit" 或 "Yes/No"**。这些懒惰且模糊。

| 差 | 好 | 为什么 |
|----|----|----|
| OK | Save changes | 说明会发生什么 |
| Submit | Create account | 结果导向 |
| Yes | Delete message | 确认操作 |
| Cancel | Keep editing | 澄清"取消"意味着什么 |
| Click here | Download PDF | 描述目的地 |

**破坏性操作**：命名破坏
- "Delete" 不是 "Remove"（delete 是永久的，remove 暗示可恢复）
- "Delete 5 items" 不是 "Delete selected"（显示计数）

### 7.2 错误消息：公式

每条错误消息应回答：
1. 发生了什么？
2. 为什么？
3. 如何修复？

**示例**: "Email address isn't valid. Please include an @ symbol." 而非 "Invalid input"。

**错误消息模板**:

| 情况 | 模板 |
|------|------|
| **格式错误** | "[Field] needs to be [format]. Example: [example]" |
| **缺少必填** | "Please enter [what's missing]" |
| **权限拒绝** | "You don't have access to [thing]. [What to do instead]" |
| **网络错误** | "We couldn't reach [thing]. Check your connection and [action]." |

**不要责怪用户**:
"Please enter a date in MM/DD/YYYY format" 而非 "You entered an invalid date"。

### 7.3 空状态是机会

空状态是引导时刻：
1. 简要确认
2. 解释填充的价值
3. 提供清晰操作

"No projects yet. Create your first one to get started." 不是仅 "No items"。

### 7.4 Voice vs Tone

**Voice** 是你品牌的个性——到处一致。
**Tone** 适应时刻。

| 时刻 | Tone 转变 |
|------|----------|
| 成功 | 庆祝、简短："Done! Your changes are live." |
| 错误 | 同理心、有帮助："That didn't work. Here's what to try..." |
| 加载 | 安心："Saving your work..." |
| 破坏性确认 | 严肃、清晰："Delete this project? This can't be undone." |

**永不使用幽默处理错误**。用户已经沮丧。要有帮助，不要卖萌。

### 7.5 翻译写作

德语文本比英语长约 30%。分配空间：

| 语言 | 扩展 |
|------|------|
| 德语 | +30% |
| 法语 | +20% |
| 芬兰语 | +30-40% |
| 中文 | -30%（字符更少，但宽度相同） |

**翻译友好模式**:
- 分开数字（"New messages: 3" 不是 "You have 3 new messages"）
- 使用完整句子作为单字符串
- 避免缩写

### 7.6 一致性：术语问题

选择一个术语并坚持：

| 不一致 | 一致 |
|--------|------|
| Delete / Remove / Trash | Delete |
| Settings / Preferences / Options | Settings |
| Sign in / Log in / Enter | Sign in |
| Create / Add / New | Create |

**建立术语词汇表并执行它**。变化创建困惑。

## 八、绝对禁令总结

### 8.1 BAN 1: 侧条纹边框

```css
/* 禁止 */
border-left: 3px solid red;
border-left: 4px solid var(--color-warning);
border-left: 5px solid oklch(...);
```

**原因**: 这是管理面板、仪表板和医疗 UI 中最过度使用的"设计点缀"。

**替代**: 使用完全不同的元素结构——全边框、背景色调、前导数字/图标，或无视觉指示器。

### 8.2 BAN 2: 渐变文字

```css
/* 禁止 */
background-clip: text;
-webkit-background-clip: text;
background: linear-gradient(...);
```

**原因**: 渐变文字是装饰性而非有意义的，是前三大 AI 设计特征之一。

**替代**: 使用单一纯色。如需强调，使用粗细或大小。
