# 反模式检测系统深度技术分析

> 本文档深入分析 Impeccable 的 24 种反模式检测规则及其实现机制

---

## 一、系统架构

### 1.1 检测器设计哲学

反模式检测器是一个**通用文件**，能够自动检测环境（浏览器 vs Node）并适配：

```javascript
// 环境检测
const IS_BROWSER = typeof window !== 'undefined';
const IS_NODE = !IS_BROWSER;

// Node 环境动态加载
if (!IS_BROWSER) {
  fs = (await import('node:fs')).default;
  path = (await import('node:path')).default;
}
```

### 1.2 使用方式

**Node 环境**:
```bash
node detect-antipatterns.mjs [file-or-dir...]   # jsdom 解析 HTML
node detect-antipatterns.mjs https://...         # Puppeteer
node detect-antipatterns.mjs --fast [files...]   # 仅正则
node detect-antipatterns.mjs --json              # JSON 输出
```

**浏览器环境**:
```html
<script src="detect-antipatterns-browser.js"></script>
<!-- 重新扫描 -->
window.impeccableScan()
```

### 1.3 退出码

- `0` = 干净（无问题）
- `2` = 发现问题

## 二、反模式规则详解

### 2.1 AI Slop 类 (AI 生成特征)

#### 2.1.1 side-tab - 侧边条纹边框

**ID**: `side-tab`
**类别**: `slop`
**检测**: `checkBorders()`

**描述**: 卡片一侧的粗色边框——最易识别的 AI 生成 UI 特征

**检测逻辑**:
```javascript
// CSS 模式匹配
border-left: 3px solid ...   // 宽度 > 1px
border-right: 4px solid ...  // 单侧
```

**修复方案**: 使用完全不同的元素结构——全边框、背景色调、前导数字/图标，或无视觉指示器

**技能映射**: `Visual Details` → "colored accent stripe"

---

#### 2.1.2 overused-font - 过度使用的字体

**ID**: `overused-font`
**类别**: `slop`
**检测**: `checkFonts()`

**描述**: Inter、Roboto、Open Sans、Lato、Montserrat、Arial 在数百万网站上使用

**检测逻辑**:
```javascript
const OVERUSED_FONTS = new Set([
  'inter', 'roboto', 'open sans', 'lato', 'montserrat', 'arial', 'helvetica',
]);

// 品牌字体域名白名单
const BRAND_FONT_DOMAINS = {
  'roboto': ['google.com', 'youtube.com', 'android.com', ...],
  'google sans': ['google.com', ...],
};
```

**修复方案**: 选择独特的字体，赋予界面个性

**技能映射**: `Typography` → "overused fonts like Inter"

---

#### 2.1.3 gradient-text - 渐变文字

**ID**: `gradient-text`
**类别**: `slop`
**检测**: `checkColors()`

**描述**: 渐变文字是装饰性而非有意义的——常见 AI 特征

**检测逻辑**:
```javascript
// CSS 模式匹配
background-clip: text;
-webkit-background-clip: text;
background: linear-gradient(...);  // 或 radial/conic-gradient
```

**修复方案**: 使用单一纯色。如需强调，使用粗细或大小

**技能映射**: `Color & Contrast` → "gradient text for"

---

#### 2.1.4 ai-color-palette - AI 色彩调色板

**ID**: `ai-color-palette`
**类别**: `slop`
**检测**: `checkColors()`

**描述**: 紫色/紫色渐变和深色背景上的青色是最易识别的 AI 生成 UI 特征

**检测逻辑**:
```javascript
// 检测紫色/青色渐变组合
// 深色背景 + 高饱和度青色/紫色强调
```

**修复方案**: 选择独特、有意图的调色板

**技能映射**: `Color & Contrast` → "AI color palette"

---

#### 2.1.5 nested-cards - 嵌套卡片

**ID**: `nested-cards`
**类别**: `slop`
**检测**: DOM 结构遍历

**描述**: 卡片中的卡片创建视觉噪音和过度深度

**修复方案**: 扁平化层次——使用间距、排版和分隔线

**技能映射**: `Layout & Space` → "Nest cards inside cards"

---

#### 2.1.6 icon-tile-stack - 图标瓦片堆叠

**ID**: `icon-tile-stack`
**类别**: `slop`
**检测**: `checkIconTile()`

**描述**: 标题上方的小圆角方形图标容器——通用 AI 功能卡片模板

**检测逻辑**:
```javascript
// 检测结构: 
// <div class="icon-tile">  // 圆角方形容器
//   <icon>
// </div>
// <heading>
```

**修复方案**: 尝试并排图标和标题，或让图标在流中无容器

**技能映射**: `Typography` → "large icons with rounded corners above every heading"

---

#### 2.1.7 bounce-easing - 弹跳缓动

**ID**: `bounce-easing`
**类别**: `slop`
**检测**: `checkMotion()`

**描述**: 弹跳和弹性缓动感觉过时俗气。真实物体会平滑减速

**检测逻辑**:
```javascript
// CSS 检测
animation-timing-function: bounce;
animation-timing-function: elastic;
// cubic-bezier 弹跳模式
```

**修复方案**: 使用指数缓动 (ease-out-quart/quint/expo)

**技能映射**: `Motion` → "bounce or elastic easing"

---

#### 2.1.8 dark-glow - 深色模式发光强调

**ID**: `dark-glow`
**类别**: `slop`
**检测**: `checkGlow()`

**描述**: 深色背景配彩色阴影发光是 AI 生成 UI 的默认"酷"外观

**修复方案**: 使用微妙、有目的的照明——或跳过深色主题

**技能映射**: `Color & Contrast` → "dark mode with glowing accents"

---

#### 2.1.9 flat-type-hierarchy - 扁平类型层次

**ID**: `flat-type-hierarchy`
**类别**: `slop`
**检测**: `checkPageTypography()`

**描述**: 字体大小太接近——没有清晰的视觉层次

**修复方案**: 使用更少的大小，更多的对比（至少 1.25 比例）

**技能映射**: `Typography` → "flat type hierarchy"

---

#### 2.1.10 single-font - 单一字体

**ID**: `single-font`
**类别**: `slop`
**检测**: `checkFonts()`

**描述**: 整个页面只使用一个字体家族

**修复方案**: 配对独特的显示字体和精致的正文字体

**技能映射**: `Typography` → "only one font family for the entire page"

---

#### 2.1.11 monotonous-spacing - 单调间距

**ID**: `monotonous-spacing`
**类别**: `slop`
**检测**: CSS 分析

**描述**: 到处使用相同的间距值——无节奏，无变化

**修复方案**: 相关项目紧凑分组，部分之间慷慨分离

**技能映射**: `Layout & Space` → "same spacing everywhere"

---

#### 2.1.12 everything-centered - 一切居中

**ID**: `everything-centered`
**类别**: `slop`
**检测**: CSS 分析

**描述**: 每个文本元素都是居中对齐

**修复方案**: 左对齐文本配不对称布局感觉更有设计感

**技能映射**: `Layout & Space` → "Center everything"

---

#### 2.1.13 border-accent-on-rounded - 圆角元素边框强调

**ID**: `border-accent-on-rounded`
**类别**: `slop`
**检测**: `checkBorders()`

**描述**: 圆角卡片上的粗边框强调——边框与圆角冲突

**技能映射**: `Visual Details` → "colored accent stripe"

---

### 2.2 Quality 类 (设计质量问题)

#### 2.2.1 pure-black-white - 纯黑背景

**ID**: `pure-black-white`
**类别**: `quality`

**描述**: #000000 背景看起来刺眼不自然

**修复方案**: 略微向品牌色调倾斜 (e.g., oklch(12% 0.01 250))

**技能映射**: `Color & Contrast` → "pure black (#000)"

---

#### 2.2.2 gray-on-color - 彩色背景上的灰色文字

**ID**: `gray-on-color`
**类别**: `quality`
**检测**: `checkColors()`

**描述**: 灰色文字在彩色背景上看起来褪色

**修复方案**: 使用背景色的深色调，或白色/近白色

**技能映射**: `Color & Contrast` → "gray text on colored backgrounds"

---

#### 2.2.3 low-contrast - 低对比度文字

**ID**: `low-contrast`
**类别**: `quality`
**检测**: `checkColors()` - 对比度计算

**描述**: 文字不满足 WCAG AA 对比度要求（正文 4.5:1，大字 3:1）

**检测逻辑**:
```javascript
function relativeLuminance({ r, g, b }) {
  const [rs, gs, bs] = [r/255, g/255, b/255].map(c =>
    c <= 0.03928 ? c/12.92 : ((c+0.055)/1.055)**2.4
  );
  return 0.2126*rs + 0.7156*gs + 0.0722*bs;
}

function contrastRatio(c1, c2) {
  const l1 = relativeLuminance(c1);
  const l2 = relativeLuminance(c2);
  return (Math.max(l1,l2) + 0.05) / (Math.min(l1,l2) + 0.05);
}
```

---

#### 2.2.4 layout-transition - 布局属性动画

**ID**: `layout-transition`
**类别**: `quality`
**检测**: `checkMotion()`

**描述**: 动画 width/height/padding/margin 导致布局抖动和性能问题

**修复方案**: 使用 transform 和 opacity，或 grid-template-rows 用于高度动画

**技能映射**: `Motion` → "Animate layout properties"

---

#### 2.2.5 line-length - 行长度过长

**ID**: `line-length`
**类别**: `quality`

**描述**: 超过 ~80 字符的文本行难以阅读

**修复方案**: 添加 max-width (65ch 到 75ch) 到文本容器

**技能映射**: `Layout & Space` → "wrap beyond ~80 characters"

---

#### 2.2.6 cramped-padding - 拥挤的内边距

**ID**: `cramped-padding`
**类别**: `quality`

**描述**: 文本离容器边缘太近

**修复方案**: 至少 8px（理想 12-16px）内边距

---

#### 2.2.7 tight-leading - 紧凑行高

**ID**: `tight-leading`
**类别**: `quality`

**描述**: 行高低于 1.3x 字体大小使多行文本难以阅读

**修复方案**: 正文使用 1.5 到 1.7 行高

---

#### 2.2.8 skipped-heading - 跳过的标题级别

**ID**: `skipped-heading`
**类别**: `quality`

**描述**: 标题级别不应跳过（如 h1 后直接 h3）

**原因**: 屏幕阅读器使用标题层次进行导航

---

#### 2.2.9 justified-text - 两端对齐文本

**ID**: `justified-text`
**类别**: `quality`

**描述**: 无连字符的两端对齐创建不均匀的单词间距（"白色河流"）

**修复方案**: 正文使用 text-align: left，或启用 hyphens: auto

---

#### 2.2.10 tiny-text - 微小正文

**ID**: `tiny-text`
**类别**: `quality`

**描述**: 低于 12px 的正文难以阅读，尤其是高 DPI 屏幕

**修复方案**: 正文至少 14px，16px 理想

---

#### 2.2.11 all-caps-body - 全大写正文

**ID**: `all-caps-body`
**类别**: `quality`

**描述**: 长段落全大写难以阅读

**原因**: 我们通过形状识别单词（升部和降部），全大写移除了这些

**技能映射**: `Typography` → "long body passages in uppercase"

---

#### 2.2.12 wide-tracking - 宽字母间距

**ID**: `wide-tracking`
**类别**: `quality`

**描述**: 正文上超过 0.05em 的字母间距破坏自然字符分组

**修复方案**: 仅对短大写标签使用宽字距

## 三、检测函数架构

### 3.1 核心检测函数

```
┌─────────────────────────────────────────────────────┐
│                  检测函数层级                        │
├─────────────────────────────────────────────────────┤
│                                                      │
│  checkColors(opts)     → 颜色相关检测               │
│      ├── low-contrast                                │
│      ├── gradient-text                               │
│      ├── ai-color-palette                            │
│      ├── gray-on-color                               │
│      └── pure-black-white                            │
│                                                      │
│  checkBorders(opts)    → 边框相关检测               │
│      ├── side-tab                                    │
│      └── border-accent-on-rounded                    │
│                                                      │
│  checkMotion(opts)     → 动效相关检测               │
│      ├── bounce-easing                               │
│      └── layout-transition                           │
│                                                      │
│  checkGlow(opts)       → 发光效果检测               │
│      └── dark-glow                                   │
│                                                      │
│  checkIconTile(opts)   → 图标瓦片检测               │
│      └── icon-tile-stack                            │
│                                                      │
│  checkPageTypography() → 页面级排版检测             │
│      └── flat-type-hierarchy                        │
│                                                      │
│  checkFonts()          → 字体检测                   │
│      ├── overused-font                              │
│      └── single-font                                │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 3.2 适配器模式

每个检测函数有两个适配器：

```javascript
// 浏览器适配器 - 使用 getComputedStyle + getBoundingClientRect
function checkElementXxxDOM(el) {
  const style = getComputedStyle(el);
  const rect = el.getBoundingClientRect();
  return checkXxx({ tag, style, rect, ... });
}

// Node (jsdom) 适配器 - 使用 window.getComputedStyle + parseFloat(style.width)
function checkElementXxx(el, tag, window) {
  const style = window.getComputedStyle(el);
  // jsdom 不做布局！必须读取显式像素值
  const width = parseFloat(style.width);
  return checkXxx({ tag, style, width, ... });
}
```

### 3.3 jsdom 约束

**关键**: jsdom 不执行布局！

| 问题 | 解决方案 |
|------|----------|
| `getBoundingClientRect()` 返回 0×0 | 使用 `parseFloat(style.width)` |
| `background:` 简写不分解读 | 使用 `resolveBackground()` 帮助函数 |
| 颜色未标准化 | 使用 `parseGradientColors()` 处理两种格式 |

## 四、规则定义模式

### 4.1 完整规则结构

```javascript
{
  id: 'icon-tile-stack',                    // kebab-case，唯一，稳定
  category: 'slop',                          // 'slop' 或 'quality'
  name: 'Icon tile stacked above heading',  // 人类可读，用于扩展 UI
  description: '...',                        // 1-2 句话，用于 CLI 输出
  skillSection: 'Typography',               // 可选但强烈推荐
  skillGuideline: 'large icons with...',    // 必须匹配 SKILL.md 中的 DON'T 行
}
```

### 4.2 类别定义

| 类别 | 定义 |
|------|------|
| **slop** | "AI 特征"。模式显示"AI 生成了这个"。如紫色渐变、渐变文字、深色发光、粗侧边框、图标瓦片堆叠。标记这些是关于品味和新鲜感，不是正确性。 |
| **quality** | 真正的设计或可访问性问题，无论谁写的代码。WCAG 对比度、行长度、内边距、行高、两端对齐、跳过标题等。 |

**判断标准**: *"一个细心且有品味的人类设计师会发布这个吗？"*
- 否 → `quality`
- 会（因为它工作正常，只是看起来模板化）→ `slop`

## 五、测试驱动开发流程

### 5.1 测试夹具约定

**两列约定**:
```html
<!DOCTYPE html>
<html>
<head>
  <style>
    .grid { display: grid; grid-template-columns: 1fr 1fr; gap: 32px; }
    .col h2 { font-size: 14px; text-transform: uppercase; }
    /* 显式像素尺寸（jsdom 无法布局） */
  </style>
</head>
<body>
  <div class="grid">
    <div class="col" data-col="flag">
      <h2>Should flag</h2>
      <!-- 应标记的案例，每个有唯一 <h3> 文本 -->
    </div>
    <div class="col" data-col="pass">
      <h2>Should pass</h2>
      <!-- 应通过的案例：覆盖你能想到的每个假阳性形状 -->
    </div>
  </div>
  <script src="/js/detect-antipatterns-browser.js"></script>
</body>
</html>
```

### 5.2 测试模式

```javascript
describe('detectHtml — {rule-id}', () => {
  const SHOULD_FLAG = ['Heading One', 'Heading Two', /* ... */];
  const SHOULD_PASS = ['Pass Heading One', /* ... */];

  it('{rule-id}: flags only the should-flag column', async () => {
    const f = await detectHtml(path.join(FIXTURES, '{rule-id}.html'));
    const flagged = new Set();
    for (const r of f) {
      if (r.antipattern !== '{rule-id}') continue;
      const m = (r.snippet || '').match(/"([^"]+)"/);
      if (m) flagged.add(m[1]);
    }
    for (const text of SHOULD_FLAG) {
      assert.ok(flagged.has(text), `expected "${text}" to be flagged`);
    }
    for (const text of SHOULD_PASS) {
      assert.ok(!flagged.has(text), `"${text}" should NOT be flagged`);
    }
  });
});
```

### 5.3 TDD 顺序（不可协商）

1. 编写夹具（两列约定）
2. 编写失败的测试
3. 添加规则定义
4. 实现纯检查函数
5. 添加两个适配器
6. 接入两个元素迭代循环
7. 添加 SKILL.md DON'T 行（如果规则不复用现有行）
8. 运行构建（重新生成一切并验证）
9. 运行测试套件
10. 在浏览器中实时页面验证

## 六、同步机制

### 6.1 五个同步点

| 位置 | 内容 | 同步方式 |
|------|------|----------|
| `src/detect-antipatterns.mjs` | 规则元数据 + 检测逻辑 | **手动编辑**（真相源） |
| `src/detect-antipatterns-browser.js` | 浏览器打包引擎 | `bun run build:browser` 生成 |
| `extension/detector/detect.js` | Chrome 扩展引擎 | `bun run build:extension` 生成 |
| `extension/detector/antipatterns.json` | 扩展规则列表 | `bun run build:extension` 生成 |
| `source/skills/impeccable/SKILL.md` | 技能 DON'T 行 | **手动编辑** |

### 6.2 构建验证

构建过程会交叉验证每个规则：

```bash
bun run build && bun run build:browser && bun run build:extension
```

**验证内容**:
- 检查每个有 `skillGuideline` 的规则
- 对 `source/skills/impeccable/SKILL.md` 进行 `String.includes()` 检查
- 如果漂移存在，**构建失败**

## 七、常见问题与陷阱

### 7.1 常见错误

| 错误 | 症状 | 解决方案 |
|------|------|----------|
| 忘记运行 `build:extension` | 扩展 JSON 过期，缺少新规则切换 | 始终运行所有三个构建命令 |
| 忘记更新两个循环 | 测试在 jsdom 通过但浏览器静默（或反之） | grep 现有规则的适配器调用并复制位置 |
| `skillGuideline` 不匹配 | 验证器失败 | 子字符串必须在某 `**DON'T**:` 行中逐字出现 |
| 错误的 `skillSection` 名 | 解析错误 | 使用精确值：`Typography`, `Color & Contrast` 等 |
| 夹具无显式像素 | jsdom 返回 0×0，规则永不匹配 | 在 CSS 中设置 `width: Npx; height: Npx` |

### 7.2 Snippet 约定

测试提取标题文本使用正则 `/"([^"]+)"/`：

```javascript
// 好的 snippet
'80x80px icon tile above h3 "Lightning Fast"'
'4.5:1 (need 4.5:1) — text #808080 on #3b82f6'

// 用直双引号包装识别标题文本
```

## 八、预提交清单

在提交新规则前，**所有**必须为真：

- [ ] 测试通过：`bun run test` 是绿色
- [ ] 构建通过：所有三个构建命令成功（验证器说 `✓ Validated N/N anti-pattern rules`）
- [ ] 实时验证：规则在真实页面上触发，首页零假阳性
