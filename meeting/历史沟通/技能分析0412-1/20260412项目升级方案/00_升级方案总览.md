# Flow Harness 项目升级方案总览

**制定日期**: 2026-04-12  
**当前版本**: v0.1.0  
**目标版本**: v0.3.0  
**整体进度基线**: ~48%  
**升级目标**: 稳定核心、补全能力、提升可靠性，不破坏任何已有规则

---

## 核心约束（不可变）

以下规则来自 `AGENTS.md`，升级过程中**严禁修改**：

| 约束项 | 内容 |
|--------|------|
| 架构固定 | 1 CEO (Supervisor) + 4总监：Explore / Plan / General-Purpose / Inspector |
| CEO禁止 | 不写代码、不改文件、不跑命令、只做决策和调度 |
| 核心链路保护 | 不修改 schema、不修改支付/认证/鉴权、不删生产数据、不修改 API 契约 |
| 安全原则 | Deny-by-Default、Inspect Before Trust、Single Responsibility |
| 6步闭环 | 分析→拆解→分工→执行→检查→复盘（不可删减步骤） |

## 当前问题诊断

### 关键数字
```
总运行次数: 610
成功: 245 (40.2%)
失败: 365 (59.8%)   ← 核心问题
```

### 主要失败类型
1. **"部分任务失败"** — 模拟执行随机失败率过高（feature/general/refactor 失败率接近100%）
2. **真实执行缺失** — 所有 Agent 依然是模拟执行，没有连接真实 LLM
3. **技能系统缺失** — 没有可复用的技能库，每次从零开始
4. **Token 管理粗放** — `token-tracker.js` 存在但策略层薄弱
5. **Hooks 系统薄弱** — `config.yml` 只有 echo 级别的钩子
6. **跨端调度未实现** — `cross-platform-dispatcher.js` 待完善

---

## 升级设计原则

1. **稳定优先**: 所有改动必须向后兼容，现有测试不能退化
2. **渐进增强**: 新能力通过增量方式叠加，不重构已有模块
3. **借鉴最佳实践**: 结合 RTK（Token压缩）、ECC（技能/钩子体系）、Impeccable（反模式检测）
4. **不改变规则**: AGENTS.md 架构、config.yml 结构、6步闭环逻辑保持不变

---

## 升级分6个阶段

| 阶段 | 名称 | 重点 | 预计时间 |
|------|------|------|----------|
| **Phase A** | 稳定基础 | 修复高失败率、降低模拟随机性 | 1周 |
| **Phase B** | 技能系统 | 添加可复用技能库（ECC 风格） | 1-2周 |
| **Phase C** | 钩子与命令 | 增强钩子生命周期 + 命令系统 | 1-2周 |
| **Phase D** | Token 智能管控 | RTK 风格输出压缩 + 预算看板 | 1周 |
| **Phase E** | Agent 真实执行 | 连接真实 LLM，去模拟 | 2-3周 |
| **Phase F** | 外部能力接入层 | 以受治理方式接入低风险外部 API 能力 | 1-2周 |
