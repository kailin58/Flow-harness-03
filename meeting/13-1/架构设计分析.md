# Everything Claude Code 架构设计与最佳实践分析

## 一、项目设计哲学

### 1.1 核心设计原则

**1. 实战驱动 (Battle-Tested)**
- 所有组件经过10个月以上日常高强度使用
- 从实际问题中提炼，而非理论假设
- 持续迭代优化，响应真实需求

**2. 关注点分离 (Separation of Concerns)**
```
Agents (做什么) → Skills (怎么做) → Rules (边界约束)
     ↓               ↓                  ↓
   代理执行       技能指导           规则限制
```

**3. 防御性编程 (Defensive Programming)**
- 钩子失败不阻塞工具执行 (exit 0原则)
- 配置保护防止意外修改
- 多层安全检查

**4. 渐进增强 (Progressive Enhancement)**
- 基础功能开箱即用
- 高级功能可选配置
- 通过环境变量控制行为

### 1.2 架构分层

```
┌────────────────────────────────────────────────────────────┐
│                      用户交互层                             │
│  (Commands: /tdd, /plan, /code-review, etc.)              │
├────────────────────────────────────────────────────────────┤
│                      编排协调层                             │
│  (planner, orchestrator, loop-operator)                   │
├────────────────────────────────────────────────────────────┤
│                      专业代理层                             │
│  (code-reviewer, security-reviewer, tdd-guide, etc.)      │
├────────────────────────────────────────────────────────────┤
│                      技能知识层                             │
│  (Skills: patterns, standards, best practices)            │
├────────────────────────────────────────────────────────────┤
│                      基础设施层                             │
│  (Hooks, MCP, Scripts, Rules)                             │
└────────────────────────────────────────────────────────────┘
```

## 二、代理设计模式分析

### 2.1 代理提示词架构

项目采用结构化的代理提示词设计，遵循一致的模板：

```markdown
# 代理名称

## Your Role (角色定义)
明确代理的职责边界和专业领域

## Workflow (工作流程)
分步骤的执行指导，编号清晰

## Checklist (检查清单)
分类整理的检查项，优先级明确

## Examples (示例)
PASS/FAIL对比代码示例

## Output Format (输出格式)
结构化输出模板
```

### 2.2 代理协作模式

**串行协作模式:**
```
planner → code-reviewer → security-reviewer → tdd-guide
   ↓           ↓                ↓                ↓
 规划方案    质量检查         安全审查         测试验证
```

**并行协作模式:**
```
          ┌─→ python-reviewer
          │
planner ──┼─→ security-reviewer
          │
          └─→ performance-reviewer
                 ↓
            综合报告
```

**层级委托模式:**
```
chief-of-staff (协调者)
    ├── planner (规划)
    ├── architect (架构)
    └── code-reviewer (审查)
```

### 2.3 代理权限控制

```json
{
  "tools": ["@builtin"],           // 基础工具集
  "allowedTools": [                // 特定允许的工具
    "fs_read",                     // 只读文件
    "fs_write",                    // 写入文件
    "shell"                        // 执行命令
  ]
}
```

**最小权限原则:**
- planner: 仅 fs_read (只规划，不修改)
- code-reviewer: fs_read + shell (读取+运行检查命令)
- tdd-guide: fs_read + fs_write + shell (完整权限)

## 三、技能系统设计

### 3.1 技能触发机制

**触发条件设计原则:**

| 原则 | 说明 | 示例 |
|------|------|------|
| 明确性 | 条件清晰无歧义 | "Code imports `anthropic`" |
| 可检测性 | 基于代码特征 | 导入语句、函数调用 |
| 互斥性 | 避免技能冲突 | 不同语言模式分开 |
| 层次性 | 支持技能组合 | 基础模式 + 安全模式 |

**触发条件分类:**

```yaml
# 代码特征触发
- Code imports `anthropic`
- Code imports `@anthropic-ai/sdk`

- User says "research", "deep dive"

# 操作类型触发
- Implementing authentication
- Handling user input
- Creating new API endpoints

# 项目阶段触发
- Writing new features
- Fixing bugs
- Refactoring existing code
```

## 四、钩子系统设计

### 4.1 钩子生命周期

```
SessionStart
     ↓
     ┌──────────────────────────────────────┐
     │           工具执行循环                │
     │  PreToolUse → Tool → PostToolUse     │
     │       ↑                    ↓         │
     │       └────────────────────┘         │
     └──────────────────────────────────────┘
                    ↓
                  Stop
                    ↓
              SessionEnd
```

### 4.2 钩子匹配器设计

**匹配器类型:**

| 匹配器 | 匹配工具 | 典型用途 |
|--------|----------|----------|
| `*` | 所有工具 | 全局观察、日志 |
| `Bash` | 命令执行 | Git操作监控 |
| `Write` | 文件写入 | 配置保护 |
| `Edit` | 文件编辑 | 质量检查 |
| `Edit\|Write` | 编辑组合 | 综合检查 |

### 4.3 钩子执行策略

**同步执行 (默认):**
```json
{
  "type": "command",
  "command": "node script.js",
  "timeout": 10
}
```

**异步执行 (非阻塞):**
```json
{
  "type": "command", 
  "command": "node script.js",
  "async": true,
  "timeout": 30
}
```

**执行原则:**
- PreToolUse: 必须快速 (<200ms)
- PostToolUse: 可异步执行
- Stop: 可长时间运行 (批量处理)

### 4.4 钩子链式处理

**run-with-flags.js 包装器:**

```bash
node "run-with-flags.js" \
  "hook-id" \
  "actual-script.js" \
  "standard,strict"
```

**功能:**
1. 环境变量控制 (ECC_HOOK_PROFILE)
2. 钩子开关 (ECC_DISABLED_HOOKS)
3. 统一日志格式
4. 错误处理隔离

### 4.5 安全钩子设计

**配置保护钩子 (pre:config-protection):**

保护的配置文件:
- `.eslintrc*`
- `.prettierrc*`
- `biome.json`
- `tsconfig.json`
- `.markdownlint*`

**设计理念:**
- 不允许AI降低代码质量标准
- 引导AI修复代码而非修改配置
- 保持团队规范一致性

## 十一、最佳实践总结

### 11.1 代理设计最佳实践

1. **单一职责**: 每个代理专注一个领域
2. **明确边界**: 通过allowedTools控制权限
3. **结构化提示**: 使用一致的提示词模板
4. **示例驱动**: 提供PASS/FAIL代码对比

### 11.2 技能设计最佳实践

1. **触发明确**: 清晰定义激活条件
2. **内容实用**: 提供可直接使用的代码示例
3. **层次分明**: 从基础到高级逐步深入
4. **交叉引用**: 关联相关技能形成知识网络

### 11.3 钩子设计最佳实践

1. **非阻塞优先**: 避免影响用户体验
2. **快速执行**: PreToolUse < 200ms
3. **优雅降级**: 失败时exit 0
4. **可配置**: 支持环境变量控制

### 11.4 配置管理最佳实践

1. **版本控制**: 所有配置纳入git
2. **环境隔离**: 敏感信息用环境变量
3. **验证优先**: 修改前先验证格式
4. **文档同步**: 配置变更同步文档
