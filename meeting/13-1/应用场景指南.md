# Everything Claude Code 应用场景与实践指南

## 一、典型应用场景

### 1.1 新项目启动

**场景描述**: 从零开始创建新项目

**推荐工作流**:
```
1. /plan → 生成实施方案
2. /setup-pm → 配置包管理器
3. /tdd → 开始TDD开发
4. /code-review → 代码审查
5. /security-review → 安全检查
```

**示例对话**:
```
用户: 我要创建一个新的TypeScript后端项目，使用Express和PostgreSQL

Claude: 让我先规划一下实施方案...
[/plan 执行中]

# Implementation Plan: TypeScript Express Backend

## Phase 1: 项目初始化
1. 初始化项目结构
2. 配置TypeScript
3. 设置ESLint和Prettier

## Phase 2: 核心功能
...
```

### 1.2 功能开发

**场景描述**: 在现有项目中添加新功能

**推荐工作流**:
```
1. /feature-dev → 功能开发工作流
2. /tdd → TDD模式开发
3. /code-review → 审查变更
4. /verify → 验证功能
5. /prp-commit → 提交变更
```

**钩子自动触发**:
- pre:bash:commit-quality → 提交前质量检查
- post:quality-gate → 文件编辑后质量门禁
- stop:format-typecheck → 响应结束后格式化

### 1.3 代码审查

**场景描述**: 审查PR或代码变更

**推荐工作流**:
```
1. /review-pr <PR号> → PR审查
2. 或: /code-review → 本地代码审查
3. /security-review → 安全审查(如涉及敏感代码)
```

**审查输出示例**:
```markdown
## Review Summary

| Severity | Count | Status |
|----------|-------|--------|
| CRITICAL | 0     | pass   |
| HIGH     | 2     | warn   |
| MEDIUM   | 3     | info   |
| LOW      | 1     | note   |

Verdict: WARNING — 2 HIGH issues should be resolved before merge.
```

### 1.4 安全审计

**场景描述**: 对代码进行安全检查

**推荐工作流**:
```
1. /security-review → 运行安全审查代理
2. npm audit → 检查依赖漏洞
3. 根据发现修复问题
4. 验证修复效果
```

**检测范围**:
- 硬编码密钥
- SQL注入风险
- XSS漏洞
- 认证/授权问题
- CSRF保护
- 敏感数据暴露

### 1.6 Bug修复

**场景描述**: 修复发现的Bug

**推荐工作流**:

## 二、团队协作场景

### 2.1 代码规范统一

**配置步骤**:

1. **安装项目规则**
```bash
# 将ECC规则复制到项目
cp -r .cursor/rules/ /path/to/project/.cursor/
```

2. **配置钩子**
```json
// .claude/hooks.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{"type": "command", "command": "..."}],
        "id": "pre:bash:commit-quality"
      }
    ]
  }
}
```

3. **团队约定**
- 所有成员使用相同配置
- 提交前必须通过钩子检查
- 定期同步规则更新

### 2.2 PR审查流程

**自动化审查流程**:
```
PR创建
    ↓
自动触发 /review-pr
    ↓
生成审查报告
    ↓
[CRITICAL问题] → 必须修复
    ↓
[HIGH问题] → 建议修复
    ↓
[通过] → 可以合并
```

### 2.3 知识沉淀

**持续学习系统**:
```
日常开发
    ↓
钩子观察 (observe.sh)
    ↓
模式积累 (≥20条)
    ↓
自动分析
    ↓
提取新技能
    ↓
/learn 查看学习成果
```

### 4.2 CI/CD集成

**GitHub Actions配置**:
```yaml
name: Code Review

on: [pull_request]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Run Code Review
        run: |
          npm install
          npm run harness:audit
          
      - name: Security Scan
        run: npm audit --audit-level=high
```

### 4.3 代码质量门禁

**质量门禁配置**:
```json
{
  "qualityGate": {
    "coverage": 80,

    "typecheck": "pass"
  }
}
```

**执行时机**:
- pre:bash:commit-quality → 提交前
- post:quality-gate → 编辑后

## 七、高级使用技巧

### 7.1 多代理协作

**并行执行**:
```
用户: 审查这个PR，检查代码质量、安全性和性能

Claude: 启动多个代理并行审查...
[code-reviewer] 检查代码质量...
[security-reviewer] 检查安全问题...
[performance-reviewer] 检查性能...

生成综合报告
```

### 7.2 自定义钩子

**添加项目特定钩子**:
```javascript
// scripts/hooks/project-check.js

module.exports = {
  run(input) {
    // 项目特定检查逻辑
    const data = JSON.parse(input);
    
    // 检查API密钥格式
    if (data.content?.includes('API_KEY')) {
      return {
        exitCode: 0,
        message: '警告: 检测到API密钥引用，请确保使用环境变量'
      };
    }
    
    return { exitCode: 0 };
  }
};
```

### 7.3 环境变量控制

```bash
# 完全禁用钩子
export ECC_DISABLED_HOOKS=*

# 禁用特定钩子
export ECC_DISABLED_HOOKS=pre:bash:tmux-reminder,stop:desktop-notify

# 切换配置文件
export ECC_HOOK_PROFILE=strict   # strict | standard | minimal

# 启用治理捕获
export ECC_GOVERNANCE_CAPTURE=1
```

## 八、常见问题解决

### Q1: 钩子执行太慢怎么办?

**解决方案**:
1. 设置`ECC_HOOK_PROFILE=minimal`
2. 使用异步钩子
3. 增加超时阈值

### Q2: 如何添加自定义技能?

**步骤**:
1. 创建技能目录: `.claude/skills/my-skill/`
2. 编写SKILL.md文件
3. 在agent.yaml中引用

### Q3: 如何在不同项目间共享配置?

**方案**:
1. 创建共享仓库
2. 使用符号链接
3. 或使用ECC插件安装

### Q4: 如何禁用特定语言的规则?

**方法**:
```bash
# 只保留TypeScript规则
rm .cursor/rules/python-*
rm .cursor/rules/golang-*
```
