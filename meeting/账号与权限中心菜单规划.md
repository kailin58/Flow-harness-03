# 账号与权限中心菜单规划

## 目的

在已经确定“全部账号管理统一收口到平台后台”的前提下，进一步明确 `账号与权限中心` 的菜单结构、职责边界与一期范围，避免后续只知道“要做”，但不知道具体承接哪些能力。

## 定位

`账号与权限中心` 是平台后台中的统一治理工作域。

它不是普通系统配置页，也不是某个业务端域临时挂出来的管理页，而是平台对“人、角色、权限、数据范围、字段范围、审批权限、登录安全、审计”进行统一治理的主工作台。

需要区分：

- `权限与组织中台 / 权限中台`：后端真相源，输出统一身份、授权、鉴权决策与数据范围约束
- `账号与权限中心`：平台后台治理台，负责管理面、配置面与审计面

## 核心承接范围

`账号与权限中心` 统一承接：

- 员工账号治理
- 供应商账号统一治理（含主账号与子账号的身份授权体系）
- 组织与岗位治理
- 角色与权限治理
- 数据范围治理
- 字段范围治理
- 审批权限治理
- 登录与安全策略治理
- 操作审计与账号状态治理

`账号与权限中心` 不承接：

- C 端会员“我的/账号安全”自助页
- 供应商入口内的子账号日常维护操作页面
- 渠道店铺授权凭证管理
- API Key、模型密钥、外部平台密钥
- 各业务模块自己的规则真相源

## 建议一级菜单

建议在 `账号与权限中心` 下拆为以下一级菜单：

1. 员工账号
2. 供应商账号
3. 组织与岗位
4. 角色权限
5. 数据权限
6. 字段权限
7. 审批权限
8. 登录与安全策略
9. 审计日志

## 各菜单职责

### 1. 员工账号

负责：

- 员工账号创建、启用、停用、冻结、解冻
- 账号与组织、岗位、角色绑定
- 初始密码重置、找回、二次验证开关
- 在职、离职、转岗状态管理

不负责：

- 员工个人资料自助编辑
- 业务规则配置

### 2. 供应商账号

负责：

- 供应商主账号开通、冻结、停用
- 供应商子账号统一身份模型与授权模板
- 主账号可用状态与登录策略
- 主账号与 `supplier_id` 绑定关系
- 主账号基础权限模板下发

不负责：

- 供应商子账号的日常新增、停用、重置操作页面
- 跨供应商数据授权

说明：

- 供应商子账号的日常管理入口应留在供应商协同入口
- 但其身份与授权模型仍应纳入统一治理

### 3. 组织与岗位

负责：

- 平台组织架构
- 岗位定义
- 岗位与默认角色模板关系
- 转岗后的权限收敛与继承

不负责：

- 具体业务审批规则内容本身

### 4. 角色权限

负责：

- 角色创建与维护
- 菜单权限
- 页面权限
- 操作权限
- 字段权限
- 高危动作权限

不负责：

- 每个业务模块内部自定义一套角色表

统一要求：

- 角色必须复用统一权限模型
- 不允许商品、订单、售后、客服、供应商等模块各自再造一套权限体系

### 5. 数据权限

负责：

- 按组织、岗位、角色配置数据范围
- 按供应商、品牌、品类、店铺、区域、渠道等维度配置范围
- 为查询与导出生成统一的数据范围约束

不负责：

- 业务模块自己硬编码数据可见范围

说明：

- 这是权限中台在管理面的最核心体现之一
- 后台“谁能看哪些订单、哪些供应商、哪些店铺”都应从这里收口

### 6. 字段权限

负责：

- 成本价是否可见
- 结算金额是否可见
- 用户手机号是否可见全量
- 敏感字段是否可导出

说明：

- 原文已明确字段权限不是补充项，而是权限体系核心组成
- 至少要做到“可看掩码 / 可看全量 / 不可导出”的分层

### 7. 审批权限

负责：

- 谁能审商品
- 谁能审退款
- 谁能审发货异常
- 谁能审账单
- 谁能审批高风险操作

说明：

- 审批权限不应散在业务页面各自维护
- 应与角色、资源、动作、数据范围一并纳入统一权限模型

### 8. 登录与安全策略

负责：

- 密码策略
- 验证码策略
- 登录失败锁定策略
- 设备信任策略
- 二次验证策略
- 异地登录提醒
- 强密码策略
- IP 白名单预留
- 单点登录预留
- 高风险操作二次确认策略

不负责：

- 模型密钥、渠道密钥、第三方接口密钥

### 9. 审计日志

负责：

- 账号变更日志
- 角色与权限变更日志
- 登录日志
- 高危操作日志
- 导出与批量操作日志

不负责：

- 业务流水替代
- 业务报表替代

## 与其他模块的边界

### 与 `会员管理中心` 的边界

- `会员管理中心` 关注会员档案、标签、等级、生命周期
- `账号与权限中心` 关注员工账号、供应商主账号、权限、登录策略、审计
- C 端会员账号安全仍在前台“我的/账号安全”

### 与 `权限中台` 的边界

- `权限中台` 是后端真相源与鉴权执行层
- `账号与权限中心` 是治理工作台与配置入口

### 与 `供应商协同入口` 的边界

- 平台后台治理供应商主账号
- 供应商协同入口治理本供应商范围内的子账号自助能力

### 与 `系统配置` 的边界

- `系统配置` 承接通用系统参数
- `账号与权限中心` 承接账号、权限、数据范围、安全策略、审计
- 两者不应再次混并

## 面向未来模块的接口预留

`账号与权限中心` 不只是当前模块的管理台，还必须承担未来新增模块的统一接入治理。

建议预留以下接口能力：

### 1. 模块注册接口

用于新模块接入统一权限模型。

至少应支持注册：

- `module_code`
- 资源清单（resource）
- 动作清单（action）
- 默认角色模板
- 是否高风险模块
- 是否需要字段权限
- 是否需要审批权限

### 2. 统一鉴权接口

供未来模块复用，不允许各模块自造权限判断。

至少应支持：

- 身份主体识别
- 角色查询
- 资源动作鉴权
- 数据范围返回
- 字段范围返回
- 高风险动作二次校验

### 3. 菜单装配接口

用于未来模块接入平台后台或其他工作台时，动态装配菜单、页面入口、按钮级能力。

### 4. 统一审计接口

未来模块必须统一写入审计中心。

至少覆盖：

- 登录
- 导出
- 批量操作
- 审批
- 配置变更
- 高风险动作

### 5. 生命周期事件接口

用于未来模块响应账号状态变化。

建议预留事件：

- 入职
- 离职
- 转岗
- 冻结
- 解冻
- 供应商停用
- 角色变更
- 数据范围变更

### 6. 安全策略复用接口

未来模块应统一复用：

- 密码策略
- 二次验证
- 设备信任
- 异地登录提醒
- IP 白名单预留

## 新模块接入约束

后续新增模块如果需要接入账号与权限体系，必须满足：

- 先在 `权限与组织中台 / 权限中台` 注册资源与动作模型
- 再在 `账号与权限中心` 配置角色、数据范围、字段范围、审批权限
- 最后由模块消费统一鉴权、统一审计、统一安全策略接口

不允许：

- 模块内部自建独立账号主表
- 模块内部自建独立角色表
- 模块内部绕过统一审计
- 模块内部绕过高风险操作校验

## 一期范围建议

### 一期必须做

- 员工账号管理
- 供应商账号统一治理基础版（主账号开通/冻结 + 子账号身份授权模型）
- 组织与岗位基础结构
- 角色权限基础模型
- 数据权限基础模型
- 字段权限基础模型
- 审批权限基础模型
- 登录与安全基础策略
- 审计日志查询
- 模块注册接口预留
- 统一鉴权接口预留
- 统一审计接口预留
- 生命周期事件接口预留

### 一期预留但不展开

- 更复杂的审批链与授权流
- SSO 深度接入
- 多组织多法人复杂结构
- 菜单动态编排深度能力

### 二期再做

- 更复杂的 ABAC 策略
- 临时授权与到期回收自动化
- 更精细的风险策略编排
- 权限模拟与权限冲突分析

## 当前阶段结论

- `账号与权限中心` 应作为平台后台独立一级菜单存在
- 它是账号与权限统一治理主工作台，不是系统配置子页
- 它承接员工账号、供应商主账号、角色权限、数据权限、安全策略与审计
- 各入口只保留本角色的账号自助页，不替代统一治理
