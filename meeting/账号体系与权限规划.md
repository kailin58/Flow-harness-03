# 账号体系与权限规划

## 目的

明确“全部账号管理”应该放在哪里体现，避免后续把会员账号、员工账号、供应商账号、子账号、权限配置混成一套页面。

本文件只回答三件事：

- 账号分哪几类
- 统一治理放在哪里
- 各入口分别暴露什么账号能力

## 总原则

账号管理不要做成一个所有角色共用的杂糅页面，而应采用：

`统一治理中心 + 各入口自助页`

也就是：

- 平台后台负责统一治理账号、角色、权限、数据范围、审计
- 各入口只暴露与本角色强相关的账号自助能力
- 消费者账号、员工账号、供应商账号共用底层身份与权限能力，但前台表现不同

## 账号分类

### 1. 会员账号

服务对象：

- 普通用户
- 注册会员
- 推广员

核心特征：

- 以会员 ID 作为主身份
- 可绑定手机号、微信、支付宝、小程序 `openid` 等多端凭证
- 账号安全与凭证合并属于前台“我的”体系

### 2. 员工账号

服务对象：

- 平台运营
- 审核
- 财务
- 风控
- 内容运营
- 客服
- 仓配管理人员
- 管理员

核心特征：

- 以组织、岗位、角色、数据范围为核心
- 权限控制比个人资料更重要
- 应由平台后台统一治理，不分散到各业务页面里各做一套

### 3. 供应商账号

服务对象：

- 供应商主账号
- 供应商子账号

核心特征：

- 必须带 `supplier_id` 数据范围
- 主账号可管理本供应商子账号
- 不能进入平台级治理视图

### 4. 系统与集成账号

服务对象：

- 外部渠道店铺授权
- 系统对系统调用
- 机器人/服务账号

核心特征：

- 这类不是“人账号”
- 不应混入会员、员工、供应商账号页面
- 应分别归属到渠道店铺、开放接口、系统集成配置中管理

## 最合理的体现方式

### 一、统一治理层：放在平台后台的 `账号与权限中心`

这是“全部账号管理”的主体现位置。

需要特别区分两层：

- `权限与组织中台 / 权限中台`：后端真相源，负责统一身份、授权、鉴权决策与数据范围约束输出
- `账号与权限中心`：平台后台治理台，负责配置、管理、审计与运营侧操作入口

建议作为平台后台中的独立一级菜单，而不是埋在“系统配置”下面。

原因：

- 账号、角色、权限、数据范围、审计是平台基础能力，不是普通配置项
- 员工账号与供应商账号治理都依赖权限中台
- 如果放在系统配置里，后续很容易越做越乱、越做越隐蔽

`账号与权限中心` 建议承接：

- 员工账号管理
- 平台组织与岗位管理
- 角色管理
- 菜单权限、操作权限、数据权限、字段权限
- 审批权限与高风险动作审批策略
- 供应商主账号开通与冻结
- 供应商子账号统一身份与授权策略
- 登录策略（密码、验证码、设备信任、二次验证、SSO 预留）
- 审计日志与账号状态管理

不建议放在这里的内容：

- 会员前台“我的/账号安全”日常操作
- 供应商子账号在供应商入口内的日常维护操作页面
- 渠道店铺授权凭证
- API Key、模型密钥、第三方平台密钥

### 二、入口自助层：放在各自入口里

#### 1. 用户统一入口

放在：

- `我的/账号安全`

建议承接：

- 手机号绑定/换绑
- 微信/支付宝/小程序 `openid` 绑定与合并提示
- 登录设备与安全提示
- 会员基础资料
- 注销、找回、登录态管理

不建议承接：

- 角色权限配置
- 员工/供应商账号治理

#### 2. 平台后台入口

放在：

- 独立一级菜单 `账号与权限中心`

建议承接：

- 员工账号治理
- 平台岗位与组织
- 角色与权限
- 数据范围与审批策略
- 供应商主账号开通/冻结

#### 3. 供应商协同入口

放在：

- `账号设置`
- `子账号管理`

建议承接：

- 本供应商主账号资料维护
- 本供应商子账号新增、停用、重置
- 子账号角色分配（仅限本供应商范围）
- 高风险操作的二次验证配合

不建议承接：

- 平台员工账号管理
- 跨供应商账号管理
- 平台级权限策略定义

#### 4. 客服工作台入口

放在：

- 个人设置

建议承接：

- 修改本人密码
- 修改本人资料
- 查看本人登录状态

不建议承接：

- 独立做一套客服账号体系
- 独立做角色与权限配置

客服账号本质上仍属于员工账号，由 `账号与权限中心` 统一治理。

#### 5. 发货执行入口

放在：

- 极简个人设置

建议承接：

- 当前账号查看
- 退出登录
- 极少量设备绑定/解绑

不建议承接：

- 复杂账号管理
- 角色与权限配置

发货执行入口重点是履约动作，不应演变成账号后台。

## 统一边界结论

- “全部账号管理”的统一治理主入口，应放在平台后台的 `账号与权限中心`
- `权限与组织中台 / 权限中台` 是后端真相源，`账号与权限中心` 是治理工作台
- 用户端只放 `我的/账号安全`
- 供应商端只放 `账号设置/子账号管理`
- 客服与发货端只放极简个人设置
- 渠道授权账号、API Key、模型密钥不归入人账号体系
- 供应商子账号的身份与授权归统一治理，日常维护操作放在供应商入口承接

## 与现有口径的关系

- 不新增一级入口
- 不新增新的业务端域
- 这是平台后台入口下新增的独立治理工作域
- 与 `会员管理中心`、`权限中台`、`渠道店铺`、`AI模型管理中心` 分工明确

## 面向未来模块的接口预留原则

账号管理必须预留未来新增模块的统一接入能力，不能每增加一个模块就新增一套独立账号、角色与权限逻辑。

统一原则：

- 新模块接入统一身份体系，不新建独立账号主表
- 新模块接入统一授权模型，不新建独立角色权限体系
- 新模块接入统一审计体系，不新建独立日志口径
- 新模块接入统一安全策略，不绕过登录与二次验证规则

建议预留的接入接口能力：

- 模块注册接口：新模块注册 `module_code`、资源清单、动作清单、默认角色模板
- 身份查询接口：按 `user_id`、员工账号、供应商账号查询统一身份主体
- 授权决策接口：按“角色 + 资源 + 动作 + 数据范围 + 字段范围”返回鉴权结果
- 数据范围接口：返回当前账号可见的供应商、品牌、店铺、渠道、组织范围
- 菜单与路由装配接口：按角色下发可见菜单、页面入口、按钮权限
- 审批能力接入接口：新模块可复用统一审批权限与高风险动作审批流
- 审计写入接口：新模块统一写入登录、变更、导出、审批、批量操作等审计事件
- 登录安全策略接口：统一复用密码策略、设备信任、二次验证、异地登录提醒
- 账号生命周期事件接口：支持入职、离职、冻结、解冻、转岗、供应商停用等事件通知

未来模块增加时必须满足：

- 资源模型先注册到 `权限与组织中台 / 权限中台`
- 管理入口再挂接到 `账号与权限中心`
- 不允许跳过中台直接在模块内写死权限判断
