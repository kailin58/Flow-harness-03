# 5个一级入口设计

## 设计原则

采用：

`5个一级入口 + 9个业务端域 + 后台中心/中台支撑`

目标：

- 对用户入口尽量合一，避免前台割裂
- 对高频岗位入口独立，避免后台臃肿
- 对系统能力隐藏在后面，不把中心/中台直接暴露成"端"

## 口径先锁定

本文件讨论的是"入口设计"，不是"模块总数"或"中心总数"。

- 沟通口径：按 `9个业务端域`
- 入口口径：按 `5个一级入口`
- 实现口径：由业务中心与中台在后端支撑

也就是说：

- 不是每个业务端域都单独做一个一级入口
- 也不是每个中心或中台都要暴露成前端工作台
- 入口设计优先考虑用户感知、岗位频率、权限边界、实施复杂度

## 三层口径澄清

为避免后续继续把"入口"、"端域"、"治理台"、"增强模块"混在一起，当前统一采用三层口径：

### 第一层：一级入口

对外只按 `5个一级入口` 沟通：

- 用户统一入口
- 平台后台入口
- 供应商协同入口
- 客服工作台入口
- 发货执行入口

### 第二层：业务端域

业务层仍然冻结为 `9个业务端域`，当前没有新增第 10 个业务端域：

1. C端交易端
2. 推广员端
3. 积分商城端
4. 平台运营后台端
5. 内容创造端
6. 外部渠道与店铺管理端
7. 供应商协同端
8. 客服工作台端
9. 发货执行端

### 第三层：治理工作域 / 增强模块 / 后台支撑能力

以下能力重要，但当前**不新增为业务端域**：

- `账号与权限中心`：平台后台下的统一治理工作域
- `AI模型管理中心`：平台后台下的统一治理工作域
- `对话模块`：`C端交易端` 内部增强模块
- `衣橱模块`：`C端交易端` 内部增强模块
- `行程规划模块`：`C端交易端` 内部增强模块
- 商品中心、库存中心、订单中心、结算中台、物流中台、消息中心等：后台支撑能力

统一判断原则：

- `能独立服务一类角色 + 承担相对完整流程职责`，才考虑定义为业务端域
- `只负责治理、配置、审计、统一能力输出`，应归为治理工作域
- `只服务某个端域内部体验增强`，应归为端内增强模块
- `只提供规则、数据、引擎、同步、日志等底座能力`，应归为后台支撑能力

## AI模型治理口径

AI 属于平台级能力，必须统一模型治理。

- 平台后台中的 `AI模型管理中心` 负责统一治理模型能力
- 用户端对话模块、客服工作台端、内容创造端都是模型能力消费端
- 各消费端只负责调用模型能力，不负责治理模型能力
- AI模型管理中心采用三层定位：治理层（模型注册/状态/标签/配额/健康度）、语料层（知识底座/版本管理）、调用层（统一接口/路由/降级/成本归集）
- 详细规划见 `meeting/AI模型管理中心规划.md`

## 账号管理口径

账号管理采用：

`统一治理中心 + 各入口自助页`

- `权限与组织中台 / 权限中台` 是后端真相源；`账号与权限中心` 是平台后台治理台
- 平台后台中的 `账号与权限中心` 负责统一治理员工账号、供应商账号、角色、菜单权限、操作权限、数据权限、字段权限、审批权限、审计与登录策略
- `账号与权限中心` 预留未来模块统一接入接口：模块注册、统一鉴权、统一审计、生命周期事件
- 用户统一入口中的账号能力收口到 `我的/账号安全`
- 供应商协同入口中的账号能力收口到 `账号设置/子账号管理`
- 客服工作台入口、发货执行入口只保留极简个人设置，不单独发展成账号管理后台
- 渠道授权账号、API Key、模型密钥不归入人账号体系
- 详细规划见 `meeting/账号体系与权限规划.md`
- 菜单结构见 `meeting/账号与权限中心菜单规划.md`
- 对外接口草案见 `meeting/账号与权限中心对外接口草案.md`

| 一级入口 | 入口形态 | 核心服务对象 | 设计定位 |
|---|---|---|---|
| 用户统一入口 | H5 / 小程序 / App | 普通用户、会员、推广员 | 前台统一交易与会员入口 |

|---|---|---|

| 推广员端 | 用户统一入口 | 作为"我的/推广中心"子域进入 |
| 积分商城端 | 用户统一入口 | 作为"权益/积分商城"子域进入 |
| 平台运营后台端 | 平台后台入口 | 后台主端，承载平台运营与配置 |
| 内容创造端 | 平台后台入口 | 作为独立一级菜单域存在 |
| 外部渠道与店铺管理端 | 平台后台入口 | 作为独立一级菜单域存在 |
| 供应商协同端 | 供应商协同入口 | 独立入口，不挂平台后台角色视图 |
| 客服工作台端 | 客服工作台入口 | 独立入口，不并入运营后台 |
| 发货执行端 | 发货执行入口 | 独立轻入口，不走复杂后台 |

## 一级入口与业务端域映射

### A. 用户入口群

- 用户统一入口
  - C端交易端
  - 推广员端
  - 积分商城端

设计含义：

- 对普通用户只有一个前台系统
- 商城仍然是主入口与主交易场
- 对话模块、衣橱模块、行程规划模块属于用户端增强层，不单独升级为一级入口
- 推广员能力作为会员能力的延伸，不单独做第二个主前台
- 积分商城作为权益子域存在，不做独立登录体系

### B. 平台入口群

- 平台后台入口
  - 平台运营后台端
  - 内容创造端
  - 外部渠道与店铺管理端

设计含义：

- 平台内部统一登录
- 内容创造与渠道店铺必须是独立工作域，但不必拆成第二、第三个后台域名
- `账号与权限中心`、`AI模型管理中心` 属于平台后台入口下的治理工作域，不新增业务端域数量
- 角色通过权限控制看到不同菜单与数据范围

### C. 协同与执行入口群

- 供应商协同入口
  - 供应商协同端
- 客服工作台入口
  - 客服工作台端
- 发货执行入口
  - 发货执行端

设计含义：

- 供应商、客服、仓配都是高频岗位，应该独立入口
- 三者的使用频率、页面结构、数据权限完全不同，不适合并进同一个后台入口
- 发货执行入口必须保持极简，不能被后台导航拖慢

## 各入口建议结构

### 1. 用户统一入口

建议一级导航：

- 首页
- 分类/搜索
- 购物车
- 订单
- 我的

建议核心子入口：

- 对话入口
- 衣橱入口
- 行程规划入口
- 推广中心
- 积分商城
- 会员权益
- 客服入口
- 活动会场入口（可配置开关）

**IA 挂载表、登录默认落地、一期边界切片、游客合并与归因约束**：见分册 `meeting/5-1用户统一入口.md`（「一级导航与核心子入口挂载」「登录默认落地」「一期边界切片」「游客态与登录合并最小规则」「外部导流参数与归因最小口径」）。

设计要求：

- 用户只感知一个前台系统
- 推广员能力按资格显隐
- 积分商城不做独立登录体系
- 客服入口可从商品页、订单页、我的页进入
- 推广中心、积分商城、活动会场入口采用配置化开关，避免频繁发版
- 用户端第一优先级始终是交易主链路，增长入口不能压过交易主链路
- 首页搜索下方模块快捷栏按“固定坑位 + 配置化内容”治理，新增模块需满足准入规则且不得改写交易主链路层级

建议补充说明：

- 推广员准入入口在"我的/推广中心"下承载，且准入流程必须强绑定学习平台（先学习→测验≥80分→协议签署→提交审核），任一步骤未完成则申请入口置灰
- 会员分享、推广员分享、活动会场流量最终仍回到同一用户统一入口
- 用户统一入口应统一承接淘宝、抖音、小红书、京东、唯品会、快手、百度等外部渠道的导流、回站与参数透传，不把第三方渠道流量单独拆成新的前台入口
- 短视频素材导流页属于用户统一入口的流量承接页，不单独算一级入口
- **短剧与短视频素材的边界必须明确**：短视频素材导流（AI/模板生成，目标是转化）≠ 短剧产品形态（章节化内容消费，目标是留存）；一期 C 端只承接短视频素材带来的流量转化，不在同期引入短剧产品形态
- **对话模块**：用户端增强交互入口，以穿搭顾问为主，同时兼容售后、客服等服务分流入口
- **衣橱模块**：用户端个人穿搭资产底座，用于沉淀已有单品、风格偏好与搭配基础
- **行程规划模块**：用户端场景与时间编排模块，用于承接出行、场景、时间安排与后续穿搭规划
- 对话模块、衣橱模块、行程规划模块当前先确定归属和定位，具体功能与布局后续单独展开
- **游客态处理**：游客可浏览商品、加购；登录触发时需补全身份并合并购物车上下文与来源参数，不得丢失归因链路
- **归因最小口径**：外部导流链路至少保留 `source_platform / campaign_id / content_id / promoter_id / landing_page / trace_id`，参数缺失需有补偿标记与兜底策略
- **多端凭证合并**：手机号/微信/支付宝/小程序 openid 合并提示入口在"我的/账号安全"下承载，合并完成后积分、等级、推广关系统一归集到主会员 ID
- **SEO/渲染策略**：投流落地页、商品详情页等有 SEO 需求的页面，技术上可选 SSR/预渲染（Nuxt 路线），交易页优先稳定渲染策略，活动页允许更高灵活度，但不影响主链路 SLA

### 2. 平台后台入口

建议一级菜单：

- 工作台
- 商品
- 订单
- 售后
- 会员
- 营销（活动/优惠券）
- 分销
- 积分
- 内容创造
- 账号与权限中心
- AI模型管理中心
- 渠道店铺
- 风控/合规
- 结算报表
- 发票/财务
- 系统配置（含审计检索）

**仅 IA 阶段说明**：见分册 `meeting/5-2平台后台入口.md`（「当前阶段范围（仅 IA）」）。

设计要求：

- 一个后台统一登录
- `内容创造` 独立一级菜单
- `AI模型管理中心` 独立一级菜单
- `账号与权限中心` 独立一级菜单
- `渠道店铺` 独立一级菜单
- `风控/合规` 独立一级菜单，不并入系统配置
- 角色不同，看到的菜单和数据范围不同
- 佣金规则、积分规则、渠道策略、内容审核都在后台配置，不在前台端侧分散配置

建议补充说明：

- 后台是平台内部管理入口，不承担供应商、客服、仓配的一线高频操作
- 内容创造端虽然挂在后台入口下，但应视为独立业务域，不与普通商品运营混写
- AI模型管理中心应作为平台后台中的独立工作域存在，负责模型注册、能力标签、路由策略、限额和健康度管理
- 账号管理不应散落在多个后台模块中，应统一收口到平台后台的 `账号与权限中心`
- 内容创造端负责“素材生产、审核、版本与发布前准备”，AI模型管理中心负责“模型注册、能力治理、路由策略、额度与健康度管理”
- 用户端对话模块、客服工作台端、内容创造端都消费模型能力，但模型治理统一收口到 AI模型管理中心
- `账号与权限中心` 负责员工账号、供应商主账号、角色、权限、数据范围与审计；各入口仅保留各自的账号自助页
- 外部渠道与店铺管理端应独立承接平台接入、店铺授权、能力矩阵、映射管理，不要只当成"系统配置"子页面
- `渠道店铺` 覆盖的平台范围可统一表述为“如淘宝、抖音、小红书、京东、唯品会、快手、百度等第三方平台”，具体一期试点平台在范围冻结文档中单独约定
- 售后作为独立一级菜单，承载退款审核、售后单处理、逆向物流协同，不并入订单菜单
- 营销中心（活动/优惠券）作为独立一级菜单，承载活动编排、券模板、人群圈选，与分销和积分保持独立配置域

### 3. 供应商协同入口

建议一级菜单：

- 商品提报
- 库存
- 待发货
- 售后协同
- 账单对账
- 账号设置/子账号管理
- 通知消息

**与发货执行入口 IA 边界、默认落地、一期边界与评审清单**：见分册 `meeting/5-3供应商协同入口.md`。

设计要求：

- 独立入口
- 默认只看本供应商数据
- 不能触达平台级配置
- 高危动作有审计和权限控制
- 一期边界遵循“提报/协同/对账必须做，扩展报表与复杂经营分析后置”

建议补充说明：

- 供应商入口可采用 `supplier.xxx.com`，也可按场景补充独立小程序
- 供应商端的工作重点是协同，不是平台经营总控
- 供应商账单查看、售后协同、发货协同可以在同一入口内，但都必须带 `supplier_id` 数据范围

### 4. 客服工作台入口

建议一级菜单：

- 会话
- 工单
- 待处理
- 升级协同
- SLA看板
- 知识库（可后续）
- 个人设置（顶栏账户区进入，不替代账号与权限中心）

**仅 IA 与功能清单范围说明、一期边界与评审清单**：见分册 `meeting/5-4客服工作台入口.md`。

设计要求：

- 独立入口
- 可联查订单、售后、物流
- 不直接承担规则真相源
- 侧重处理效率、留痕、协同
- 一期边界遵循“工单/会话闭环优先，IM 聚合与机器人后置”

建议补充说明：

- 客服入口独立于平台后台，避免与运营后台混杂
- 若后续接入第三方 IM，仍以自有客服工作台和工单闭环为准
- 客服入口重点解决"处理"和"协同"，消息中心仅负责通道触达，不替代客服工作台

### 5. 发货执行入口

建议主导航（极简 IA）：

- 发货任务（默认首屏：扫码拣货、打包、打印、发货确认、批量发货等主路径）
- 异常（上报与进度）
- 我的（极简设置与设备绑定）

建议核心功能（归属在「发货任务」流程内，不拆成供应商后台级多级菜单）：

- 扫码拣货
- 打包
- 打印面单（支持蓝牙打印机/云打印）
- 发货确认
- 批量发货
- 异常上报

**与供应商协同入口 IA 边界、一期边界与评审清单**：见分册 `meeting/5-5发货执行入口.md`、`meeting/5-3供应商协同入口.md`。

设计要求：

- 极简
- 快速
- 适合现场操作
- 支持扫码枪
- 不要复杂后台导航
- 一期边界遵循“履约执行必需动作先落地，批量与多设备增强预留”

建议补充说明：

- 发货执行入口优先采用独立发货小程序或 PDA 形态
- 支持扫码枪、蓝牙打印机/云打印，按场景选型
- 一期先明确：是"只发货"还是同时"查订单+扫码+打印面单"，两者复杂度差异较大
- 若后续扩展，也应围绕履约动作展开，不应演变成第二个供应商后台

## 哪些能力明确不单列一级入口

以下能力重要，但在当前阶段不建议独立暴露为一级入口：

- 商品中心
- 库存中心
- 订单中心
- 支付与退款中心
- 会员管理中心
- 会员营销中心
- 分销与佣金中心
- 积分与权益中心
- 价格中心
- 权限与组织中台
- 结算中台
- 物流中台
- 消息中心
- 发票中心
- 渠道同步中心

原因：

- 这些能力更多是后端真相源、规则源、执行引擎或中台能力
- 它们支撑入口运行，但不适合作为沟通层面的一级入口
- 入口面向角色和流程，中台面向规则和能力，不应混为一谈

## 多环境与安全沙箱原则

入口设计必须在多环境下保持隔离，不允许测试/演练误伤生产：

- **环境隔离**：`dev / test / stage / prod` 全隔离；第三方平台对接必须有沙箱店铺或仿真桩
- **数据隔离**：测试环境使用脱敏数据（手机号、地址、身份证、支付信息）
- **操作隔离**：批量改价、全量库存回写、店铺停用、结算调整等高危动作必须有"演练模式"，演练模式只输出影响预估，不落真实外部平台
- **密钥安全**：各入口涉及的 API Key、Token、密钥统一走 KMS/密文存储，禁止明文落库或出现在入口配置页面
- **导出安全**：所有入口的导出功能须加水印、走审批或限频，超量导出自动告警

## 后台支撑原则

前面看见的是入口，后面真正运转的是中心和中台。

## 当前结论

- 对外沟通：按 `9个端`
- 入口设计：按 `5个一级入口`
- 内部实现：由中心与中台提供支撑
- 入口分布：`1个用户入口 + 1个平台后台入口 + 3个独立工作入口`

这样后续再展开页面、权限、排期和模块设计时，不容易混乱。
