# 5个入口评审一页版

## 用途

本文件用于评审会快速对齐 `5个一级入口` 的当前冻结状态、待决策项与主要风险，不替代各分册真源。

分册真源仍为：

- `meeting/5-1用户统一入口.md`
- `meeting/5-2平台后台入口.md`
- `meeting/5-3供应商协同入口.md`
- `meeting/5-4客服工作台入口.md`
- `meeting/5-5发货执行入口.md`

## 全局已冻结

- 入口层统一按 `5个一级入口` 沟通
- 业务层统一按 `9个业务端域` 沟通
- 分册负责 IA 冻结与边界细则，总览负责索引摘要
- 各分册已统一具备：`一期边界切片 + 文档验收标准（DoD） + 评审清单（Checklist）`

## 1. 用户统一入口

**已冻结**

- 底部一级导航固定为：`首页 / 分类搜索 / 购物车 / 订单 / 我的`
- 子入口挂载已冻结：消息中心、对话、衣橱、行程规划、推广中心、会员权益、账号安全、客服入口、活动会场
- 一期必须做聚焦：交易闭环、推广中心极简入口、会员权益基础入口、客服入口、外部导流承接
- 高变入口遵循“固定坑位 + 配置化内容”，不得冲击交易主链路
- 首页快捷栏当前冻结为：单栏展示、横向滑动，不预设固定数量上限
- 游客转登录合并、外部导流最小归因字段已补入口级约束

**待决策**

- `商城` 分流到“客户端商城”还是“分销自己商城首页”的更细命中规则
- 登录后强业务引导的默认打断策略
- 对话、衣橱、行程规划的一期展示深度

**风险点**

- 快捷栏若缺准入治理，容易重新长成第二套首页导航
- 导流参数若前后链路不一致，会直接影响归因与分佣
- 游客购物车与来源参数合并若处理不清，会影响转化和归因

## 2. 平台后台入口

**已冻结**

- 平台内部统一登录，不拆多个割裂后台域名
- 一级菜单已冻结到平台主控所需范围：商品、订单、售后、会员、营销、分销、积分、内容创造、账号与权限中心、AI模型管理中心、渠道店铺、风控合规、结算报表、发票财务、系统配置
- `账号与权限中心`、`AI模型管理中心` 口径已明确为治理工作域，不新增业务端域
- `渠道店铺` 与 `渠道同步中心` 已明确治理层/执行层分离
- 一期边界已冻结：平台主控能力先落，BI 与重型运营中台后置

**待决策**

- 平台后台默认首屏是工作台还是按角色首屏配置
- `风控/合规` 与 `系统配置` 的更细菜单分界
- `AI模型管理中心` 一期是否显式露出语料层入口还是只保留接口预留

**风险点**

- 后台最容易继续膨胀，重新把供应商、客服、仓配现场能力卷进来
- `渠道店铺` 若被实现成普通配置页，后续多平台多店治理会返工
- `账号与权限中心` 若职责不稳，容易和各入口自助页重复

## 3. 供应商协同入口

**已冻结**

- 独立入口，不并入平台后台
- 一级菜单固定为：商品提报、库存、待发货、售后协同、账单对账、账号设置/子账号管理、通知消息
- 数据权限以 `supplier_id` 为主边界，默认只见本供应商数据
- 与发货执行入口边界已冻结：本入口负责“协同视图”，发货入口负责“现场执行视图”
- 一期边界已冻结：提报、协同、对账先做；复杂经营分析后置

**待决策**

- 默认落地页是 `待发货` 还是 `通知消息`
- 一期是否补独立小程序形态，还是先只保留 Web
- 对账确认是否一期支持简单异议反馈链路

**风险点**

- 若协同入口混入现场扫码/打印主路径，会和发货入口边界打架
- 若主账号/子账号授权模型不清，会出现越权或权限碎片化
- 若对账只展示不留痕，后续财务争议难追溯

## 4. 客服工作台入口

**已冻结**

- 独立入口，不并入平台后台
- 一级菜单主路径固定为：会话、工单、待处理、升级协同；`SLA看板` 与 `知识库` 一期可占位或预留
- 一期必须做聚焦：工单/会话入口、分派与升级、订单/售后/物流上下文联查、处理留痕
- AI 能力口径已冻结为“辅助理解/摘要/回复建议/分流”，不替代规则裁定
- 外发消息治理、违禁词与联系方式拦截已明确为入口级约束

**待决策**

- 默认落地页是 `待处理` 还是 `会话`
- 一期 `SLA看板` 是否只保留菜单占位，还是做基础可视化
- 基础知识库是一期开启弱版，还是仅保留结构预留

**风险点**

- 若客服入口和消息中心边界不清，容易把“消息通道”误做成“处理系统”
- 若高风险动作审批链没冻结，普通客服权限容易失控
- 若知识库、AI 辅助、人工回复不共用同一发送治理规则，存在合规风险

## 5. 发货执行入口

**已冻结**

- 极简入口，建议形态为独立发货小程序 / PDA / 极简 Web
- 主导航固定收敛为：发货任务、异常、我的
- 一期必须做聚焦：扫码拣货、打包、打印面单、发货确认、异常上报
- `查订单 + 扫码 + 打印` 若一期纳入，必须收敛在 `发货任务` 流程内
- 与供应商协同入口边界已冻结：本入口负责执行，不负责供应商协同总控

**待决策**

- 一期到底是“只发货”还是“查订单 + 扫码 + 打印一体化”
- 先上哪种设备形态：小程序、PDA 还是极简 Web
- 批量发货是一期开灰度还是明确后置

**风险点**

- 若主导航继续扩张，很快会变成第二个供应商后台
- 设备形态若不先定，会反复影响扫码、打印、登录与操作流设计
- 若异常处理链路只上报不闭环，会影响现场执行效率

## 当前建议评审顺序

1. 先确认 5 个入口的边界是否接受，不先抠页面细节
2. 再确认每个入口的一期必须做/预留/二期边界
3. 最后只留下“待决策”项进入下一轮专项评审

## 评审后建议产出

- 一份“已冻结决议清单”
- 一份“待决策问题单”
- 一份“跨入口风险跟踪表”
