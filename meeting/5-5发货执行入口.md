# 发货执行入口

## 入口定位

发货执行入口是仓库、打包、履约人员的现场执行入口，服务于拣货、打包、打印、发货确认与异常上报。

该入口必须轻量、快速，不应做成另一个复杂后台。

## 服务对象

- 仓库人员
- 打包人员
- 履约人员
- 部分供应商发货人员

## 挂载的业务端域

- 发货执行端

## 口径澄清

本入口当前挂载的**业务端域**只有 1 个：

- 发货执行端

本入口**不**承担供应商全量协同后台职责；与 `供应商协同入口` 的划分见下文「与供应商协同入口的 IA 边界」。

## 建议入口形态

- 独立发货小程序
- PDA
- 极简Web

## 当前阶段范围（仅 IA）

本文件在本阶段**仅冻结信息架构（IA）**：极简主导航、与供应商协同入口的边界、导航形态上限。扫码、打印、批量等**能力**在主导航下的归属见「建议核心功能」，不在本阶段展开为独立后台菜单树。

下列内容**不在本阶段展开**：设备认证协议、打印 SDK 选型、与仓库 WMS 对接细节等。

## 一期边界切片（与范围冻结对齐）

为避免发货入口在一期内膨胀为“仓配管理后台”，补充范围边界：

**一期必须做（MVP）**

- 扫码拣货
- 打包
- 打印面单
- 发货确认
- 异常上报

**一期预留但不展开**

- 批量发货
- 多设备适配

**二期再做**

- 高级 WMS
- 波次、路径优化、复杂仓配调度

## 建议主导航（极简 IA）

顶层路径建议**不超过 3～4 个**并列入口（底部 Tab、侧栏或等价形式），且首屏心智为履约执行：

| 主导航项 | IA 职责 |
| --- | --- |
| 发货任务 | **默认首屏 / 默认进入**：承载扫码拣货、打包、打印面单、发货确认、批量发货等主路径 |
| 异常 | 异常上报、处理进度查询 |
| 我的 | 极简个人设置、设备与打印机绑定；不承接复杂账号治理（账号真相源仍在平台后台 `账号与权限中心`） |

「查订单 + 扫码 + 打印面单」若一期纳入，应收敛在**发货任务**首屏内的步骤流或二级页，**不**单独拉起供应商后台级左侧菜单树。

## 与供应商协同入口的 IA 边界

- **发货执行入口**：现场执行、扫码、打印、确认发货；详见 `meeting/5-3供应商协同入口.md` 对称说明。
- **供应商协同入口 / 待发货**：供应商视角的发货协同与单量处理；**不**替代仓内扫码打印主路径。

## 建议核心功能

以下能力归属在主导航 **「发货任务」** 下的流程与页面，**不作为**与供应商后台同构的多级菜单：

- 扫码拣货
- 打包
- 打印面单（支持蓝牙打印机/云打印）
- 发货确认
- 批量发货
- 异常上报（与主导航「异常」联动：上报入口可在任务流内，列表与进度在「异常」聚合）

## 关键设计要求

- 极简
- 快速
- 适合现场操作
- 支持扫码枪
- 不要复杂后台导航

## 关键补充说明

- 发货执行入口优先采用独立发货小程序或 PDA 形态
- 支持扫码枪、蓝牙打印机/云打印，按场景选型
- 一期先明确：是“只发货”还是同时“查订单 + 扫码 + 打印面单”，两者复杂度差异较大；**IA 上**无论哪种，均不得演化为多级业务后台，应保持在「发货任务」步骤流内消化增量
- 后续扩展也应围绕履约动作展开，不应演变成第二个供应商后台
- 发货执行入口只保留极简个人设置与设备绑定能力，不承接复杂账号管理；相关账号治理仍统一收口到平台后台的 `账号与权限中心`

## 本入口不负责

- 不负责物流规则配置
- 不负责库存规则定义
- 不负责售后裁定与经营分析主界面
- 不负责平台级配置

## 依赖的后台能力

- 订单中心
- 库存中心
- 物流中台
- 权限中台
- 消息中心
