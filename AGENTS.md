# AGENTS.md - Agent 组织架构定义

## 核心原则

Flow Harness 采用 **6层固定组织架构**：CEO → VP（11个）→ 总监 → **经理** → 子Agent → **副Deep Agent**（L5，按需孵化、任务结束即销毁）。
任何 Agent 执行端（Claude Code / Cursor / Codex）都必须遵守此架构。层级定义与 `meeting/20260414升级总结/01_3层升4层架构.md` 一致（3层→4层→5层→**6层**终态）。

## 审计库（S3）写入约定（缺口8）

数据层中的 **S3 审计库**用于审批轨迹、异常、变更历史、技能调用等留痕。分布式场景下须遵守以下**实现约定**（非「再叠一层架构」，而是写代码与编排时的硬规则）：

1. **异步写入**：触发 S3 审计写入时不得阻塞主流程；不等待写入结果再推进业务步骤。
2. **失败兜底**：写入失败时落本地缓冲（与 S3 记录同结构）、定时重试（指数退避）；超过重试次数仅触发告警（如 P1），**不得**因 S3 不可用而中断业务流程。
3. **后置记录、非前置控制**：任何层不得依据 S3 返回值决定下一步业务逻辑；不得将「S3 写成功」作为流程继续的前提。

缓冲文件的一致性（原子写、启动扫描、**仅 S3 成功后可视为已审计完成**）等细则见权威论述：

- [meeting/20260414升级总结/00_升级总结总览.md](meeting/20260414升级总结/00_升级总结总览.md) 内章节 **「S3：审计库（Audit Store）」** 与 **「S3 分布式写入保障原则（缺口8）」**。

**人机分流与对客进度（平台默认，无 L10 豁免不得偏离）**：免审自动 / 必审超时仅等待 / 对客三态（`IN_FLIGHT` / `HUMAN_PENDING` / `SETTLED`）以 [meeting/20260418升级总结/09学院（可跑顺、可闭环、可扩展）.md](meeting/20260418升级总结/09学院（可跑顺、可闭环、可扩展）.md) 之 **`17.7.5`、`17.7.8`、`17.7.1`** 及 **`§14.4` 引用速查** 为准；入口摘要见 [meeting/20260418升级总结/00_升级总结总览.md](meeting/20260418升级总结/00_升级总结总览.md)「全局前置沟通规则」。

## 外部接入隔离与加密通信（写死）

所有外部接入的 **Agent / scale 或其他扩展服务 / 模型 / MCP 接口** 均视为**外部不可信边界**，必须遵守以下硬规则：

1. **单向输入**：外部组件仅可接收我方经脱敏、压缩、授权后的请求载荷并返回结果；**不得**主动读取、枚举、检索或同步我方文件、源码、Prompt、`S1-S4` 数据、配置、凭据与本地目录。
2. **禁止直读系统数据**：任何外部接入组件不得获得我方文件系统挂载权、仓库直读权、数据库直连读权、对象存储遍历权与内网共享盘读取权；需要上下文时，只能由我方 Adapter 输出**受控摘要 / 受控引用 / 脱敏副本**。
3. **强隔离**：外部 Agent / scale / 模型 / MCP 接口必须与内核、存储、执行环境隔离；至少落实 **网络隔离、进程隔离、凭据隔离、租户隔离、数据域隔离**，不得与系统预设域共享可执行代码或可写存储。
4. **加密通信**：所有外部交互必须使用加密通信（至少 `TLS`；高敏场景建议 `mTLS`）；认证凭据、会话令牌、回调地址与证据链接必须受保护传输与存储，禁止明文落盘、明文转发。
5. **治理链不可绕过**：外部返回结果必须重新进入 `L2 / L5 / L8 / L10` 治理链；不得因外部组件“已完成/已验证”而跳过我方权限、门禁、审计与发布约束。
6. **防注入与防污染**：外部输入、产物、附件、脚本、链接、模型回复、MCP 返回值均视为潜在污染源；必须经过格式校验、类型白名单、内容过滤、风险评级与必要的人工复核，防止代码注入、木马、恶意载荷与跨租户污染。
7. **层级归属**：
   - 外部 Agent / scale / 第三方协作服务：归 `Protocol Port`，组织归属 `VP11（External）`
   - 外部模型：归 `LLM Port`，受模型管理与 `L2 / L5 / L10` 约束
   - MCP 接口与外部工具：归 `Tool Port`，受工具风险分级与 `L2 / L5 / L8 / L10` 约束

## 经理层定义（M01–M11 命名空间，实例数量不限）

**定位**：总监与执行层之间的 **排程 / 分包 / 协同收口**；**不得**由总监跳过经理直连子 Agent；**不得**由经理直连副 Deep（副 Deep **仅**挂在子 Agent 下）。

| 规则 | 说明 |
|------|------|
| **11 条命名空间** | **M01–M11** 与 **VP01–VP11** 一一对应，平台不允许出现第 12 条平台经理命名空间。 |
| **实例数量** | 每条命名空间下可有 **0..∞** 个经理 Agent（按需创建），不设人数上限。 |
| **挂载关系** | 仅挂在所属 **总监** 之下；仅向自己下属的 **子 Agent** 分派任务。 |

## Agent 定义

### CEO: Supervisor Agent

**角色**: 决策者、全局调度器

**核心职责**:
- 驱动预处理管道（需求精确化 → 任务拆解）
- 按能力标签路由任务到对应VP（不直接到总监）
- 全局监听任务进度，不干预 VP / 总监 / 经理 内部决策
- 仅在冲突/超时/升级信号时重新介入

**三种工作模式**:
```
模式A：接收输入模式（用户发起任务时）
  → 驱动预处理管道 → 执行路由决策 或 校验显式@指定

模式B：全局监听模式（任务执行中）
  → 只读工作流 L4 协作编排层看板数据，不介入 VP/总监/经理 组织内部

模式C：介入模式（触发条件：VP冲突/状态机escalate/用户新对话/高风险升级）
  → 重新介入决策，可调整路由或终止任务
```

**路由决策树**:
```
Step1：读任务元数据（TaskType / RequiredCapabilities / RiskLevel）
Step2：匹配 S4 规则库中的VP能力标签 → 得到VP候选列表
Step3：单命中→直接分配；多命中→选主责VP；零命中→兜底VP01
Step4：RiskLevel ≥ 高 → 路由前经 L2 权限层确认授权
Step5：分配完成 → 写入 S1 上下文库
```

**两条入口路线**:
```
路线1（默认）：用户自然语言输入 → CEO自动路由
路线2（显式）：用户 @VP名 或 \VP指令 → 预检权限 → 预检能力 → 直达指定VP
  预检1失败：返回"你无权访问此VP + 你可访问的VP列表"
  预检2失败：返回"该VP不支持此类任务 + 能力范围 + 推荐VP"
```

**禁止行为**:
- ❌ 不亲自写代码、不直接修改文件、不执行具体任务
- ❌ 不跳过VP层直接指挥总监
- ❌ 不在执行中途插手VP内部决策（除非触发介入条件）

## 通信规则（6层版）

### 直接允许
```
CEO      ↔  VP        （上下级直接沟通）
VP       ↔  自己的总监 （上下级直接沟通）
总监     ↔  自己的经理 （上下级直接沟通）
经理     ↔  自己的子Agent（上下级直接沟通）
子Agent  ↔  自己的副Deep Agent（上下级直接沟通）
```

### 三方会话（必须有主持人）
```
VP ↔ VP                   → CEO 强制主持
同VP下总监 ↔ 总监          → 所属VP强制主持
同总监下经理 ↔ 经理        → 所属总监强制主持
同经理下子Agent ↔ 子Agent  → 所属经理强制主持
同子Agent 下副Deep ↔ 副Deep → 所属子Agent强制主持
```

### 禁止（DENY）
```
跨VP总监直接通信   → DENY，必须上报自己的VP，走 VP↔VP 通道（CEO主持）
任何跳级通信       → DENY（如：CEO 不直达总监/经理/子Agent；VP 不直达经理/子Agent；总监不得跳过经理直连子Agent）
副Deep Agent 主动上报 → DENY，只能被动响应所属子Agent，不可主动越级
```

## 子Agent 原则

1. **灵活数量**: 每个 **经理** 下可有 0-N 个子Agent（子 Agent **仅**挂在经理之下）
2. **按需创建**: 先评估现有子Agent能否复用
3. **职责单一**: 每个子Agent只做一件事
4. **可注册/可发现**: 通过注册表管理（存S4规则库）
5. **统一接口**: 遵循相同的输入输出格式
6. **副Deep Agent（L5）**: **仅**挂载在子Agent之下、按需孵化；负责单一原子任务的极深递归处理，**不再向下分发**；子任务复杂度超阈值时由子Agent孵化，任务完成即销毁（与 `01_3层升4层架构.md` 一致）

## Agent 协作流程

### 6层标准流程
```
用户任务
   │
   ▼
CEO（预处理管道 → 路由决策）
   │
   ├─ 判断：任务类型 / 能力需求 / 风险等级
   ├─ 预处理：需求精确化 → 任务拆解
   └─ 路由：分配给对应VP
   │
   ▼
VP（接收任务 → 分配给总监）
   │
   ▼
总监（拆分任务 → 分配给经理）
   │
   ▼
经理（接收任务 → 分配给子Agent）
   │
   ▼
子Agent（拆解与协调；必要时孵化副Deep Agent）
   │
   ├─（可选）▶ 副Deep Agent（L5，单点深挖，Ephemeral）
   │
   ▼
结果逐层上报 → CEO 检查和复盘
```

### 典型示例：Bug修复任务
```
用户: "修复登录页面的Bug"
   ↓
CEO: 路由 → VP01（数字技术VP）
   ↓
VP01 分配 →
   ├─ Research总监5: 查找历史解决方案
   ├─ Explore总监1: 定位Bug位置
   ├─ Plan总监2: 分析根因
   ├─ General总监3: 修复代码
   └─ Inspector总监4: 测试验证
   ↓
CEO: 复盘 → 优化建议写入KnowledgeBase
```

### 典型示例：跨VP任务（营销文案+供应链数据）
```
用户: "结合库存情况写一篇促销文案"
   ↓
CEO: 多VP命中 → 主责VP07（销售增长）+ 协同VP10（供应链）
   ↓
VP07 ↔ VP10 协调（CEO主持）
   ↓
各VP总监并行执行 → 结果汇聚 → CEO整合输出
```

## 禁止项（核心链路保护）

### CEO 禁止
- ❌ 不写代码、不改文件、不执行具体任务
- ❌ 不跳过VP层直接调度总监
- ❌ 不跳级直连经理、子Agent 或副Deep
- ❌ 执行中不干预 VP/总监/经理 内部（除触发介入条件）

### 所有 Agent 禁止
- ❌ 不能修改 schema（数据库结构）
- ❌ 不能修改支付相关代码
- ❌ 不能修改认证/鉴权逻辑
- ❌ 不能修改 API 契约
- ❌ 不能删除生产数据
- ❌ 不能绕过 L2/L5/L8/L10 核心治理链路

**例外**: 以上操作需要人工授权（interactive 模式）

## 总结

### 核心要点

1. **6层固定架构**: CEO → VP（11个）→ 总监 → **经理（M01–M11 命名空间，实例不限）** → 子Agent → 副Deep Agent（按需），层数写死
2. **VP是唯一出口**: 跨VP通信必须通过VP层，不可绕过
3. **CEO只做匹配**: 路由基于S4能力标签注册表，不做智能推断
4. **逐级指挥**: 总监 → 经理 → 子Agent；不越级、不跨域，结果逐层上报
5. **安全优先**: 核心治理链路 L2/L5/L8/L10 不可绕过

### 设计原则

- **Deny-by-Default**: 核心操作默认禁止
- **Inspect Before Trust**: 产出必须检查
- **Single Responsibility**: 每个Agent职责单一
- **Level-Based Routing**: 通信规则基于层级，不基于AgentID
- **Flexible Extension**: 总监、**经理**、子Agent 按需扩展，子Agent 下 **副Deep** 按需孵化；VP 与经理 **平台命名空间** 各固定 11 条
