# 数据流与跨系统协作架构

## 文档定位

本文档冻结平台级数据流架构设计，包括：
- 五个一级入口间的数据传输与处理
- 第三方系统接入（支付、物流、电商渠道、AI模型等）
- 企业系统对接（ERP、WMS、财务等）
- AI基础设施统一管理

本文档是架构决策真源，后续所有开发以此为基准。

---

## 一、核心架构

### 一句话定义

> **核心交易用Java稳住，入口协作用事件驱动，AI能力统一供给，外部系统适配隔离。**

## 二、三层核心设计

### 2.1 第一层：事件总线（跨入口协作的神经中枢）

#### 职责

- 入口间状态同步（订单支付 → 通知供应商 → 通知客服）
- 跨端协同编排（售后申请 → 客服处理 → 供应商协同 → 退款）
- AI任务触发（工单关闭 → 触发知识沉淀Agent）
- 外部系统回调（支付回调 → 订单状态更新 → 多入口通知）

#### 事件分类

| 事件域 | Topic | 典型事件 |
|--------|-------|----------|
| 交易事件 | trade.events | OrderPaid, OrderShipped, OrderRefunded, OrderCancelled |
| 客服事件 | cs.events | TicketCreated, TicketEscalated, TicketClosed |
| 供应商事件 | supplier.events | GoodsSubmitted, StockUpdated, BillGenerated |
| 履约事件 | fulfillment.events | ShipmentConfirmed, ExceptionRaised |
| AI事件 | ai.events | AgentTaskTriggered, KnowledgeCreated |
| 外部事件 | external.events | PaymentCallback, LogisticsUpdated, ChannelSynced |
| 权限事件 | iam.events | RoleChanged, ScopeChanged |

#### 关键原则

1. **所有跨入口通信必须走事件**，禁止入口间直接调用
2. **所有事件必须带 trace_id**，全链路可追踪
3. **所有事件写入 Outbox 表**，与业务事务原子提交
4. **事件消费者幂等**，支持重试和回放

#### 事件格式

```json
{
  "event_id": "uuid-v7",
  "event_type": "OrderPaid",
  "aggregate_type": "Order",
  "aggregate_id": "order-uuid",
  "trace_id": "trace-uuid",
  "occurred_at": "2026-04-09T14:30:00.000Z",
  "payload": {
    "order_id": "order-uuid",
    "member_id": "member-uuid",
    "amount": 299.00,
    "supplier_ids": ["supplier-1", "supplier-2"]
  },
  "metadata": {
    "source": "bff-user",
    "version": "1.0"
  }
}
```

### 2.2 第二层：业务真相源（Java核心）

#### 职责

- **交易主链路**：订单、支付、退款
- **规则真相源**：价格、库存、佣金、积分
- **权限治理**：角色、权限、数据范围
- **结算核算**：供应商结算、分销佣金

#### 关键原则

1. **所有写入必须走真相源**，禁止旁路篡改
2. **AI/Agent 只读**，不允许直接写核心表
3. **高风险动作必须人工确认或规则引擎裁定**

#### 模块划分

| 模块前缀 | 模块名 | 职责 |
|----------|--------|------|
| iam | 权限与身份 | 认证、授权、审计 |
| catalog | 商品 | SPU/SKU、类目、品牌 |
| inventory | 库存 | 库存管理、流水 |
| pricing | 价格 | 定价、快照 |
| trade | 交易 | 订单、购物车 |
| payment | 支付 | 支付、退款 |
| aftersale | 售后 | 售后单、退款审核 |
| member | 会员 | 会员信息、等级 |
| settlement | 结算 | 供应商结算、分销佣金 |
| logistics | 物流 | 物流追踪、发货 |

### 2.3 第三层：统一AI基础设施（平台级能力）

#### 定位

**AI模型管理中心是平台级统一基础设施**，未来所有系统（C端、B端、内部、外部）都通过统一接口消费AI能力。

#### 职责

- **模型统一管理**：本地部署 + 第三方API
- **智能路由**：按场景/成本/延迟/安全自动选择模型
- **知识库统一**：商品/客服/内容知识集中管理
- **Agent Runtime**：工具注册、执行追踪、人工确认

#### 调优层触发规则

```yaml
自动触发调优:
  效果下降触发:
    - 条件: 采纳率连续7天下降超过10%
    - 动作: 自动创建调优任务，通知知识管理员
    
  周期调优触发:
    - 条件: 知识发布满90天且使用量>100次
    - 动作: 加入调优队列，按优先级排序
    
  投诉触发:
    - 条件: 单条知识收到3次以上负面反馈
    - 动作: 紧急进入调优，优先处理

手动触发调优:
  - 运营主动发起调优任务
  - 业务变更后批量更新知识
  - 新产品/活动上线前预调优

调优优先级:
  1. 高频知识：日均使用量 > 100
  2. 高风险知识：涉及退款/赔付
  3. 高投诉知识：负面反馈率 > 5%
  4. 过期知识：超过180天未更新
```

#### 知识检索优先级

```yaml

  1. 公司层知识：已发布、已验证的官方知识，最高优先级
  2. 验证层知识：通过审核正在验证，次高优先级

  4. 个人收藏：用户个人收藏的常用知识

  - 默认从公司层开始检索，逐层向下
  - 高风险场景（退款/赔付）仅检索公司层已发布知识
  - 部门专属问题优先检索部门层知识
  - 个人收藏可作为快捷入口，不参与通用检索

敏感知识控制:
  - 敏感字段（手机号、身份证等）不在知识中明文存储
  - 高风险知识仅授权人员可检索
  - 供应商相关知识按 supplier_id 隔离
```

### 3.3 提示词分类

```yaml
按用途分类:

  1. 系统提示词（System Prompt）:
    定义: 定义AI的角色、行为边界、输出格式、安全约束
    示例: |
      你是ACmodus平台的客服助手。
      你的职责是帮助用户解决购物和售后问题。
      
      行为约束：
      - 不讨论竞品
      - 不承诺未授权的优惠
      - 不泄露用户隐私
      - 遇到无法处理的问题转人工
      
      输出格式：
      - 友好、专业、简洁
      - 优先使用知识库中的标准话术
    层级: 平台层
    权限: 仅AI管理员可编辑

  2. 场景提示词（Scene Prompt）:
    定义: 针对特定业务场景的提示词模板
    示例: |
      场景：售后退款咨询
      
      当前用户信息：
      - 会员等级：{{member_level}}
      - 订单状态：{{order_status}}
      - 订单金额：{{order_amount}}
      
      相关知识：
      {{knowledge_base}}
      
      用户问题：{{user_question}}
      
      请根据以上信息，给出退款方案建议。
    层级: 场景层
    权限: 场景负责人可编辑

  3. Agent提示词（Agent Prompt）:
    定义: 定义Agent的行为、工具调用、决策逻辑
    示例: |
      你是客服副驾驶Agent。
      
      你的能力：
      - 查询订单信息（工具：order_query）
      - 查询物流状态（工具：logistics_query）
      - 检索知识库（工具：knowledge_search）
      - 创建工单（工具：ticket_create）
      
      工作流程：
      1. 分析用户意图
      2. 调用相应工具获取信息
      3. 生成回复建议
      4. 等待人工确认后发送
      
      高风险操作必须人工确认：
      - 退款承诺
      - 赔偿承诺
      - 改地址
    层级: 场景层/平台层
    权限: AI管理员审核

  4. 工具提示词（Tool Prompt）:
    定义: 定义工具调用的参数格式和行为约束
    示例: |
      工具：订单查询
      
      输入参数：
      - order_id: 订单ID（必填）
      - fields: 返回字段列表（可选）
      
      使用场景：
      - 用户询问订单状态
      - 客服需要查看订单详情
      
      返回格式：
      {
        "order_no": "订单号",
        "status": "订单状态",
        "items": [...]
      }
    层级: 平台层
    权限: 仅AI管理员可编辑
```

```yaml
联动机制:

    提示词中可引用知识库内容作为上下文

        ## 用户问题
        {{user_question}}

  2. 动态注入:
    运行时自动检索相关知识并注入提示词
    
    流程:
      用户问题 → 知识检索 → 匹配Top-K → 注入{{knowledge_base}} → 模型推理

  3. 版本关联:
    提示词版本与知识库版本关联，确保一致性
    
    规则:
      - 提示词引用的知识库更新时，自动创建新版本提示词
      - 发布前检查知识库版本有效性

  4. 效果联动:
    提示词效果与知识库效果共同评估
    
    指标:
      - 知识命中率
      - 答案采纳率
      - 用户满意度
```

### 3.7 提示词A/B测试

```yaml
测试设计:

  测试场景:
    - 新版本提示词上线前测试
    - 效果下降时的优化测试
    - 不同模型适配效果对比

  测试流程:
    1. 创建测试版本：
       - 基于现有版本创建变体
       - 定义测试假设和预期效果
       
    2. 配置分流：
       - 设置流量比例（如对照组50%，实验组50%）
       - 按用户ID哈希分流，保证同一用户一致
       
    3. 数据采集：
       - 响应质量评分
       - 用户满意度
       - 问题解决率
       - 异常率
       
    4. 效果分析：
       - 统计显著性检验
       - 效果差异分析
       - 最佳版本推荐
       
    5. 发布决策：
       - 实验版本优于对照组 → 灰度发布
       - 无显著差异 → 继续观察或调整
       - 实验版本劣于对照组 → 回滚

  测试指标:
    - 响应质量：人工评估 / 自动评估
    - 用户满意度：点赞率 / 评分
    - 问题解决率：无后续咨询比例
    - 安全合规：违规率 / 投诉率
    - 成本效率：Token消耗 / 响应时间
```

## 五、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 入口间通信 | 事件驱动 | 解耦、可扩展、天然支持AI触发 |
| 核心交易 | Java模块化单体 | 稳定、事务、审计、规则确定性 |
| AI能力 | 统一基础设施 | 避免重复建设、统一治理、成本可控 |
| 外部系统 | 适配器隔离 | 变更不影响核心、能力差异透明化 |
| 数据敏感度 | 本地优先 | 安全合规、数据不出域 |
| Agent定位 | 辅助编排 | 不替代规则引擎，高风险需人工确认 |

### 8.2 升级方向详细说明

#### 方向1：事件溯源（Event Sourcing）

```yaml
当前状态:
  - 使用Outbox CDC模式保证事件不丢失
  - 事件作为状态变更的副产品

升级目标:
  - 事件作为真相源，状态从事件重放
  - 完整的审计追溯能力
  - 时间旅行查询（任意历史时刻状态）

升级收益:
  业务价值:
    - 合规审计：满足财务/法务审计要求
    - 问题排查：回放历史事件定位问题
    - 数据修复：重放事件修复数据不一致

  技术价值:
    - 解耦读写：独立优化读性能
    - 灵活扩展：新增读模型无需改写逻辑
    - 灾难恢复：从事件流重建完整状态

升级路径:
  Phase 1: 选择核心聚合试点
    - 订单聚合：OrderCreated → OrderPaid → OrderShipped → OrderCompleted
    - 会员聚合：MemberCreated → MemberUpgraded → PointsEarned

  Phase 2: 事件存储优化
    - 事件快照：定期快照减少重放事件量
    - 分区策略：按聚合ID分区保证顺序
    - 过期归档：历史事件冷存储

  Phase 3: 读模型独立
    - CQRS分离：写模型走事件，读模型独立投影
    - 多读模型：为不同查询场景建不同读模型

技术选型:
  - EventStore / Axon Framework
  - 自建基于Kafka + 事件存储表
```

#### 方向2：CQRS读写分离

```yaml
当前状态:
  - 读写共用同一数据库
  - 复杂查询可能影响写入性能

升级目标:
  - 命令模型（写）与查询模型（读）分离
  - 读模型可独立扩展和优化

架构演进:
  ┌─────────────────────────────────────────────────────────────────────────────┐
  │                           CQRS架构演进                                        │
  ├─────────────────────────────────────────────────────────────────────────────┤
  │                                                                             │
  │  一期（当前）                      二期（CQRS）                              │
  │  ┌─────────────┐                 ┌─────────────┐                            │
  │  │   写入      │                 │   命令模型   │                            │
  │  │   读取      │ ──────────────▶ │   (写)      │                            │
  │  │  同一DB     │                 └──────┬──────┘                            │
  │  └─────────────┘                        │ 事件投影                           │
  │                                         ▼                                   │
  │                                  ┌─────────────┐                            │
  │                                  │   查询模型   │                            │
  │                                  │   (读)      │                            │
  │                                  └─────────────┘                            │
  │                                  Elasticsearch / Redis                      │
  │                                                                             │
  └─────────────────────────────────────────────────────────────────────────────┘

应用场景:
  - 订单列表查询：读写分离，支持复杂筛选
  - 商品搜索：写入MySQL，读取Elasticsearch
  - 数据报表：读模型预聚合，避免实时计算

实施步骤:
  1. 识别高并发读取场景
  2. 建立读模型投影逻辑
  3. 双写验证一致性
```

### 8.5 升级风险评估

```yaml
风险分类:

  技术风险:
    事件溯源:
      - 风险：事件格式变更导致历史事件无法重放
      - 缓解：事件版本管理 + Upcasting

    CQRS:
      - 风险：读写数据不一致
      - 缓解：最终一致性窗口 + 补偿机制

    多Agent:
      - 风险：Agent协作失败导致任务卡死
      - 缓解：超时机制 + 人工介入

  业务风险:
    模型微调:
      - 风险：微调后效果反而下降
      - 缓解：灰度发布 + A/B测试 + 回滚机制

    开放平台:
      - 风险：API滥用导致系统过载
      - 缓解：限流 + 熔断 + 监控告警

  运维风险:
    多云部署:
      - 风险：跨云数据同步延迟
      - 缓解：异步同步 + 冲突检测

    服务网格:
      - 风险：Sidecar故障影响业务
      - 缓解：健康检查 + 自动重启
```
